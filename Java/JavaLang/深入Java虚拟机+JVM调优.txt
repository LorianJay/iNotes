如果我在main方法里面new了一个新的对象,这个对象有个String类型的属性name.此时我要直接给这个name赋值的内存分析:
在栈里面,拥有一个指针(引用变量)指向这个对象,通过引用变量.name=...来赋值,但实际上name也是指向一个内存,相当于把一个字符串的内存赋值给对里面的name属性.这个字符串的内存实际上存储在方法区里面(方法区属于堆).

·java虚拟机的内存可以分为三个区域:栈stack、堆heap、方法区method area.		注意:方法区实际上也在堆里,本质上只有栈和堆.

·栈的特点如下:
1.堆内存用于存储局部变量、临时变量(定义在代码块里的变量(类型是八种基本数据类型))和引用变量.大小固定
2.栈描述的是方法执行的内存模型.每个方法被调用都会创建一个栈帧(这个栈帧会存储当前方法的存储局部变量、操作数、方法出口等).
3.JVM为每个线程创建一个栈,用于存放该线程存放方法的信息(实际参数,局部变量等)
4.栈属于线程私有,不能实现线程间的共享.
5.栈的存储模式是"先进后出,后进后出"		//注意!!!不是先进先出,后进后出!!!
6.栈是由系统自动分配,速度快!栈是一个连续的内存空间.
7.堆的大小是固定固定

关于栈帧的解释:任何方法调用的时候都会创建一个栈帧,满足"先进后出,后进先出"的道理.一个程序肯定有mian方法,当我们调用main方		法的时候,就会创建一个栈帧,如果这个main方法里面实例化了一个新的对象(new),此时该对象就是一个引用变量.在堆中会把这个对象的内存	地址从堆中获取然后赋值给mian方法中的变量.这个引用变量实际上就相当于c语言中的指针,它是用来寻找平对象的.
比如int a = 1;	这个a就是局部变量
Object obj = new Onject();		//new Object(即Object对象的实例化是存储在堆里面的) obj就相当于一个指针(引用变量)用于						Object对象的内存.我们在调用obj.方法的时候实际上就是顺着这个指针找到这个对象,然后通过这						个对象来调用它的方法.
如果存储的是基本数据类型,那么直接就存储在栈里面,改多少个字节就多少个字节.


·堆的特点如下:
1.堆用于存储创建好的对象和数组(数组也是对象).
2.JVM只是一个堆,被所有线程共享.
3.堆是一个不连续的内存空间,分配灵活速度慢.
4.堆大小不固定

·方法区(又称静态区)特点如下:
1.JVM只有一个方法区,被索要线程共享!
2.方法区实际也是堆,只是用于存储类、常量相关的信息!
3.用来存放程序中永远是不变或唯一的内容.(类信息[Class对象]、静态成员、字符串常量等.)
 
	关于字符串常量的解释说明:当我们一个类被加载到方法区里面后,这个类里面所有的字符串都在方法区里,如果我们要对一个变量进行赋		值的话,实际上也就是把方法区里面的内存,赋值给对里面的对象的具体变量.

·当mian方法执行结束的时候,实际上也就是栈关闭栈帧的时候,此时虚拟机也就关闭了,那么虚拟机内存也没有了,那么堆也就响应的关闭了.

1.1.1:方法区
·介绍:
程序计数器是一块较小的内存空间.在虚拟机的概念模型里,字节码解释器工作时就是通过改变这个计数器的值来选取执行下一条需要执行的字节码指令,分支、循环、跳转、异常、线程恢复等基础功能都需要依赖这个计数器来完成.
由于java虚拟机是通过线程轮流切换并分配处理器执行时间的方式来实现的,在任何一个确定的时刻,一个处理器(对于多核处理器来说是一个内核)都会执行一条线程中的指令.因此为了线程切换好能恢复到正确的执行位置,每条线程都需要有一个独立的程序计数器,各线程之间计数器互不影响,独立存储,我们称这类内存为"线程独有"的内存.
如果线程正在执行的是一个java方法,这个计数器记录的是正在执行的虚拟机字节码指令的地址,如果正在执行的是native(本地)方法,这个计数器值则为null.此内存区域是唯一一个在java虚拟机规范中没有规定任何OutOfMemoryError情况的区域.

·总结来说方法区的作用就是:记录每个线程当前执行到哪条执行,以便线程恢复时能继续执行上次结尾的地方.



1.1.2:java虚拟机栈(线程独有的)
·介绍:
java虚拟机栈也是线程独有的,它的声明周期与线程相同.(注意,因为每个线程都对应一个java虚拟机栈,可以说当线程结束的时候java虚拟机也结束了)
虚拟机栈描述的是java方法执行的内存模型:每个方法在执行的同时都会创建一个栈帧,用于存储局部变量表、操作数栈、动态链接、方法出口等信息.每一个方法从调用直至执行完成的过程,就对应着一个栈帧在虚拟机栈中入栈到出栈的过程.

局部变量表:
存放着八大基本数据类型、对象引用、returnAddress类型(指向了一条字节码指令的地址).

其中64位长度的long和double类型的数据会占用两个局部变量空间,其余的数据类型只占用1个.局部变量表所需的内存空间在编译期间完成分配,当进入一个方法时,这个方法需要在栈中分配多大的局部变量是完全确定的,在方法运行期间不会改变局部变量表的大小.

在java虚拟机规范中,对站规定了两种异常状况:如果线程请求的栈深度大于虚拟机所允许的深度,将抛出StackOverflowError异常.如果扩展时无法申请到足够的内存,就会抛出OutOfMenmroyError异常.


1.1.3:本地方法栈(线程独有的)

·介绍:
本地方法栈与虚拟机栈所发挥的作用是非常相似的,它们之间的区别不过是虚拟机栈为虚拟机执行java方法,而本地方法栈则为虚拟机执行了native方法.


1.1.4:java堆
·介绍:
堆内存是java虚拟机所管理的内存中最大的一块.堆内存的唯一目的就是存放对象实例,几乎所有的对象实例(包括数组)都在都在这里分配内存.(但随着一些优化,所有的对象都在堆中分配内存也变得不那么绝对了)

堆内存是垃圾回收器的主要工作区域.从回收的角度来看还可以将堆内存进行进一步划分以便垃圾回收器能更好的工作(具体的分析放在第二大节2)

根据java虚拟机规定,java堆可以处于无力上不连续的内存空间中,只要逻辑上时连续的即可.如果在堆中没有内存完成实例分配,并且堆也无法在扩展时,将会抛出OutOfMemoryError异常.

                                                                                                                                                1.1.5:方法区(线程共享)                                                                                                                          ·介绍:
该内存用于存储已被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等数据.方法区除了和堆一次不需要连续的内存和可以选择固定大小或者可扩展外,还可以选择不实现垃圾收集.相对GC而言,垃圾收集行为在方法区是比较少出现的,但并不代表数据一旦进入了方法区就永久存在了.方法区的内存回收目标主要是针对常量池的回收和对类型的卸载,一般来说这个区域的回收成绩比较难以令人满意,尤其是类型的卸载,条件想当苛刻,但是方法区的回收确实是有必要的.
根据java虚拟机规范的规定,当方法区无法满足内存分配需求时,将抛出OutOfMenmoryError异常.                                                             

1.1.6:运行时常量池                                                                                                                              
·介绍:
运行时常量池是方法区的一部分.class文件中除了有类的版本、字段、方法、接口等描述信息外,还有一项信息是常量池,常量池用于存放编译期间生成的各种字面量和符号引用,这部分内容将在类加载后进入方法区的运行时常量池中存放.
java虚拟机规范没有对常量池做任何细节要求,不过,一般来说常量池除了保存Class文件中描述的符号引用外,还会把翻译出来的直接引用也存储在运行时常量池中.
常量并不一定只有在编译时才会产生,在运行时也会产生常量,比如String的intern方法(由于String是不可变对象,产生任何新字符串的操作,都会在方法区的常量池中产生一个新的常量).所以既然运行时常量时方法区的一部分,自然受到方法区内存的限制,当常量池无法再申请到内存时会抛出OutOfMomoryError异常.


1.1.7:直接内存
·介绍:
直接内存并不是虚拟机运行时数据区的一部分,也不是java虚拟机规范中定义的内存区域.但是这部分内存也被频繁使用,而且也可能导致OutOfMemoryError异常出现
在JDK1.4中新加的NIO类,可以使用native函数库分配堆外内存,然后通过一个存储在java对中的DirectByteBuffer对象作为这块内存的引用进行操作.显然本机直接内存的分配不会受到java堆大小的限制,但既然是内容还是会受到本机总内存的限制.所以有时如果忽略直接内存,可能就会导致动态扩展时出现OutOfMemoryError异常.

知识相关:
新生代和来年代的比例=1:2 新生代占1/3 老年代占2/3
新生代Eden和两个新生代Survivor的比例=8:2 Eden占新生代的8/10 一个Survivor1/10
JIT即时编译器,将热点代码通过即时编译器编译为本地代码.
Full GC(整个堆和方法区的GC) Minor GC(新生代GC) Major GC(老年代GC)

除了CMS、G1、ZGC收集器外,别的收集器在进行可达性分析时都需要用户线程到达安全点.在枚举GC Roots的时候用户线程是肯定需要暂停的.
对于G1收集器而言,实际上在初始标记和最终标记也是需要暂停用户线程的,以及在最终的筛选回收(标记-整理)时也要暂停用户线程.
Full GC后会对内存碎片进行整理
在并发清理老年代的时候CMS是可以和用户线程一起运行的.但这就导致了CMS会产生浮动垃圾从而导致会更快地出现下一次Full GC
G1收集器相较于CMS收集器会花费更多的内存、以及更多的额外负载(当下CMS和G1还不能完全分高下),所以CMS更适合小内存而G1更适合大内存.
Shenandoah相对于G1的优点:
支持并发的标记-整理(不需要挂起用户线程)
Shenandoah收集器比G1更猛,但不支持OracleJDK,只支持OpenJDK12以上的版本.
ZGC:
ZGC收集器是一款基于Region内存布局的,(暂时)不设分代的,使用了读屏障、染色指针和内存多重映射等技术来实现可并发的标记-整理算法的
ZGC目前只支持64位Linux不支持32位,在JDK14中可以支持Windows、Mac OS
ZGC的全过程都支持并发,但有些阶段会有小挺短.例如和之前一样枚举GCRoots时会停顿.

优化知识相关:
可以通过强制设置例如堆空间的初始大小和堆空间的最大值一致避免因为扩容而导致的GC,扩容的时间可以忽略不计.包括元数据的大小.
如果是运行在客户端模式下的虚拟机而且物理内存较小的情况下可以选择Serial的新生代收集器.如果线程数CPU核心数和线程数较低则不要使用CMS收集器.

-XX:MaxMetaspaceSize=128M -XX:MetaspaceSize=128M,不管这两个值怎么设置.MetaSpace区容量范围为[20.8m,MaxMetaspaceSize) 设置-XX:MetaspaceSize值代表当元空间达到该值时才进行扩容.
元空间默认的大小就是20.8M所以建议将这两个值设置一样大.

日志名词相关解释:
user:进程执行用户态代码所耗费的处理器时间
sys:进程执行核心态代码所耗费的处理器时间
real:执行动作从开始到结束耗费的时钟时间

Parallel Time:GC Stop World时间
GC Workers:并发执行GC的线程数量

GC Worker Start (ms):GC开始时间
GC Worker End (ms):GC结束时间
Min:任务最短时长(最好和Max差不多)
Max:任务最长时长(最好和Min差不多)
Avg:平均值
Diff:偏移平均时间的值,越小越好

Ext Root Scanning:直接GCroot链的关联时间

Update RS (ms):更新RSet的时间信息,一般这个时间小于目标时间的10%即可(目标时间指的是当前GC的时长)
Processed Buffers:已处理缓冲区.这个阶段处理的是在优化线程中处理dirty card分区扫描时记录的日志缓冲区.

Scan RS:关于RSet的粒度,如果RSet中的Bitmap是粗粒度的,那么就会增加RSet扫描的时间.

Code Root Scanning:代码根的扫描,只有在分区的RSet有强代码根时会检查CSet的对内引用,例如常量池.

Object Copy:该任务主要是对CSet中存活对象进行转移(复制).对象拷贝的时间一般占用暂停时间的主要部分.如果拷贝时间和"预期暂停时间"有相差很大,也可以调整年轻代尺寸大小.

Termination:Work线程在工作终止前会检查其他工作线程的任务,如果其他work线程有没完成的任务,会抢活.如果终止时间较长,可能是某个work线程在某项任务执行时间过长.

GC Worker Other:花在GC之外的工作线程的时间,比如因为JVM的某个活动,导致GC线程被停掉.这部分消耗的时间不是真正花在GC上,只是作为log的一部分记录.

GC Worker Total:并行阶段的GC汇总,包含了GC以及GC Worker Other的总时间.

[Code Root Fixup: 0.1 ms]	GC串行化活动
[Code Root Purge: 0.0 ms]	GC串行化活动
[Clear CT: 0.1 ms]		GC串行化活动

GCOther:一般这个值超过GC暂停的10%就需要注意
    Choose CSet:选择CSet,因为年轻代的所有分区都会被收集,所以CSet不需要选择,消耗时间都是0ms.Choose CSet任务一般都是在mixed gc的过程中触发.
    Ref Proc、Enq:引用处理主要针对弱引用,软引用,徐引用,final,JNI引用.将这些引用排列到相应的reference队列中.
    Redirty Cards:重新脏化卡片.排队引用可能会更新RSet,所以需要对关联的Card重新脏化(Redirty Cards).
    Humongous Register、Reclaim:主要是对巨型对象回收的信息,youngGC阶段会对RSet中有引用的短命的巨型对象进行回收,巨型对象会直接回收而不需要进行转移(转移代价巨大,也没必要).
    Free CSet:释放CSet,其中也会清理CSet中的RSet.
