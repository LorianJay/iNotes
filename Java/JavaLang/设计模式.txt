							设计模式
总体来说设计模式分为三大类：

创建型模式,共五种:工厂方法模式、抽象工厂模式、单例模式、建造者模式、原型模式.

结构型模式,共七种:适配器模式、装饰器模式、代理模式、外观模式、桥接模式、组合模式、享元模式.

行为型模式,共十一种:策略模式、模板方法模式、观察者模式、迭代子模式、责任链模式、命令模式、备忘录模式、状态模式、访问者模式、中介者模式、解释器模式.

另类：并发型模式和线程池模式

						    设计模式6大原则
1、开闭原则(Open Close Principle)(貌似又称作单一职责)
开闭原则就是说对扩展开放,对修改关闭.在程序需要进行拓展的时候,不能去修改原有的代码,实现一个热插拔的效果.所以一句话概括就是：为了使程序的扩展性好,易于维护和升级.想要达到这样的效果,我们需要使用接口和抽象类,后面的具体设计中我们会提到这点.

2、里氏代换原则(Liskov Substitution Principle)
里氏代换原则(Liskov Substitution Principle LSP)面向对象设计的基本原则之一. 里氏代换原则中说,任何基类可以出现的地方,子类一定可以出现. LSP是继承复用的基石,只有当衍生类可以替换掉基类,软件单位的功能不受到影响时,基类才能真正被复用,而衍生类也能够在基类的基础上增加新的行为.里氏代换原则是对"开-闭"原则的补充.实现"开-闭"原则的关键步骤就是抽象化.而基类与子类的继承关系就是抽象化的具体实现,所以里氏代换原则是对实现抽象化的具体步骤的规范

3、依赖倒转原则(Dependence Inversion Principle)
这个是开闭原则的基础,具体内容：针对接口编程,依赖于抽象而不依赖于具体.

4、接口隔离原则(Interface Segregation Principle)
这个原则的意思是：使用多个隔离的接口,比使用单个接口要好.还是一个降低类之间的耦合度的意思,从这儿我们看出,其实设计模式就是一个软件的设计思想,从大型软件架构出发,为了升级和维护方便.所以上文中多次出现：降低依赖,降低耦合.

5、迪米特法则(最少知道原则)(Demeter Principle)
为什么叫最少知道原则,就是说：一个实体应当尽量少的与其他实体之间发生相互作用,使得系统功能模块相对独立.

6、合成复用原则(Composite Reuse Principle)
原则是尽量使用组合的方式,而不是使用继承.
-个人理解:因为java是双继承,所以java的继承基本没什么用.比如在开发Android的时候,一个Activity会默认继承自一个类.


							创造者模式
·意义:帮助我们构建对象(关注对象的创建过程)
·分类:单例模式、工厂模式、抽象工厂模式、建造者模式、原型模式

{
·单例模式:保证一个类只有一个实例,并且提供一个访问该实例的全局访问点.
·分类:
·主要:
	饿汉式:线程安全,调用效率高.但是,不能延迟加载
	懒汉式:线程安全,调用效率不高.但是,可以延迟加载.
·其他:
	双重检测锁式:由于jvm底层内部模块原因,偶尔会出现问题.不建议使用
	静态内部类式:线程安全,调用效率高.但是,但是可以延迟加载
	枚举单例:线程安全,调用效率高,不能延迟加载.并且可以天然的防止反射和反序列化漏洞.

{
饿汉式:类初始化的时候立即加载,构造器私有化,暴露一个方法用于被返回
懒汉式:声明类而不加载,当你真正需要这个类的时候再加载.
双重检测锁式:双重if判断
}

{
·工厂模式:
实例化对象,用工厂方法代替new操作
将选择实现类、创建对象统一管理和控制
从而将调用者跟我们的实现类解耦.实现创造者和调用者的分离
·分类:
	简单工厂模式:用来生产同一等级结构中的任意产品.(对于增加新的产品,需要修改已有的代码)
	工厂方法模式:用来生产同一等级结构中的固定产品(支持增加任意产品)
{
简单工厂:增加新产品时需要修改已有代码
工厂方法模式:对于一个新的产品,需要增加产品类以及产品的工厂实现类.增加新的产品不需要修改已有代码.但是调用者还是需要知道有哪个工厂,所以和直接new感觉没什么区别..真的没什么用..
}

·抽象工厂模式:适用于一个接口有好多实现类.这些实现类重写方法,这些方法返回的又是针对单个产品不同的实现类.

{
·建造者模式:真的很复杂
 -分离了对象子组件的单独构造器(由builder来负责)和装配(由Director)负责.从而可以构造出复杂的对象.这个模式适用于:某个对象的构建过程复杂的情况下使用.
 -由于实现了构造和装配的解耦.不同的构造器,相同的装配,也可以做出不同的对象;相同的构造器,不同的装配顺序也可以做出不同的对象.也就是实现了构建算法、装配算法的解耦,实现了更好的复用.
}

{
·原型模式
 -通过new产生一个对象需要非常繁琐的数据准备或访问权限,则可以使用原型模式.
 -就是java中的克隆技术,以某个对象为原型,复制出新的对象.显然,新的对象具备原型对象的特点
 -优势有:效率高(直接克隆,避免了重新执行构造过程步骤)
 -克隆类似于new,但是不同于new.new创建新的对象属性采用的是默认值.克隆出的对象的属性值完全和原型对象相同.并且克隆出的    新对象改变不会影响原型对象.让后再修改克隆对象的值.

·实现方法:在需要克隆的对象里面重写Cloneable接口,重写clone方法.首先实例对象,再调用该对象的clone方法,用该类接手,虽然是不同的对象,但是所有的值都相同.

}


							结构型模式

·意义:是从程序的结构上实现松耦合,从而可以扩大整体的类结构,用来解决更大的问题.(关注对象的组织关系)
·分类:适配器模式、代理模式、桥接模式、装饰模式、组合模式、外观模式、亨元模式.
·共同特点:都有另外类的引用
{
·适配器:
-目标接口(Target):客户所期待的接口.目标可以是具体的或抽象的类,也可以是接口.
-需要适配的类(Adaptee):需要适配的类或适配者类
-适配器(Adapter):通过包装一个需要适配的对象,把原接口转换成目标接口.
·适用:比如说有A类中有一个方法需要传入B接口的实现类C完成.但是我想调用D类的方法.怎么办?
在实现类C类里面,使用组合,调用D类的方法.
}

{
·代理模式:
·核心作用:通过代理,控制对对象的访问!
	   可以详细控制访问某个对象的方法,在调用这个方法前做前置处理,调用这个方法后做后置处理.
·核心角色:
 -抽象角色:定义代理角色和真实角色的公共对外方法
 -真实角色:实现抽象角色,定义真实角色所要实现的业务逻辑供代理角色调用.关注真正的业务逻辑.
 -代理角色:实现抽象角色,是真实角色的代理,通过真实角色的业务逻辑方法来实现抽象方法,并可以附加自已的操作.

·应用场景:
 -安全代理:屏蔽对真实角色的直接访问
 -远程代理:通过代理类处理远程方法调用(RMI)
 -延迟加载:先记载轻量级的代理对象,真正需要再加载真实对象.

·分类
 -静态代理(静态定义代理类,static proxy)		//持有真实角色的引用.通过静态代理可以找到真实角色.实现了抽象角色.
 -动态代理(动态生成代理类)
  动态dialing相比于静态代理的优点:抽象角色中声明的所以方法都被转移到调用处理器一个集中的方法中处理,这样,我们可以更  加灵活和统  一的处理众多的方法.
	·静态代理和动态代理的区别:静态代理是有我们自已定义,动态代理有程序自动生成代理类(常用).
	·动态代理又可分为:
		JDK自带的动态代理
		javaassist字节码操作库实现
		CGLIB
		ASM(底层使用之类,可维护性较差)	
{
·静态代理:
·具体实现:什么操作都是在代理模式里面实现的,一旦要有真实角色的实现的话,直接在代理里面调用真实角色的方法.
}

{
·桥接模式(bridge):
 -用于归纳商品的层次结构.便于添加新的商品元素.
 -这样我们添加新的商品品牌或商品类型时.就可以分别添加.
·桥接模式总结:
 -桥接模式可以取代多层继承的方案.多层继承违背了单一职责原则.复用性较差,类的个数也非常多.桥接模式可以极大的减少类的  个数.,从而降低管理和维护的成本.
 -桥接模式极大的提高了系统可扩展性,在两个变化维度中,任意扩展一个维度,都不需要修改原有的系统,符合开辟原则.
·特点:
当一个产品存在多种性质的时候,(比如手机存在:Android IOS操作系统的性质,又存在iPhone,HUAWEI,OPPO,VIVO等品牌性质),此时就可以使用桥接模式.这个存在一个继承类,和多个实现接口类.对于每个单独的性质都需要一个接口,建造新的类重写该接口的方法,来具体实现性质的差异化.(实现接口类).在继承类里面,需要创建一个父类,该父类拥有所有性质接口的引用,并且在构造方法里面传入具体性质的实现类,是一个回调.并且还需要一个方法,该方法是调用对于性质的具体操作的代码.(也就是要在这个方法里面,通过声明的属性调用接口里声明的方法).对于剩下的一个性质,每个性质都需要继承自这个父类.(继承父类类)

}

{
·组合模式(composite):		//组合模式不是组合
 -使用组合的常见:把部分和整体的关系用树形结构来表示,从而使客户端可以使用统一的方式处理部分对象和整体对象.
 ·组合模式核心:
  -抽象构件角色(Component):定义了叶子和容器构件的共同点.
  -叶子构件角色(Leaf):无字节点
  -容器构件角色(Composite):有容器特征,可以包含字节点.
·组合模式工作流程分析:
 -组合模式为处理树形结构提供了完美的解决方案,描述了如何将容器和叶子进行递归组合,使得用户在使用时可以一致性的对待容  器和叶子.
 -当容器对象的指定方法被调用时,将遍历整个树形结构,寻找也包含这个方法的成员,并调用执行.其中,使用了递归调用的机制对整  个结构进行处理.
·只要想到树形结构,就用组合模式.(最常见的就是文件夹结构)
·好处:无论是叶子还是容器都有统一的处理方式.
·特点:叶子和容器都继承自同一接口,并且在容器构建角色内要定义一个容器,这个容器的add方法,可以添加当前实现的接口的实现类.也就是说一个节点的下面可以放置节点或者容器.(有点类似于容器里面放置容器)

}

{
·装饰器模式:
 -职责:
  -动态的为一个对象增加新的功能.
  -装饰模式是一种用于代替继承的技术,无需通过继承增加子类就能扩展对象的新功能.使用对象的关联关系代替继承关系,更加灵   活,同时避免类型体系的快速膨胀.

·实现细节:类似于IO流中的BufferedInPutStream的特点.

 -Component抽象构件角色(真实角色和抽象角色都需要实现的接口):
  ·真实对象和装饰对象有相同的接口.这样,客户端对象就能够以与真实对象相同的方式同装饰对象交互.真实对象和抽象对象都需要实现的接口

 -ConcreteComponent具体构件角色(真实角色):
  ·io流中的FileInputStream、FileOutputStream

 -Decorator装饰角色:
 ·持有一个真实对象的引用.装饰对象接受所有客户端的请求,并把这些请求转发给真实的对象.这样,就能在真实对象调用前后增加新的功能.	   持有对真实对象的引用

 -ConcreteDecorator具体装饰角色:		//继承装饰角色
  ·负责给构件对象增加新的功能.

·总结:
 -装饰模式(Decorator)也叫包装器模式(Wrapper)
 -装饰模式降低系统的耦合度,可以动态的增加或删除对象的职责,并使得需要装饰的具体构件类和具体装饰类可以独立变化,一遍增  加新的具体构件类和具体装饰类.

·优点:
 -扩展对象功能,比继承灵活,不会导致类个数急剧增加.
 -可以对一个类进行多次装饰,创造出不同行为的组合,得到功能更加强大的对象.
 -具体构件类和具体装饰类可以独立变化,用户可以根据需要自已增加新的具体构件子类和具体装饰子类.

·缺点:
 -产生很多小对象.大量小对象占据内存,一定程度上影响性能.
 -装饰模式易于出错,调查排查比较麻烦.

·装饰模式和桥接模式的区别:
 -两个模式都是为了解决过多子类对象问题.但他们的诱因不一样.桥模式是对象自身现有机制沿着多个维度变化,是既有部分不稳定  .装饰模式是为了增加新的功能.
·特点:如果要实现一个功能可以通过基本的类之间的关系进行组合,就可以用装饰者模式.换句话说,就是通过类之间关系的组合实现更多的功能.
}

  -类似于封装的套路

{
·享元模式:用于很多相似的对象

 ·场景:
  -内存属于稀缺资源,不要随便浪费.如果有很多个完全相同或相似的对象,我们可以通过享元模式,节省内存.
 
·核心:
  -享元模式以贡献的方式高效地支持大量细粒度对象的重用.
  -享元对象能做到贡献的关键是区分了内部状态和外部状态.
   内部状态:可以贡献,不会随环境变化而改变
   外部状态:不可以共享,会随环境变化而变化.
 
 ·享元模式实现:一般和工厂模式搭配起来

 -FliweightFactory享元工厂类:
  ·创建并管理享元对象,享元池一般设计成键值对.

 -FlyWeight抽象享元类:(享元类)
  ·通常是一个接口或抽象类,声明公共方法,这些方法可以向外界提供对象的内部状态,设置外部状态.

 -ConcreteFlyWeight具体享元类:(代表:内部状态,可共享)
  ·为内部状态提供成员变量进行储存

 -UnsharedConcreteFlyWeight非共享 享元类(代表:外部状态,不可共享)
  ·不能被共享的子类可以设计为非共享享元类

 ·优点
  -极大减少内存中对象的数量
  -相同或相似对象内存中只存一份,极大的节约资源,提高系统性能
  -外部状态相对独立,不影响内部状态
 ·缺点
  -模式较复杂,是程序逻辑复杂化
  -为了节省内存,共享了内部状态,分离出外部状态,而读取外部状态使运行时间变长,用时间换取了空间.

							行为型模式
·核心作用:
 -关注系统中对象之间的相互交互,研究系统在运行时对象之间的相互通信和协作,进一步明确对象的职责,共有11中模式.
 -分类:职责链模式、迭代器模式、中介者模式、命令模式、解释器模式、策略模式、模板方式(钩子方法)模式、状态模式、观察者模式、备忘录模式、访    问者模式.
 -不常用的三种模式:
  -解释器模式:一般用于构建新的脚本语言才使用(java可以动态执行脚本语言)
  -命令模式:用于底层
  -访问者模式:没讲???
{
·责任链模式: 
 -定义:将能够处理同一类请求的对象连成一条链,所提交的请求沿着传递,连上的对象逐个判断是否有能力处理该请求,如果能则处理,如果不能则  传递给链的下一个对象.
·场景:
 -打牌时,轮流出牌
 -接力赛跑
·个人理解:当我们要处理一个请求时,我们就沿着这条链,挨个找链子上的对象做处理.
·对于为什么不用if else语句的解释:其一如果我们要修改流程的话,就要修改源代码.其二这会使得整个程序结构非常复杂,不利于扩展,不利于维护.
·特点:拥有一个抽象处理者抽象类,其余的具体处理者都需要继承这个抽象类.其余的具体处理者就相当于链子的每一个节点.需要注意的是抽象处理者抽象类拥有一个特殊的属性和方法,有一个当前抽象类类型的属性nextLeader,该属性就指代下一级的处理者(我们需要对该属性提供set方法).有一个handleRequest抽象方法,是处理请求的核心.这个方法应当定义成抽象方法,形参类型是请求类对象.这样的好处是,可以直接通过nextLeader.handleRequest(请求对象).来进行下一请求.handleRequest是核心!!!

·好处:当增加新的流程的时候不需要修改已有的代码,只需要在客户端设置下家即可.
}

·策略模式:
·场景:
 -某个市场人员接到单后的报价策略.报价策略很复杂,可以简单作如下分类:

·模板方法模式:注意模板方法模式的核心是模板方法!!!是一个用final修饰无法更改的方法.该方法内的操作内容就是模板.
·场景:
 -客户到银行办理业务:
 ①取号排队
 ②办理具体现金/转账/企业/个人/理财业务
 ③给银行工作人员评分
·模板方法模式介绍:
  -模板方法模式是编程中经常用得到的模式.它定义了一个操作中的算法骨架,将某些步骤延迟到子类中实现.这样,新的子类可以在不改变一个      算法结构的前提下重新定义该算法的某些特定步骤.
·核心:
  -处理某个流程的代码已经都具备,但是其中某个节点的代码暂时不能确定.因此,我们采用工厂方法模式,将这个节点的代码实现转移给子类完     成.即:处理步骤父类中定义好具体实现延迟到子类中定义.
·方法回调(钩子方法)
  -好莱坞原则"Don't call me ,we'll call you back"
·在好莱坞中,当艺人把建立递交给好莱坞娱乐公司时,所能做的就是等待,整个过程由娱乐公司控制,演员只能被动的服务安排,在需要的时候  再  由公司安排具体环节的演出.
   -在软件开发中,我们可以将call翻译为调用.子类不能调用父类,而通过父类调用子类.这些调用步骤已经在父类中写好了,完全有父  类控制整  个过程.
·什么时候用到模板方法模式:
   -实现一个算法时,整体步骤很固定.但是,某些部分易变.易变部分可以抽象成出来,供子类实现.
   -实际上很类似于回调的应用场景,做一件事步骤一步骤二步骤四都知道怎么实现,就是不知道步骤三怎么办的时候用回调.
·开发中常见的场景:
   -非常频繁.各个框架、类库中都有他的影子.
·个人理解:
首先要定义一个抽象类,在这个抽象类里面有一个模板方法,这个方法里面执行的操作都是写死的.所以我们在模板方法里面可以调用任意方法,因为是抽象类,所以调用的方法也不一定要有实现,所以在这个模板方法里面调用没有实现的方法就称为钩子方法.当我们在客户端调用的时候有两种方法,一种是写一个抽象类的子类,实现钩子方法,在new的时候new实现的子类.或者直接new当前的这个抽象类,然后用匿名内部类的方式去重写这个钩子方法,此时调用模板方法,就会调用重写的钩子方法.
}

·观察者模式:稍微有点难,有两种方式
·适用:
 -适用于广播机制的场景,需要群体广播信息的时候采用.
·介绍:
 -我们可以把多个订阅者、客户称之为观察者.需要同步给多个订阅者的数据封装到对象中,称之为目标.
·核心:
 -观察者模式主要用于1对N的通知.当一个对象(目标对象Subject或Objservable)的状态变化时,他需要及时告知一系列对象(观察者对    象,Observer),令他们做出响应
 -通知观察者的方式:
 ·推:
   -每次都会把通知以广播方式发送给所有观察者,所有观察者只能被动接受.
 ·拉
   -观察者只要直到有情况即可.置于什么时候获取内容,获取什么内容,都可以自主决定.
·角色:
以广播为例子,这里有播音员和收音机两种角色
收音机:观察者,观察目标,一旦目标发生改变所有的观察者都能接收到
播音员:消息发送者(目标)目标发生改变
·个人理解:
-观察者接口:里面只定义了一个方法,需要传入一个参数,类型是目标对象(接口)
-目标对象(可以是类,也可以是抽象类):里面需要定义一个容器加上泛型,泛型类型为观察者接口,提供add和remove方法和发送方法(send).这个发送方法需要遍历容器,并且调用每一个观察者里面的那个唯一的方法,传入参数直接填入this.
-具体目标:当前的类要继承目标对象,而且所有需要发送的内容都需要定义在这个类里面,比如说定义一个字符串再定义一个int值,等等,想定义什么定义什么,但是一定要对每个需要发送的内容提供get和set方法,这个set方法除了修改内容之外还要this.send(),(在每一个set方法里面都需要this.send())调用父类的方法,把自已遍历给每一个观察者对象.
具体观察者:需要实现观察者接口,重写唯一的一个方法.在这个具体的观察者类里面也需要定义所有接收的内容,要和具体目标对应,比如这里就是一个String+int.通过这个方法的形参来获取数据,我们需要将这个形参强制转换为具体目标类,然后再调用get方法,使得当前类里面所有接收的内容与具体目标发出的内容一致.然后在这个类里面也要提供所有接收内容的get方法.

·通过javase自带的类来实现观察者模式 
·介绍:JAVASE提供了java.util.Observable和java.util.Observer来实现观察者模式
·个人理解:注意这两个类在jdk11+中已经被废弃了.我们的理念是能实现的尽量不继承(讲个笑话java单继承233)
目标对象类:Observable
真实对象类:继承Observable
观察者接口:Observer
真实观察者:实现Observer
注意:这个真的是要结合练习的代码来看了.但要理解上面一种方式的内容.
