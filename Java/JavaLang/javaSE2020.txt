security:
数字签名、数字证书
公钥是公开的,私钥只存在用户这一端,用公钥加密出来的内容可以用私钥完全解密.
数字签名:拥有私钥的用户,现在想发布一些内容,但是怎么能保障这个内容就是该用户发出来的呢?用户首先对要发布的原文进行私钥加密(这一步就已经生成数字签名了)
然后再对生成的数字签名进行摘要运算(防止内容过大),将原文和摘要后的文本一并传给接受者.接受者拿到摘要后的签名后先解码出原先的数字签名
最后接受者根据原文以及自身的公钥去验证这个数字签名是否正确.
数字证书:但假设现在有人把接受者的公钥修改为自已的公钥,那么此时篡改者只要用自已的私钥加密数字签名再发送给接受者,因为接受者的公钥已经改为和篡改者的一致
但是接受者并不知情,还以为自已的公钥是发送者的公钥,那么此时篡改者发布的内容同样可以被解密(因为公钥一致).现在为了避免这种情况出现了数字证书
数字证书(CA)是做公钥验证的,证书中心用自已的私钥对内容发送者的公钥和一些相关信息做加密生成数字证书.
这样以后内容发布者只要发布内容的时候同时带上数字签名和数字证书发送即可,接受者拿到数字证书后就直接从数字证书中获取公钥(这一步也就是通过数字证书确认公钥/或者叫获取公钥,但前提是这个证书接受者也必须得有,需要通过这个数字证书来解密发送过来的公钥)
那么如何保证数字证书不被伪造呢?数字证书是存储在电脑中的,相当于数字证书现在就是这个公钥,虽然可以获取到公钥,但是无法伪造.

但是在正常的网络通行中,非对称加密(比如RSA)这种加密解密是非常耗费性能的,但是对称加密是不怎么耗费性能的.那么SSL/TLS的思想就是,先生成一个对称加密的密钥
然后通过非对称加密加密这段对称加密的密钥.然后传输给对方,那么此时两者通信就还是通过对称加密来通信,但是对称加密的密钥的传输还是通过非对称加密来完成.
这基本上就是SSL/TLS的工作流程了

java中私钥签名可以用公钥验证签名.公钥加密可以用私钥解密.加密是加密签名是签名.但实际上私钥签名的本质是私钥加密,公钥验签的本质是公钥解密.这个和公钥加密/私钥解密并不冲突(另外记得公钥加密只允许加密117字节,私钥解密只能解密128字节.如果超出需要分段加密/解密)

网站申请SSL的步骤和配置:https://pay.weixin.qq.com/wiki/doc/api/tools/cash_coupon.php?chapter=10_4
另外SSL协议并不安全,最好使用TLS1.2协议,或者使用SSL3

摘要算法:
MD5是一种摘要算法,它不是加密算法,长度固定,是不可逆的.它实际上也就是HASH算法,会存在HASH碰撞的问题.
BASE64是一种摘要算法,它也不是加密算法,只是一种编码方式.Base64编码之所以称为Base64,是因为其使用64个字符来对任意数据进行编码,同理有Base32、Base16编码.
在BASE64中,如果最后有等于号=,那么这个等于号起到的作用只是补位,没有实际意义,可以省略,所以一般遇到等于号=就代表结束符


在微信V2中:
我们拥有:商户证书(API证书)该证书可通过权威CA证书颁布.该证书包含:商户号、证书序列号、公司名称、公钥(其中的公钥可以通过X509Certificate获得)+私钥(私钥单独存在apiclient_key.pem文件中)、支付密钥(32位单独到微信上设置)
商户证书可以用于双向验证、私钥解密.当我们在服务器上安装商户证书时需要用到证书密码(这个密码很可能就是商户ID).当我们请求的时候直接带上证书是没有的,还需要一并带上证书密码,获得到KeyManager后带上KeyManager去请求.
当我们生成签名的时候,先根据约定好的规则拼接请求参数,以及将支付密钥拼接进去.然后对内容进行MD5或者HMACSha256摘要计算,并添加到请求体中(而且要告诉微信当前使用的哪种签名算法).对于微信而言,验证签名只需要将请求体中的参数也按照约定拼接,并在最后加上支付密码,进行MD5或者HMACSha256摘要计算出签名,并验证签名是否一致即可.
对于微信回调而言,同样也会遵循上述规则生成数字签名,作为开发第一步不管是回调还是响应内容都要先验证数字签名(同样微信也会告诉你当前使用哪种签名算法)

在微信V3中:V3中的特性完全兼容V2,相当于从V2中继承过来的
我们拥有:商户证书、平台证书(该证书包含:公钥、平台标识、证书序列号)、APIV3密钥(32位对称密钥单独到微信上设置,采用AES256GCM算法)

平台证书在微信支付上获取,平台证书是针对V3接口使用的,平台证书会定期更换,不同商户对应的平台证书是不同的.平台证书可以用于对微信的响应报文或者微信回调的请求报文进行签名校验(和V2不同的是V3要用私钥签名公钥验签)
APIV3密钥可以用于微信回调和平台证书的下载中,对关键信息进行对称加密.
当我们通过微信接口获取平台证书的时候(同样要先验证签名),为了安全该证书会经过微信APIV3密钥(AES256GCM算法即AES算法)进行加密,当拿到该证书之后,再通过APIV3密钥解密即可(对称加密).
平台证书并没有一个确切的文件,只是当我们https://api.mch.weixin.qq.com/v3/certificates接口时,该请求的响应信息就是我们的平台证书(该响应报文中就包含了公钥,但该公钥需要处理,接着往下看)
同样的对于平台证书的响应信息我们也要验签,但现在就出现了一个矛盾.因为验签是需要平台证书的,而我们现在正是要获取平台证书(我们没有平台证书),那怎么办?对于第一次证书的下载,微信提供了下载工具:https://github.com/wechatpay-apiv3/CertificateDownloader
之后当更替平台证书的时候,首先我们也得先验签,如果验签成功了,则从响应报文中得到String类型的公钥,同理该公钥得先BASE64解码成字节数组.解码之后由于微信对该公钥进行APIV3加密(AES-256-GCM/AES),所以得先通过APIV3密钥对其进行解密,解密后是字节数组.然后再对该字节数组BASE64编码为字符串保存到本地即可(此时就是字符串形式的PublicKey)
同样如果后续用到该公钥验签的时候,先通过BASE64解码为字节数组后再通过KeyFactory+KeySpec得到PublicKey实例.
响应信息可能是一个数组(因为平台证书可能不止一个),其中响应信息中也包含了每一个证书的证书序列号(这个需要对应上).

当我们生成签名的时候,先根据约定好的规则拼接一些参数(注意这里和V2不同,这里包括:请求方法、请求URL绝对路径、请求时间戳、请求随机串、请求报文(这个请求报文就是我们将要请求微信的JSON串))
然后用商户证书(API证书)中的私钥(apiclient_key.pem)对内容进行私钥签名(采用SHA256-RSA签名算法),注意签名出来的内容是字节数组,还需要通过BASE64将其编码为String.此时生成出来的叫签名值(还不叫签名)
然后拼接以下内容:WECHATPAY2-SHA256-RSA2048(认证类型,该值目前固定) mchid="商户ID",serial_no="商户证书序列号",nonce_str="请求随机串",timestamp="时间戳",signature="签名值" 将拼接出来的内容添加到请求头Authorization中.注意在V3接口请求中不需要带上TLS证书(貌似是因为直接通过CA证书交互了?)
然后微信拿到请求头中的信息拿到响应正文,按照同样的规则拼接出原文!然后根据Authorization指定的商户证书序列号得到对应的公钥,然后使用公钥进行验签.其中微信拿到的签名值signature不能直接使用,因为signature被BASE64编码过,所以signature要通过BASE64解码为字节数组,然后再验证(而且验证的时候签名正好得要字节数组形式)
对于微信的响应或者微信回调,作为开发同样需要去验证签名,在V3接口中的签名验证和V2不同.
首先微信平台同样也会根据约定好的规则拼接一些参数(包括:应答时间戳、随机串、响应报文).然后微信通过平台证书中的私钥对内容进行私钥签名.然后微信会在请求头Wechatpay-Timestamp中指定时间戳、Wechatpay-Nonce指定随机串、Wechatpay-Serial指定当前使用的平台证书序列号(因为平台证书可能有多份)
当我们拿到请求信息后,根据同样的规则拼接这些参数,并根据平台证书序列号找到证书获得公钥,然后使用公钥进行验签(采用SHA256-RSA签名算法)在微信V3接口中指出,签名算法采用SHA256-RSA!!!验证的时候对于微信的签名同样要BASE64解码为字节数组!

当我们在请求中如果要对敏感信息进行加密时,使用RSA加密算法,加密算法的填充方案选择RSAES-OAEP.我们需要使用平台证书的公钥对敏感信息加密,然后微信拿到加密后的内容后使用私钥对其解密.(私钥只保存在微信上)
当微信在响应信息中要响应敏感信息时,微信会使用商户证书(API证书)中的公钥对敏感信息进行加密,然后我们根据证书序列号使用对应的公钥进行验签.证书序列号可以在请求中获得.
微信的回调通知相当复杂,首先微信会对所有的敏感信息使用APIV3对称密钥进行加密(采用AEAD_AES_256_GCM算法),加密后是字节数组所以微信会对加密后的内容进行BASE64编码.在加密的时候需要用到associatedData、nonce(这两个会作为字符串出现在响应体中,使用的时候应该也要BASE64解码为数组,这两个值有没有被对称加密过还不清楚,如果有就记得解密再使用)
首先还是先验证签名,然后从响应体中获得associatedData、nonce、密文,然后使用APIV3对称密钥进行解密(采用AEAD_AES_256_GCM算法).

如何从apiclient_key.pem证书中得到商户的私钥?(不是直接获取的!!)因为要指定密钥的加密算法(就算是解析证书中的密钥你也需要告诉程序当前使用的什么加密算法)
首先通过IO流读取到证书,证书会以特定的字符串作为开头和结尾,将这些特定的开头和结尾替换掉.然后通过PKCS8EncodedKeySpec将字符串转为PrivateKey对象.
之前一直在强调的,PrivateKey取到的密钥是以byte的形式存放的,所以还要将该字节数组通过BASE64编码后的结果才是密钥的字符串形式!!

有的时候微信需要回调(调用)我们的接口,微信会对回调请求进行签名(微信平台使用密钥数字签名),我们的接口必须通过平台证书的公钥去验证签名.

证书类型:
.DER、.CER:文件是二进制格式,只保存证书(公钥),不保存私钥.
.PEM:一般是文本格式,可保存证书(公钥)，可保存私钥.
.CRT:可以是二进制格式,可以是文本格式,与 .DER 格式相同,不保存私钥.
.PFX、P12:二进制格式,同时包含证书(公钥)和私钥,一般有密码保护.
.JKS:二进制格式,同时包含证书(公钥)和私钥,一般有密码保护.

缓冲区中有四个核心变量:
容量Capacity:缓冲区能够容纳的数据元素的最大数量(大小在创建的时候确定,不能改变)
上界Limit:缓冲区里的数据的总数,代表了当前缓冲区中一共有多少数据
位置Position:下一个要被读或写的元素的位置.
标记Mark:一个备忘位置,用于记录上一次读写的位置.

因为Buffer分为读和写,所以这些变量要分情况讨论,也就是一个变量往往代表多重含义,在一开始写的时候,Limit和Capacity一样,因为一开始Buffer是什么都没有的,那么
你指定的缓冲区大小(Capacity)就是这个缓冲区能存放数据的上限(也就是Limit),对于写而言能写的上限不就是容量吗?对于写而言,每写一个值它的位置Position就自增
当我们调用flip​()方法将缓冲区改为读模式时,对于读而言上界Limit此时的大小就是我们写入的数据量(元素个数),这个也很容易理解因为你现在缓冲区就这么多数据,那么
Limit自然就是缓冲区能读取到的元素的个数,此时位置Position为0,因为是才切换到读模式的,所以肯定是从0开始往后读(就是我们说的一样类似数组),缓冲区的容量Capacity
是自始至终不变的.

FileLock:文件锁(通过FileChannel获得该对象)详细内容https://www.cnblogs.com/chy18883701161/p/10940390.html 这篇博客写的不错
文件锁是进程级别的,不是线程级别的.文件锁可以解决多个进程并发访问、修改同一个文件的问题,但不能解决多线程并发访问、修改同一文件的问题.
文件锁是当前程序所属的JVM实例持有的,一旦获取到文件锁(对文件加锁),要调用release(),或者关闭对应的FileChannel对象,或者当前JVM退出,才会释放这个锁.
一旦某个进程(比如说JVM实例)对某个文件加锁,则在释放这个锁之前,此进程不能再对此文件加锁,就是说JVM实例在同一文件上的文件锁是不重叠的(进程级别不能重复在同一文件上获取锁)
排它锁:又叫独占锁.对文件加排它锁后,该进程可以对此文件进行读写,该进程独占此文件,其他进程不能读写此文件,直到该进程释放文件锁.
共享锁:某个进程对文件加共享锁,其他进程也可以访问此文件,但这些进程都只能读此文件,不能写.线程是安全的.只要还有一个进程持有共享锁,此文件就只能读,不能写.

下面是一个对文件事件监听的Demo:
public class Test {
    public static void main(String[] args) throws IOException, InterruptedException {

        //创建一个文件系统的监听服务
        WatchService watchService= FileSystems.getDefault().newWatchService();

        Path path=Paths.get("D:/untitled/test");

        //为该文件夹注册监听,监听新建、修改、删除事件。只能为文件夹（目录）注册监听，不能为单个文件注册监听
        path.register(watchService,StandardWatchEventKinds.ENTRY_CREATE,StandardWatchEventKinds.ENTRY_MODIFY,StandardWatchEventKinds.ENTRY_DELETE);

        //编写事件处理
        while (true){  //一直监听
            //拉取一个WatchKey。当触发监听的事件时，就会产生一个WatchKey，此WatchKey封装了事件信息。
            WatchKey watchKey=watchService.take();

            //使用循环是因为这一个WatchKey中可能有多个文件变化了，比如Ctrl+A全选，然后删除，只触发了一个WatchKey，但有多个文件变化了
            for (WatchEvent event:watchKey.pollEvents()){
                System.out.println(event.context()+"发生了"+event.kind()+"事件！");
                /*
                watchKey.pollEvents()  获取此次WatchKey中所有变化了的文件的信息，以List（列表）形式返回，一个WatchEvent就是一个元素，封装了一个变化了的文件的信息
                event.context()  获取文件名
                event.kind()  获取发生的事件类型

                因为只能为文件夹注册监听，如果要监听某些指定的文件，可以在增强的for循环中，先根据event.context()判断是否是指定的文件，是才处理。
                 */
            }

            //虽然是while()循环，但WatchKey和ByteBuffer一样，使用完要重置状态，才能继续用。
            watchKey.reset();    //如果不重置，WatchKey使用一次过后就不能再使用，即只能监听到一次文件变化。
        }
    }
}

如何使用模块管理机制?在一个模块下新建module-info.java文件表示当前项目采用模块管理机制,文件的格式如下:
module [modelName] {				//模块名称随便起一个就可以,但是注意必须是唯一的,以及模块内的包名也得是唯一的,即使模块不同(也就是不同模块的包名也必须是唯一的)
    
}

这样使用之后发现程序可以正常运行但是某些类无法使用了,这里无法使用Java的Logger类了,这是因为Java的java.logging包是作为一个单独的模块存在,要想使用必须先导入该模块.

module [modelName]{
    requires java.logging;		//将日志模块导入到当前项目,如果需要导入JavaSE的所有依赖,像之前一样的话,直接 requires java.se;  即可
}

所以实际上,全新的模块化机制提供了另一个级别的Java代码可见性、可访问性的控制.除了JDK的一些常用包之外,只有我们明确需要的模块才会导入依赖库.所以模块说道最后就是对包的可访问性的控制.
反射 API 的 Java 9 封装和安全性得到了改进,如果模块没有明确授权给其他模块使用反射的权限,那么其他模块是不允许使用反射进行.什么意思呢?
之前说过JDK常用的包会自动导入,假设我现在要获取String的value变量,由于value变量是private的,所以我们是无法直接获取这个属性的属性值的,那么我们就可以用反射来获取该变量.但是现在由于java.lang模块(系统模块)没有明确授权给别的模块的反射权限,所以就不允许使用反射获取.
模块机制共有四种类型:
1. 系统模块:来自JDK官方的模块,可以使用java --list-modules命令来列出所有的模块.
2. 应用程序模块:我们自已写的Java模块
3. 自动模块:对于那些不是Java9以上版本的模块项目,这种情况下是需要做兼容的,默认情况下是直接导出所有的包,可以访问所有其它模块提供的类,不然这些库就无法使用了.
4. 未命名模块:我们自己创建的一个Java项目,如果没有创建module-info.java,那么会按照未命名模块进行处理,未命名模块同样可以访问所有其他模块提供的类,这样我们之前写的Java 8代码才能正常地在Java 9以及之后的版本下运行.不过,由于没有使用Java 9的模块新特性,未命名模块只能默认暴露给其他未命名的模块和自动模块,应用程序模块无法访问这些类.

我们创建Maven项目之后如果没有module-info.Java文件则该模块为未命名模块还是按照之前的开发方式可以使用依赖库的任何类,但如果我们要采用模块化的方式来开发,新建好module-info.java文件后.
我们需要去对应的包下面找到它的module-info.java文件看看这个依赖当前暴露了哪些模块供我们使用.

	requires java.logging;		//如果模块A依赖于某个模块,那么会不会传递给依赖于模块A的模块呢?事实上是不会的.也就是说哪个模块导入的依赖只能哪个模块用.
}

moduleA{
	exports com.test to moduleB;
	requires transitive java.logging;	//通过添加参数transitive来向别的模块传递此依赖
}

--------------------------------------------------------------------------------------------------------------------------------------------------------------------------
moduleA{
	exports com.test to moduleB;
	uses com.test.Test;			//使用uses指定当前模块需要Test接口的实现类
}

moduleA{
	requires moduleA;				//将模块moduleA导入进当前项目,这样就能导入模块A的Test类了
	provides com.test.Test with com.main.TestImpl; 	//声明此模块提供了Test的实现类
}
