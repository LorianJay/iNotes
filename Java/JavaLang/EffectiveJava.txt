@1.1:考虑用静态方法代替构造器

·杂税知识点:
一个类想要通过new关键字来实例的话,那么在new这个类的这个方法必须有权限能够访问到这个构造器.

·一个静态方法实例:

public static Boolean valueOf(boolean b){
    return b ? BOOLEAN.TRUE : BOOLEAN.FALSE;
}

这个静态方法将boolean基本数据类型转换为Boolean对象引用.
BOOLEAN.TRUE的源码  public static final Boolean TRUE = new Boolean(true);
BOOLEAN.FALSE的源码 public static final Boolean FALSE = new Boolean(false);
通过创建静态final对象也可以直接将对象返回出去 

·静态工厂的优势在哪里?

1.静态工厂方法的名称可以修饰对象
静态工厂方法可以通过自身的名称来修饰将要返回的对象.但是每个构造器的名称都相同,构造器只可以通过重载的方式来修饰将要返回的对象,但通过重载的方式显然不利于描述对象.

2.不必再每次调用静态工厂方法的时候都创建一个对象(大巴黎模式)
就例如上面的valueOf()方法连续两次的(true)并不会创建重复的对象.如果程序经常请求创建相同的对象,并且创建对象的代价很高,则这项技术可以极大的提升性能.

3.它们可以返回原返回类型的任何子类的对象
这里有两个例子

多态:
public class Father {

    public void say(){
        System.out.println("Father");
    }

    private static class Child extends Father{			//此时的子类是被隐藏起来的,因为无法被访问,只能被Father类访问
        @Override
        public void say() {
            System.out.println("Child");
        }
    }

    public static Father newIn(){
        return new Child();					//返回的时候是返回父类的子类
    }

}

接口:
public interface NeedImp {
    void say();
}


public class Father2 {

    public static NeedImp newIn(){
        return new NeedA();					//返回的是接口的实现类
    }

    private static class NeedA implements NeedImp{		//同样隐藏实现类

        @Override
        public void say() {
            System.out.println("Hello");
        }
    }

}

这样做的好处是:使得API变得非常的简洁,程序员只需要关注静态工厂方法返回的类型,即使这个类型是接口类型的.也可以直接使用,注重应用,不需要关注实现

·静态工厂方法的缺点在哪里?
1.类如果不含有公有的或者受保护的构造器就不能被子类化

类如果不含有公有的或者受保护的构造器就不能被子类化: 也就是父类不能只有私有构造器
public class Father{
    private Father (){
    
    }
}

class Child extends Father{		//此时会报错

}

如果一个类没有写构造器会默认加上公开的无参构造器,但只要这个类有构造器了,无论是什么构造器,就不会加上公开的无参构造器了.

2.它们与其它的静态方法没有任何区别

2.附加:
为此我们可以通过遵守标准的命名习惯来弥补这一劣势:

valueOf--不太严格的将,该方法返回的实例与它的参数具有相同的值.这样的静态工厂方法实际上是类型转换的方法.

of--valueOf的一种更为简洁的替代.

getInstance--返回的实例是通过方法的参数来描述的,但是不能够说与参数具有相同的值,对于单例模式来说,该方法没有参数,并返回唯一实例

newInstance--想getInstance,但是newInstance是多利的.每次返回不同的对象.

getType--像getInstance一样,但是在工厂方法处于不同的类中的时候使用,Type表示工厂方法所返回的对象类型.

newType--像newType一样,但是在工厂方法处于不同的类中的时候使用,Type表示该工厂方法所返回的对象类型.

·总结:
不要第一反应就是提供公有的构造器,需要优先考虑静态工厂.我们需要理解构造器和静态工厂各自的长处,选择合适的.







@1.2遇到多个构造器参数时要考虑构建器

·为什么要使用构建器?(何时选择构建器)
如果一个对象的构造需要很多的参数,那么此时无论是使用静态工厂还是构造器都显得参数列表很长.一般当构造器需要3个以上的参数时就考虑使用构建器.
而且构建器所创造的对象是不可变的.
	
	*例如:构造器调用构造器,设置默认值为0.
	public class Constructor {
    		private int age;
   		private int fat;
    		private int sodium;

    		public Constructor() {
        		this(0, 0, 0);
    		}

    		public Constructor(int age) {
        		this(age, 0, 0);
    		}

    		public Constructor(int age, int fat) {
        		this(age, fat, 0);
    		}

    		public Constructor(int age, int fat, int sodium) {
        		this.age = age;
        		this.fat = fat;
        		this.sodium = sodium;
    		}
	}
	
	*解决这种方法的一种比较好的方式是使用javabean.也就是对每一个参数提供公开的getter/setter方法
	使用javabean方法的缺点是:因为构造过程被分到了几个调用中,在构造过程中javabean可能处于不一致的状态.而且使用javabean阻止了把类作为不可		变的可能,这就需要程序员付出额外的努力确保线程安全.

·如何使用建造者模式(builder模式)以及注意事项?
*介绍:建造者模式不直接生成想要的对象,而是让客户端在builder对象上调用类似于setter的方法,来设置每个相关的可选参数.最后客户端调用无参的build方法来生成不可变的对象.这个builder是它构建的类的静态成员类.另外builder像个构造器,可以对其参数强加约束条件(也就是调用构建方法的时候对传进来的参数进行校验,看是否满足要求).如果违反了约束条件应该抛出IllegalAccessException异常.在考虑设计的时候,如果后期添加了参数有原来使用构造器或静态工厂方法转为构建器时会无法控制,所以最好在一开始设计的时候就使用构建器.

public class ConstructorBuilder {			//该类没有可以直接修改内部数据的方法

    private int initialization;

    private int age;
    private int fat;
    private int speed;

    public ConstructorBuilder(Bulider bulider) {		//这里甚至可以把public改成private
        this.initialization = bulider.initialization;		
        this.age = bulider.age;
        this.fat = bulider.fat;
        this.speed = bulider.speed;
    }

    public static class Bulider {			//一定要是静态的以便被外部的类获取 !!!
        private int initialization;		//如果不声明权限默认是

        private int age = 0;
        private int fat = 0;
        private int speed = 0;

        public Bulider(int initialization) throws IllegalAccessException {		//构造器
            if (initialization == 0) {			//检验参数是否为0,一旦为0则抛出异常
                throw new IllegalAccessException();	//抛出IllegalAccessException异常
            }
            this.initialization = initialization;
        }

        public Bulider age(int age) {		
            this.age = age;
            return this;			//对于每一个类似setter方法,都需要返回当前对象		
        }

        public Bulider fat(int fat) {
            this.fat = fat;
            return this;
        }

        public Bulider speed(int speed) {
            this.speed = speed;
            return this;
        }

        public ConstructorBuilder build() {	//build方法,调用外部类的构造器.只有在调用这一方法的时候外部类的实例才真正被创建.
            return new ConstructorBuilder(this);
        }

    }
    
}

·构建器的微弱优势:
与构造器相比,构建器可以有多个可变参数方法(直到每个setter方法都有一个可变参数).
Builder模式十分灵活,可以利用单个builder构建多个对象.builder的参数可以在创建对象期间进行调整,也可以随着不同的对象而改变.

·构建器的缺点:
为了创建对象必须先创建这个对象的构建器,这可能会导致性能的问题.

·总结:
如果类的构造器或者静态工厂方法具有多个参数,设置这种类时Builder模式就是种不错的选择,特别是大多数参数都是可选的时候.与使用传统的构造器相比,使用Builder模式的客户端代码将易于阅读和编写,构建器也比javabena更安全.






@1.3:用私有构造器或者枚举类型实现单例模式

·如何使用私有构造器来实现单例模式以及注意事项?

实现方法1:
public class Elvis {

    public static final Elvis INSTANCE = new Elvis();		//公有的静态的final变量.可以直接被访问到

    public Elvis() {						//提供私有构造器,防止实例化
    }

}

实现方法2:
public class Elvis2 {

    private static final Elvis2 INSTANCE = new Elvis2();	//私有的静态final变量,不可以被访问到

    private Elvis2() {

    }

    public static Elvis2 getInstance() {			//提供公有的访问单例实例的方法
        return INSTANCE;
    }

}

使用方法2的优势:静态工厂方法提供了灵活性,再不改变其API的情况下,我们可以改变该类是否应该为单例的想法.另外一个优势与泛型有关(见27条)§

使用方法1和方法2,如果这个类是可以被序列化的那就要格外注意(见77条)§	

实现方法3:枚举
public enum Elvis3 {
    INSTANCE;
    public void add(){
        System.out.println("add");
    }
}

关于枚举理解的附加内容:
可以把Elvis3想成一个接口,在这个接口里面定义的所有枚举变量都可以认为是这个接口的实现.(具体见30条)§

·总结:
优先使用枚举的实现方式,枚举提供了天然的单例机制以及序列化机制,绝对防止多次实例化,即使是面对复杂的攻击时.



@1.4:通过私有构造器强化不可实例化的能力

·什么样类不需要被实例化?
例如Math、Arrays、以及各种工具类,这些类就不需要被实例化.

·如何让一个类不被实例化?
public class UtilClass {
    private UtilClass() {				//提供私有的构造器,否则JVM就会自动生成一个公开的构造器.
        throw new AssertionError();			//一旦调用了构造器则直接报错.报错是不需要抛出的.
    }
}

附加内容:
如果一个类不提供任何构造器,则JVM就会提供一个公有的默认构造器.但是只要这个类有一个构造器,不管是什么构造器,则JVM都不会提供构造器.

·使用这种实现的缺点(总结):
缺点就是这个类不能被继承.(因为:一个类如果想要继承另外一个类,那么这个类就必须能够调用父类的构造器.以及这个父类不能是final的)




@1.5:避免创建不必要的对象

·一个不值得被效仿的例子:
String str = new String("string");			//这个方法创建了两个对象,永远不要这样
改进后的版本:
String str = "string";					//这个方法创建了一个对象,而且如果重复创建会使用单例模式.

·静态工厂总是优先于构造器:
例如Boolean.valueOf(boolean b)是单例的,而构造器在每次调用的时候都会创建一个新的对象.
*实际上通过构造器new Boolean();的方法以及被废弃了.推荐使用Boolean aBoolean = true;

·可以重用那些已知不可被修改的类:
public class Person {

    private final Date birthday;

    public Person(Date birthday) {
        this.birthday = birthday;			//一旦Date被设置了,就无法更改.
    }

    private static final Date START;
    private static final Date END;

    static {			//在类初始化的时候初始化STRAT和END变量.而且重用这些对象.
        Calendar gmtCal = Calendar.getInstance(TimeZone.getTimeZone("GMT"));
        gmtCal.set(1946, Calendar.JANUARY, 1, 0, 0, 0);
        START = gmtCal.getTime();
        gmtCal.set(1956, Calendar.JANUARY, 1, 0, 0, 0);
        END = gmtCal.getTime();
    }

    public boolean isBabyBoomer() {
        return birthday.compareTo(START) >= 0 && birthday.compareTo(END) < 0;		//判断birthday对象的值是否在START和END之间
    }
    
}

START和END变量不必再每次调用isBabyBoomer方法的时候都创建一个新的实例.

·在适配器模式中对象的重用就不是那么的明显了:
例如调用Map的keySet方法,每次所返回的Key都是同一个对象的引用.

·总结:
在提倡保护性拷贝的时候,因重用对象而付出的代价要远远大于因创建对象而付出的代价.必要时如果没能实施保护性拷贝,将会导致潜在的错误和安全漏洞.而不必要的创建对象只会影响程序的风格和性能.

·附加内容:
要优先使用基本类型而不是装箱基本类型,要当心无意识的自动装箱即不要将基本数据类型和装箱类型进行混用.




@1.6:消除过期的对象引用

·内存泄露的例子:
public class Stack {
    private Object[] elements;
    private int size = 0;
    private static final int DEFAULT = 16;

    public Stack() {
        elements = new Object[DEFAULT];
    }

    public void push(Object e) {
        ensureCapacity();
        elements[size++] = e;				1
    }

    public Object pop() {
        if (size == 0) {
            throw new EmptyStackException();
        }
        return elements[size--];			2
    }

    private void ensureCapacity() {
        if (elements.length == size) {
            elements = Arrays.copyOf(elements, 2 * size + 1);
        }
    }

}

这段代码的问题在于:
当步骤2将数组里面的Object弹出之后,在步骤1中将刚才弹出的那个位置的又被覆盖了.此时GC会认为仍然有引用指向刚才被弹出的对象,所以数组的元素覆盖,实际上就是索引从一个对象转移到另一个对象上,但原本被引用的对象并不会被回收.


*解决这个问题的办法是当o1被替换的时候后,就立即消除这个引用.
public Object pop() {
    if (size == 0) {
	throw new EmptyStackException();
    }
    Object result = elements[--size];
    elements[size] = null;
    return result;
}

·何时应该清除过期引用?
简而言之造成Stack类安全泄露的原因是,Stack类自已管理自已的内存.对于程序员而言只有数组里面的元素才是正真有用的,但是对于垃圾回收器而言,elements数组中的所有对象引用都等同有效.但是程序员可以把这个情况告诉垃圾回收期,做法就是:一旦数组元素变成了非法活动的一部分,程序员就手动清除这些数组元素.

·内存泄露的另一个来源是缓存:
一旦你把对象放到缓存区,他就很容易被遗忘,从而使得它不在有用之后很长一段时间内仍然留存在缓存中.
 ·解决缓存导致的内存泄露的方法:
-使用WeakReference类:注意WeakReference只对一个对象的引用进行管理
Date date = new Date(100000);					//date引用指向堆内存
WeakReference<Date> wr = new WeakReference<>(date);		//将date放入WeakReference对象中
System.out.println("gc运行之前"+wr.get());			//此时可以获得数据
date = null;							//断开date引用
System.gc();							//通知垃圾回收器进行回收
System.runFinalization();					//通知GC回收
System.out.println(wr.get());					//输出null

-使用WeakHashMap类:注意WeakHashMap对多个对象以键值段的形式管理对象
Date date = new Date(100000);
WeakHashMap<Date, String> dateStringWeakHashMap = new WeakHashMap<>();
dateStringWeakHashMap.put(date, "233");				//可以存放多个键值对,任然拥有Map的特性
System.out.println(dateStringWeakHashMap.size());		//输出1
date = null;							//断开引用
System.gc();							//通知GC
System.runFinalization();					//通知GC
System.out.println(dateStringWeakHashMap.size());		//输出0,date对象被回收

-注意:这两种方式只对指向堆内存的引用进行回收,如果引用指向方法区(包含常量).
例如:String str = "String";		//则该对象无论使用上述那种方法都不会被回收


·总结:
清空对象引用是一种例外,而不是一种行为规范.
一般而言,只要是类自已管理内存,程序员就应该警惕内存泄露问题,一旦元素被释放掉,则该元素中包含的任何对象的引用都应该被清空.




@1.7:避免使用终结方法		§		已经被弃用了

·容易混淆:
终结方法:类似于FileInputStream的close方法
终止方法:try-finally块

·终结方法的缺点?
1.不能保证终结方法会被及时的执行,注重时间的任务不应该由终结方法来完成.
2.一种情形:
如果未被捕获的异常在终结过程中被抛出,那么这种异常可以被忽略,并且该对象的终结过程也会终止.例如:
        try {
            Integer.parseInt("23333333333333");
        } catch (NumberFormatException e) {
	    e.printStackTrace();			//打印出栈轨迹
            int i = 1/0;				//此时在catch模块中发生异常
            System.out.println("发生错误");		//这段话并不会被打印出来
        }finally {
            System.out.println("最终执行");
        }
3.使用终结方法有非常严重的性能损失

·如果对象中的资源确实需要终结该怎么办?
1.只需提供一个公开的终止方法,并要求该客户端在每个实例不在有用的时候调用这个方法.并且该实例需要记录自已是否已经被终止了,公开的终止方法必须在一个私有的变量中记录"该对象已经不在有用了",如果这些方法是在终止之后被调用的,其他的方法就必须检查这个变量,如果是则抛出IllegalAccessException.
*公开终止方法的典型例子是InputStream、OutputStream、和java.sql.Connection上的close方法.
2.公开的终止方法通常与try-finally结构结合起来使用,以确保及时终止.在finally子句内部调用公开的终止方法,可以保证即使在使用对象的时候有异常抛出,该终止方法也会执行.
try{

}finally{
 xxx.xxx		//调用公开的终止方法
}

·使用终止方法的好处:(不是终结方法)
1.当对象所有者(程序员)忘记调用该对象的公开的终止方法时,终结方法可以充当"安全网".虽然这样做并不能保证终结方法会被及时的调用,但是迟一点释放关键资源总比不释放要好.
*注意:如果终结方法发现资源还未被终止,则应该在日志中记录一条警告.
公开的终止方法模式的实例中所示的四个类(FileInputStream、OutputStream、Timer、Connection),都具有终结方法,当他们的终止方法未被调用的时候这些终结方法都充当了安全网.
2.第二个用途与本地对等体有关,本地对等体是一个本地对象,因为本地对等体不是一个普通对象,所以垃圾回收器不会知道它,所以不会被回收.通过终止方法应该完成所有必要的工作以便释放关键资源.

·使用终结方法需要注意的点:(不是终止方法)
1.终结方法链并不会被自动执行.如果类有终止 方法,并且子类覆盖了终结方法,子类的终结方法就必须手动调用父类的终结方法,


·总结:
1.不应该依赖终结方法来更新重要的持久对象

@2.8:覆盖equals时请遵守的通用规定

·equals方法必须满足以下的所有条件:
1.类的每个实例本质上都是唯一的
2.不关心类是否提供了"逻辑相等"的测试功能
3.父类已经覆盖了equals方法,从父类继承过来的行为对于子类也是合适的
4.类是私有的或者包级私有的,可以确定它的equals方法永远不会被调用:在这种情况下,无疑是应该覆盖equals方法的,以防它被以外的调用
@Override
public boolean equals(Object obj) {
    throw new AssertionError();
}

·什么时候应该覆盖equals方法?
如果这个类具有自已的某种逻辑概念,而且父类还没有覆盖equals以实现期望的行为,此时我们就应该覆盖equals方法.这种类通常就是值类,值类仅仅是表示一个值的类,程序要要通过equals方法来比较两个对象的逻辑是否相等,而不是是否是同一个对象.
有一种情况不需要被覆盖,就是确保"这个值至多只存在一个对象"的类.枚举类型就属于这种类(见30条)(不仅仅是枚举类)


·覆盖equals的通用规定:
自反性:对于任何非null的引用值x,x.equals(x)必须返回true
对称性:对于任何非null的引用值x和y,当且仅当y.equals(x)返回true时,x,equals(y)必须返回true.
传递性:对于任何非null的引用值x、y和z,如果x.equals(y)返回true,并且y.equals(z)也返回true,那么x.equals(z)也必须返回true.
一致性:对于任何非null的引用值x和y,当多次调用x.equals(y)必须一致的返回true,或者一致的返回false.
对于任何非null的引用值,x.equals(null)必须返回false.

·写出高质量equals方法的诀窍:
1.使用==操作符检查"参数是否为这个对象的引用"
public boolean equals(Object obj) {    
    return this == obj;
}           	`
2.使用instanceof操作符检查"参数是否为正确的类型",如果不是则返回false
@Override
public boolean equals(Object obj) {
    if (!(obj instanceof MyEquals)) {
	return false;
    }
    //...
}

3.把参数装换成正确的类型
    @Override
    public boolean equals(Object obj) {
        if (!(obj instanceof MyEquals)) {
            return false;
        }
        MyEquals myEquals = (MyEquals)obj;
        //...
    }

4.对于该类中的每个关键变量(即需要比较的变量),检查传进来的参数里面的变量是否与该对象中对对应的变量相匹配.如果这些测试全部成功,则返回true,否则返回false.
对于不是float也不是double类型的基本类型域,可以使用==操作符进行比较,对于对象引用,可以递归的调用equals方法,对于float可以使用Float.compare方法,对于double域,则使用Double.compare.
变量的比较顺序可能会影响到equals方法的性能.为了获得最佳的性能,应该最先比较最有可能不一致的变量,或者开销最低的变量.

5.当你完成equals方法之后,就应该问自已三个问题:它是否是对称的、传递的、一致的?并且不是自问,还需要进行测试.

*不要让equals方法过于智能:equals方法最好是判断变量的值是否相等
*不要将equals声明中的object对象替换为其他类型.


·总结:一个小模板

            return true;
        }
        if (!(obj instanceof String)) {		//不一定是String

    }






@2.9:覆盖equals方法时总要覆盖hashCode

·为什么覆盖equals方法时总要覆盖hashCode?
原因1:基于约定
在引用程序执行期间,只要对象的equals方法的比较操作所用到的信息没有被修改,那么对这同一个对象调用多次,hashCode方法都必须始终如一地返回同一个整数.在同一个应用程序多次执行过程中,每次执行所返回的整数可以不一致.
如果两个对象根据equals(Object)方法比较是相等的,那么调用者两个对象中任意一个对象的hashCode方法都必须产生同样的结果.
如果两个对象根据equals(Object)方法比较是不相等的,那么调用这两个对象中任意一个对象的hashCode方法,则不一定要产生不同的整数结果.
原因2:相等的对象必须拥有相等的散列码(如果不覆盖默认调用的是父类的hashCode方法,父类的hashCode方法返回的是这个对象的内存地址的十进制表示)
原因3:如果不覆盖hashCode法法,则所有基于散列码的集合将无法使用.


·如何重写一个hashCode方法?
1.把某个非零的常数值,比如说17,保存在一虐名为result的int类型的变量中.
2.对于对象每个关键变量f(指equals方法中涉及比较的每个变量),完成以下步骤:
 i.如果该变量是boolean类型,则计算(f ? 1 : 0)
 ii.如果该变量是byte、char、short或者int类型,则计算(int)f.
 iii.如果该变量是long类型,则计算(int)(f^(f>>>32)).
 iv.如果该变量是float类型,则计算Float.floatToIntBits(f)
 v.如果改变量是double类型,则计算Double.doubleToLongBits(f),然后按照步骤iii,为得到的long类型值计算散列值.
 vi.如果该变量是一个对象引用,并且该变量的equals方法通过递归的调用equals的方式来比较这个变量,则同样为这个变量递归的调用hashCode方法.如果需要更复    杂的比较,则为这个变量计算一个范式,然后针对这个范式调用hashCode.如果这个变量的值为null,则返回0.(如果要比较一个对象引用,则调用这个对象的    equals方法)
 vii.如果该变量是一个数组,则要把每一个元素当作单独的变量来处理.也就是说对于数组中的每一个元素,递归地调用上述的方法,对每一个元素都计算出一个散列     码,然后把它们组合起来.

3.返回result
4.当你写完hashCode方法,要反过来问自已"相等的实例是否具有相等的散列码".

*一个示例:
public class Demo8 {
    private final short areaCode;
    private final short prefix;
    private final short lineNumber;

    public Demo8(short areaCode, short prefix, short lineNumber) {
        this.areaCode = areaCode;
        this.prefix = prefix;
        this.lineNumber = lineNumber;
    }

    @Override
    public int hashCode() {
        int result = 17;
        result = 31*result+areaCode;
        result = 31*result+prefix;
        result = 31*result+lineNumber;
        return result;
    }
}

*如果一个类是不可变类,并且计算散列码开销较大,可以考虑把散列码缓存在对象内部.
    @Override
    public int hashCode() {
        int result = hashCode;
        if (result == 0) {
            result = 31*result+areaCode;
            result = 31*result+prefix;
            result = 31*result+lineNumber;
            hashCode = result;
        }
        return result;
    }

·附加:
不要试图从散列码计算中移除一个对象的关键部分来提高性能.




@2.10:始终要覆盖toString

·toString应该包含哪些信息?
在实际情况中,toString方法应该返回对象中包含所有值得关注的信息.

·在实现toString的时候,指定返回值格式的好处:
它可以被用作一种标准的、明确的、适合人阅读的对象表示法.这种表示法可以用于输入和输出,以及用在永久的适合人类阅读的数据对象中,例如XML文档.

·在实现toString的时候,指定返回值格式的坏处:
如果这个类已经被广泛使用,一旦指定格式,就必须始终如一的坚持这种格式.程序员将会编出相应的代码来解析这些字符串表示法、产生字符串表示法,以及把字符串表示法嵌入到持久的数据中.如果将来的发行版本改变了这种表示法,将会破坏他们的代码和数据.如果不指定格式,就可以保留灵活性,便于在将来的发行版本中增加信息,或者改进格式.

·另外,无论是否指定格式,都需要为toSting方法返回值中包含的所有信息,提供一种编程式的访问途径.(即,无论是否指定格式,在上面Demo8类里面都要为那三个变量提供可访问的方法)




@2.11:谨慎覆盖clone

·克隆方法的一种机制:
clone方法实际上也是一种构造器.clone方法可以在不调用对象原本构造器的情况下创建对象.

·覆盖clone的注意事项:
1.如果你覆盖了非final类中的clone方法,则应该返回一个通过调用super.clone而得到的对象.如果类的所有父类都遵守这条约定,最后会调用Object类的clone方法
2.通常情况下,除非该类的所有父类都提供了行为良好的clone实现(下面的这点),无论是公有的还是守保护的,否则都不应该这么做.
3.永远不要让客户去做任何类库能够替客户端完成的事情.


·如何做到一个行为良好的clone实现?
1.如果这个将要被克隆的类,只有基本类型和指向不可变对象的引用的话,clone不需要做额外的步骤.
2.如果对象中包含的变量引用了可变参数,则clone方法需要做额外的步骤.
 2.1对于数组需要单独的clone
   *例如:
public class PhoneNumber implements Cloneable {
    private Object[] elements;

    public PhoneNumber() {
        elements = new Object[16];
    }

    @Override
    public PhoneNumber clone() {
        PhoneNumber result = null;
        try {
            result = (PhoneNumber) super.clone();
            result.elements = elements.clone();
            return result;
        } catch (CloneNotSupportedException e) {
            throw new AssertionError();
        }
    }
}

  2.2对于散列表要单独将每个元素clone.(因为,虽然数组是一个新的数组,但是数组里面元素还是指向同一个引用的.对于散列表而言,各个容器指向的引用是不相       同的)


3.在上述例子中,数组不能为final.
4.clone方法不应该在构造的过程中,调用新对象中任何非final的方法.
5.如果决定用线程安全的类实现Cloneable接口,要记得它的clone方法必须得到很好的同步,就像任何其他方法一样.

·总结:
所有实现了Cloneable接口的类都应该用一个公有的方法覆盖clone.次公有方法首先调用super.clone,然后修正任何需要修正的变量.(所谓的需要深克隆,而不是浅克隆)

*另外一个实现对象拷贝的好办法是提供一个拷贝构造器或拷贝工厂.(拷贝构造器只是一个构造器,他唯一的参数类型是包含该构造器的类)例如:
    public PhoneNumber phoneNumber(PhoneNumber phoneNumber){
        return new PhoneNumber();
    }
拷贝构造器的优势在于:
它们不依赖于某一种很有风险的,语言之外的对象创建机制,它们不要求遵守尚未制定好的文档规范,它们不会与final变量的正常使用发生冲突,它们不会抛出不必要的受检查时异常,它们不需要进行类型转换.

·总结:
既然Cloneable具有上述那么多问题,可以肯定的说,其他的接口都不应该扩展这个接口,为了继承而设计的类(见17条§)也不应该实现这个接口.所以很多人不会覆盖这个方法,也不会去调用clone方法.必须清楚,对于一个专门为了继承而设计的类,如果你未能提供行为良好的守保护的clone方法,它的子类就不可能实现Cloneable接口.



@2.12:考虑实现Comparable接口

·为什么要实现这个接口?
一旦类实现了Comparable接口,它就可以跟许多泛型算法以及依赖于该集合的集合实现进行协作.
Arrays.sort();

·compareTo约定的条款:
1.如果颠倒了两个对象引用之间的比较方向,则如果第一个对象小于第二个对象,则第二个对象一定大于第一个对象,如果第一个对象等于第二个对象,则第二个对象一定等于第一个对象,如果第一个对象大于第二个对象,则第二个对象一定小于第一个对象.
2.如果第一个对象大于第二个对象,第二个对象又大于第三个对象,则第一个对象大于第三个对象.
3.在比较式被认为相等的对象,它们跟别的对象做比较时一定会产生同样的结果.(这一条是强烈建议,而不是真正的规则,在通常情况下应该返回与equals方法同样的结果)

·接口特性:
1.与equals不同的是,如果类跨越了不同了类进行比较,compareTo可以抛出ClassCastException异常.
2.比较整数型基本类型的变量,可以使用关系操作符 < 和 > .不过浮点变量使用Double.compare或者Float.compare.

@3.1:使类和成员的可访问性最小化

·好处:
1.设计良好的模块会隐藏所有的实现细节,把它的API与它的实现清晰地隔离开来.然后模块之间只通过它们的API进行通信.
2.它可以有效解除组成系统各模块之间的耦合关系,使得这些模块可以独立开发、测试、优化、使用、理解和修改.

·如何设计?
1.尽可能地使每个类或者成员不被外界访问.
对于顶层的类和接口,只有两种访问权限(public和default(default就是可以被修饰权限的内容不加上任何修饰符)).如果类和接口能够被做成default就要被做成default.如果被做成公有的,你就有责任永远支持它.
2.四种访问权限:
private		只能被当前类访问到(不会被导出API)
default		不对任何属性进行修饰就是default.只能被当前包下的类访问到(不会被导出API)
protected	只能被当前类和当前类的子类访问到(会被导出API)
public		任何地方都可以访问(会被导出API)
3.只有当同一个包的另一个类真正需要访问一个成员的时候,你才应该将private换成default.(删掉private)
4.如果这个类实现了Serializable接口(见74条和75条),这些变量有可能会被泄露导出到API中
5.少用protected关键字,一旦使用代表你要对其进行支持.
6.实例变量不能是公有的,因此包含公有可变变量的类并不是线程安全的.
7.长度非0的数组总是可变的,所以类具有公有的静态地拿了数组变量,或者返回这种变量的访问方法,这几乎总是错误的.

·总结:
你应该始终尽可能地降低可访问性.你在仔细地设计了一个最小的公有API之后,应该放置把任何散乱的类、接口和成员编程API的一部分.除了公有静态final变量的特殊情形之外,公有类都不应该包含公有变量.并且要确保公有静态final变量所引用的对象都是不可变的.


@3.14:在公有类中使用访问方法而非公有变量

·解释:
对一个公有类中的变量提供set/get方法.

·总结:
公有类永远都不应该暴露可变的变量.

·附加:
如果包是default或者私有的成员类,则直接暴露它的数据变量并没有本质的错误.




@3.15:使可变性最小化(不可变类:只是其实例不能被修改的类,例如:String、BigInteger)

·编写一个不可变类需要遵守的约定:
1.不要提供任何会修改对象状态的方法
2.保证类不会被继承:这样可以放置粗心或者恶意的子类假装对象的状态已经改变,从而破坏该类的不可变行为.为了防止子类化,一般做法是加上final(当然还有其他的方法)
3.使所有的变量都是final.
4.使所有的变量都成为私有的.
5.确保对于任何可变组件的互斥访问:如果类具有指向可变对象的变量,则必须确保该类的客户端无法获得指向这些对象的引用.并且永远不要用客户端提供的UI想引用来初始化这样的变量,也不要从任何访问方法中返回该对象的引用.在构造器、访问方法readObject方法中请使用保护性拷贝技术(见39条)§

·不可变对象的优点:
1.不可变对象比较简单
2.不可变对象本质上是线程安全的,它们不要求同步.
3.不可变对象可以被自由的共享.
4.不需要为不可变对象提供clone方法和拷贝构造器.(所以String类任然具有拷贝构造器,但是应该尽量少使用)(见5条)
5.不仅可以共享不可变对象,甚至也可以共享它们的内部信息.
6.不可变对象为其他对象提供了大量的构件.

·不可变对象的缺点:
1.对于每个不同的值都需要一个单独的对象(创建这种对象的代价可能很高,特别是对于大型的对象)

·附加:
如果你执行一个多步骤的操作,并且每个步骤都会产生一个新的对象,除了最后的结果之外,其他的对象都会被丢弃,此时性能问题就会显露出来.
处理这种问题的两种办法:
1.先猜测一下经常用到哪些多步骤操作,然后将他们作为基本类型提供.如果某个多步骤操作已经作为基本类型提供,不可变的类就可以不必再每个步骤单独创建一个对象.
2.如果无法预测,最好的办法就是提供一个共有的可配套类.在java平台中,这种方法的主要例子就是String类,它的可变配套类是StringBuilder(和基本已经被废弃的StringBuffer).

·使类不可被继承的另外方法:
对应-编写一个不可变类需要准守的约定2
让类的所有构造器都编程私有的或者包级私有的,并添加公有的静态工厂来代替公有构造器.

·总结:
1.坚决不要为每个get方法编写一个相应的set方法.除非有很好的理由要让类称为可变的类,否则就应该是不可变的.
2.对于有些类而言,其不可变性是不切实际的.如果类不能被做成是不可变的,任然应该尽可能地限制它的可变性.因此除非要有令人信服的理由要使变量变成是非final的,否则要使每个变量都是final的.
3.不要在构造器或者静态工厂之外再提供公有的初始化方法,除非有令人信服的理由.同样的也不应该提供"重新初始化"方法.


@3.16:复合优先于继承

·使用继承带来的问题:
1.如果子类的部分实现依赖于其父类,那么如果父类在新版本更新了,其子类也必须随父类的更新而演变.否则子类可能会遭到破坏.
2.被重写的方法优先级大于父类方法.例:现在子类重写了两个父类方法.其中父类的A方法会调用B方法.此时子类的A方法通过super调用了父类的A方法,现在调用子类的A方法会:子类A->父类B->子类B	注意:没有调用父类的A方法.
3.就算子类的实现不依赖于父类.而是只对父类做扩展,同样也会带来问题.如果父类更新后恰有一个新方法的名称和参数列表和子类原有方法相同,仅是返回值不同时,该之类根本无法通过编译.(因为子类重载错误)


@3.17:要么为继承而设计,并提供文档说明,要么就禁止继承

·为了继承而设计的类,在设计时因注意哪些问题:
1.为了继承而设计的类,则该类(父类)文档必须精确地描述每个方法所带来的影响.
2.为了继承而设计的类,实现Cloneable和Serializable接口都不是好主意.
3.如果一个普通类想要被继承.那么必须确保这个类永远不会调用它的任何可覆盖方法,换句话说.完全消除这个类中可覆盖方法的自用特性(这个类的不可覆盖方法不能调用可覆盖方法)



@3.18:接口优于抽象类

·使用接口的优点:
1.如果一个类想要通过接口或抽象类扩展这个类.那实现接口的扩展方式将比继承抽象类容易的多.
2.接口是定义mixin类型的理想选择.这代表当一个类实现一个接口时,代表这个类拥有这个接口的行为特性.例如Comparable接口,实现类就代表这个类可以比较对象进行排序.但是抽象类不可以(因为一个类不可以继承多个抽象类)
3.如果设计得当,骨架实现可以使程序员很容易地实现这个接口(模板方法模式)

·使用抽象类的优势:
1.抽象类的演变比接口演变容易很多

·附加:
接口一旦倍公开发行,并且已经被广泛实现,在想改变这个接口几乎是不可能的.


3.19:接口只用于定义类型

·解释:
1.接口的定义只是充当实现类的类型引用
2.有一种接口叫常量接口.这种接口不包含任何抽象方法,只包含静态的final变量要使用这些常量的类,就必须实现这个接口.
3.使用常量接口是不良的使用
4.如果要导出常量.如果这些常量与某个现有的类或者接口紧密相关,就应该把这些类=常量添加到这个类或者接口中.如果这些常量最好被看做枚举类型的成员,就应该用枚举类型.

·总结:
接口应该只被用来定义类型,它们不应该被用来导出常量.

@3.22:优先考虑静态成员类

·嵌套类的分类:
1.静态成员类(最好把静态成员类看成普通类)
2.非静态成员类
3.匿名类
4.局部类

·附加:
1.嵌套类存在的目的应该只是为它的外围类提供服务.

·静态成员类和非静态成员类的区别是:
1.非静态成员类的每个实例都隐含着与外围类的一个外围实例相关联.
2.在非静态成员实例方法内部,可以调用外围实例上的方法,或者刘勇修饰过的this构造获得外围实例的引用.
3.如果嵌套类的实例可以在它外围类的实例之外单独存在,这个嵌套类就必须是静态成员类:在没有外围实例的情况下,要想创建非静态成员类的实例时不可能的.
4.如果声明成员类不要求访问外围实例,就始终要把static修饰符放在它的声明当中(否则会消耗性能).

·匿名类:
1.匿名类的客户端(程序员)无法调用任何成员(该类的变量以及方法)
2.由于匿名类出现在表达式当中,他们必须保持简短,大约10行或者更少一些,否则会影响程序可读性.

·局部类:局部类是四种类中使用的最少的
1.在任何"可以声明局部变量"的地方,都可以声明局部类,并且局部类也遵守同样的作用变量规则.
2.局部类与其他三种嵌套类中的每一种都有一些共同属性.与成员类一样,局部类有名字,可以被重复使用.与匿名类一样,只有当局部类是在非静态环境中定义的时候,才有外围实例,它们也不能包含静态成员.与匿名类一样,它们必须非常简短,以便于不会影响到可读性.

·总结:
1.如果一个嵌套类需要在单独方法之外忍让可见的,或者他太长了,不适合放在方法内部,就应该使用成员类.如果成员类的每个实例都需要一个指向其外围实例的引用,就要把成员类做成非静态的,否则就做成静态的.假设这个嵌套类属于一个方法的内部,如果你只需要在一个地方创建实例,并且已经有了一个预置的类型可以说明这个类的特性,就要把它做成匿名类,否则就做成局部类.

@4.23:不要在新代码中使用原生态泛型

·什么是原生态泛型:
就是不加泛型约束的类型,例如List list = new ArrayList();

·使用泛型的优点:
1.在编译期间检查元素是否正确.有安全性和表述性的优势.



·附加:
1.使用List<Object>比List更安全.
2.List<String>可以作为List的参数.但List<String>不可以作为List<Object>的参数.(因为泛型没有多态)例如:

public class Generic {
    
    private static void unsafeAdd(List<Object> list, Object o) {
        list.add(o);
    }
    
    public static void main(String[] args) {
        List<String> list = new ArrayList<>();
        unsafeAdd(list,new Integer(100));		//此时会编译报错
    }
}
3.永远记住通配符就相当于Object只不过有了通配符可以约束泛型的上限和下限.
4.通配符和原生态泛型相比,通配符是安全的.

·总结:
使用原生态类型会在运行时导致异常,因此不要在新代码中使用.原生态类型 只是为了与引入泛型之前的遗留代码进行兼容和互用而提供的.

2.每当使用@SuppressWarnings注解时,都要添加一条注释,说明为什么这么做是安全的.


@4.25:列表优于数组

·数组和泛型集合的区别:
1.泛型没有多态.例如:List<Father> list = new ArrayList<Child>();		这样是错误的
2.数组有多态.例如:
Object[] objects = new Long[100];		//编译合法
objects[0] = "233";				//编译合法,运行时产生异常
3.什么叫声明看左边,运行看右边.现在对javaSE中的这句话进行理解
上面的这段代码没有在编译期间发生故障的原因是,数组声明时(编译期间)是Object.但在真正运行时是具体的类型是看右边的,右边是Long类型的,所以Long类型的数组显然不能放入String类型的数据.把这段代码改变一下:
Object[] objects = new String[10];
objects[0] = ""233;
此时编译和运行都不会发生错误.

·数组和泛型集合的区别2:
1.数组是具体化的.因此数组会在运行时才知道并检查它们的元素类型约束.
2.泛型是通过擦除来实现的.因此泛型只是在编译时强化它们的类型信息,并在运行时丢弃它们的元素类型信息.
什么是泛型的擦除?
例如:List<String> list = new ArrayList<>();当运行期间泛型擦除之后,就相当于把String类型换成了Object类型.因此泛型的擦除就是使泛型可以与没有使用泛型的代码随意进行互用.

·不可以创建泛型数组或者泛型集合数组(那为什么ArrayList底层实现是泛型数组呢?后面会讲到):
1.由于上述这些根本的区别,因此数组和泛型不能很好地混合使用.例如 创建泛型、参数化类型或者类型参数化的数组是非法的.这些数组创建的表达式没有一个是合法的:new List<E>、new List<String>、new E[].
2.当你得到泛型数组创建错误时,最好的办法是优先使用集合类型List<E>.

·总结:
数组和泛型有着非常不同的类型规则.数组是协变且可以具体化的.泛型是不可变的且可以被擦除的.因此数组提供了运行时的类型安全,但是没有编译时的类型安全,反之对于泛型也一样.一般来说,数组和泛型不能很好的地混合使用.如果发现自已混用并且得到了编译时错误或者警告,应该使用泛型集合来代替.

·附加内容泛型的擦除:
public class Father<T>{
	T name;
}

1.子类声明时父类指定具体类型.属性类型为具体类型
public Child extends Father<String>{
	//此时从父类继承过来的name为String类型
}
2.子类为泛型类
public Child<T> extends Father<T>{
	//只要父类和子类同时声明泛型,则父类有的泛型子类必须有
}
3.子类执行泛型,父类不指定泛型,则泛型擦除.用Object替换.
public Child<T1,T2> extends Father{
	//此时name类型为Object
}
4.子类和父类同时不指定泛型
public Child extends Father{
	//name为Object.子类泛型无作用.
}

 
@4.26:优先考虑泛型

·什么是类型参数:
例如:其中的E为类型参数
class List<E>{

}

·将类泛型化的步骤:
1.给这个类声明添加一个或多个类型参数

·不能创建不可具体化的类型数组,如new E[];解决这个问题的两种方法 优先使用第二种方法
方法一:
E[] elements = new E[16];			这种声明方法是错误的
E[] elements = (E[])new Object[16];		这样创建虽然编译器不能证明你的程序是类型安全的,但是我们自已清除这是安全的.所以此时我们就可以在						小范围内使用禁止警告.

方法二:
将elements数组的类型从E[]改为Object[].		Object[] elements = new Object[16]
并且在数组的增减操作中,将获取的元素转为类型参数
E result = (E)elements[0];

优先使用第二种方法

·总结:
使用泛型比使用需要在客户端代码中进行转换的类型更加安全,也更加容易.在设计新类时,要确保它们不需要转换就可以使用这通常意味着要把类作成泛型的.




@4.27:优先使用泛型方法

·使用泛型方法的优势:
1.与构造器不同的是,调用泛型方法可以不指定泛型类型,而是根据传入的类型进行类型推导.如:
public <E> E result(E e){
    return e;
}

如果调用result("Test").此时根据类型推导可知E为String.

·总结:
泛型方法就像泛型一样,使用起来比要求客户端传入参数并返回值的方法来得更加安全,也更容易.就像类型一样,你应该确保新方法可以不用转换就能使用,这通常意味着要将它们泛型化.并且就像类型一样,还应该将现有的方法泛型化,使新用户使用起来更轻松,不会破坏现有的客户端.




@4.28:利用有限通配符来提升API的灵活性

·什么是通配符?
List<?>	List<? extends Number> List<? super Number> List<? extends E> 这些都算是通配符

·如何使用,一般使用通配符是发生在方法形参,不知道传入的对象的具体类型,但是给出约束:
			//extends直接使用add方法受限
public class Item28 {
    public static void main(String[] args) {
        firstPrint();
        secondPrint();
    }

    public static void firstPrint() {
        List<Float> list1 = new ArrayList<>();//使用 Float作实参类型
        list1.add(12.3f);
        list1.add(23.4f);
        print(list1);
    }

    public static void secondPrint() {
        List<Integer> list2 = new ArrayList<>();//使用Integer作实参类型
        list2.add(1);
        list2.add(2);
        list2.add(3);
        print(list2);
    }

    public static void print(List<? extends Number> list) {// 通配符作形参
        for (int i = 0; i < list.size(); i++) {
            System.out.println(list.get(i));// 使用get方法
        }
    }
}

			//super直接使用get方法受限
public class Main {
    public static void main(String[] args) {
        List<Number> list = new ArrayList<>();
        firstInflate(list);
        for (int i = 0; i < list.size(); i++) {
            System.out.println(list.get(i));
        }
        list.clear();
        secondInflate(list);
        for (int i = 0; i < list.size(); i++) {
            System.out.println(list.get(i));
        }
    }

    public static void firstInflate(List<? super Integer> list) {// 装填整数
        list.add(1);
        list.add(2);
        list.add(3);
    }

    public static void secondInflate(List<? super Float> list) {// 装填浮点数
        list.add(1.1f);
        list.add(2.2f);
        list.add(1.3f);
    }
}

·List<? extends String>和List<T extends String>的区别:
List<? extends String>是针对调用方法而言的,通配符规定了客户端传入的参数的泛型类型.(比如这里只能是String类型)(用于规定泛型类型)
List<E extends String>是针对创建对象而言的,表名创建对象时,泛型只能是String.(用于规定具体类型)

·使用通配符应注意的事项:
1.不要用通配符类型作为返回类型
2.如果类的用户必须考虑通配符类型,类的API或许就会出错
3.如果编译器不能进行类型推导,则可以提供一个显示的类型参数:
4.对于Comparable接口始终应该是Comparable<? super T>优先于Comparable<T>.
因此使用时始终应该是Comparator<? superT>优先于Comparator<T>
比如:
public static <T extends Comparable<? super T>> T max(List<? extends T> list){

}
5.对于:
public static <E> void swap(List<E> list)和public static void swap(List<?>)而言.
一般来说,如果类型参数只在方法声明中出现一次,就可以用通配符取代它.

·使用通配符的好处是:
1.使得泛型可以有多态.

@5.30:用enum代替int常量

·什么是int枚举类型(int枚举模式):
public static final int APPLE_FUJI = 0;
public static final int APPLE_PIPPIN = 1;

public static final int ORANGE_NAVEL = 0;
public static final int ORANGE_TEMPLE = 1;

·使用枚举模式的缺点:(包括String枚举模式等其它的类型)
1.使用枚举模式在类型安全性和使用方便性方面没有任何帮助.如果你将apple放到需要orange方法中,编译器不会出现警告还会用==操作符将apple与orange比较.
2.int枚举式编译器常量被编译到使用它们的客户端中,如果与枚举常量关联的int发生了变化,客户端就必须重写编译.
3.将int枚举常量翻译成可打印的字符串,要遍历一个组中的所有int枚举常量,甚至获得int枚举组的大小,这些都没有可靠方法.

·使用枚举的优点:
1.使用toString方法,将枚举转换成可打印的字符串(返回每个枚举值的声明名称)
2.枚举类型还运行添加任意的方法和变量
3.提供了Object的高级实现,实现了Comparable和Serializable接口.
个人对枚举的理解:如
public enum Plant {				//每个枚举类相当于一个接口
    MERCURY(1, 1), VENUS(2, 2);		//枚举中的每个值相当于一个具体实现,调用构造器,实际上就是调用Plant

    Plant(int mass, int redius) {	//定义在Plant里的构造器就是每个实例的构造器,定义在Plant里面的变量也是共享的.
        
    }

}

4.枚举天生就是不可变的,因此所有的变量都应该为final的.
5.如果每个枚举都有一个不同的操作,则使用一个抽象方法
public enum Operation {
    PLUS{
        @Override
        double apply(double x, double y) {		//实现抽象方法
            return 0;
        }
    },
    MINUS {
        @Override
        double apply(double x, double y) {
            return 0;
        }
    };
    abstract double apply(double x, double y);		//Operation类的抽象类
}
6.枚举类型有一个自动产生的valueOf(String)方法,它将常量的名字转为常量本身.
7.在枚举中,构造器不能访问静态变量
8.double不适合薪资应用场景(见48条)§
9.枚举类中的switch语句适合于给外部的枚举类型增加特定于常量的行为
10.每当需要一组固定常量时.但它也包括你在编译时就知道其所有可能值的集合.枚举类型中的常量集并不一定要始终保持不变.

·总结:
与int常量相比,枚举类型要易读得多,也更加安全,功能更强大.许多枚举都不需要显示的构造器或者成员,但许多其他枚举则受益于"每个常量与属性的关联"以及"提供行为受这个属性影响的方法".只有极少数的枚举受益于将多种行为与单个方法关联.在这种相对少见的情况下,特定于常量的方法要优先于启动自由值的枚举.如果多个枚举常量同时共享相同的行为,则考虑策略枚举.

@5.34:用接口模拟可伸缩的枚举

·解释:

·总结:

@5.37:用标记接口定义类型

·什么是标记接口:
标记接口是没有包含方法声明的接口.例如Serializable接口.通过实现这个接口,表明它的实例可以被写到ObjectOutputStream.

·标记接口相对标记注解的优势:
1.标记接口定义的类型是有被标记类的实例实现的,标记注解则没有定义这样的类型.
2.标记接口可以被更加精确的锁定.
3.标记接口可以通过默认的方式添加一个或者多个注解类型元素,给已被使用的注解类型添加更多的信息.

·什么时候用标记注解?
如果标记是应用到任何程序元素而不是类或者接口,就必须使用注解.

·什么时候用标记接口?
如果标记值引用给类和接口,如果是要编写一个或者多个只接受这种标记的方法,则应该优先使用标记接口而非注解.

@6.38:检查参数的有效性

·解释:
1.检查构造器参数的有效性是非常重要的,这样可以避免构造出来的对象违反了这个类的约束条件.
2.在方法执行它的计算任务之前,应该先检查它的参数.
3.如果不检查数据,在执行过程中必然会出现异常,则可以不检查这些数据.
4.假如方法对于它能接受的所有参数值都能够完成合理的工作,对参数的限制就应该是越少越好.

·总结:
每当编写方法或者构造器的时候,应该考虑它的参数有哪些限制.应该吧这些限制写到文档中,并且在这个方法开头处,通过显示的检查来实施这些限制.养成这样的习惯非常重要的.



@6.39:必要时进行保护性拷贝

·何时对保护性拷贝:
1.假设类的客户端会尽其所能来破坏这个类的约束条件,因此你必须保护性的设计程序.
2.对于构造器的每个可变参数进行保护性拷贝是必要的.
3.保护性拷贝是检查类型 的有效性之前进行的,并且有效性检查时针对拷贝之后的对象,而不是针对原始对象.
4.对于参数类型可以被不可信任方子类化的参数,请不要使用clone方法进行保护性拷贝.
5.对于get/set方法返回可变内部变量的保护性拷贝.例如:
public class Period {
    private final Date start;

    public Period(Date start) {
        this.start = start;
    }

    public Date start() {
        return new Date(start.getTime());
    }
}

·总结:
如果类具有从客户端得到或者返回到客户端的可变组件,类就必须保护性地拷贝这些组件.如果拷贝的成本受到限制,并且类行人它的客户端不会不恰当地修改组件,就可以在文档中指明客户端的职责是不得修改受到影响的组件,以此来代替保护性拷贝.



@6.40:谨慎设计方法签名

·什么是方法签名:
由方法名称和参数列表组成(决定方法重载的各项参数)

·设计方法签名的注意点:
1.谨慎的选择方法名称.方法的名称应该始终遵循标准的命名习惯.首要的目标应该是选择易于理解的,并且与同一个包中的其他名称风格一致的名称.第二个目标应该是选择与大众认可的名称.如果还有疑问,请参考java类库的API.
2.不要过于最追求遍历的方法.每个方法应该尽其所能.方法太多会使类难以学习、使用、文档化、测试和维护.对于接口而言,这无疑是正确的,方法太多会使接口是想着和接口用户的工作变得复杂起来.对于类和接口所支持的每个动作,都提供一个功能相当齐全的方法.只有当一项操作被经常用到的时候,才考虑为它提供快捷方式.如果不能确定还是不要提供快捷为好.
3.避免过长的参数列表:目标是四个参数或者更少.大多数程序员都无法记住更长的参数列表.如果你编写的许多方法都超过了这个限制,你的API就不太便于使用,除非用户不同的参考它的文档.现代的IDE会有所帮助,但最好还是使用简单的参数列表.相同类型的长参数序列格外有害.API用户不仅无法记住参数的顺序,而且当他们不小心弄错了参数顺序时他们的程序任然可以编译和运行,只不过这些程序不会按照作者的意图进行工作.
 *三种方法可以缩短参数列表:
 3.1:将方法分解成多个方法.
 3.2:创建辅助类,这些辅助类一般为静态成员类.
 3.3使用Builder模式
4.对于参数类型,要优先使用接口而不是类(见52条)§
5.对于Boolean参数,要优先使用两个元素的枚举类型.



@6.41:慎用重载

·介绍:
1.调用哪个重载方法是在编译时决定的
2.对于重载方法的选择是静态的,而对于被覆盖的方法的选择的动态的.
3.使用重载的保守策略:永远不要到处两个具有相同参数数目的重载方法.如果方法使用可变参数保守的策略是根本不要重载它.
4.如果显然不能把一种类型的实例转换为另一种类型.这两种类型就是根本不同的.
5.只要当这两个重载方法在同样的参数上被调用时,它们执行相同的功能,重载就不会带来危害.
6.不必改造具有final数组参数的每个方法,只当确实是在数量布丁的值上指向调用才使用可变参数.
7.使用可变参数会导致性能问题,但可以这样:
public void foo(){}
public void foo(int a1){}
public void foo(int a1,int a2){}
public void foo(int a1,int a2,int... rest){}



@6.42:慎用可变参数

·总结:
在定义参数数目不定的方法时,可变参数方法是一种很方便的方式,但是它们不应该被过度滥用.如果使用不当会产生混乱的后果.

@7.45:将局部变量的作用域最小化

·如何使作用域最小化:
1.最有利的方法就是在第一次使用它的地方声明
2.几乎每个局部变量的声明都应该包含一个初始化表达式

·附加:
1.如果在循环终止之后不再需要循环变量的内容,for循环就优先于while循环.


@7.46:for-each循环优先于传统的for循环

·使用for-each的好处:
for-each循环不仅让你遍历集合和数组,还让你遍历任何实现iterable接口的对象

·三种无法使用for-each的情况:
1.过滤:如果需要遍历列表或者集合,并删除选定的元素,就需要使用显示的迭代器,以便可以调用它的remove方法
2.转换:如果需要遍历列表或者数组,并取代它部分或者全部的元素值,就需要列表迭代器或者数组索引,以便设定元素的值.
3.平行迭代:如果需要并行的遍历多个集合,就需要显示地控制迭代器或者索引变量,以便所有迭代器或者索引变量都可以得到同步前移.


@7.48:如果需要精确的答案,请避免使用float和double

·为什么不用float和double?
因为float和double会产生精度丢失,所以需要使用BigDecimal保证精度.

·使用BigDecimal的两个缺点:
1.与使用基本运算类型相比,这样做很不方便
2.性能不高


@7.49:基本类型优先于装箱类型

·使用装箱基本类型需要注意的点:
1.对于装箱基本类型不能使用==进行值比较
2.不要将基本类型和装箱混用

·什么时候应该使用装箱基本类型?
1.最为集合中的元素、键和值.因为不能将基本类型放在集合中,因此必须使用装箱基本类型.

·总结:
当可以选择的时候,基本类型要优先于装箱基本类型.基本类型更加简单,也更加快速.如果必须使用装箱基本类型,要特别小心.自动装箱减少了使用装箱基本类型的繁琐性,但是并没有减少它的风险性.当程序员使用==操作符比较两个装箱基本类型时,他做了同一性比较,这几乎肯定不是你所希望的.当程序员进行设计装箱和拆箱基本类型的混合类型计算时,它会进行拆箱,当程序进行拆箱时,会抛出NullPointException.最后当程序装箱了基本类型值时,会导致高开销和不必要的对象创建.


@7.50:如果其他类型更适合因避免使用字符串

·字符串不适用在哪些场景:
1.字符串不适用代替其他的值类型:假如一段数据是以字符串形式保留,但实际上是别的类型如:int、double、true等.此时要将这些数据转为它们对应的类型.
2.字符串不适合代替枚举类型
3.字符串不适合代替聚集类型,如:string key = name+"#"+address;这样是错误的.key代表了两个数据,而且还有#.
4.字符串不适合代替能力表

·总结:
如果可以使用更加合适的数据类型,或者可以编写更加适当的数据类型,就应该避免使用字符串来表示对象.若使用不当,字符串会比其他类型更加笨拙、更加不灵活、速度更慢、也更容易出错.进程被错误地使用字符串来代替的类型包括基本类型、枚举类型和聚集类型.




@7.51:当心字符串的连接性能

·避免字符串连接操作:
实际上 string = stringA + "A"; 像这种操作,每执行一次都会返回一个新的string.

·总结:
不要使用字符串合并操作,而应该使用StringBuilder的append方法.


@7.52:通过接口引用对象

·何时通过接口引用对象:
1.如果有合适的接口类型存在,那么对于参数、返回值、变量和域来说,都应该使用接口类型进行声明.
2.如果没有合适的接口存在,完全可以用类而不是借口引用对象.



@7.53:接口优先于反射机制

·使用反射的缺点:
1.丧失了编译时类型检查的好处
2.执行反射访问所需要的代码非常笨拙和冗长.
3.性能损失
4.需要捕获大量的异常

·附加:
1.通常普通应用程序在运行时不应该以反射方式访问对象.


·何时使用反射?
除了一些复杂的应用程序外,平常对于有些程序,它们必须用到在编译时无法获取的类,但是在编译时存在适当的接口或者父类引用这个类.如果是这种情况,就可以以反射方式创建实例,然后通过它们的接口或者父类,以正常的方式访问这些实例.


@7.54:谨慎使用本地方法

·使用本地方法来提高性能的用法不提倡



@7.55:谨慎的进行优化

·在优化方面应该遵守的链条规则:
1.不要进行优化
2.还是不要进行优化-也就是说,在你还没有绝对清晰的未优化方案之前请不要进行优化.

·总结:
不要费力去编写快速程序,应该努力编写好的程序,速度自然会随之而来.

@8.57:只针对异常的情况才使用异常

·使用try-catch的注意事项:
1.因为异常机制的设计初衷是用于不正当的情形,所以很少会有JVM事项视图对它们进行优化,使得与实现的测试一样快速.
2.把代码放在try-catch块中阻止了现代JVM实现本来可能要执行的某些特定优化.
3.对数组进行遍历的标准模式并不会导致冗余的检查.有些现代的JVM实现会将它们优化掉.

·如果把正常的代码放到try-catch中的缺点:
1.使代码意图变得模糊.
2.捕获到不正确的异常

·总结:
异常是为了在异常情况下使用而设计的.不要将它们用于普遍的控制流,也不要编写迫使它们这么做的API.





@5.58:对于可恢复的情况使用受检异常,对编程错误使用运行时异常


·什么是受检异常,什么是非检查异常?
受检异常:在客户端调用的时候需要try-catch进行处理或者throws的异常.(检查时异常)
非检查异常:分为运行时异常和错误.这两种异常都不需要捕获.(运行时异常)

·什么时候抛出受检异常,什么时候抛出非检查异常?
1.如果调用者能适当的恢复代码,此时就应该使用视察时异常.(如果不清楚是否可以恢复,最好使用未检查异常)
如果程序抛出未检查异常或者错误,往往结束与不可恢复的情形,继续执行下去有害无益.

·什么时候抛出运行时异常?
当编程错误时使用运行时异常

·什么时候抛出错误?
当资源不足、约束失败、或者其他程序无法继续执行的条件时抛出错误.

·附加:
1.另外实现所有的为检查异常都应该是RunntimeException的子类
2.需要在异常中定义能够捕获异常的代码而提供额外信息的方法
3.检查异常必须捕获



@5.59:避免不必要地使用受检异常

·不必要的受检异常的缺点:
1.不必要的受检异常会导致过多的try-catch
2.异常受检的本质 并没有为程序员提供任何好处,它反而需要付出努力,而使程序更为复杂.
 


@5.60:优先使用标准异常

·专家级程序员与缺乏经验的程序员的区别:专家级程序追求并通常能够实现代码的重用.

·重用现有异常的好处:
1.它使你的的API更加易于学习和使用,因为它与程序员已经熟悉的习惯用法是一致的.
2.对于用到这些API的程序而言,它们的可读性会更好,因为它们不会出现很多程序员不熟悉的异常.

·最常被使用的异常:
IllegalStateException,如果因为接受对象的状态而使调用非法,通常就会抛出这个异常.
如果调用者在某个不允许null值的参数中传递了null,习惯性的做法是抛出NullPointException.
如果调用者在表示序列下标的参数中传递了越界的值,应该抛出的解释IndexOutBoundsException.

·附加:
1.可以说所有错误的方法调用都可以被归结为非法参数或者非法状态.
2.如果某个异常能满足你的需要,就不要由于,使用就是.(不过一定要确保抛出遗产规定条件与该异常的文档中描述的条件一致)
3.有时候选择哪个异常并不总是那么清楚.



@5.61:抛出与抽象相对应的异常

·异常转译:
如果方法抛出的异常与它执行的任务没有明显的的联系,这种情况会使人不知所措.
为了解决这个问题,更高层的实现应该捕获低层的遗产,同时抛出可以按照高层抽象进行解释的异常.例如:
try{
					
} catch(LowerLevelException e){			//在catch中抛出对当前异常进行解释的异常.
    throw new HIgherLevelException(...);
}

·异常链:一种特殊的异常转译被称为异常链
如果底层的异常对于调试导致高层异常的问题非常有帮助,使用异常链就很合适.低层的异常被传到高层的异常,高层的异常提供访问方法来获得低层的异常:
try{

} catch (LowerLevelException cause){
	throw new HigherLevelException(cause);	//将低层异常参数传递给高层异常,前提是高层异常很需要这个低层的参数.
}

高层异常的构造器将原因传到支持链的超级构造器,因此它最终将被传给Throwable其中一个运行异常链的构造器,例如Throwable:
class HigherLevelException extends Exception{    
    HigherLevelException(Throwable cause){
    	super(cause);
    }
}

大多数标准的异常都有支持链的构造器.对于没有支持链的异常,可以利用Throwable的initCuse方法设置原因.

·注意:
尽管异常转译与不加选择地从低层传递异常的做法相比有所改进,但是它也不能被滥用.如果可能,处理来自低层异常的最好做法是,在调用低层方法之前确保他们会执行成功,从而避免他们抛出异常.

·总结:
如果不能阻止或者处理来自更低层的异常,一般的做法是使用异常转译,除非低层方法碰巧可以保证它抛出的所有异常对高层也适合才可以将异常从低层传播到高层.
异常链对高层和低层异常都提供了最佳的功能:它允许抛出适当的高层异常,同时又能捕获底层的原因进行失败分析.




@5.62:每个方法抛出的异常都要有文档

·
1.始终要单独地声明受检异常,并且利用javadoc的@throw标记,准确的记录下抛出每个异常条件.(例如IO流找不到文件的异常)
2.如果一个方法可能抛出多个异常,则不要将它抛出.(之前说到的要,对异常进行捕获).
3.永远不要抛出throw Exception,或者更糟糕的是声明它throw Throwable.

·文档注意事项:
1.每个方法的文档应该描述它的前提条件,这是很重要的,在文档中记录下未收件的异常时满足前提条件的最佳做法.
2.对于接口中的方法,在文档中记录下他可能抛出的未受检异常显得尤为重要.这份文档构成了该接口的通用约定的一部分,他指定了该接口的多个实现必须遵守的公共行为.
3.使用javadoc的@throw标签记录下一个方法可能抛出的每个未受检的异常,但是不要使用throw关键字将未受检的异常包含在方法的声明中.
4.如果一个类中的许多方法处于同样的原因而抛出同一个异常,在该类的文档注释中对这个异常建立文档,这是可以接受的,而不是为每个方法单独建立文档.



@5.63:在细节消息中包含能捕获异常失败的信息

·总结:
为异常的"失败捕获"信息提供一些访问方法是何时的.提供这样的访问方法对于受检的异常,比对于未受检的异常更为重要,因为失败-捕获信息对于从失败中恢复是非常有用的.



@5.64:努力使失败保持原子性

·注意:
1.一般而言,失败的方法调用应该使对象保持在被调用之前的状态.具有这种属性的方法被称为具有失败原子性.
2.当方法抛出错误时,它们不需要努力保持失败原子性.

·实现失败原子性的方法:
1.设计一个不可变对象
2.先计算只有计算结果正确时对象才会被持久化
3.编写一段恢复代码,使得对象回滚到操作开始之前的状态.(不常用)
4.在对象的一份临时拷贝上操作,当操作完成之后在用临时拷贝中的结果代替对象的内容.

·总结:
一般而言,作为方法规范的一部分,产生的任何异常都应该让对象保持在该方法调用之前的状态.如果违反这条规定,API文档就应该清除地指明对象将会处于什么样的状态.



@5.65:不要忽略异常

·例如:
try{

}catch(SomeException e){

}

1.不应该使用空的catch块,你需要处理这个异常.当你发现有空的catch块时,应该注意.至少要说明为什么可以忽略这个异常.
2.有一种情况可以忽略,即关闭FileInputStream的时候.即使在这种情况下,把异常记录下来还是明智的做法.
3.作为客户端而言,只要将异常传播给外界,至少会导致程序迅速地失败,从而保留了有助于调试该失败条件的信息.

@9.66:同步访问共享的可变数据

·什么是同步?
正确的使用同步可以保证没有任何方法会看到对象处于不一致的状态中.如果没有同步,一个线程的变化就不能被其他线程看到.

·注意事项:
1.读取一个非double或float类型的变量,可以保证返回的值是某个线程保存在该变量中,即使在多个线程没有同步的情况下并发地修改这个变量也是如此.
2.为了在线程之间进行可靠的通信,也为了互斥访问,同步是必要的.
3.不要使用Thread.stop
4.要阻止一个线程访问另一个线程,建议做法是让第一个线程轮询一个Boolean变量,这个变量一开始为false,但是可以通过第二个线程设置为true,以表示第一个线程将终结自已.
5.增量操作符++不是原子的(如果一个线程中用到了++操作符,就需要格外注意!)
6.让一个线程在短时间内修改一个数据对象,然后与其他线程共享是可以的.

·如何避免高并发:
高并发解决的最佳策略是不共享可变数据.要么共享不可变数据,要么压根不共享.换句话说,将可变数据限制在单个线程中.

·总结:
当多个线程共享可变数据的的时候,每个读或者写数据的线程都必须执行同步.如果没有同步就无法保证一个线程所做的修改可以被另一个线程获知.未能同步共享可变数据会造成程序的活性失败和安全性失败.如果只需要线程之间的交互通信,而不需要互斥,volatile修饰符就是一种可以接受的同步形式,但要正确的使用它可能需要一些技巧.



@9.67:避免过度同步

·过度同步存在的问题:
过度同步会造成性能降低,死锁.甚至不确定的行为.

·同步区域内注意事项:
1.在一个被同步的区域内部,不要调用设计成要被覆盖的方法或者是有客户端以函数对象的形式提供的方法.
2.锁是可重入的
3.在同步区域之外被调用的外来方法被称作"开放调用",处理可以避免死锁之外,开放调用还可以增加并发性.
4.通常你应该在同步区域内做尽可能的工作.

·总结:
为了避免死锁和数据破坏,千万不要从同步区域内部调用外来方法.要尽量限制同步区域内部的工作量.当你在设计一个可变类的时候,要考虑一下它们是否应该自已完成同步操作.只有当你有足够的理由一定要在内部同步类的时候,才应该这么做.




@9.68:executor和task优于线程

·不同的线程池用于不同的情况:
1.如果编写的是小程序,或者是请在的服务器,使用Executors.newCachedThreadool是个不错的选择.
2.在大负载的产品服务器中,最好使用Executors.newFixedThreadPool

·附加:
尽量不要编写自已的工作队列,以及尽量不直接使用线程(优先使用线程池)



@9.69:并发工具优先于wait和notify

·并发工具分类:
java.util.concurrent被分为三类:线程池、并发容器、同步容器.

·使用并发集合注意事项:
1.并发集合不可能排除并发活动,将它们锁定并没有什么作用,只会使线程的速度变慢.
2.除非不得已,否则应该优先使用ConcurrentHashMap,而不是使用Collections.synchronizedMap或者Hashtable.
3.应该优先使用并发集合,而不是使用外部的同步的集合.

·同步器;
1.同步器是一些使线程能够等待另一个线程的对象,允许它们协调动作.最常用的同步器是CountDownLatch和Semaphore.

·附加:
1.对于门歇式的定时,始终应该优先使用System.nanoTime而不是使用System.currentTimeMills.System.nanoTime更加准确也更加精确,它不受系统的实时时针的调整所影响.
2.总是应该使用notifyAll(但有例外,下面一条)
3.如果处于等待状态的所有线程都在等待同一个条件,而每次只有一个线程可以从这个条件中被唤醒,那就应该使用notify.

·总结:
没有理由在新代码中使用wait和notify.使用并发容器优先于使用wait和notify.一般情况下,你应该优先使用nofityAll而不是使用notify.



@9.70:线程安全性的文档化

·附加:
1.通过查看文档中是否出现synchronized修饰符,可以确定一个方法是否是线程安全的.这种说法是错误的.因为在一个方法声明中出现synchronized修饰符,这是个实现细节,并不是导出API的一部分.
2.一个类为了可被多个线程安全地使用,必须在文档中清除的说明它所支持的线程安全性级别.

·线程安全性级别分类:
1.不可变的:这个类的实例时不可变的,所以,不需要外部的同步.这样的例子包括String、Long、BigInteger.
2.无条件的线程安全:这个类的实例时可变的,但是这个类有着足够的内部同步,所以它的实例可以被并发使用,无序任何外部同步.其例子包括:Random和ConcurrentHashMap.
3.有条件的线程安全:除了有些方法为进行安全的并发使用而需要外部同步之外,这种线程安全级别与条件的线程安全相同.这样的例子包括Collections.synchronized包装返回的集合,它们的迭代器要求外部同步.
4.非线程安全:这个类的实例是可变的.为了并发地使用它们,客户必须利用自已选择的外部同步保卫每个方法调用.这样的例子包括通用的集合实现,例如ArrayList和HashMap.

·私有锁对象模式:§
1.私有锁对象模式只能用在无条件的线程安全类上,有条件的线程安全了不能使用这种模式.
2.私有锁对象模式特别适用于那些专门为继承而设计的类.
3.出于不同目的而使用相同的锁,子类和基类可能会"相互绊住对象的脚"

·总结:
每个类都应该利用字斟酌句的说明线程安全注解,清除地在稳定当说明它的线程安全属性.synchronized修饰符与这个文档毫无关系.有条件的线程安全类必须在文档中指出"哪个方法调用序列需要外部同步,以及在执行这些序列的时候要获取哪把锁".如果你编写的是无条件的线程安全类,就应该考虑使用私有锁对象来代替同步方法.这样可以防止客户端和子类的不同步干扰,让你能够在后续的版本中灵活的对并发控制采用更加复杂的方法.


  
@9.71:慎用延迟初始化

·延迟初始化的缺点:
1.对于延迟初始化,最好建议"除非绝对必要",否则就不要这么做.
2.延迟初始化降低了初始化类或者创建实例的开销,却增加了访问被延迟初始化域的开销.
3.延迟初始化实际上降低了性能
4.延迟初始化需要被同步
5.在大多数情况下,正常的初始化要优先于延迟初始化.
6.如果利用延迟初始化优化来破坏初始化的循环,就要使用同步访问方法.
7.如果处于对性能的考虑而需要对静态变量使用延迟初始化则:
public class SingLetonDeme3 {
	
	private SingLetonDeme3(){
		
	}
	
	private static class SingLetonClassInstance{
		private static SingLetonDeme3 instance = new SingLetonDeme3();
	}
	
	public static synchronized SingLetonDeme3 getinstance() {
		return SingLetonClassInstance.instance;
	}
}
8.如果处于性能的考虑而需要对实例变量使用延迟初始化,将使用双重检查模式:
private volatile FieldType field;
FieldType getField(){
    fieldType rsult = field;
    if(result == null){
        synchronized(this){
		result = field;
	if(result == null){
		field = result = computeFieldValue();
	    }
        }
    }	
}


·什么时候应该使用延迟初始化:
1.如果变量只在类的实例部分被访问,并且初始化这个变量的开销很高时,就值得进行延迟初始化.

·总结:
大多数的变量应该正常地进行初始化,而不是延迟初始化.如果为了达到性能目标,或者为了破坏有害的初始化循环,而必须延迟初始化一个变量,就可以使用相应的延迟初始化方法.对于实例变量,就使用双重检查模式,对于静态变量则使用上述方法.对于可以接受重复初始化的变量,也可以考虑使用单重检查模式.



@9.72:不要依赖于线程调度器

·线程调度器的缺点:
1.任何依赖于线程调度器来达到正确性或者新呢个要求的程序,很有可能都是不可移植的.

·使用线程要注意的事项:
1.要编写健壮的、响应良好的、可移植性的多线程应用程序,最好的办法是确保可运行线程的平均数量不明显多于处理器的数量.(有一个线程池)
2.保持可运行线程数量尽可能少的主要方法是,让每个线程做些有意义的工作,然后等待更多有意义的工作.
3.如果线程没有在做有意义的工作,就不应该运行.
4.线程不应该一直处于忙-等的状态,即反复地检查一个共享对象,以等待某些事情发生.
5.不要调用Thread.yield来修正程序.
6.线程优先级是java平台上最不可移植的特性了.
7.Thread.yield的唯一用途是在测试期间人为的增加程序的并发性.

·总结:
1.不要让应用程序的正确性依赖于线程调度器.
2.不要依赖Thread.yield或者线程优先级.

@10.74:谨慎地实现Serializable接口

·实现Serializable接口的代价:
1.一旦一个类被发布了,就大大降低了"改变这个类的实现"的灵活性.(这是最大的代价)
2.它增加了出现BUG和安全漏洞的可能性
3.随着类发行新的版本,相关的测试负担也增加了

·序列化的注意事项:
1.不应该使用默认的序列化形式.如果你接受了默认的序列化形式,并且以后又要改变这个类的内部表示法,结果可能导致序列化形式的不兼容.
2.如果你接受了默认的序列化形式,这个类中私有的和包级私有的实例变量将都变成导出的API的一部分.
3.一开始就设计高尚品质的序列化了类,并且在很长时间内都愿意使用这种形式.
4.设计良好的序列化形式也许会给类的演变带来限制,但是设计不好的序列化形式则可能会使类根本无法演变.
5.要声明一个SerivlVersionUID的私有静态final的long变量,否则一旦类修改,兼容性会遭到破坏.(因为JVM会自动更具类的状态生成一个版本标识的值)
			private static final long serialVersionUID = 100;
6.实现Serializable接口并不是一个很轻松就可以做出的决定.

·什么情况下实现Serializable接口由必要?
1.如果一个类要加入到某个框架中,并且该框架依赖于序列化来实现对象或者持久化,对于这个类来说,实现Serializable接口是非常有必要的.
2.静态成员类可以实现Serializable接口


·什么情况最好不要实现Serializable接口:
1.为了继承而设计的类应该尽可能少地去实现Serializable接口,用户的接口也应该尽可能少地继承Serializable接口.
2.内部类不应该实现Serializable接口

·附加:
特别是,如果父类没有提供可访问的无参构造器,子类也不可能做到可序列化.因此对于为继承而设计的不可变序列化的类,你应该考虑提供一个无参构造器.



·总结:
千万不要认为实现Serializable接口会很容易,除非一个类在用了一段时间之后会被抛弃(而且这个类不能使用默认的实现),否则实现Serializable接口就是个很严肃的承诺.如果一个类是为了继承而设计的,则更加需要加倍小心.对于这种为了继承而设计的类而言,可以提供一个可访问的无参构造器.这种设计方案运行但不要求子类实现Serializable接口.



@10.75:考虑使用自定义的序列化形式

·如何实现自定义的序列化形式?
public class TestSerializable implements Serializable {
	
	private transient int size = 0;
	
	private void writeObject(ObjectOutputStream s) throws IOException {		//写出时会自动调用该方法
		s.defaultWriteObject();		//必须首先调用defaultWriteObject
		s.writeInt(size);		//即使size加上了transient修饰符仍然可以被序列化
	}

	private void readObject(ObjectInputStream s) throws ClassNotFoundException, IOException {	//读也会自动调用该方法
		s.defaultReadObject();		//必须首先调用defaultReadObject
		size = s.readInt();		//即使size加上了transient修饰符仍然可以被序列化
	}
	
}


·使用默认序列化形式注意事项:
1.如果没有先认真考虑默认的序列化形式是否合适,则不要贸然接受.一般情况下,只有当你自行设计的自定义序列化形式和默认的序列化形式基本相同时,才能接受默认的序列化形式.
2.如果一个对象的无力表示法等同于它的逻辑内容,可能就适合于使用默认的序列化形式.
3.即使你确定了默认的序列化形式是合适的,通常还必须体统一个readObject方法以保证约束关系和安全性.
4.在决定将一个变量做成非transient的之前,请一定要确信它的值将是该对象逻辑状态的一部分.
5.无论你是否使用默认的序列化形式,如果在读取整个对象状态的任何其他方法上强制任何同步,也必须在对象序列化上强制这种同步.
		private static final long serialVersionUID = 100;
6.如果你想修改一个没有序列化版本UID的现有的类,并希望新的版本能够接受现有的序列化的实例,就必须使用哪个自动为旧版本生成的值.
7.如果一个类声明中加上了implements Serializable的字样,它就不再是一个单例了.
8.任何一个readObject方法,不管是显示的还是默认的,它都会返回一个新建的实例,这个新建的实例不同于该类初始化时创建的实例.


·当一个对象的物理表示法与它的逻辑数据内容有实质性区别时,使用默认序列化形式会有以下四个缺点:
1.它使这个类导出的API无法永远地束缚在该类的内部表示法上.
2.它会消耗过多的空间.
3.它会消耗过多的时间.
4.它会引起栈溢出.


·总结:
当你决定将一个类作成可序列化的时候,请仔细考虑应该采用什么样的序列化形式.只有当默认的序列化形式能够合理地描述对象的逻辑状态时,才能使用默认的序列化形式,否则就要设定一个自定义的序列化形式,通过它合理的描述对象的状态.你应该分配足够多的时间来设计类的序列化形式.你也不能去掉序列化形式中的变量,它们必须被永久的保留下去,以确保序列化兼容性.选择一个错误的序列化形式对于一个类的复杂性和性能都会有永久的负面影响.


@10.76:保护性地编写readObject方法

·使用readObject方法需要注意的事项:
1.readObject方法实际上相当于一个公有的构造器.和其他构造器一样,readObject必须检查其参数的有效性,并且在必要的时候对参数进行保护性拷贝.
2.readObject不可以调用可被覆盖的方法,无论是直接调用还是间接调用都不可以.

·如何避免仿造的字节流被反序列化?
为了修正这个问题,你可以为Period提供一个readObject方法,该方法首先调用defaultReadObject,然后检查被反序列化之后的对象的有效性.如果有效性检查失败,readObject方法就会抛出一个InvalidObjectException异常,使得反序列化过程不能成功的完成.

·如果恶意的字节流满足有效性检查,但是在这个字节流内部有指向原类的变量,则可以轻易修改这个变量:
1.当一个对象被反序列化的时候,对于客户端不应该拥有对象的引用,如果哪个变量包含了这样的对象引用,就必须要做保护性拷贝,这是非常重要的.
2.因此,对于每个可序列化的不可变类,如果它包含了私有的可变组件,那么在它的的readObject方法中,必须要对这些组件进行保护性拷贝.
注意保护性拷贝不要使用clone,具体的在上面.

·总结:
每当你编写readObject方法的时候,都要这样想:你正在编写一个公有的构造器,无论给它传递什么样的字节流.它都必须产生一个有效的实例.(不要假设这个字节流一定代表着一个被真正序列化过的实例)
一些指导方案:
1.对于对象引用变量必须保持为私有的类,要保护性拷贝地拷贝这个变量中的每个对象.不可变类的可变对象就属于这一类别.
2.对于任何约束条件,如果检查失败,则抛出一个InvalidObjectException异常.这些检查动作应该跟在所有的保护性拷贝之后.
3.如果整个对象图在被反序列化之后必须进行验证,就应该使用ObjectInputValidation接口.
4.无论是直接方式还是间接方式,都不要调用类总任何可被覆盖的方法.





@10.77:对于实例控制,枚举类型优先于readResolve

·readResolve有什么用?(可以保证单例)
readResolve特性允许你用readObject创建的实例代替另一个实例.对于一个正在被反序列化的对象,如过它的类定义了一个readResolve方法,并且具备正确的声明,那么在反序列化之后,新建对象上的readResolve方法就会被调用.然后该方法返回的对象引用将被返回,取代新建的对象.

·readResolve如何使用:
public vlass Elvis{
    public static final transient Elvis INSTANCE = new Elvis();
    private Elvis(){
    }
    private Object readResolve(){
	return INSTANCE;
    }
}

·使用readResolve的注意事项:
如果依赖readResolve进行实例控制,带有对象引用类型的所有实例变量则都必须声明为transient.

·附加:
1.使用枚举是有天然保障的单例,所以推荐使用枚举.
2.readResolve的可访问性很重要.如果把readResolve方法放在一个final类上,他就应该是私有的.如果把readResolve方法放在一个非final的类上,就必须认真考虑它的可访问性.如果它是私有的,就不适用于任何子类.如果它是包级私有的,就只是用于同一个包中的子类.如果它是守保护的或者公有的,仅适用于所有没有覆盖它的子类.如果readResolve方法是守保护的或者公有的,并且子类没有覆盖它,对于序列化的子类实例进行反序列化,就会产生一个父类实例,这样有可能导致ClassCastException.

·总结:
应该尽可能地使用枚举类型来实施实例控制的约束条件.如果做不到,同时又需要一个即可序列化又是实例受控的类,就必须提供一个readResolve方法,并确保该类的所有实例都为基本类型,或者是transient.



@10.78:考虑用序列化代理代替序列化实例

·使用序列化代理减少风险

·如何实现序列化代理模式:
1.为可序列化的类设计一个私有的静态嵌套类,精确地表示外围实例的逻辑状态
2.这个嵌套类被称作序列化代理,它应该有一个单独的构造器.,其参数就是那个外围类.这个构造器只要从它的参数中赋值数据:他不需要进行任何一致性检查或者保护性拷贝.从设计角度来看,序列化代理的默认序列化形式是外围类最好的序列化形式.外围类及其序列化代理都必须声明实现Serializable接口.

·如何使用:
private static class SerializableProxy implemenst Serializable{

    private Object writeReplace(){			//在写出时调用该方法
	return new SerializableProxy(this);
    {

}

writeReplace方法的存在导致序列化系统产生一个SerializableProxy实例,代替外围类的实例.换句话说,writeReplace方法在序列化之前,将外围类的实例转变成了它的序列化代理.
有了writeReplace方法之后,序列化系统永远不会产生外围类的序列化实例.但是攻击者任有可能伪造,所以需要在外围类中添加readObject方法:
private void readObject(ObjectInputStream stream){
    throw new InvalidObjectException("Proxy required");
}

最后再提供一个readResolve方法,它返回一个逻辑上想当的外围类的实例.这个方法的出现,导致序列化系统在反序列化时将序列化代理转变回外围类的实例.

private Object readResolve(){
    return new Period();
}

·使用序列化代理模式的局限性:
1.他不能与科研被客户端扩展的类兼容.
2.它也不能与对象图中包含循环的某些类兼容.

·总结:
每当你发现自已必须在一个不能被客户端扩展的类上编写readObject或者writeObject方法的时候,应该考虑使用序列化代理模式.要想稳健的将带有重要约束条件的对象序列化时,这种模式可能是最容易的方法.
