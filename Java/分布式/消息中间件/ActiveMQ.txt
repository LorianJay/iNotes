1.4:为什么要引入MQ?(MQ结局了什么问题)这个在springboot整合rabbitMQ的时候讲到过
1.异步处理:
以前处理事件时,客户端和服务端都是同步的方式运行的,客户端一个请求过来必须等到你服务端有响应,不然就在那干等着.现在是客户端一个请求过来,这个请求不直接与服务器交互,而是将自已的请求内容交给MQ,交给MQ之后客户端该干嘛干嘛,然后服务端从MQ里面拿数据进行处理,一旦处理成功就把处理成功的数据返回给对应的客户端.实际上就是异步的一种体现.

2.流量削峰:
根据之前说的,当有大量的请求访问的时候不是直接与服务器交互,而是与MQ交互.那么MQ就会容纳这些请求,防止服务器在某一刻遭遇很大的流浪,可以让服务器专心处理当前的业务.

3.解耦:
请求不再直接访问服务器,而是与MQ交互实现应用的解耦.模块之间也不是直接互相调用,从而实现模块直接的解耦.

1.6.1:JMS规范
JMS规范:
基于JVM消息代理的规范.ActiveMQ、HornetMQ是JMS实现


1.6.2:整体架构
通过Connection Factory创建一个Connection与ActiveMQ建立连接,通过这个Connection创建一个Session,通过这个Session可以创建Message、可以创建目的地(目的地可以指定是队列或主题)、可以创建消息生产者(通过消息生产者向队列或主题发送消息)、可以创建消费者(消费者从Destination(目的地)消费).(和之前JDBC类似)

Destination(目的地):分为两个区域:队列(点对点通讯)、主题(一对多,订阅/发布)

说一下订阅的概念:订阅必须要先有消费者(必须要先订阅),生产者再生产.如果没有人订阅则生产者的消息就是废消息(言外之意就是主题模式下,生产者生产的消息只会发送给正等着消费的消费者).JMS规范允许用户创建持久订阅(在3.8)

1.6.3:JMS组成的四大元素 
JMS组成的四大元素:
JMS provider:实现JMS接口和规范的消息中间件产品(也就是ActiveMQ)
JMS producer:消息生产者
JMS consumer:消费者
JMS Message☆:一个消息由 消息头、消息体、消息属性 组成
消息头:Message的五种设置内容
消息体:封装具体的消息数据、5种消息体格式(也就是Message的五个直接子接口)、发送和接收的消息体类型必须一致对应
消息属性:如果有很多消息,要找到特殊的消息,就可以通过消息属性 识别/去重/重点标注一个消息.

1.6.4:JMS的可靠性(持久性、事务、签收(Acknowledge))

·持久性:
1.生产者的持久性(3.4):
1.1:如果服务器宕机了,服务器重启后如果消息还在的话就成为持久,如果消息不存在了称为非持久的.
1.2:要想指定生产者发送的消息是否是持久化可以有至少三种方式 第一种是针对消息而言的,通过Message的顶级接口的setJMSDeliveryMode(int)方法指定当前消息是否是可持久化的(该方法是针对单个消息有效)
第二种是针对生产者而言的,通过生产者的setDeliveryMode(int)方法指定该生产者发送的消息是否是可持久化的(该方法针对该生产者发出的所有消息有效) 第三种是针对生产者发出消息这个动作而言的,可以通过生产者的send方法的重载方法指定当前发出消息是否是可持久化的.在设置的时候传入DeliveryMode接口的静态常量,如果是1代表持久化的,如果是2代表是持久化的.
1.3:队列的生产者发出的消息默认就是持久化的,主题的生产者发出的消息默认是非持久化的.

·事务:在我们通过Connection创建Session的时候说到过事务(agrs0)
来回顾一下通过Connection创建Session:connection.createSession(boolean,int);

1.生产者提交时的事务(也就是说生产者是靠有事务的Session创建出来的)
1.1:如果是false,相当于数据库的自动提交,每生产一条消息就自动提交到MQ中
1.2:如果是true,相当于数据库的手动提交,最终需要通过session.commit();来提交当前session的所有操作(生产者的所有生产).保证当前生产者生产的消息要么全部生产成功要么全部不生产成功(一致性).

2.消费者消费事务时(同样,消费者是靠有事务的Session创建出来的)
1.1:如果是false,相当于自动提交,每消费一条消息就自动提交到MQ中
1.2:如果是true,相当于手动提交,最终需要通过session.commit;来提交当前消费的操作.保证当前消费者要么所有消息全部消费,要么所有消息全部不消费.如果开启了事务还忘记了commit则会造成重复签收问题.

3.签收:(args1)
3.1:共有三种签收方式 Session.AUTO_ACKNOWLEDGE(0,自动签收) Session.CLIENT_ACKNOWLEDGE(1,手动签收) Session.DUPS_OK_ACKNOWLEDGE(2,允许按副本地重复签收,这个用得少不介绍)
3.2:当采用手动签收时,通过消费者获得的每个Message消息都需要调用该方法,告诉MQ该消息被签收了,否则会造成消息重复接收的问题

4.事务偏生产者/签收偏消费者

5.当既有手动提交又有手动签收时(这里只针对消费者,因为生产者不存在什么签收机制)!!!!!!!
5.1:开启手动提交和手动签收后,忘记调用acknowledge()方法,但是调用了commit方法.结论是 当调用了commit方法之后会自动签收所有消息.
5.2:开启手动提交和手动签收后,忘记调用commit()方法,但是调用了acknowledge()方法.结论是 当只调用acknowledge()方法后会产生重复消费的问题.
5.3:一句话概括只要调用commit肯定签收 如果没有调用commit肯定不签收 如果调用了rollback肯定也是不会签收的

1.7:ActiveMQ特性
1.当多个消费者消费同一个队列时,ActiveMQ会保证每个消费者按照启动顺序均匀分配这个队列的信息.(这就是负载均衡)
2.队列的消息会被持久化保存(即使没有消费者消息也不会消失),主题的消息不会被持久化.

3.3:创建消息生成者,以及向目的地(队列)中发送消息

public class JmsProduce {

    public static final String ACTIVE_MQ = "tcp://localhost:61616";
    public static final String QUEUE_NAME = "sukidayo";

    public static void main(String[] args) {
        //1.创建连接工厂,按照给定的URL地址,采用默认的账号和密码
        ActiveMQConnectionFactory factory = new ActiveMQConnectionFactory(ACTIVE_MQ);
        Connection connection = null;
        try {
            //2.通过连接工厂,获得Connection连接 并启动访问
            connection = factory.createConnection();
            connection.start();
            //3.创建Session args0:是否开启事务 args1:签收
            Session session = connection.createSession(false, Session.AUTO_ACKNOWLEDGE);
            //4.创建目的地(指定是队列还是主题)
            Queue queue = session.createQueue(QUEUE_NAME);
            //5.创建消息的生产者
            MessageProducer producer = session.createProducer(queue);
            //6.创建消息
            TextMessage textMessage = session.createTextMessage("msg...");
            //7.发送消息
            producer.send(textMessage);

            //8.关闭
            producer.close();
            session.close();
            connection.close();
        } catch (JMSException e) {
            System.out.println("连接失败");
            e.printStackTrace();
        }

    }
}

·BytesMessage:类似于StreamMessage,不过内部是byte[]数组
write*(*) return void;					//按照流的形式写出数据,看API就清楚了 agrs0:八大基本数据类型+String+Object+bytes[]等
read*() return *;					//获得这个流的数据

·StreamMessage:类似于DataIn/OutPutStream流,以二进制流的形式写出数据,写出和获取的时候按照类型获取.

·ObjectMessage:
setObject(Serializable) return void;			//写出一个可被序列化的对象
getObject() return Serializable;			//获得这个序列化对象

public class JmsConsumer {

    public static final String ACTIVE_MQ = "tcp://localhost:61616";
    public static final String QUEUE_NAME = "sukidayo";

    public static void main(String[] args) {
        //1.创建连接工厂,按照给定的URL地址,采用默认的账号和密码
        ActiveMQConnectionFactory factory = new ActiveMQConnectionFactory(ACTIVE_MQ);
        Connection connection = null;
        try {
            //2.通过连接工厂,获得Connection连接 并启动访问
            connection = factory.createConnection();
            connection.start();
            //3.创建Session args0:是否开启事务 args1:签收
            Session session = connection.createSession(false, Session.AUTO_ACKNOWLEDGE);
            //4.创建目的地(指定是队列还是主题)
            Queue queue = session.createQueue(QUEUE_NAME);			//这里的队列名称和3.3的队列名称一致
            //5.创建消息的消费者 
            MessageConsumer consumer = session.createConsumer(queue);		//由于你会发现ActiveMQ是根据消息队列的名称,来区分各个队列的.就是K-V一样.			
            //6.接收消息
            while (true) {
                TextMessage message = (TextMessage) consumer.receive();
                if (message != null) {
                    System.out.println(message.getText());
                } else {
                    break;
                }
            }
            //7.关闭
            consumer.close();
            session.close();
            connection.close();
        } catch (JMSException e) {
            System.out.println("连接失败");
            e.printStackTrace();
        }

    }
}


3.6:消费者监听器

作用:那你就猜呗,之间以及讲过很多的监听器了(包括rabbitMQ的时候),作用无非就是监听一个消息队列,当这个队列有消息的时候就会触发监听方法,然后监听方法去消费.
public class JmsConsumerListener {

    public static final String ACTIVE_MQ = "tcp://localhost:61616";
    public static final String QUEUE_NAME = "sukidayo";

    public static void main(String[] args) {
        //1.创建连接工厂,按照给定的URL地址,采用默认的账号和密码
        ActiveMQConnectionFactory factory = new ActiveMQConnectionFactory(ACTIVE_MQ);
        Connection connection = null;
        try {
            //2.通过连接工厂,获得Connection连接 并启动访问
            connection = factory.createConnection();
            connection.start();
            //3.创建Session args0:是否开启事务 args1:签收
            Session session = connection.createSession(false, Session.AUTO_ACKNOWLEDGE);
            //4.创建目的地(指定是队列还是主题)
            Queue queue = session.createQueue(QUEUE_NAME);
            //5.创建消息的生产者
            MessageConsumer consumer = session.createConsumer(queue);
            //6.利用监听器
            consumer.setMessageListener(message -> {
                if (message instanceof TextMessage) {
                    TextMessage textMessage = (TextMessage) message;
                    try {
                        System.out.println(textMessage.getText());
                    } catch (JMSException e) {
                        e.printStackTrace();
                    }
                }
            });
            //保证控制台不关闭
            System.in.read();
            //7.关闭
            consumer.close();
            session.close();
            connection.close();
        } catch (JMSException e) {
            System.out.println("连接失败");
            e.printStackTrace();
        } catch (IOException e) {
            e.printStackTrace();
        }

    }
}

3.8:持久化订阅主题

和3.7的区别:如果是通过普通的主题发送和接收消息就存在一个问题,虽然说绑定了同一个主题的消费者可以同时获得生产者发送的消息.但是必须保证生产者和消费者同时在线的情况下,生产者的消息才会被消费者消费.(JMS1.6.2整体架构的时候提到过)

在ActiveMQ图形化界面的最上面有一项  Subscribers 该页面展示当前持久化订阅,其中Active Durable Topic Subscribers表示在线的用户,Offline Durable Topic Subscribers表示不在线的用户

现在,只要消费者先启动一次(代表有消费者订阅了指定的主题),哪怕现在消费者暂时关闭了.在这关闭期间生产者生产了内容,当消费者再次启动时就可以获得生产者的信息了(不想之前会丢失)

生产者代码:有改动
public class JmsProduce_Topic_Persist {

    public static final String ACTIVE_MQ = "tcp://localhost:61616";
    public static final String TOPIC_NAME = "myTopic";

            connection.start();

            TextMessage textMessage = session.createTextMessage("Hello");
            //8.发送消息
            producer.send(textMessage);

            //9.关闭
            producer.close();
            session.close();

5.5.1:插一嘴个人理解的特性
通过5.3和5.5,你会发现并没有什么所谓的生产者消费者,到底是生产还是消费完全取决于调用的是JmsTemplate类的哪个方法.再回顾之前创建生产者和消费者的时候,都是绑定队列或者主题,主题和队列又都是通过名称来标识自已到底是哪个目的地.一般一个生产者就有与之对应的消费者,那么这两个对应的生产和消费使用的目的地当然是同一个目的地,所以spring也就全封装在JmsTemplate里了,相当于对一个目的地创建了一个JmsTemplate,然后再在真正调用方法的时候去决断是生产还是消费.不同的队列就注入不同配置的JmsTemplate.

7.1:传输协议
ActiveMQ传输的时候支持很多种协议,打开ActiveMQ的配置文件,找到
<transportConnectors>
	<transportConnector name="openwire" uri="tcp://127.0.0.1:61616?maximumConnections=1000&amp;wireFormat.maxFrameSize=104857600"/>	
	<transportConnector name="amqp" uri="amqp://127.0.0.1:5672?maximumConnections=1000&amp;wireFormat.maxFrameSize=104857600"/>
	<transportConnector name="stomp" uri="stomp://127.0.0.1:61613?maximumConnections=1000&amp;wireFormat.maxFrameSize=104857600"/>
	<transportConnector name="mqtt" uri="mqtt://127.0.0.1:1883?maximumConnections=1000&amp;wireFormat.maxFrameSize=104857600"/>
	<transportConnector name="ws" uri="ws://127.0.0.1:61614?maximumConnections=1000&amp;wireFormat.maxFrameSize=104857600"/>
</transportConnectors>

可以看到第一行的值 openwire-tcp 这就是默认的TCP协议,启用的端口也是61616.你会发现一点除了第一行是 openwire-tcp 其余行的内容是一样的 amqp-amqp stomp-stomp.这是因为ActiveMQ默认就是TCP协议.

7.2:协议的选择

点击可视化界面标题栏的Connections,里面展示的内容正好和7.1的配置所对应.

·Tcp:
1.这是默认的Broker配置,TCP的Client监听端口61616
2.虽然底层传输的时候采用TCP协议传输,但在传输之前要先通过wire protocol协议将数据序列化成字节流,(所以这也是为什么TCP被称作openwire原因),这样做的目的是促使网络上的效率和数据快速交互.
3.TCP的URI形式:tcp:hostname:port?key=value&key=value,后面的参数是可选的
4.TCP传输的优点:
4.1:TCP协议传输可靠性高,稳定性强
4.2:高效性 字节流方式传递,效率很高
4.3:有效性、可用性 应用广泛,支持任何平台

·NIO:
1.NIO表面上和TCP类似,但NIO比TCP有更高的性能,能承受更大的负载.
2.NIO的URI形式: nio:hostname:port?keey=value&key=value		
3.添加配置<transportConnector name="nio" uri="nio://127.0.0.1:61616?maximumConnections=1000&amp;wireFormat.maxFrameSize=104857600"/>	以开启NIO

·AMQP:注意不同的协议在生产和消费的时候的代码也是不同的
1.这个协议可以更高地兼容更多的语言

·MQTT:
1.支持更多的平台

·WS:
1.支持H5(HTML5)

7.4:NIO加强

原本TCP的时候使用的IO模型是BIO(也就是TCP+BIO),现在使用NIO(也就是NIO+TCP).现在要加强NIO让NIO支持多种协议,让NIO同时支持TCP+AMQP+MQTT+WS
<transportConnector name="auto+nio" uri="auto+nio://127.0.0.1:61608?maximumConnections=1000&amp;wireFormat.maxFrameSize=104857600&amp;
org.apache.activemq.transport.nio.SelectorManager.corePoolSize=20&amp;org.apache.activemq.transport.nio.SelectorManager.maximumPoolSize=50"/>
		
·这里直接借鉴3.3和3.5的代码,只把做了改变的地方列出来:
public static final String ACTIVE_MQ = "nio://localhost:61608";				//现在的端口是61608

此时如果URI是 tcp://localhost:61608 照样可以访问,由此就可以得出一个结论,具体使用的是什么IO模型,以及该IO模型支持哪些协议完全是取决于你访问的是哪个端口.而一开始指定的TCP NIO AMQP就是用来指定是那种传输协议的. 但是如果直接换可能会报错,就目前来看索然TCP和NIO的代码可以共用,但是不代表AMQP等别的协议就可以.

8.1:持久化概念介绍
之前讲1.6.4的JMS的高可用(可靠性)时,讲到MQ支持事务、持久、签收.但是这三种特性是ActiveMQ直接支持的,假如说现在ActiveMQ自已都挂了就不能保障这三个特性了,所以现在需要借助外力,将MQ的数据备份一份到数据库上.(而且MQ的服务器要和SQL服务器物理上独立)


8.2:持久化方式介绍
ActiveMQ的持久化方式主要就是:KahaDB、LevelDB、JDBC

8.2.1:KahaDB
基于日志文件,从ActiveMQ5.4开始默认的持久化插件.也就是通过日志文件记录当前的所有操作.

怎么知道当前版本使用的哪种持久化方式呢?查看ActiveMQ的配置文件有一项
<persistenceAdapter>
	<kahaDB directory="${activemq.data}/kahadb"/>		//这个配置代表持久化数据保存在activemq/data目录下的kahadb文件夹下
</persistenceAdapter>

找到data文件夹下果然有kahadb文件夹,说明当前持久化的方式是kahadb.

在kahadb文件夹下有四个文件:db.data、db.free、db.redo、db-1.log、lock

·KahaDB的存储机制:使用一个事务日志和仅用一个索引文件来存储它所有的地址.

db-1.log:
该文件是日志文件用来存储数据记录(也就是上面说的事务日志),文件名为db-<Number>.log 当数据文件存储内容达到上限时会创建一个新的日志文件(默认每个文件32M),每新建一个文件Number都会+1.
当不在有引用到数据文件中的任何消息时,文件会被删除或归档.

db.data:
该文件包含了持久化的BTree索引,使用该文件的索引来定位db-<number>.log日志文件

db.free:
当前db.data文件里哪些页面是空闲的,文件具体内容是所有空闲页的ID.

db.redo:
用来进行消息恢复,如果KahaDB消息存储在强制退出后启动,用于恢复BTree索引.

lock:
文件锁,类似mysql里面的读写锁.

8.2.4:JDBC增强

之前使用JDBC(数据库)会产生什么问题呢?生产者生产消息要生产到数据库中(写库),消费者消费也要从数据库中读取(读+写),这样频繁地操作数据库很浪费时间.
现在在AMQ访问时速局库之前加一层缓存,如果说消费者消费的速度能跟上生产者,那么直接从缓存里面拿.剩下的没有消费掉的内容就一次性批量写入数据库.

将原来的KahaDB或LevelDB或JDBC配置修改为:
<persistenceFactory>
	<journalPersistenceAdapterFactory
	journalLogFiles="4"
	journalLogFileSize="32768"
	useJournal="true"
	useQuickJournal="true"
	dataSource="#mysql-ds"				//这个地方还是对应的
	dataDirectory="activemq-data"/>
</persistenceFactory>

此时利用8.3的知识,先针对队列而言,当调用生产者之后消息不会立即同步到mysql中,而是要等大约7-8分钟才会同步到mysql中,再次期间消费者消费是没有任何问题的.

10.2:异步投递Async Sends
大多数情况下投递消息都是异步(这之前已经讲过了),ActiveMQ支持同步和异步.(不过异步的性能更好)
在不修改配置的情况下,如果不使用事务而且还需要持久化地发送信息就会造成类似同步的效果(阻塞).

在高性能的应用下允许有少量数据的丢失(也就是不会发送到MQ服务器上).如果使用异步发送就不能有效的确保消息的发送成功,在useAsyncSend=true的情况下客户端需要容忍消息丢失的可能(但这样可以保证更高的性能)

如何开启异步投递:
方法一:
之前在创建连接工厂或者spring配置或者springboot配置的时候,都有一个 tcp://localhost:61616 之前讲这个协议的时候说过,可以在后面加K-V,在创建连接的时候通过该地址开启异步投递:
tcp://localhost:61616s?jms.useAsyncSend=true
方法二:
在3.4的时候有说到过,通过调用ActiveMQConnectionFactory对象的setUseAsyncSend(true)方法开启异步投递.注意由于是投递所以是生产者那端调用此方法.

·追问,如何确认异步投递成功:

public class JmsProduce_AsyncSend {

    public static final String ACTIVE_MQ = "nio://localhost:61618";
    public static final String QUEUE_NAME = "sukidayo";

    public static void main(String[] args) {
        ActiveMQConnectionFactory factory = new ActiveMQConnectionFactory(ACTIVE_MQ);
        factory.setUseAsyncSend(true);		//设置异步投递
        Connection connection = null;
        try {
            connection = factory.createConnection();
            connection.start();
            Session session = connection.createSession(false, Session.AUTO_ACKNOWLEDGE);
            Queue queue = session.createQueue(QUEUE_NAME);
            ActiveMQMessageProducer activeMQMessageProducer = (ActiveMQMessageProducer) session.createProducer(queue);		//!!!一定要强转为ActiveMQMessageProducer类
            TextMessage textMessage = session.createTextMessage("HelloJDBC");
            for (int i = 0; i < 1; i++) {
                textMessage.setJMSMessageID(UUID.randomUUID().toString());			//给当前消息对象设置ID
                String UID = textMessage.getJMSMessageID();					//获得到这个ID
                activeMQMessageProducer.send(textMessage, new AsyncCallback() {
                    @Override
                    public void onSuccess() {			//当消息发送成功后会调用该方法
                        System.out.println("UID为:" + UID + "消息发送成功");
                    }

                    @Override
                    public void onException(JMSException exception) {		//当消息发送失败后会调用该方法,如果有异常还会传入异常 
                        System.out.println("UID为:" + UID + "消息发送失败");
                    }
                });
            }

            activeMQMessageProducer.close();
            session.close();
            connection.close();
        } catch (JMSException e) {
            System.out.println("连接失败");
            e.printStackTrace();
        }

    }
}


10.3:延迟投递和定时投递

首先修改配置文件:
将<broker xmlns="http://activemq.apache.org/schema/core" brokerName="localhost" dataDirectory="${activemq.data}">修改为:
<broker xmlns="http://activemq.apache.org/schema/core" brokerName="localhost" dataDirectory="${activemq.data}" schedulerSupport="true">

生产者:
public class JmsProduce_SchedulerSupport {

    public static final String ACTIVE_MQ = "nio://localhost:61618";
    public static final String QUEUE_NAME = "sukidayo";

    public static void main(String[] args) {
        ActiveMQConnectionFactory factory = new ActiveMQConnectionFactory(ACTIVE_MQ);
        Connection connection = null;
        try {
            connection = factory.createConnection();
            connection.start();
            Session session = connection.createSession(false, Session.AUTO_ACKNOWLEDGE);
            Queue queue = session.createQueue(QUEUE_NAME);
            MessageProducer producer = session.createProducer(queue);

            long delay = 3 * 1000;          //延时投递的时间,3s
            long period = 4 * 1000;         //重复投递的间隔,4s
            int repeat = 5;                 //重复多少次,5次

            TextMessage textMessage = session.createTextMessage("HelloJDBC");
            textMessage.setLongProperty(ScheduledMessage.AMQ_SCHEDULED_DELAY, delay);
            textMessage.setLongProperty(ScheduledMessage.AMQ_SCHEDULED_PERIOD, period);
            textMessage.setIntProperty(ScheduledMessage.AMQ_SCHEDULED_REPEAT, repeat);

            producer.send(textMessage);

            producer.close();
            session.close();
            connection.close();
        } catch (JMSException e) {
            System.out.println("连接失败");
            e.printStackTrace();
        }

    }
}

通过TextMessage对象的setxxxProperty来设置投递规则(详解可看3.4的API),值得注意的是agrs1传入的类型必须和方法名保持一致,例如setLongProperty这个方法虽然可以传入int类型的不会报错,但是执行起来会出现问题.这些set方法是针对当前这个Message实例而言的,当设置完之后程序不会阻塞,只是告诉MQ这里需要定时投递,所以MQ会后台自已运行.

10.5:ActiveMQ消费重试机制

哪些情况下会引起消息重发?如果发生了重复消费AMQ会检测到并记录次数.
1.Client用了事务且在session中调用了rollback()
2.Client用了事务且在调用commit之前关闭或者没有commit,也就是消费者开启了事务 消费消失后没有commit,这个之前讲过的呀
3.Client在手动签收的传递模式下,在session中调用了recover

默认情况下如果需要消息重发会每秒重发一次,共重发6次.
如果某个消息重发了6次还没有重发到,那么这个消息就是有毒消息Poison ACK,此时broker就会把这个消息放到DLQ(死信队列,这个队列名称叫ActiveMQ.DLQ,也就是相当于把这些消息挪到了另一个队列)

有一些配置属性到时候可以回视频中再看一下.(比如指定重发间隔、重发次数等)

还可以通过代码指定重发次数、重发间隔等.

ActiveMQConnectionFactory factory = new ActiveMQConnectionFactory(ACTIVE_MQ);			//建立连接,这个设置是针对消费者而言的
RedeliveryPolicy redeliveryPolicy = new RedeliveryPolicy();
redeliveryPolicy.setMaximumRedeliveries(3);
factory.setRedeliveryPolicy(redeliveryPolicy);				//通过调用连接共产的setRedeliveryPolicy方法传入一个RedeliveryPolicy类来设置这些内容


10.6:死信队列

当某个消息重发次数达到指定次数(默认是6次),此时该消息就会进入死信队列,然后人工进行干预死信队列.

AMQ默认会将所有的死信保存到一个共享的队列,这是AMQ的默认策略,共享队列默认就是Active.MQ 可以通过"deadLetterQueue"属性来设定.(通过配置文件指定,这里没有写)


10.7:如何保证消息不被重复消费?谈谈幂等性问题
准备一个第三方来做消费记录,以redis为例,给消息分配一个全局ID,只要消费过该消息,将<id,message>以K-V的形式写入redis.那消费者开始消费前,先去redis中查询有没有消费记录即可.(如果有就不消费,没有就消费).

幂等性就是重复性消费问题.
