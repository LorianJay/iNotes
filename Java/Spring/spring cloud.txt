2.1:什么是微服务?
微服务提倡将单一应用程序划分成一组小的服务,每个服务运行在其独立的进程中(注意是进程),服务之间互相协调、互相配合;应尽量避免统一的、集中式的服务管理机制.各个服务可以独立部署;可以使用不同的语言来编写服务.
根据业务拆分成多个服务(不绝对),每个服务还有自已的单独的数据库.

2.2:微服务架构
也就是相当于Eclipse里通过maven创建的一个个model.

2.3:微服务优缺点
优点:
1.每个服务足够内聚,足够小,代码更容易理解这样能聚焦一个指定的业务功能或业务需求.
2.开发简单、开发效率高,一个服务可能是专一的只干一件事.
3.微服务能够被小团队独立开发,团队2-5人
4.微服务是松耦合的,是有功能意义的服务,无论是在开发阶段或部署阶段都是独立的.
5.微服务能使用不同的语言开发.
6.抑郁和第三方继承,微服务运行容易且灵活的方式集成自动部署,通过持续集成工具,如Jenkinns,Hudson,bamboo
7.微服务易于被一个开发人员理解,修改和维护,这样小团队也能够更关注自已的工作成果.无需通过合作才能体现价值.
8.微服务允许你利用融合最新技术
9.微服务只是业务逻辑代码,不会和HTML,CSS或其他界面组件混合.(最强大的特性)
10.每个微服务都有自已的存储能力,可以有自已的数据库.也可以有统一数据库.(这句的体现在6.3)
由第9点引申出前后端分离:
java后端程序员,只需要管理后端,给前端的H5工程师按照约定,Rest地址+输入参数格式和报文约定+输出参数.(通过Restful接口交互,和dubbo模块间交互一样本质上也是模块间通过接口交互)

缺点:
1.服务之间通信成本增加、调用超时
2.运维成本增加


2.4:微服务技术栈有哪一些?
微服务技术栈是多种技术的集合体.
服务治理、服务注册、服务调用、服务负载均衡、服务监控.实现这些技术内容需要不同的框架,现在springcloud一站式地全部解决了.

2.5:为什么选择springcloud?
微服务是一种理念,实现微服务的有dubbo、springcloud但为什么要选择springcloud?

springcloud是完整的微服务框架,一站式解决了微服务,使用的Restful风格,而dubbo是RPC的.

3.1:什么是springcloud?
springcloud是分布式微服务架构下的一站式解决方法,是各个微服务架构落地技术的结合体,俗称微服务全家桶.

springcloud和springboot是什么关系?
springboot关注内部一个个的细节,而springcloud是将springboot整个整合起来对外.springboot可以单独使用,而springcloud需要依赖springboot.
springboot专注于快速、方便的开发单个微服务个体,springcloud关注全局的服务治理框架.

·springcloud分布式开发五大常用组件:(这个之前讲springboot的时候提到过)
注册中心-Netflix Eureka

客户端负载均衡-Netflix Ribbon

断路器-Netflix Hystrix
如果模块A调用模块B,模块B调用模块C,模块C调用模块D 调用D模块的时候发生了错误,那么下个用户同样访问该功能时就会直接失败不会等待很长时间.

服务网关-Netflix Zuul
过滤请求

分布式配置-Spring cloud config

public interface DeptDao {

@Service
public class DeptServiceImpl implements DeptService {

    DeptDao deptDao;

    public boolean addDept(Dept dept) {
        return deptDao.addDept(dept);
    }

    @Override
    public Dept findById(long id) {
        return deptDao.findById(id);
    }

    @Override
    public List<Dept> findAll() {
        return deptDao.findAll();
    }
}


·controller层:

@RestController
public class DeptController {
    @Autowired
    DeptService service;

    @RequestMapping(value = "dept/add")
    public boolean add(Dept dept) {
        return service.addDept(dept);
    }

    @RequestMapping(value = "dept/get/{id}")
    public Dept get(@PathVariable("id") Long id) {
        return service.findById(id);
    }

    @RequestMapping(value = "dept/list")
    public List<Dept> getAll() {
        return service.findAll();
    }
}

·编写主类:springbootapplication

大体的内容没有什么好说的,基本上还是controller层调用service层,service层调用dao层,这里需要详细说明controller层:
可以看到修饰DeptController类的注解是@RestController(这个之前讲过,此时该类下面方法的返回值是json串),此时如果在浏览器访问localhost:port/dept/list.会显示数据库dept表中被封装成json的数据.
所以说这里加强前后端概念,一个请求地址来了 只有前端拦截(然后前端去决定是转发还是重定向),然后前端给一个地址到后端,后端通过controller拦截到这个地址,然后返回一个json(一定是json)给前端.前端拿到这个json去做处理.

4.8:编写microservicecolud-consumer-dept-80消费者模块的代码

·RestTemplate:这个类提供了多种便捷访问远程HTTP服务的方法,是一种简单便捷的访问restful服务模板类,是spring提供的用于访问rest服务的客户端模板工具集.
postForObject(String,Object,Class<T>) return T;			//带参数地请求一个地址 agrs0:请求地址 agrs1:请求参数 args2:Http响应被转换成的类型 return:返回值就是agrs2的类型
getForObject(String,Class<T>,Object...) return T;		//该方法主要用来请求一个地址

·编写Config配置文件:
@Configuration
public class ConfigBean {
    @Bean
    public RestTemplate getRestTemplate(){			//将RestTemplate添加到IOC容器中供我们使用
        return new RestTemplate();
    }
}

·编写消费者:
@RestController
public class DeptController_Consumer {

    private static final String REST_URL_PREFIX = "http://localhost:8001";

    @Autowired
    RestTemplate restTemplate;

    @RequestMapping(value = "consumer/dept/add")
    public boolean add(Dept dept) {
        return restTemplate.postForObject(REST_URL_PREFIX + "/dept/add", dept, Boolean.class);		
	//第一个参数是要访问的地址
	//第二个参数就是访问这个地址时要传入的参数
	//第三个参数就是请求了这个地址后返回值的类型
    }

    @RequestMapping(value = "consumer/dept/get/{id}")
    public Dept get(@PathVariable("id") Long id) {
        return restTemplate.getForObject(REST_URL_PREFIX + "/dept/add" + id, Dept.class);
    }

    @RequestMapping(value = "consumer/dept/list")
    public List<Dept> get() {
        return restTemplate.getForObject(REST_URL_PREFIX + "/dept/list", List.class);
    }


}

对于RestTemplate的理解:
就是通过这个类再去请求一个地址,这个地址就是后端服务器的地址.

5.1.1:什么是Eureka
Eureka遵守的就是AP原则(之前在springboot进阶的时候也讲过),它就是一个注册中心.(类似dubbo的zookeeper)

5.1.2:Eureka架构
Eureka采用C-S架构,Eureka Server作为服务注册功能的服务器.
Eureka包含两个组件:Eureka Server和Eureka Client.Eureka Server提供服务注册服务,Eureka Client是一个java客户端.

5.1.3:三大角色
Eureka:提供服务注册和发现
Service Provider:服务提供方将自身服务注册到Eureka,从而使服务消费方能够找到
Eureka Consumer:服务消费方从Eureka获取注册服务列表,从而能够消费服务

5.2.5:eureka自我保护

如果现在8001的服务提供者长时间不提供服务,当我们刷新springcloud的管理页面时会出现红色的错误字体(这个就是springcloud的自我保护机制).
如果8001模块的配置文件这一项内容为eureka.instance.instance-id=microservicecolud-dept8001,此时8001模块一次,然后将配置文件改为eureka.instance.instance-id=microservicecolud-dept8001xxx再启动模块一次,然后再将配置改回去并重启一次8001,此时刷新管理页面同样也会出现错误信息.(据我估测是因为springcloud是根据spring.application.name这一项配置来区分服务的,现在一个服务下机后又上限,但是他们的eureka.instance.instance-id名不同就会发生错误.)当出现冲突时springcloud的默认处理机制,不是将这两个服务直接注销,而是暂时先保留着.

·某时刻某个服务不可用,eureka不会立刻清理,依旧会对该微服务的信息进行保存.

·产生自我保护的原因:
1.默认情况下,如果eureka在一定时间内没有接受到某个微服务实例的心跳,eureka将会会注销该实例(默认90s).但是当网络分区故障发生时,此时就不会注销该微服务.

eureka设计的原则就是宁可保留错误的服务信息也不盲目地注销服务,在springcloud中可以使用eureka.server.enable-self-preservation=false关闭自我保护机制.(同样也可以通过配置修改默认等待的时间),当然这两个配置都不建议去修改.

@Autowired
DiscoveryClient client;

@RequestMapping("url")
public Object discovery() {
    List<String> services = client.getServices();					//1
	for (String s : services) {
            List<ServiceInstance> instances = client.getInstances(s.toUpperCase());	//2
            for (ServiceInstance instance : instances) {

5.4:eureka比zookeeper好在哪里?关于CP和AP在redis1.10.2中查看

zookeeper保证的是CP
eureka保证的是AP

zookeeper的问题在于,虽然保证了强一致性但是,zookeeper master瘫痪重新选举的这段期间(大约30-120s),这个注册的服务全部瘫痪不可用.
eureka吸取了zookeeper的教训,各个eureka节点都是平等的,如果某几个节点挂掉剩余的节点依然可以提供注册和查询服务.eureka客户端(Client)在想某个eureka服务端(Server)注册或者发现连接失败时,则会自动切换至其它节点,只要有一台eureka服务器就能保证服务可用,只不过信息可能不是最新的(因为保证AP丢掉了C(强一致性)).另外eureka还有自我保护机制(也就是前面的自我保护机制),因此eureka可以很好的应对因网络故障导致部分节点失去联系的情况,而不会像zookeeper那样使整个注册服务瘫痪.

比如在淘宝双十一的时候,首先应该保证可用性,因为数据就算误差了一点也比整个网站瘫痪要好.

6.1:概述

·什么是ribbon?
springcloud ribbon 是基于Netflix ribbon实现的一套客户端 负载均衡工具.
所以说Netflix ribbon是规范,springcloud ribbon 是实现.具体的作用是,当一个请求来到eureka时,此时可能有多个服务(生产者)给该请求服务,那么eureka该选用哪个服务呢?此时就是通过负载均衡算法来选择具体的服务.

springcloud ribbon的负载均衡算法可以自定义,

LB(负载均衡):分为 集中式、进程内
集中性:在服务的消费方和提供质检使用独立的LB设施.
进程内:将负载均衡的逻辑继承到消费方,消费方从服务注册中心获知有哪些地址可用,然后自已再从这些地址中选择出一个合适的服务器.Ribbon就属于进程内的LB,它只是一个类库,集成于消费方进程,消费方通过它来获取到服务提供方的地址.

6.4:ribbon核心组件IRule

IRule:根据特定算法中从服务列表中选取一个要访问的服务.下面是7种算法

RoundRobinRule:轮询
RandomRule:随机
AvailabilityFilteringRule:会先过滤掉由于多次访问故障而处于断路器跳闸状态的服务,还有并发的连接数量超过阈值的服务,然后对剩余的服务列表按照轮询策略进行访问.
WeightedResponseTimmeRule:根据平均响应时间计算所有服务的权重,响应时间越快服务权重越大被选中的概率越高,刚启动时如果统计信息不足,则使用轮询策略,等待统计信息足够时再切换到该策略.
RetryRule:先按照轮询的策略获取服务,如果获取服务失败则在指定时间内会进行重试.获取可用的服务.
BestAvailableRule:会先过滤由于多次访问故障而处于断路器跳闸状态的服务,然后选择一个并发量最小的服务.
ZoneAvoidanceRule:默认规则,符合判断server所在区域的性能和server可用性选择服务器.


如何选择别的负载均衡算法?之前说过选择具体的哪个服务的权利在消费者手上,所以来到消费者80模块,在ConfigBean配置类中添加一个方法:

@Bean
public IRule myIRule(){
    return new RandomRule();
}
    
因为一个消费者只能选择一个算法,所以当我们自已在IOC容器中添加一个IRule接口的实现后,该实现就会把默认采用的算法给覆盖.要想算法生效必须将所有 eureka和所有服务提供者都重启一遍.(不过运维的兄弟可以写脚本批量启动)


重点强调RetryRule算法:
在没有异常的情况下RetryRule算法和轮询算法一样,假设现在有三个服务 8001 8002 8003 ,一开始还是按照正常的方式轮询,但是中间突然把8002宕了,RetryRule此时还是会按照轮询算法,但是当访问到8002服务时,由于没有这个服务,所以会出现404.然后紧接着还是 3->1->失败->3->1->.... 经过几次访问8002失败后,IRule就不会去访问8002,而是只访问 8001和8003(访问正常的服务).

//5.RandomRule_MY类:

RandomRule_MY类实现的算法是:依然按照轮询算法,不过每台机器调用5次后才进入下一台机器.模板可以拷贝RandomRule这类实现类

public class RandomRule_MY extends AbstractLoadBalancerRule {

    private volatile int total;						//考虑到多线程的情况,加上volatile关键字.		
    private volatile int index;

    public Server choose(ILoadBalancer lb, Object key) {
        if (lb == null) {
            return null;
        }
        Server server = null;

        while (server == null) {
            if (Thread.interrupted()) {
                return null;
            }
            List<Server> upList = lb.getReachableServers();			//获取到当前在线的服务   1
            List<Server> allList = lb.getAllServers();			//获取所有的服务		 2

            int serverCount = allList.size();
            if (serverCount == 0) {
                /*
                 * No servers. End regardless of pass, because subsequent passes
                 * only get more restrictive.
                 */
                return null;
            }
            if (total == 5) {
                total = 0;
                if (index < upList.size() - 1) {
                    index++;
                } else {
                    index = 0;
                }
            }
            total++;
            server = upList.get(index);				//从所有在线的服务中选择一个服务返回.	3

            if (server == null) {
                /*
                 * The only time this should happen is if the server list were
                 * somehow trimmed. This is a transient condition. Retry after
                 * yielding.
                 */
                Thread.yield();
                continue;
            }

            if (server.isAlive()) {
                return (server);
            }

            // Shouldn't actually happen.. but must be transient or a bug.
            server = null;
            Thread.yield();
        }

        return server;

    }

    @Override
    public void initWithNiwsConfig(IClientConfig clientConfig) {

    }

    @Override
    public Server choose(Object key) {
        return choose(getLoadBalancer(), key);
    }
}


说重点:
实际上就是通过 public Server choose(ILoadBalancer lb, Object key) 方法来确定服务,这个方法最终返回提供本次服务的实例.主要是对upList这个集合进行操作,实际上这里就是数据结构与算法具体的算法可以自已看下.

8.1:概述
·分布式系统面临的问题:如果服务A调用服务B,服务B又调用服务C,这就是所谓的扇出.如果服务A调用某个服务的时候出现长时间未响应或者不可用,对微服务A的调用就会占用越来越多的系统资源,进而引起系统崩溃,所谓的雪崩效应.所以需要对故障进行隔离和管理,以便单个依赖关系的失败,不能取消整个应用程序或系统.

·Hystrix就是用于处理分布式系统的延迟和容错的开源库,Hystrix能保证在一个依赖出问题的情况下,不会导致整体服务失败,避免级联故障,以提高分布式系统的弹性.
熔断器本省是一种开关装置,通过熔断器的故障监控,向调用方发那会一个符合预期的、可处理的备选响应,而不是长时间的等待或者抛出调用方无法处理的异常.

8.3:服务降级

什么是服务降级:整体资源快不够了,忍痛将某些服务先关掉,待渡过难关,再开启回来.

服务的降级是在客户端进行完成和服务端没有关系.

我们8.2的代码存在一个问题,每一个业务方法都伴随着一个异常方法,这样会导致方法膨胀,并且异方法和业务方法耦合.

在思考一下,调用8001模块的service层的方法,找到源头只有可能是80端口通过RestTemplate接口或者通过Feign(面向接口)这两个防水来调用的,所以如果8001服务调用异常,那么80模块(消费者模块)也会出现异常,所以只要我们检测到80模块出现异常就相当于检测8001模块.

8.3.1:服务熔断和服务降级

代码层面:
服务熔断就是为了避免服务之间互相调用时,某个服务超时引起占用系统大量资源.但是在技术上来说异常方法和业务方法耦合了,所以就需要服务降级.
服务降级很类似服务熔断都是为了避免异常导致的系统资源占用,服务降级从技术上来说,就是让异常方法和业务方法解耦,以及避免方法膨胀. 

术语层面:

服务熔断:一般是某个服务故障或者异常引起,类似显示世界中的"保险丝",当某个异常条件被出发,直接熔断整个服务,而不是一直等到此服务超时.
服务降级:所谓降级,一般是从整体符合考虑.就是当某个服务熔断之后,服务器不再被调用,此时客户端可以自已准备一个本地父fallback回调,返回一个缺省值.这样做,虽然服务水平下降,但好歹可用,比直接挂掉强.

10.1:概述

·分布式系统面临的配置问题:
由于每个服务都需要必要的配置信息(一个服务要有一个application配置文件),springcloud提供了configServer来解决这个问题.

springcloudconfig为微服务架构中的微服务提供集中化的外部配置支持,配置服务器为各个不同微服务应用的所有环境提供了一个中心化的外部配置.
springcloudconfig本身也是一个微服务,之前每个微服务都需要自已管理配置文件,现在全部交给springcloudconfig,然后springcloudconfig在个GitHub交互统一管理.

springcloudconfig分为服务端和客户端两部分
服务端也称为分布式配置中心,它是一个独立的微服务应用,用来连接配置服务器并未客户端提供获取配置信息,加密/解密信息等访问接口.
客户端是通过制定的配置中心来管理应用资源,springcloudconfig推荐采用git来存储信以及与业务相关的配置内容,并在启动的时候从配置中心获取和加载配置信息配置服务器默认采用git来存储配置信息,这样就有助于对环境配置进行版本管理,并且可以通过git客户端工具来方便的管理和访问配置内容。

·能干嘛:
1.集中管理配置文件。
2.不同环境, 不同配置动态化的配置更新分环境部署,比如,dev/test/prod/beta/release
3.运行期间动态调整配置不再需要在每个服务部署的机器上编写配置文件服务,会向配置中心统一拉取配置自己的信息.
4.当配置发生变动时,服务不需要重启即可感知到配置的变化,并应用新的配置.
5.将配置信息以rest接口的形式暴露
