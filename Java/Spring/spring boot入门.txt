1.6:主程序类、主入口类(@SpringBootApplication注解详解)
·介绍:
springboot应用标注在某个类上说明这个类是springboot的主配置类,springboot就应该运行这个类的main方法来启动springboot应用.

·查看springboot注解:这是一个组合注解

@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Inherited
@SpringBootConfiguration
@EnableAutoConfiguration
@ComponentScan(excludeFilters = { @Filter(type = FilterType.CUSTOM, classes = TypeExcludeFilter.class),
		@Filter(type = FilterType.CUSTOM, classes = AutoConfigurationExcludeFilter.class) })
public @interface SpringBootApplication {
	@AliasFor(annotation = EnableAutoConfiguration.class)
	Class<?>[] exclude() default {};
	@AliasFor(annotation = EnableAutoConfiguration.class)
	String[] excludeName() default {};
	@AliasFor(annotation = ComponentScan.class, attribute = "basePackages")
	String[] scanBasePackages() default {};
	@AliasFor(annotation = ComponentScan.class, attribute = "basePackageClasses")
	Class<?>[] scanBasePackageClasses() default {};
}

这里所说的组件就是bean.bean就对应java类.

@SpringBootConfiguration:springboot的配置类
	标注在某个类上,表示这是一个springboot的配置类
	@Configuration:配置类上来标注这个注解
	配置类=配置文件,配置类也是容器中的一个组件 @Component

@EnableAutoConfiguration:开启自动配置功能
	以前我们需要配置的东西,springboot帮我们自动配置.@EnableAutoConfiguration告诉springboot开启自动配置功能,这样自动配置才能生效.
	@AutoConfigurationPackage:自动配置包
		@Import(AutoConfigurationPackages.Registrar.class):spring底层注解,给容器中导入一个组件,导入的组件是AutoConfigurationPackages.Registrar.class
		AutoConfigurationPackages.Registrar这个类的作用,就是将@SpringBootApplication标注的类所在包及子包下面所有的组件(拦截器)扫描到spring容器中.所以如果你的拦截器类放在主类所在的包		或子包的话就可以被加载到,如果不是的话拦截器就无法被加载.(意思就是和springboot主类同级或子级的类会被自动加载)

	@EnableAutoConfiguration被@Import注解导入了AutoConfigurationImportSelector.class组件
		AutoConfigurationImportSelector:这个类会导入META-INF/spring.factories文件里的所有类(注意这个文件里面全是类的全路径名,所以应该是用反射加载的,然后这些类全是自动配置的类).
		这个文件就是jar包里面常见的资源文件.

2.2:YML的基本语法

2.2.1基本语法:
K: V 			表示一堆键值对,K后面的冒号和V之间要有一个空格,只要左对齐的一列数据都是同一层级的.属性和值大小写敏感

2.2.2值的写法:

·
字面量:普通的值(数字,字符串,布尔)
K: V 字面直接通过KV写	字符串默认不需要加上单引号或者双引号

"":双引号,不会转义字符串里面的特殊字符,特殊字符会作为本身想表达的意思.也就是支持转义
	name: "换行 \n"		输出:换行 [换行]
'':单引号:不支持转义,直接输出原来的东西
	name: '换行 \n'		输出:换行 \n

·
对象、Map:
K: V		对象还是KV对的形式,在下一行写上对象和值的关系.只要注意空格.
friends:	表名 lastName和age形成映射关系
  lastName:张三
  age:20

行内写法:
friedns: {lastName:张三,age:20}		用个大括号括起来

·
数组(List、Set):	通过:- V		通过-[空格][元素]的形式表示一个元素
list:
  - cat
  - dog 
  - fish: {Name:三文鱼,KG:1}

行内写法:
list: [cat,dog,fish: {},]

properties的写法:
list[0]=cat
list[0]=dog

2.3:@ConfigurationProperties和@Value的区别

之前在2.2.3的时候是使用@ConfigurationProperties进行映射注入的.现在使用@Value

@Component
@ConfigurationProperties(prefix = "person")
public class Person {
    @Value("${person.lastName}")		//通过EL表达式引用配置文件中的内容
    private String lastName;
    @Value("#{20*2}")				//SPEL表达式,查看spring4.5.4
    private Integer age;
    @Value("2018/1/1")
    private Date birthday;
}

区别:
@ConfigurationProperties:支持批量注入属性、支持松散绑定、不支持EL表达式、支持JSR303校验、支持复杂类型的封装.
@Value:一个个指定、不支持松散绑定、支持EL表达式、不支持JSR303校验、不支持复杂类型的封装.

两种注解分别适应什么场景:
如果说我们只是在业务逻辑中需要获取文件中的某项值,使用@Value.也就是说@ConfigurationProperties更适合配置文件和实体类之间批量的映射操作.
@Value是非映射的,只是对于某个属性而言,直接从配置文件中获取一下.

松散绑定:
现在Person实体类中lastName在配置文件中可以是:last-name last_name lastName 这三种写法都是等价的.不过只有@ConfigurationProperties支持.@Value必须完全对应.

JSR303校验:其实之前讲过
@Value("${person.lastName}")
@Emial				//JSR303校验.在赋值的时候判断其值是否满足注解格式.比如这里的lastName就必须是邮箱格式的值.	
private String lastName;

复杂的封装:
比如说之前的集合,从配置文件中将一个集合注入给一个集合属性.@Value就做不到.

//1.编写注解类:有了配置类就不需要配置文件了.
/*
@Configuration:指明当前类是一个配置类;就是用来代替之前的spring配置文件.现在在springboot中所有的@Around方法所处的类必须是被@Configuration注解修饰的类中.
 */
@Configuration
public class MyAppConfig {
    //将返回值添加到容器中,容器中这个组件默认的id就是方法名称
    @Bean
    public HelloService helloService(){
        return new HelloService();
    }
}

//2.此时再次测试代码

这里和之前的@Component又有什么区别呢?@Component修饰的bean是可以被注入的类,注入的信息是在配置文件中获得的.当然类里面的方法是自已定义的,这个bean也会被加载到容器中.
@Configuration又是什么呢?以后就可以简单理解为,只要一个类被@Configuration修饰,那么这个类就是一个xml文件(配置文件).在这个配置类里面可以定义被@Bean修饰的方法(注意@Bean注解也只能作用在方法和注解上),一个方法被@Bean修饰代表什么呢?表明springboot在初始化的时候会调用每一个被@Bean修饰的方法,这些方法返回的对象就是一个bean,这些bean全部被加载到ApplicationContext对象(IOC容器中),我们只需要通过这个全局的IOC容器去获取到这个bean,就像以前获取到配置文件中的bean,我们需要给IOC传入的id就是被@Bean修饰的方法的方法名.

@Configuration和@ConfigurationProperties的区别:
@ConfigurationProperties:该注解修饰的类的属性都是从配置问价中获取的,然后springboot会帮助我们创建这个对象,以便之后配置@Autowired使用.
@Configuration:该注解修饰的类,该类中被@Bean修饰的方法,该方法要返回一个对象(这个对象是我们自已创建的,自已去new出来的).而且有时这个对象并不一定需要被获取,加载到容器中之后便于我们的使用.比如该对象实现了某个springboot的接口,那么该对象就会把springboot自动配置的实现给顶替掉.
什么时候用什么注解?
那就需要自已考量了
如果你想springboot帮忙创建,并且只是为了给自已别的地方用的话你就用@Configuration
如果你想springboot用你创建的对象的功能那就用@ConfigurationProperties

另外如果你想springboot帮助我们创建对象,而不与springboot配置文件进行绑定的话.可以使用@Component @Repository @Service @Controller 这些都是讲当前类添加到容器中,所以@Component只是将当前类添加到容器中@ConfigurationProperties注解的意思实际上是和配置文件进行绑定.另外想在别的项目别的地方使用些好的类遇到@Repository @Service @Controller这三层一定要反映过来.

@Component和@Bean的区别:
@Component:
1.利用spring类路径扫描,
2.写在类上,和类定义不分离 
3.不需要配合@configuration注解

@Bean:
1.当类不受spring控制的时候,写一个方法,返回某个不受spring控制的类的对象.这样就能在spring里注入一个bean.
2.配合configuration注解

不过使用@Bean比使用@Component更加灵活

2.10:自动配置原理

自动配置原理:
1.springboot启动的时候加载主配置类,通过@EnableAutoConfiguration这个注解开启了自动配置功能.
2.在@SpringBootApplication这个注解内部又有@Import(AutoConfigurationImportSelector.class)注解.@Import这里的作用是为@SpringBootApplication这个注解导入AutoConfigurationImportSelector这个类,这个类是一个选择器类.进入选择器,主要通过selectImports方法获取候选配置.这个方法返回一个String数组,然后这个方法又调用了getAutoConfigurationEntry()方法.最终返回一个List<String>集合.这个list集合是通过getCandidateConfigurations()方法获取的.getCandidateConfigurations()方法返回的集合是通过SpringFactoriesLoader.loadFactoryNames()这个静态方法获取的.进入这个静态方法,它调用了classLoader.getResources(FACTORIES_RESOURCE_LOCATION)这个方法真正获取到了资源.
这个资源就是:public static final String FACTORIES_RESOURCE_LOCATION = "META-INF/spring.factories";这个静态方法会扫描所有jar包类路径下的
META-INF/spring.factories文件.把扫描到的这些文件的内容包装成properties对象,然后从properties获取一些值,将这些值加到最终需要返回的List<String>集合.从properties中获取到的EnableAutoConfiguration.class类对应的类名,然后把它们添加在容器中.这些值就是一个个的类(bean).
解释:因为spring.factories这个文件中有很多的内容,并不一定全是我们需要的,我们如何定位到我们需要的值?看一眼加载的spring.factories文件的内容

3.每一个自动配置类进行自动配置功能

4.以HttpEncodingAutoConfiguration(Http编码自动配置)为例解释自动配置原理:

//1.分析这个类被修饰的注解:
@Configuration												1
@EnableConfigurationProperties(HttpProperties.class)							2
@ConditionalOnWebApplication(type = ConditionalOnWebApplication.Type.SERVLET)				3
@ConditionalOnClass(CharacterEncodingFilter.class)							4
@ConditionalOnProperty(prefix = "spring.http.encoding", value = "enabled", matchIfMissing = true)	5
public class HttpEncodingAutoConfiguration {

	private final HttpProperties.Encoding properties;						6

	@Bean		//添加一个bean
	@ConditionalOnMissingBean
	public CharacterEncodingFilter characterEncodingFilter() {
		CharacterEncodingFilter filter = new OrderedCharacterEncodingFilter();
		filter.setEncoding(this.properties.getCharset().name());
		filter.setForceRequestEncoding(this.properties.shouldForce(Type.REQUEST));
		filter.setForceResponseEncoding(this.properties.shouldForce(Type.RESPONSE));
		return filter;
	}

}

1:表名当前类是一个配置类,效果类似spring配置文件
2:启用指定类的ConfigurationProperties功能,实际上就是把HttpProperties这个对象添加到容器中,然后就可以通过@Autowired注入进来,注意被注入进来的类应该是被@ConfigurationProperties注解修饰的类.查看HttpProperties类:

@ConfigurationProperties(prefix = "spring.http")		//将当前类的属性值和配置文件进行全部映射(绑定)
public class HttpProperties {
}

@ConfigurationProperties注解:把springboot配置文件里面的内容注入到当前类里的.

3:spring底层@Conditional注解,根据不同的条件,如果满足指定条件整个配置类里面的配置才会生效.这里的意思是判断当前应用是否是web应用
一旦这个配置类生效,这个配置类就会给容器中添加各种组件,这些组件的属性是从对应的properties类中获取的,properties对应的类的属性又是从配置文件中映射过来的.
4:判断当前项目有没有传进来的这个类.(CharacterEncodingFilter:springmvc中解决乱码的拦截器)
5:判断配置文件中是否存在某个配置(不是判断是否存在某个配置文件),就类似判断是否存在server.port这个配置.如果不存在,判断也是成立的.
6:HttpProperties这里类已经和springboot的配置文件形成映射关系了.可以简单理解,当前类通过@EnableConfigurationProperties这个注解启动了HttpProperties类的功能,然后HttpProperties这个类又和springboot的配置文件形成映射关系.那么在启动这个注解的HttpEncodingAutoConfiguration类中的HttpProperties对象的引用就是被配置好的对象.

知道了HttpProperties的属性是从配置文件中映射过来的,那么我们对配置文件进行配置就可以影响全局设置,或者说直接影响HttpProperties的属性,以及需要用到这个类属性的类.
application.yml:

spring:
  http:
    encoding:
      enabled: true
      charset: UTF-8
      force: true

所以说我们在配置文件中能配置的东西全都来源于这些properties类.因为我们配置的内容功能只有一个就是映射到对应的properties类的属性上.

自动配置类@Configuration必须满足@Conditional才能生效,那我们怎么知道哪些配置类生效了呢?
在全局配置文件application.yml/application.properties:我们可以通过启用debug=true属性,让控制台打印自动配置报告,这样我们就可以知道哪些配置类生效了.哪些没有生效都有显示.

对3、4、5的细节讲解:
@Conditional的作用:必须是@Conditional指定的条件成立,才能给容器中添加组件,配置类(Configuration类)里面的所有内容才生效.
@Conditional扩展注解:
@ConditionalOnJava			系统的java版本是否符合条件
@ConditionalOnBean			容器中是否存在指定的bean
@ConditionalOnMissingBean		容器中不存在执行bean
@ConditionalOnExpression		满足SPEL表达式指定
@ConditionalOnClass			系统中有指定类
@ConditionalOnMissingClass		系统中没有指定类
@ConditionalOnSingleCandidate		容器中只有一个指定的bean,或者这个bean是首选的bean
@ConditionalOnProperty			系统中指定的舒适性是否有指定的值
@ConditionalOnResource			系统路径下是否存在指定资源文件
@ConditionalOnWebApplication		当前是web环境
@ConditionalOnNotWebApplication		当前不是web环境
@ConditionalOnJndi			JNDI存在指定项




springboot的流程(精髓):
1.首先我们编写配置文件,我们所编写的配置都对应一个properties类.
2.properties类加载配置文件中对应自身类属性的内容.
3.定义在spring.factories里面的Configuration类(spring配置类),里面所有的bean对象的产生的内容都是通过properties属性得到的.
4.将大量配置类统一规划到Configuration注解下面.在这个Configuration注解里面通过@Import(AutoConfigurationImportSelector.class)导入这个注解加载这个注解下所有对应的Configuration类.

·介绍:

市面上的日志框架:

SLF4J						Log4j2 
jboss-logging(用的太少)				Logback

左边选择一个门面(抽象层)、右边来选一个实现:
日志门面:SLF4J
日志实现:Logback

springboot:底层是spring框架,spring框架默认是JCL,springboot选用SLF4J和logback.主流还是SLF4J(Log4j2)


3.2:统一项目日志框架SLF4j
·如何在系统中使用SLF4j:
以后开发的时候,日志记录方法的调用,不应该来直接调用日志的实现类,而是调用日志抽象层里面的方法.
给系统里面导入slf4j的jar和logback的实现jar
public class HelloWorld{
	public static void mian(String[] args){
		Logger logger = LoggerFactory.getLogger(HelloWorld.class);
		logger.info("HelloWorld");
	}
}

如果你的是log4j这种老的实现类库,那么你必须在slf4j和log4j之间使用adapter(适配器).
每一个日志的实现框架都有自已的配置文件,使用log4j之后,配置文件还是做成日志实现框架的配置文件.

·统一项目框架:
假设我现在开发的项目A,项目A本身使用logback实现,但项目有依赖spring、hibernate、mybatis...然而这些jar包各自又有各自的日志框架,比如spring底层使用commons-logging,hibernate底层使用jboss-logging.那如何统一使用logback呢?
偷天换日:把我们项目原本需要调用的日志jar包给替换掉,原本spring需要调用的Commons-logging里面的类在替换的jar包里面全有,可以正常调用.不过这些类都没有实现,这些类相当于适配器一样,会转换成SLF4J的API.然后SLF4J再去调用logback的实现.

springboot底层也是使用slf4j+logback的方式进行日志记录

实现WebMvcConfigurer接口有什么用?
1.如果你想启用自已的实现类就必须在实现类上加上@Configuration
2.实现这个接口到底有什么用?那你就去看这个接口的抽象方法,比如addViewControllers()这个抽象方法,翻译过来就是增加新的拦截器,实现这个方法之后,你就可以通过传进来的这个参数,调用它addViewController和setViewName方法拦截请求并且重定向到某个页面.再比如addInterceptors方法添加拦截器,所以在这个接口里面可以添加很多的Interceptors也好Controller也好,那么什么是Interceptors(拦截器)呢?
3.那么这和我们之前用Controller控制请求有什么区别呢?唯一的区别就是在这里添加Controller会直接跳转到页面,而如果是@Controller则可以获得浏览器发送的数据以及返回特定的数据,而这里只是一个跳转页面.

//1.WebMvcAutoConfiguration是springMVC的自动配置类

public class DelegatingWebMvcConfiguration extends WebMvcConfigurationSupport {

	private final WebMvcConfigurerComposite configurers = new WebMvcConfigurerComposite();

	//从容器中获取所有的WebMvcConfigurer
	@Autowired(required = false)
	public void setConfigurers(List<WebMvcConfigurer> configurers) {
		if (!CollectionUtils.isEmpty(configurers)) {
			this.configurers.addWebMvcConfigurers(configurers);
		}

	public void addViewControllers(ViewControllerRegistry registry) {

		}
	}

spring如何转发,spring重定向是在前面加上redirect: spring的转发是forward:

转发和重定向:
1.如果不在返回内容里面加上redirect一般都是转发,如果是转发则地址栏中的地址不会发生改变.如果是重定向地址栏中的地址会改变为转发后的地址.
2.如果是转发forward,则转发之前的request和response和转发到的request和response都是同一个对象.而如果是重定向sendRedirect,重定向之前的和重定向之后的request和response不是同一个对象不能共用.
3.转发是在服务器内部进行的,也就是说一个请求来了先到该请求对应的Controller,然后由Controller返回给客户端需要的页面.如果是重定向,一个请求来了还是先找到对应的Controller,但是Controller这回不返回页面而是告诉浏览器让浏览器去访问另一个地址,浏览器接受到这个地址之后再去访问这个地址. 转发:与服务器交互一次 重定向:与服务器交互两次

1.session:session和request、response不同的地方在于session可以跨越多个页面(相当于一种全局性的),存储在服务器上生命周期很长.
2.HttpSession接口:在springboot中通过这个接口就可以获取到session对象,这个session对象是全局的.你可以看这个对象里面有什么抽象方法,比如setAttribute(String K,Object V)添加一对键值对.那在别的页面就可以根据getAttribute(String K)来获取到对应的值.
3.通过request获取session:request.getSession();此时返回的对象就是HttpSession类型的(全局的session).
4.另外如果是在request.setAttribute();只能存在当前页面

    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {

        if(user==null){			//判断用户名是否为null,不为null则跳转到mian.html路径,负责直接返回.
            //未登录,返回登录页面
            request.setAttribute("msg","没有权限请登录");

            能获得当前的request和response

            request.getRequestDispatcher("/login.html").forward(request,response);   //这句代码相当于转发到/login.html请求上,如果此时还有拦截器则还要再走拦截器,然后看条件是否满足在走控制器

            return true;

那么这个拦截器到底有什么作用呢?
1.在此之前我需要重新明确两个概念:Interceptor才是拦截器Controller是控制器.
2.主要是对我们的请求做一个预处理,Interceptor和Controller是共存的(同时也是区分下来的).springMVC在处理请求的时候是按照链式的顺序执行的,假如我现在有一个/abc请求,springMVC会依次去执行每一个拦截器的preHandle方法(因为每一个拦截器都需要实现HandlerInterceptor接口).当我们遇到某个拦截器执行完毕时,如果这个拦截器返回false则后续所有内容都不执行,如果这个拦截器返回true则会继续执行后面的拦截器,当所有拦截器的preHandle方法都执行完毕时,此时执行这个请求真正的Controller方法,然后再执行postHandle、afterCompletion方法.
3.Interceptor比Filter更灵活,但Filter是原生的,基本上能通过Filter做的事情Interceptor都可以做.这点很重要,这是区分拦截器和过滤器的要点.

/**
 * Authentication argument resolver.
 * 对自定义参数进行解析
 * 当传参是Authentication、UserDetail、User对象时就会从ThreadLocal中得到Authentication
 * 并根据当前到底需要哪个对象从Authentication中获得.
 *
 */
@Slf4j
public class AuthenticationArgumentResolver implements HandlerMethodArgumentResolver {

    public AuthenticationArgumentResolver() {
        log.debug("Initializing AuthenticationArgumentResolver");
    }

    /**
     * 用于判定是否需要处理该参数分解,返回true为需要并会调用resolveArgument方法
     * springboot会自动将参数传入controller方法的形参中.springboot在传入每个参数之前都会将这个参数
     * 封装在MethodParameter对象中,然后调用HandlerMethodArgumentResolver的supportsParameter方法
     * 将MethodParameter这个参数传进去,根据方法的返回值决定是否调用resolveArgument方法.
     * 注意该方法匹配之后会对MethodParameter形参做一个内存级别的缓存,后续同样类型的形参就不会调用该方法了.
     * isAssignableFrom()方法是从类继承的角度去判断,instanceof关键字是从实例继承的角度去判断.
     * isAssignableFrom()方法是判断是否为某个类的父类,instanceof关键字是判断是否某个类的子类.
     *
     * @param parameter
     * @return
     */
    @Override
    public boolean supportsParameter(MethodParameter parameter) {
        // 获得到方法参数是哪个类型的
        Class<?> parameterType = parameter.getParameterType();
        return (Authentication.class.isAssignableFrom(parameterType) ||
                UserDetail.class.isAssignableFrom(parameterType) ||
                User.class.isAssignableFrom(parameterType));
    }

    /**
     * 由于不知道当前传进来的参数到底是 Authentication、UserDetail、User这三个类型中的哪个.
     * 首先我们要从ThreadLocal中得到Authentication,
     * 如果当前需要的是Authentication就直接返回Authentication
     * 如果当前需要的是UserDetail就通过Authentication的getDetail方法获得
     * 如果当前需要的是User就通过authentication.getDetail().getUser()来得到.
     * 前端根本不保留Authentication、UserDetail、User.在我们的controller方法里面如果需要这三个参数实际上还是通过
     * SecurityContextHolder来获取的.
     *
     * @param parameter
     * @param mavContainer
     * @param webRequest
     * @param binderFactory
     * @return
     */
    @Override
    @Nullable
    public Object resolveArgument(MethodParameter parameter, @Nullable ModelAndViewContainer mavContainer, NativeWebRequest webRequest, @Nullable WebDataBinderFactory binderFactory) {
        log.debug("Handle AuthenticationArgument");

        Class<?> parameterType = parameter.getParameterType();
        /*
         这个地方很有讲究,前端不可能携带用户信息(也就是User对象),前端只携带admin的token.
         对于需要User对象的controller方法,该controller的入参就是从这里得到.
         对于需要admin权限的操作,需要先经过对应的过滤器,在过滤器中拿到请求头当前的token,然后去数据库查询匹配验证是否有这个admin管理员.
         如果能查询到User对象,就把该User对象放到ThreadLocal中.反正在同一个线程中,别的线程是获取不到这个User对象的,然后在这里通过SecurityContextHolder(ThreadLocal)再得到这个User对象
         可见AdminAuthenticationFilter、AbstractAuthenticationFilter
         另外实际上在全局中就可以通过SecurityContextHolder得到SecurityContext对象,从而得到User对象.如果User对象是空,则代表没有登录.
         */
        Authentication authentication = Optional.ofNullable(SecurityContextHolder.getContext().getAuthentication())
                .orElseThrow(() -> new AuthenticationException("You haven't signed in yet"));

        if (Authentication.class.isAssignableFrom(parameterType)) {
            return authentication;
        } else if (UserDetail.class.isAssignableFrom(parameterType)) {
            return authentication.getDetail();
        } else if (User.class.isAssignableFrom(parameterType)) {
            return authentication.getDetail().getUser();
        }

        // Should never happen...
        throw new UnsupportedOperationException("Unknown parameter type: " + parameterType);
    }

}

// 定制HTTP消息转换器,这里使用自定义ObjectMapper转换Json数据为例.
public void extendMessageConverters(List<HttpMessageConverter<?>> converters) {
    converters.stream()
            .filter(c -> c instanceof MappingJackson2HttpMessageConverter)
            .findFirst().ifPresent(converter -> {
        MappingJackson2HttpMessageConverter mappingJackson2HttpMessageConverter = (MappingJackson2HttpMessageConverter) converter;
        // 创建序列化格式为Json的Builder
        Jackson2ObjectMapperBuilder builder = Jackson2ObjectMapperBuilder.json();
        // 创建JsonComponentModule通过这个对象的addSerializer方法来自定义Json序列化的格式.
        JsonComponentModule module = new JsonComponentModule();
        // 对PageImpl对象的序列化格式使用PageJacksonSerializer这个对象的策略,这个类在下面.
        module.addSerializer(PageImpl.class, new PageJacksonSerializer());
	// 这边是固定写法
        ObjectMapper objectMapper = builder.modules(module).build();
	// 设置自已的ObjectMapper
        mappingJackson2HttpMessageConverter.setObjectMapper(objectMapper);
    });
}

其实这里并不是跨域请求,准确的应该叫做跨域资源共享(CORS)
跨域请求分为简单请求和非简单请求,简单请求不会触发预检请求(当然不发送预检请求并不代表目标网站支持跨域请求).所以预检请求就是非简单请求,因为只有非简单请求才会触发预检请求.
什么叫预检请求?浏览器为了确保安全,在跨域请求的时候,会先发送一个请求,询问目标网站是否支持跨域请求.如果支持才发真正的请求,并且一段时间内的请求就不会发送预检请求了.
那简单请求和非简单请求如何区分?
简单请求:
请求方式:GET、HEAD、POST
请求头只能包含:Accept、Accept-Language、Content-Language、Content-Type、DPR、Downlink、Save-Data、Viewport-Width、Width
其中Content-Type只能包含:text/plain、multipart/form-data、application/x-www-form-urlencoded
非简单请求:
请求方式:PUT、DELETE、CONNECT、OPTIONS、TRACE、PATCH
请求头只能包含:Accept、Accept-Language、Content-Language、Content-Type、DPR、Downlink、Save-Data、Viewport-Width、Width
其中Content-Type的值不属于不属于不属于以下的值:text/plain、multipart/form-data、application/x-www-form-urlencoded

public class CorsFilter extends GenericFilterBean {			//这个过滤器继承自GenericFilterBean过滤器,GenericFilterBean过滤器是spring提供的通用过滤器.

    private final static String ALLOW_HEADERS = StringUtils.joinWith(",", HttpHeaders.CONTENT_TYPE, ADMIN_TOKEN_HEADER_NAME, API_ACCESS_KEY_HEADER_NAME);

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) throws IOException, ServletException {
        HttpServletRequest httpServletRequest = (HttpServletRequest) request;				//强转每什么好说的
        HttpServletResponse httpServletResponse = (HttpServletResponse) response;

        /*
        Set customized header 设置响应头
        ACCESS_CONTROL_ALLOW_ORIGIN:该值只有两种情况要么是*代表任意网站,要么是具体的网站域名.
	这里就是具体的网站域名,该域名是直接从请求参数中得到的,意思是运行请求的网站获得本站资源.
	现在可以看到该Filter只过滤/api/下面的请求,这是因为对于附带身份信息的请求,ACCESS_CONTROL_ALLOW_ORIGIN不能设置为* 最好就设置为当前请求的请求地址
        ACCESS_CONTROL_ALLOW_HEADERS:允许请求中携带的字段
        ACCESS_CONTROL_ALLOW_METHODS:允许请求通过哪些方式请求
        ACCESS_CONTROL_ALLOW_CREDENTIALS:是否允许浏览器读取读取response的内容.这一步决定了本站是否支持跨域请求,如果为false则表示不支持(同时这个响应信息就告知了浏览器不支持跨域请求)
        ACCESS_CONTROL_MAX_AGE:浏览器请求本站时会分两次请求,第一次称为预检请求,第二次才是真实的请求.
        该参数对应的值是秒数,代表发送一次预检请求后间隔多少秒后的请求才会发送预检请求.如果设置为0代表每次请求都需要发送
        预检请求
	Access-Control-Expose-Headers:当前浏览器能读取哪些响应头信息
         */
        httpServletResponse.setHeader(HttpHeaders.ACCESS_CONTROL_ALLOW_ORIGIN, httpServletRequest.getHeader(HttpHeaders.ORIGIN));
        httpServletResponse.setHeader(HttpHeaders.ACCESS_CONTROL_ALLOW_HEADERS, ALLOW_HEADERS);
        httpServletResponse.setHeader(HttpHeaders.ACCESS_CONTROL_ALLOW_METHODS, "GET, POST, PUT, DELETE, OPTIONS");
        httpServletResponse.setHeader(HttpHeaders.ACCESS_CONTROL_ALLOW_CREDENTIALS, "true");			
        httpServletResponse.setHeader(HttpHeaders.ACCESS_CONTROL_MAX_AGE, "3600");

	/*
        判断当前请求是否是预检请求,如果是则终止过滤链,否则继续执行过滤链.
	因为预检请求不存在真正的请求信息,只是浏览器发送一个信息来判断我们当前网站是否支持跨域请求.
	那么浏览器怎么才能知道我们当前网站是否支持跨域请求呢?实际上就靠上面response设置的响应信息.
         */
        if (!CorsUtils.isPreFlightRequest(httpServletRequest)) {			//CorsUtils是springboot的工具类
            chain.doFilter(httpServletRequest, httpServletResponse);
        }
    }
}

     *

     */

    public boolean supports(MethodParameter returnType, Class<? extends HttpMessageConverter<?>> converterType) {
判断哪些响应需要拦截,如果是AbstractJackson的消息转换器就拦截,也就是如果返回结果需要被Json序列化的话就通过,就拦截
        return AbstractJackson2HttpMessageConverter.class.isAssignableFrom(converterType);
    }

    /**
     * 自定义修改响应头,这里主要是修改响应码
     *
     * @param body          body就是响应对象response中的响应体
     * @param returnType
     * @param contentType
     * @param converterType
     * @param request
     * @param response
     * @return
     */
    @Override
    @NonNull
    public final Object beforeBodyWrite(@Nullable Object body,
                                        MethodParameter returnType,
                                        MediaType contentType,
                                        Class<? extends HttpMessageConverter<?>> converterType,
                                        ServerHttpRequest request,
                                        ServerHttpResponse response) {
        // 根据这个body获得MappingJacksonValue,或者是将body封装在MappingJacksonValue中
        MappingJacksonValue container = getOrCreateContainer(body);
        // The contain body will never be null
        beforeBodyWriteInternal(container, contentType, returnType, request, response);
        return container;
    }

    /**
     * Wrap the body in a {@link MappingJacksonValue} value container (for providing
     * additional serialization instructions) or simply cast it if already wrapped.
     */
    private MappingJacksonValue getOrCreateContainer(Object body) {
        // JSONP:Js跨域请求中的一种解决方案
        return (body instanceof MappingJacksonValue ? (MappingJacksonValue) body : new MappingJacksonValue(body));
    }

    private void beforeBodyWriteInternal(MappingJacksonValue bodyContainer,
                                         MediaType contentType,
                                         MethodParameter returnType,
                                         ServerHttpRequest request,
                                         ServerHttpResponse response) {
        // Get return body 获得到响应体,body.这个body就是被封装在MappingJacksonValue中的对象,也就是上面创建的对象
        Object returnBody = bodyContainer.getValue();
        /*
         有的时候在controller中会返回自定义的BaseResponse,如果这里的returnBody是BaseResponse就不使用通用的返回结果.
         有可能是别的地方BaseResponse结果,也有可能是产生异常的BaseResponse.
         */
        if (returnBody instanceof BaseResponse) {
            // If the return body is instance of BaseResponse
            BaseResponse<?> baseResponse = (BaseResponse) returnBody;
            // 根据响应码转为其对应的HttpStatus,并将响应码设置到response中
            response.setStatusCode(HttpStatus.resolve(baseResponse.getStatus()));
            return;
        }
        /*
        Wrap the return body 如果不是BaseResponse就创建一个响应码为200的实例
        并且将响应体对象returnBody封装在BaseResponse中.
         */
        BaseResponse<?> baseResponse = BaseResponse.ok(returnBody);
        bodyContainer.setValue(baseResponse);
        response.setStatusCode(HttpStatus.valueOf(baseResponse.getStatus()));
    }

}


/**
 * Custom serializer for Page object.
 * 自定义Page对象的序列化格式
 * 需要继承JsonSerializer并指定泛型为看具体的序列化的对象.
 *
 */
public class PageJacksonSerializer extends JsonSerializer<Page> {

    /**
     * 通过JsonGenerator对象的方法来序列化对象
     * 以writeStartObject开始以writeEndObject结束.
     * @param page
     * @param generator
     * @param serializers
     * @throws IOException
     */
    @Override
    public void serialize(Page page, JsonGenerator generator, SerializerProvider serializers) throws IOException {
        generator.writeStartObject();

        generator.writeObjectField("content", page.getContent());
        generator.writeNumberField("pages", page.getTotalPages());
        generator.writeNumberField("total", page.getTotalElements());
        generator.writeNumberField("page", page.getNumber());
        generator.writeNumberField("rpp", page.getSize());
        generator.writeBooleanField("hasNext", page.hasNext());
        generator.writeBooleanField("hasPrevious", page.hasPrevious());
        generator.writeBooleanField("isFirst", page.isFirst());
        generator.writeBooleanField("isLast", page.isLast());
        generator.writeBooleanField("isEmpty", page.isEmpty());
        generator.writeBooleanField("hasContent", page.hasContent());

        // Handle comment page
        if (page instanceof CommentPage) {
            CommentPage commentPage = (CommentPage) page;
            generator.writeNumberField("commentCount", commentPage.getCommentCount());
        }

        generator.writeEndObject();
    }
}

比如现在我想将String转为枚举类型,但是现在有很多枚举类型,比如FLOWER、CAR、ANIMAL等等很多枚举类型,此时就需要通过ConverterFactory接口.我们在使用的时候只需要调用getConverter方法传入目标枚举类型,则返回的Converter就具有String与目标枚举类型的转换.

//添加转换工厂
public void addFormatters(FormatterRegistry registry) {
    registry.addConverterFactory(new StringToEnumConverterFactory());
}


/**
 * 下面的就别看了,看上面的......
 * 这个工厂实现了ConverterFactory接口并指定了泛型
 * 该工厂可以以创建Converter对象,通过调用Converter对象的convert方法将String转为枚举类型.
 * 但凡controller需要注入的类型是Enum都会调用Converter接口的convert方法.
 * 只有当controller确确实实需要Enum类型的时候才会调用convert方法.
 *
 */
@Component
public class StringToEnumConverterFactory implements ConverterFactory<String, Enum> {			//第一个泛型是浏览器传过来的类型,第二个泛型是转出去的类型.

    @SuppressWarnings("unchecked")
    @Override
    public <T extends Enum> Converter<String, T> getConverter(Class<T> targetType) {
        return new StringToEnumConverter(targetType);
    }

    private static class StringToEnumConverter<T extends Enum> implements Converter<String, T> {

        private Class<T> enumType;

        private StringToEnumConverter(Class<T> enumType) {
            this.enumType = enumType;
        }

        @SuppressWarnings("unchecked")
        @Override
        public T convert(String source) {							//转换的时候会回调这个方法.
            return (T) Enum.valueOf(this.enumType, source.toUpperCase());
        }
    }
}

HttpMessageConverter、HandlerMethodArgumentResolver、FormatterRegistry都是转换器,它们的区别在哪里?
HttpMessageConverter:这种转换器向百搭类型,提供对controller参数的Json形式的序列化和反序列化.但是一般也只适用于前端传String后端接收的是对象类型,而且要有对应关系.
AuthenticationArgumentResolver:这种转换器对controller注入的对象就不是前端传过来的,而是后端自已根据某种规则传给controller.有点后端传给后端的意思.
FormatterRegistry:这种就比较灵活了,有的时候假如前端传一个Sting然后controller里面接收的是Enum此时就可以用这个.

1.5:更好的的错误页面的处理方式(我们手动覆盖/error请求,此时如果某个异常没有被捕获就会来到该控制层,一切的一切都是基于这次异常没有被捕获)
@Controller
@RequestMapping("${server.error.path:${error.path:/error}}")
public class Test extends AbstractErrorController {			//一定要继承AbstractErrorController!!!!!

    ErrorProperties errorProperties;					//这个Bean会自动注入

    public Test(ErrorAttributes errorAttributes,
                ServerProperties serverProperties) {			
        super(errorAttributes);
        errorProperties = serverProperties.getError();			//拿到错误信息
    }

    @GetMapping
    public String handleError(HttpServletRequest request) {			//该request包含了本次异常的信息,详情在下面
        // 下面是一些常用方法的介绍
        Object attribute = request.getAttribute("javax.servlet.error.exception");       //获得到当前的异常

        Throwable throwable = (Throwable) attribute;
        if (throwable instanceof NestedServletException) {
            //得到产生当前异常的根本异常,这个方法是NestedServletException类的方法.所以要判断当前异常是不是NestedServletException
            Throwable rootCause = ((NestedServletException) throwable).getRootCause();
            // 判断这个异常是不是项目自定义的异常
            if (rootCause instanceof HaloException) {
                HaloException haloException = (HaloException) rootCause;
                request.setAttribute("javax.servlet.error.status_code", haloException.getStatus().value());     //设置当前异常的状态码
                request.setAttribute("javax.servlet.error.exception", rootCause);                               //设置当前异常到Attribute中
                request.setAttribute("javax.servlet.error.message", haloException.getMessage());                //设置当前异常的提示信息
            }
        }
        /*
         调用下面的isIncludeStackTrace方法判断当前request有没有写到异常的堆栈信息,
         然后再调用父类的getErrorAttributes方法 args0:当前的request args1:args1的request是否携带堆栈信息(需要布尔)
         最后调用Collections的unmodifiableMap方法将getErrorAttributes返回的Map转为不可变集合.
         注意getErrorAttributes返回的Map和之前一样存放那五个信息,除非是当前request如果携带堆栈异常信息才可能会多一项.
         最终把这个map设置会给request.
         */
        Map<String, Object> errorDetail = Collections.unmodifiableMap(getErrorAttributes(request, isIncludeStackTrace(request)));
        request.setAttribute("error", errorDetail);
        // 不管怎么说该方法要返回最终渲染给浏览器的页面的地址
        return "";
    }

    /**
     * 判断当前异常有没有携带堆栈信息.
     */
    private boolean isIncludeStackTrace(HttpServletRequest request) {
        ErrorProperties.IncludeStacktrace include = errorProperties.getIncludeStacktrace();
        if (include == ErrorProperties.IncludeStacktrace.ALWAYS) {
            return true;
        }
        if (include == ErrorProperties.IncludeStacktrace.ON_TRACE_PARAM) {
            return getTraceParameter(request);
        }
        return false;
    }

    /**
     * 和之前一样,该方法要填入错误页面的资源路径.注意该路径会被资源映射规则和视图解析器影响.
     * 什么意思呢?意思就是遇到错误没有任何控制器拦截的话,spring就会调用getErrorPath方法(这个方法重写父类方法).getErrorPath方法的返回值就是我们(spring)要访问的异常路径
     * 而getErrorPath方法默认就是返回ErrorProperties中Path的值(Path的值默认就是占位符${error.path:/error}也就是error)
     * 所以如果这里返回/error2 那么controller是/error2的就会作为错误方法的控制层.然后spring会把本次异常的信息,放到request中,然后我们的controller方法就可以从HttpServletRequest中拿到异常信息(具体信息也就是下面的这些内容)
     * 之所以要继承AbstractErrorController我怀疑就是因为spring会默认调用AbstractErrorController实现类的getErrorPath方法.然而这里继承AbstractErrorController好处是可以直接调用模板方法去做一些事情.
     * 比如调用getErrorAttributes方法就可以将DefaultErrorAttributes封装的5个信息转为MAP
     * @return
     */
    @Override
    public String getErrorPath() {
	return this.errorProperties.getPath();			//这个地方有点拿捏不准,先用上面这个.上面这个默认就是返回/error.当然可以通过${error.path:/error}指定(这也就是为什么遇到错误了会去/error 原因就是调用的该方法)
        return "你的错误页面路径"; 
    }

在spring中比较特殊的是,如果当前controller已经被@ControllerAdvice捕获了,那么当我们在controller层执行某些需要捕获的异常的方法时,不应该对该方法捕获
而是应该抛出,否则spring将无法捕获到该异常.且不能及时响应错误信息.(相当于spring做全局的异常捕获)
另外异常的匹配遵循这样一个原则,如果A异常包装了B异常,那么即使是B异常被抛出,spring默认还是会拦截A异常(也就是获得包装异常),然后你可以通过调用A异常的getCause方法得到真正实际匹配的异常(这也就能解释springMVC中关于参数校验为什么会抛出MethodArgumentNotValidException异常了)
另外对于@ControllerAdvice中被@ExceptionHandler注解修饰的方法,spring可以直接传入以下参数:HandlerMethod(提供对方法参数、方法返回值、方法注释等便捷访问)、WebRequest、NativeWebRequest、ServletRequest、ServletResponse、HttpSession
对于@ExceptionHandler方法的返回值可以是:@ResponseBody(响应JSON)、ResponseEntity<B>(效果类似@ResponseBody)、String(逻辑视图)、View(视图)、Map、Model、@ModelAttribute

如果说@Controller是拦截请求的标识,那么@ControllerAdvice就是拦截异常的标识.如果是@RestControllerAdvice就代表基于rest的控制器只会返回Json.这种思想要灵活.
@RequestMapping(value)里面是具体拦截的地址,@ExceptionHandler(class) 该注解需要传入一个异常的class对象.你在程序中任意处抛出的异常都会想办法调用和@ExceptionHandler注解匹配的异常的方法.

以后就要明确一个概念!!!如果我们在类似@RequestMapping的方法有形参,那么这些形参都是通过springboot传给我们的(之前的很多例子也都可以说明)
所以handlerException方法的形参exception就是springboot传给这个方法的,这个对象就是我们抛出异常时创建的对象.

关于注解@ResponseBody的解释:
当没有这个注解是,如果方法返回的是String就代表springboot会根据返回值去寻找这个页面.如果有@ResponseBody注解,springboot就会在页面上显示这个字符串内容.
如果一个控制方法的返回值是集合类型(例如List、Map)并且这个方法被@ResponseBody修饰,此时springboot就会根据这个集合返回json数据到页面.

在此联想一下之前的Map、Model、ModelMap,不要与返回值混淆.那么页面是如何显示Model这些类的数据的呢?
在之前的使用中我们从来没有单独返回过这些对象,这些对象全是在控制方法里面作为形参的.所以根据上面说的,只要是形参 那么调用这个方法时,里面的形参就是springboot传进来的.这也就代表页面上的Moduel和传进来的Module是同一个对象,所以这也不难解释为什么页面可以根据Module获取到服务器发送的数据了.(另外再次强调,只有转发的时候才使用相同的request,重定向就不可以了)

方法一存在的问题:

spring定义异常大全:
NoHandlerFoundException:首先根据请求Url查找有没有对应的控制器,若没有则会抛该异常,也就是大家非常熟悉的404异常.
HttpRequestMethodNotSupportedException：若匹配到了（匹配结果是一个列表,不同的是http方法不同,如：Get、Post等）,则尝试将请求的http方法与列表的控制器做匹配,若没有对应http方法的控制器,则抛该异常;
HttpMediaTypeNotSupportedException：然后再对请求头与控制器支持的做比较,比如content-type请求头,若控制器的参数签名包含注解@RequestBody,但是请求的content-type请求头的值没有包含application/json,那么会抛该异常（当然,不止这种情况会抛这个异常）；
MissingPathVariableException：未检测到路径参数.比如url为：/licence/{licenceId},参数签名包含@PathVariable("licenceId"),当请求的url为/licence,在没有明确定义url为/licence的情况下,会被判定为：缺少路径参数；
MissingServletRequestParameterException：缺少请求参数.比如定义了参数@RequestParam("licenceId") String licenceId,但发起请求时,未携带该参数,则会抛该异常；
TypeMismatchException: 参数类型匹配失败.比如：接收参数为Long型,但传入的值确是一个字符串,那么将会出现类型转换失败的情况,这时会抛该异常；
HttpMessageNotReadableException：与上面的HttpMediaTypeNotSupportedException举的例子完全相反,即请求头携带了"content-type: application/json;charset=UTF-8",但接收参数却没有添加注解@RequestBody,或者请求体携带的 json 串反序列化成 pojo 的过程中失败了,也会抛该异常；
HttpMessageNotWritableException：返回的 pojo 在序列化成 json 过程失败了,那么抛该异常；
BindException:参数校验异常

4.10.5:解决跨域请求的一种方案

@ControllerAdvice("run.halo.app.controller")			//在该路径下的所有controller在真正响应前都会走这个控制器
public class CommonResultControllerAdvice implements ResponseBodyAdvice<Object> {
    /**
     * 拦截条件：拦截Json数据
     *
     * @param returnType    返回类型
     * @param converterType 转换器类型
     */
    @Override
    public boolean supports(MethodParameter returnType, Class<? extends HttpMessageConverter<?>> converterType) {
        // 判断哪些响应需要拦截,如果是AbstractJackson的消息转换器就拦截.
        // 拦截转换器 AbstractJackson2HttpMessageConverter子类的Controller方法
        return AbstractJackson2HttpMessageConverter.class.isAssignableFrom(converterType);
    }

    /**
     * 自定义修改响应头,这里主要是修改响应码
     *
     * @param body          body就是响应对象response中的响应体
     * @param returnType
     * @param contentType
     * @param converterType
     * @param request
     * @param response
     * @return
     */
    @Override
    @NonNull
    public final Object beforeBodyWrite(@Nullable Object body,
                                        MethodParameter returnType,
                                        MediaType contentType,
                                        Class<? extends HttpMessageConverter<?>> converterType,
                                        ServerHttpRequest request,
                                        ServerHttpResponse response) {
        // 根据这个body获得MappingJacksonValue,或者是将body封装在MappingJacksonValue中
        MappingJacksonValue container = getOrCreateContainer(body);
        // The contain body will never be null
        beforeBodyWriteInternal(container, contentType, returnType, request, response);
        return container;
    }

    /**
     * Wrap the body in a {@link MappingJacksonValue} value container (for providing
     * additional serialization instructions) or simply cast it if already wrapped.
     */
    private MappingJacksonValue getOrCreateContainer(Object body) {
        // JSONP:Js跨域请求中的一种解决方案
        return (body instanceof MappingJacksonValue ? (MappingJacksonValue) body : new MappingJacksonValue(body));
    }

    private void beforeBodyWriteInternal(MappingJacksonValue bodyContainer,
                                         MediaType contentType,
                                         MethodParameter returnType,
                                         ServerHttpRequest request,
                                         ServerHttpResponse response) {
        // Get return body 获得到响应体,body.这个body就是被封装在MappingJacksonValue中的对象,也就是上面创建的对象
        Object returnBody = bodyContainer.getValue();
        /*
         有的时候在controller中会返回自定义的BaseResponse,如果这里的returnBody是BaseResponse就不使用通用的返回结果.
         有可能是别的地方BaseResponse结果,也有可能是产生异常的BaseResponse.
         */
        if (returnBody instanceof BaseResponse) {
            // If the return body is instance of BaseResponse
            BaseResponse<?> baseResponse = (BaseResponse) returnBody;
            // 根据响应码转为其对应的HttpStatus,并将响应码设置到response中
            response.setStatusCode(HttpStatus.resolve(baseResponse.getStatus()));
            return;
        }
        /*
        Wrap the return body 如果不是BaseResponse就创建一个响应码为200的实例
        并且将响应体对象returnBody封装在BaseResponse中.
         */
        BaseResponse<?> baseResponse = BaseResponse.ok(returnBody);
        bodyContainer.setValue(baseResponse);
        response.setStatusCode(HttpStatus.valueOf(baseResponse.getStatus()));
    }

}

2.Filter

·Filter是一个接口,充当过滤器的作用,它会抢在servlet之前执行.

·Filtr的作用:当服务器启动时就会创建Filter执行Filter的init方法.

·Filter的应用场景:自动登录、统一设置编码格式、访问权限控制、敏感字符过滤等

·Filter的执行顺序在InterceptorHandler之前.

一个实现类:

@WebFilter("/servlet")
public class MyFilter implements Filter {
    public void destroy() {
    }

    public void doFilter(ServletRequest req, ServletResponse resp, FilterChain chain) throws ServletException, IOException {  
        chain.doFilter(req, resp);
    }

    public void init(FilterConfig config) throws ServletException {
    }

}

doFilter这个方法非常关键,Filter过滤器在使用的时候采用的是责任链设计模式,如果不人为修改执行顺序,责任链的执行是无序的.我们通过传入的这个FilterChain接口的doFilter方法去执行下一个Filter.
如果当前doFilter方法没有通过FilterChain去执行doFiler方法,那么该请求就终止了(不会去执行接下来的Filter和Servlet)(注意:Tomcat给的责任链一定是和当前请求相匹配的过滤器)

·FilterChain:
doFilter(ServletRequest,ServletResponse) return void;	执行下一个过滤器

4.12:更多的注解

@RequestBody:该注解用于修饰形参,如果前端传进来一个json数据.那么被该注解修饰的形参就会得到这个数据并封装成一个集合.
如果是表单请求(POST请求),则表单的Name只要对应后端controller层对象的属性名对应即可.现在我们不用表单请求了,请求的时候使用json请求(POST)
所以对于JSON请求(可能是POST、PUT、DELETE(记住不可能是GET,因为GET没有请求体)),前端应该把请求参数放到请求体当中(而且是以JSON的形式序列化),再联合之前微信支付讲的把请求信息放到请求体当中(好好想想)
现在前端传过来的是JSON,那么对于controller层接收还是一个对象(所以这一步将前端传过来的JSON转为对象就需要使用HttpMessageConverter).对于controller层接收的对象而言,需要标注@RequestBody才能正常使用(现在前端把请求参数放到请求体中(也就是body))

所以说在前后端分离的情况下,为什么需要指定Accept的值为application/json、application/xml这些值对于用户是无所谓的.但是对于开发者而言很有必要
JSON和XML都可以用来表示一个对象(或者叫一个对象的序列化形式),那么前端在传数据给后端的时候,由于JSON和XML是两种不同的表现形式,所以在序列化为对象的时候处理方式肯定也不同.
所以本来你前端是以JSON方式传数据的,然后你却在Accept指定序列化形式是application/xml这必然报错.所以Accept的指定就能告诉后端当前这段请求体(请求报文)是以什么方式序列化的,好让后端具体操作.
同样后端在返回对象的时候,也应该告知序列化形式.好在@ResponseBody默认指定的Accept就是application/json

@RestController:该注解用于修饰类/接口/枚举.该注解由@Controller和@ResponseBody注解组合而成,被该注解标注的类下面的所有方法的返回值都应该是一个json串.

@ResponseBody:修饰在控制方法上,会将返回的对象转换为Json数据.返回的数据放到响应体中(响应体就是body),将后端返回的对象转为JSON传回给前端也需要使用HttpMessageConverter

@RequestParam:请求方法入参(属性)名的对应(就是前端表单提交时和后端参数对应,详情见springMVC)
其实未必是表单(现在要干掉表单),对于任何请求方式都有(GET、POST、DELETE、PUT).有的时候前端只想传一个简单的参数,不需要使用对象来交互.如果是使用对象那还是上面说的通过@RequestBody来完成.
比如现在前端只想传给后端一个int值(或者String),那么前端调后端的controller的时候,指定params(比如为name),那么后端再接受的时候(类型对应)@RequestParam("name")String name 将要对应上.说白了也就是和之前对应上.
因为GET的请求参数相当于POST的请求体,但是请求参数是K-V形式的.假设现在有个GET请求 localhost:8888/test?name=233 像这种想拿到name的value也需要@RequestParam("name") 用这个和请求参数对应上
如果请求URL中的参数太多,貌似可以直接用MAP接收.

所以记住!!!POST请求用@RequestBody(注解修饰的入参要和前端JSON请求格式一致)		GET请求用@RequestParam(和前端的URL请求参数对应)

@RequestHeader:该注解和@RequestParam用法类似,这里是获取请求头中的参数 如果@RequestHeader修饰的是Map、MultiValueMap、HttpHeaders参数并且没有带上value值则Spring会将请求头中的所有内容填充满Map(其实@RequestParam也是这样的,如果没有指定请求参数则会将请求参数填充满Map) 另外内置的解析器可以将用逗号,分割的字符串(请求参数)转为String[]或者List<String>

@ResponseStatus:这个注解可以标注在一切Controller上面(包括AdviceController),如果标注了该注解Spring将优先返回该注解指定的响应码到浏览器

@CookieValue:用于访问cookie

@RequestPart:该注解有点类似@RequestBody,在文件上传时如果想读取文件的元数据并且希望元数据反序列化为对象就可以使用该注解

4.13:HttpMessageConverter

在4.12的时候说到过,可以通过@RequestBody让前端传到控制方法里面的Json反序列化为对象,通过@ResponseBody方法让后端返回的对象序列化成Json.
这其实是通过HttpMessageConverter来实现的.也就是通过HttpMessageConverter来完成序列化和反序列化.

再次强调一遍,HttpMessage之间通讯全是String,web说白了就是String.但是我们在用springboot的时候,前端传过来的信息到我们controller方法里面就变成对象了.原始的请求头就是一堆字符串它是怎么变成对象的呢?实际上就是通过HttpMessageConverter将String转为对象的.同理我们在controller方法里面直接返回一个对象,怎么到前端哪里就成了字符串了呢?同样也是通过HttpMessageConverter将对象转为String的.(注意返回给浏览器的页面是需要经过视图解析器解析的)实际上这就是一个序列化和反序列化的过程.所以不要觉得这个东西是天经地义的就是这么来的,其实背后都是springboot帮你干的.它就是通过Http消息转换器来实现的.

public class MyMessageConverter extends AbstractHttpMessageConverter<User> {

        super(new MediaType("application", "xxx-junlin", Charset.forName("UTF-8")));
    }

    /*
    当传入对象或输出Json数据时,会调用该方法.传入当前需要被序列化或者被反序列化的对象,根据当前方法的返回值决定
    决定是否序列化或反序列化当前对象.
    */
    @Override
    protected boolean supports(Class<?> clazz) {
        return clazz.isAssignableFrom(User.class);
    }

    /*
    传入Json数据,将Json转为对象并返回出去.
     */
    @Override
    protected User readInternal(Class<? extends User> clazz, HttpInputMessage inputMessage) throws IOException, HttpMessageNotReadableException {
        String string = StreamUtils.copyToString(inputMessage.getBody(), Charset.forName("UTF-8"));
        return null;
    }

    /*
    传入对象,将Json数据输出到response中.(输出到响应体中)
     */
    @Override
    protected void writeInternal(User user, HttpOutputMessage outputMessage) throws IOException, HttpMessageNotWritableException {
        String out = user.toString();
        outputMessage.getBody().write(out.getBytes());
    }
}

很多时候我们要分清,有时我们的程序是作为服务端接收请求并响应内容的.有时我们的程序又是客户端,发送请求接收响应内容.
响应头和响应信息不是同一个东西!!!!!响应信息叫body 响应头是head,以前的概念一直没搞清.
如果没有这些东西就要使用原始的ServerSocket,来操作很不方便.这个地方必须要熟悉 请求协议和响应协议否则没法玩.
在使用RestTemplate请求的时候,请求行的请求方式见方法名知意,Get方式请求行的请求参数是通过方法的Map指定的,Post方式的请求体的请求参数也是通过方法的Map指定的.
注意RestTemplate的作用是请求别的地址,所以使用RestTemplate是当前程序作为客户端请求的URL作为服务端,而且接下来这些所有的设置不要和之前的内容搞混了,都只是单独针对RestTemplate的设置.

5.1:docker基本概念

·什么是docker?
实际上就是将一个软件编译成一个镜像,然后去运行这个镜像.比如我现在有两台计算机,我在其中一台上安装了mysql,进行了一大堆配置,现在我要在另外一台计算机上也安装mysql,那么也要进行一大堆配置.
docker的作用就是,把我在第一台计算机上的mysql编译成镜像.然后可以直接去别的计算机上运行这个镜像(原本的一些配置也全部生效).而且docker的性能开销很小,运行很快.

docker主机:用于执行docker进行的主机,相当于游戏主机一样,有很多游戏
docker客户端:用于连接docker主机的客户端,通过这个客户端就可以执行主机上的应用了(容器).
docker仓库:之前说过docker需要镜像文件,docker仓库就是用来提供镜像文件的.https://hub.docker.com提供了很多镜像文件.
docker镜像:安装在docker主机上的镜像文件(就相当于软件本体)
docker容器:docker主机运行镜像文件,运行出来的程序.(相当于双击exe运行出来的程序)

6.3:开启基于注解的事务
@EnableTransactionManagement
@Transactional标注一个方法为事务方法

记住事务是针对方法而言的,之前学spring的时候是通过代理的方式来实现的.也就是我真正的业务逻辑方法不需要考虑rollback和commit.事务的相关内容是代理方法去做的, 流程一般是 开启事务-->调用真实方法->commit或rollback.
在springboot中可以在方法上标注@Transactional这个注解表示该方法被代理了,会自动使用事务.

@Transactional(rollbackFor = RollbackException.class)		//这个RollBackException可以是自已定义的任意异常
如果某个事务方法被该注解标记并且有rollbackFor这一项代表,当这个事务方法(真实事务方法)在执行的过程中出现了后面指定的这个异常时才会去回滚事务.(所以这个异常应该是我们在方法里手动抛出)


6.4:自定义异常码
在我们操作任意数据库的时候,当数据库报错时一般会产生异常码(比如在mysql输入错误指令时会产生1064这个异常码,再比如重复插入一个主键相同的值时和会产生一个异常码).spring在org.springframework.jdbc.support.xml.sql-error-codes.xml这个文件里定义了很多数据库的异常码,当数据库报这些异常码时spring就可以立马反映过来.

现在的问题是如果数据库有一层代理,这层代理在运行的时候会产生一个新异常这个新异常spring是捕获不到的,那么我们就需要自已去定义这个异常.我们需要在spring配置文件中(不是springboot配置文件!)

<beans>
	<bean id="" class="">				//这个地方直接拷贝spring的配置文件
		<property name="customTranslations">
		<bean class="org.springframe.jdbc.support.CustomSQLErrorCodesTranslation">
			<property name="errorCodes" value="23001,23505">
			<proty name="exceptionClass" value="这里写你自已定义的异常的全限定名">
		</bean>
	</bean>
</beans>

注意我们自已定义的这个异常必须继承spring定义的异常,也就是说这个异常必须还是在spring定义的异常范围内.此时如果产生23001或23505就会来到我们自已定义的这个异常类中.实际上这里应该是将原本的spring定义的异常给覆盖了

6.7:使用SqlSession
当使用了Spring之后,之前的很多规则都发生改变了.之前说过SqlSession的生命周期应该是针对每一个线程(一个HTTP请求回话) 因为SqlSession不是线程安全的,所以不能被共享 而且SqlSession用完后需要我们手动地关闭.
但是在Spring中,我们的Bean可以被注入一个线程安全的SqlSession(也就是我们依赖的SqlSession是线程安全的)而且使用的过程中不需要我们手动地关闭SqlSession.
在Spring中SqlSessionTemplate是SqlSession的线程安全的实现,应该要保证全局的SqlSession的实现是SqlSessionTemplate而不是Mybatis默认的DefaultSqlSession,否则会引起数据一致性的问题

8.自定义starters&利用自动配置

·starters就是之前所说的场景启动器.在我们实际项目的开发过程中,springboot不可能有所有的场景启动器.所以我们自已编写一个场景启动器,(然后别人一导入这个场景启动器就直接把所有的jar包导入).
实际上就像是把一个pom文件压缩成一个场景启动器一样.

这部分的内容可能比较抽象.

//1.创建一个maven项目(该项目名需要满足 [模块名]-spring-boot-starter).在pom文件里引入项目所依赖的jar包以及自已写的自动配置模块.也就是引入下面的spring项目的地址

//2.创建一个spring项目.该项目就是自动配置的模块,这里仅仅以一个自动配置模块来演示

1.首先创建要和springboot配置文件绑定的类:

@ConfigurationProperties(prefix = "hello")		//这个就是可配项
public class HelloProperties {
    private String pre;
    private String end;

    public String getPre() {
        return pre;
    }

    public void setPre(String pre) {
        this.pre = pre;
    }

    public String getEnd() {
        return end;
    }

    public void setEnd(String end) {
        this.end = end;
    }
}

2.创建我们希望别人用的类(也就是希望springboot在别的项目被注入的类)

public class HelloService {

    HelloProperties helloProperties;

    public String sayHello(String name) {						
        return helloProperties.getPre() + name + helloProperties.getEnd();
    }

    void setHelloProperties(HelloProperties helloProperties) {
        this.helloProperties = helloProperties;
    }
}

·到时候别人就把这个类注入到项目里面,然后调用这个类的sayHello方法.注意这个类一定要提供HelloProperties对象的setter方法,因为HelloProperties这个对象不是通过@Autowried来获取的.
并且HelloService也不是直接通过@Component添加到容器中的,所以我们需要编写一个@Configuration类,把HelloService对象添加到容器中.

3.创建自动配置类
@Configuration
@EnableConfigurationProperties(HelloProperties.class)
public class HelloServiceAutoConfiguration {
    @Autowired
    HelloProperties helloProperties;

    @Bean
    public HelloService helloService() {
        HelloService service = new HelloService();
        service.setHelloProperties(helloProperties);
        return service;
    }
}

@EnableConfigurationProperties:这个注解的作用可以简单理解为把后面传入的class文件添加到容器中.这样就可以通过@Autowired注入了.

4.把自动配置类添加到META-INF文件夹下的spring.factories文件中.填入的格式可以参照springboot官方的spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
xxx这里填入HelloServiceAutoConfiguration类的全路径名(自动配置类的全路径名)

·总结意义:
实际上现在HelloService类是否能被添加到IOC容器中,完全取决于别的开发者是否启用了自动配置功能,如果启用则spring.factories里EnableAutoConfiguration所修饰的类全部同时生效.否则都不生效,所以自动配置类是否生效取决于用户是否开启自动配置.
