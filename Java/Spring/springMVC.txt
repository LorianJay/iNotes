2.3@RequestMapping注解详解:
@RequestMapping()
path=""			//拦截的请求
value=""
method = 	//指定请求方式.		
填入RequestMethod的枚举类型.当请求方式是xxx时才会拦截.常见的请求方式:GET、POST、DELTE、PUT.
parames={}	//填入一个String数组,就是请求的时候必须有参数才可以被拦截.
headers ={}	//填入一个String数组,指定请求头.用的少
String[] consumes();			//当请求的Content-Type类型是该数组指定的类型时才匹配 该数组还支持否定表达式,比如"!application/json"代表支持除了application/json以外的任何Content-Type

2.6@RequestParam注解详解:
@RequestParam(value = "uage", required = false, defaultValue = "18")
required:参数是否必须
如果在配置文件中没有uage这个参数,而且required值为true(默认).那么就会报错.如果是false,就算没有传入这个参数也不会报错
当然也可以通过Optional来避免报错,例如:@RequestParam("uname") Optional<String> value 
defaultValue:在上面的情况下,如果没有传值 那通过这个可以设置默认值.

3.7获取cookie信息:
·介绍:
cookie:服务端在接收到客户端第一次请求时会给该客户端分配一个session(该session包含一个sessionId),并且服务端会在第一次响应客户端时将该sessionId赋值给JSESSIONID并传递给客户端.

·详细介绍:
当一个客户端第一次访问服务器的时候,服务器会给这个客户端生成一个sessionID.(这个ID的作用相当于给这个客户端一个身份信息)然后这个sessionID会被保存到JSESSIONID这个变量里.JSESSION又会被保存到一个cookie里.

·request、Response、Cookie:
Request(中文"请求"的意思):可以理解为客户端向服务器请求的信息,就是客户端向服务器请求时,把自己的浏览器信息、HTTP变量和保存在客户端的Cookie告诉服务器,这样服务器就可以根据这些信息判断是谁请求的,之前有没有请求过,对应客户端的Session是什么等等.

Response(中文"反应、响应"的意思):可以理解为服务器对客户端请求的响应,就是服务器接收到客户端的请求后,成生页面信息、Cookie(发到客户端后就保存在客户端)等发送到客户端.

Cookie(中文"饼干",在这里不能这样理解了):就是保存在客户端上的一些信息,可以用来验证用户信息,提高用户响应速度等等.为何不把它理解为:我把饼干放在家里,我想吃的时候就拿出来吃.

session和cookie的区别:
session的数据是存储在服务器内存中的.cookie的数据是存储在浏览器当中的.服务器通过浏览器cookie值中的sessionID来确认当前是哪个用户.
由于sessionID是存储在cookie中,cookie又是存储在浏览器当中的.当浏览器关闭的时候cookie随之被删除,从而sessionID也被删除,但是保存在服务器当中的session并没有被删除,只不过服务器中session不能对应sessionID,服务器会给session设置一个过期时间,当时间到达之后浏览器就认为这个session对应的客户端已经停止工作,就会删除这个session来释放内存.
注意:客户端这边的session信息是保存在cookie里面的.

request和session传值的区别:主要的区别在于生命周期的区别
都是从A页面跳转到B页面的时候带一些值过去的操作:
request:范围小,只是一个请求,request的生命周期相对较小,当请求(跳转页面)完毕时request里面的内容也将被删除.简单来说就是你在上一个页面的操作,可以通过request.getParameter()来获取上一个页面中的url、form中获取参数.如果你的request涉及多个类,后面还要取值可以通过rquest.setAttribute()和request.getAttribute()来获取值,但当值输出之后也就消失了.
不过如果是通过request.getRequestDispatcher("xxx.jsp").forward(request,response);这种方法.查看springboot 4.8.此时页面A的request值不会消失,而是会将之前的request生命周期延续下去,此时页面A的request和页面B的request是同一个request.

session:是一种重量级别的,存储在服务器上面的.session可以跨越多个页面,当你传到下一个页面的值,当输出完毕之后也不会立即消失,要等待生命周期的结束(一般是20min-30min)才会被删除.

request和session存值取值的特点:
session.setAttribute("所存值的对象", 要存的值);			//存值类似KV对
session.getAttribute("所存值的对象");

request.setAttribute("所存值的对象", 要存的值);
request.getParameter("所存值的对象")

如果是页面要显示的话,可以使用SPEL表达式或者Thymeleaf表达式

·request获取session:可以通过request获取session但是不能通过session获取request
request.getSession();			//返回值HttpSession		这个对象就保存着当前的客户的session信息.

5.3:视图解析器详解

什么是视图解析器?我们在控制方法里面返回的内容指向一个页面,我们可以通过配置视图解析器的方式给这个页面拼接上一个前缀和一个后缀.然而如果是JSP或者是themplates模板之类的由于页面是动态的,这个页面的内容是不能直接返回给浏览器的,是需要解析后再将数据返回给浏览器的,此时就需要使用视图解析器来解析我们编写的JSP或者ftl或者HTML等.

视图解析器链
在SpringMVC中可以同时定义多个ViewResolver视图解析器,然后它们会组成一个ViewResolver链.当Controller处理器方法返回一个逻辑视图名称后,ViewResolver链将根据其中ViewResolver的优先级来进行处理.所有的ViewResolver都实现了Ordered接口,在Spring中实现了这个接口的类都是可以排序的.在ViewResolver中是通过order属性来指定顺序的,默认都是最大值.所以我们可以通过指定ViewResolver的order属性来实现ViewResolver的优先级,order属性是Integer类型,order越小,对应的ViewResolver将有越高的解析视图的权利,所以第一个进行解析的将是ViewResolver链中order值最小的那个.当一个ViewResolver在进行视图解析后返回的View对象是null的话就表示该ViewResolver不能解析该视图,这个时候如果还存在其他order值比它大的ViewResolver就会调用剩余的ViewResolver中的order值最小的那个来解析该视图,依此类推.当ViewResolver在进行视图解析后返回的是一个非空的View对象的时候,就表示该ViewResolver能够解析该视图,那么视图解析这一步就完成了,后续的ViewResolver将不会再用来解析该视图.当定义的所有ViewResolver都不能解析该视图的时候,Spring就会抛出一个异常.

基于Spring支持的这种ViewResolver链模式,我们就可以在SpringMVC应用中同时定义多个ViewResolver,给定不同的order值,这样我们就可以对特定的视图特定处理,以此来支持同一应用中有多种视图类型.注意：像InternalResourceViewResolver这种能解析所有的视图,即永远能返回一个非空View对象的ViewResolver一定要把它放在ViewResolver链的最后面.

//1.创建转换器类:需要实现接口
public class IConverter implements Converter<String, Student> {		
//第一个泛型是从浏览器传入的类型,第二个泛型是拦截方法获得的类型
    @Override
    public Student convert(String source) {     //传入的内容是 2-zs-18
        String[] split = source.split("-");
        Student student = new Student();
        student.setId(Integer.parseInt(split[0]));
        student.setName(split[1]);
        student.setAge(Integer.parseInt(split[2]));
        return student;
    }
}

            CloseAll.closeAll(inputStream);
        }

        return "success";
    }



14.拦截器:
·介绍:
之前所说的拦截器都只是拦截一次,实际上可以形成一个拦截器链结构.

要实现拦截器,必须实现一个接口HandlerInterceptor

·导入jar包:
providedCompile group: 'javax.servlet', name: 'javax.servlet-api', version: '3.1.0'

·接口详细信息:一共三个抽象方法

preHandle(HttpServletRequest request, HttpServletResponse response, Object handler);
postHandle(HttpServletRequest request, HttpServletResponse response, Object handler,
		@Nullable ModelAndView modelAndView)
afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler,
		@Nullable Exception ex)

preHandle:对浏览器发送到服务器进行的请求拦截
postHandle:当服务器响应给浏览器之前进行拦截
afterCompletion:例子:假如服务器响应跳转到一个页面.当这个页面在客户端加载完毕的时候调用该方法.


//1.编写接口的实现类
public class IInterceptor implements HandlerInterceptor {
    /**
     *
     * @param request
     * @param response
     * @param handler
     * @return
     * 当执行到返回值的时候,说明拦截的方法已经执行完毕.如果返回值为false,那么就此为止,不执行后面的代码.
     * 如果返回值为true,则放行,接着进行后面的代码.
     * @throws Exception
     */
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        System.out.println(" Interception request ");
        return true;
    }

    @Override
    public void postHandle(HttpServletRequest request, HttpServletResponse response, Object handler, ModelAndView modelAndView) throws Exception {
        System.out.println(" Interception response ");
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {
        System.out.println("Loaded");
    }
}



//2.在springMVC中配置自定义拦截器.这里的拦截指代的是被上面编写的拦截器拦截,拦截类方法拦截和这个没有关系.

<!--配置自定义拦截器,默认拦截所有请求-->

    <bean class="cn.interceptor.IInterceptor"></bean>
</mvc:interceptors>


//3.指定具体的拦截范围:

public class ISecondInterceptor implements HandlerInterceptor {
    /**
     *
     * @param request
     * @param response
     * @param handler
     * @return
     * 当执行到返回值的时候,说明拦截的方法已经执行完毕.如果返回值为false,那么就此为止,不执行后面的代码.
     * 如果返回值为true,则放行,接着进行后面的代码.
     * @throws Exception
     */
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        System.out.println("Interception request Second");
        return true;
    }

    @Override
    public void postHandle(HttpServletRequest request, HttpServletResponse response, Object handler, ModelAndView modelAndView) throws Exception {
        System.out.println("Interception response Second");
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {
        System.out.println("Loaded Second");
    }
}

        <mvc:interceptor>

解释:

//9.多个拦截器的顺序:
满足先进后出,后进先出的原则.

请求->拦截器1->拦截器2->请求方法(拦截方法)->拦截器2响应->拦截器1响应->拦截器2加载完毕->拦截器1加载完毕.

那么这个拦截器到底有什么作用呢?
主要是对我们的请求做一个预处理,Interceptor和Controller是共存的(同时也是区分下来的).springMVC在处理请求的时候是按照链式的顺序执行的,假如我现在有一个/abc请求,springMVC会依次去执行每一个拦截器的preHandle方法(因为每一个拦截器都需要实现HandlerInterceptor接口).当我们遇到某个拦截器执行完毕时,如果这个拦截器返回false则后续所有内容都不执行,如果这个拦截器返回true则会继续执行后面的拦截器,当所有拦截器的preHandle方法都执行完毕时,此时执行这个请求真正的Controller方法,然后再执行postHandle、afterCompletion方法.

//3.拦截器(控制器)类:
@Controller()
@RequestMapping(value = "PeopleController")
public class PeopleController {

    @Autowired
    @Qualifier("serviceImpl")
    private IService iService;		//持有service层的引用,通过注解的方式注入,而不是配置文件的方式.

    public void setiService(IService iService) {
        this.iService = iService;
    }

    @RequestMapping("queryPeopleById")		//拦截链接
    public String queryPeopleById(People people, Map<String, Object> map) {
        People selectPeople = iService.signIn(people);		//调用service层的代码
        if (selectPeople != null) {			//不为null,代表有返回值.
            map.put("people", selectPeople.getMessage());	//通过map,将message信息存储到request域里面.
            return "SuccessfulLogin";			//返回登录成功的页面
        } else {
            return "LogonFailure";		//如果为null说明没有查询到people,跳转到登录失败的页面.
        }
    }
}

形参一:接收从浏览器传回来的参数.name和People的属性名对应.
形参二:用于跳转页面时传递参数.

        <property name="driverClassName" value="${driver}"/>
        <property name="url" value="${url}"/>
        <property name="username" value="${username}"/>
        <property name="password" value="${password}"/>
    </bean>

    <!--在springIOC容器中配置mybatis的核心类SqlSessionFactory
    通过这个配置就可以创建SqlSessionFactory了-->
    <bean id="sqlSessionFactoryBean" class="org.mybatis.spring.SqlSessionFactoryBean">
        <!--加载数据库数据源-->
        <property name="dataSource" ref="dataSource"/>
        <!--加载mapper文件路径-->
        <property name="mapperLocations" value="classpath:mybatis/mapper.xml"/>
        <!--为org.entity包下的类取别名-->
        <property name="typeAliasesPackage" value="com.entity"/>
    </bean>

    <!--配置dao层,这里不需要配id了-->
    <bean class="org.mybatis.spring.mapper.MapperScannerConfigurer">
        <property name="sqlSessionFactoryBeanName" value="sqlSessionFactoryBean"/>
        <!--指定批量产生哪个包中的mapper对象-->
        <property name="basePackage" value="com.mapper"/>
    </bean>

    <!--将dao层的引用注入到service层里面-->
    <bean id="serviceImpl" class="cc.Service.ServiceImpl">
        <property name="iMapper" ref="IMapper"/>
    </bean>

    <!--注意!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
    将service的引用注入到web控制器层
    这里不使用标签注入的方式,原因是控制器已经使用了注解@Controller().这个注解会将控制器加入到spring容器中
    但是现在该标签又将控制器加入到spring容器中.重复了,现在统一使用注解将service层的引用注入到web层
    <bean class="cn.controller.PeopleController">
        <property name="iService" ref="serviceImpl"/>
    </bean>
    -->
</beans>




//5.web层持有service层引用的接口:
public interface IService {
    People signIn(People people);		//传入一个People,返回一个People
}


//6.service层接口的实现类:
public class ServiceImpl implements IService {

    private IMapper iMapper;			//service层持有dao层的引用,这个在spring配置文件中是动态生成的Mapper实现类.

    public void setiMapper(IMapper iMapper) {
        this.iMapper = iMapper;
    }


    @Override
    public People signIn(People people) {
        People returnPeople = iMapper.signIn(people);
        return returnPeople;
    }
}
