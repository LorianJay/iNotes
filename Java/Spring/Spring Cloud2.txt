11但现在就有问题了,当同一个服务(多个Eureka Client提供相同内容)有多个Eureka Client的时候,当我想要RPC远程调用的时候那么具体是哪个服务来处理这个远程调用呢?
我们都知道一般通过RestTemplate来进行远程调用的时候需要指定域名(IP)+端口号,但如果我们把IP和端口号写死了那么别的服务就调用不到了,但问题是现在你也不知道
到底有几个服务,它们的端口号到底是多少,怎么解决?在Eureka 第6条中有写到我们可以通过spring.application.name配置来指定当前服务的名称,所以我们只要基于约定
提供相同服务的微服务它们的服务名称相同,那么真正通过RestTemplate RPC远程调用的时候,只要使用这种形式 http://now_cloud_name/ 也就是在请求URL中填入服务的服务名称
不需要关注具体的IP和端口我们就能使用负载均衡去请求具体的服务,只不过原版RestTemplate不支持,要做一些修改:
@Configuration
public class ApplicationConfig{

    @Bean
    @LoadBalanced			//加上这个注解,让RestTemplate拥有springcloud的功能(实际上这个注解是Ribbon的内容,加上这个注解让RestTemplate可以使用Ribbon的负载均衡算法远程调用,所以与其说springcloud是使用RestTemplate去调用还不如说是Ribbon要使用RestTemplate去调用.例如负载均衡OpenFegin就不是使用RestTemplate去调用的)
    public RestTemplate getRestTemplate(){
	return new RestTemplate();
    }

}

15.Eureka Server的自我保护机制
产生原因:假如某一时刻某个服务和Eureka Server发送的心跳检测失败了,那么Eureka Server就会开启自我保护机制,因为Eureka Server认为就算服务心跳失败了也不一定就代表这个服务不能用了,如果直接删除该服务可能会造成更大的问题
具体表现:Eureka Server如果某时刻心跳检测不到一个服务,那么Eureka Server也不会立即把该服务从注册信息列表中删除,会保留信息
16.关闭Eureka Server自我保护机制(更多配置)
eureka:
    server:
	enable-self-preservation: false		//是否开启Eureka Server自我保护机制
	eviction-interval-timer-in-ms: [int/ms]	//当Eureka Client心跳连接超时多长时间后剔除服务信息(单位毫秒 默认90000ms/90s)

eureka:
    instance:
	lease-renewal-interval-in-seconds: [int]	//Eureka Client向Eureka Server服务端发送心跳的时间间隔(单位秒 默认30s)
	lease-expiration-duration-in-seconds: [int]	//Eureka Server在受到最后一次心跳后等待时间上线(单位为秒 90s),超时就剔除该服务信息(和上面的配置有点异曲同工之妙)

负载均衡之Ribbon:
1.已经有了nginx为什么还需要Ribbon?
Nginx是服务器负载均衡,是挡在最外面的所有的客户端(浏览器)请求都会先到Nginx,由Nginx选择转发到哪个Tomcat上.
Ribbon是本地负载均衡,当内部服务调用服务的时候,会先从注册中心上获取Server的注册列表信息缓存到本地,然后在本地实现RPC远程调用的选择(在本地选择具体调用哪个服务)
2.Ribbon是针对Client而言的,之前说过Server只提供服务注册的功能
3.依赖,现在当引入spring-cloud-starter-netflix-eureka-client之后下面这个依赖会自动引入
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-ribbon</artifactId>
</dependency>
4.只要把上面的依赖引入Ribbon不需要改变任何东西就已经可以负载均衡了.
但需要注意的是 在当前第1条说过,Ribbon是自已去Server上获取服务信息后自已根据负载均衡算法选择具体要调用的服务,所以实际上调用的选择权在调用者身上(这个也很容易理解,被调用者一般没有权利管你调用者调用哪个服务)
所以这个依赖只针对调用者有效,也就是一般谁调用别的服务需要负载均衡谁就加这个依赖(和被调用者无关)
5.
IRule:顶级接口,负责Ribbon负载均衡算法的规则
AbstractLoadBalancerRule:IRule接口的抽象实现
RoundRobinRule:继承自AbstractLoadBalancerRule抽象类,该实现类代表负载均衡算法之轮询算法
RandomRule:继承自AbstractLoadBalancerRule抽象类,该实现类代表负载均衡算法之随机算法
AvailabilityFilteringRule:会先过滤掉由于多次访问故障而处于断路器跳闸状态的服务,还有并发的连接数量超过阈值的服务,然后对剩余的服务列表按照轮询策略进行访问.
WeightedResponseTimmeRule:根据平均响应时间计算所有服务的权重,响应时间越快服务权重越大被选中的概率越高,刚启动时如果统计信息不足,则使用轮询策略,等待统计信息足够时再切换到该策略.
RetryRule:先按照轮询的策略获取服务,如果获取服务失败则在指定时间内会进行重试.获取可用的服务.
BestAvailableRule:会先过滤由于多次访问故障而处于断路器跳闸状态的服务,然后选择一个并发量最小的服务.(会选择故障最小的服务)
ZoneAvoidanceRule:默认规则,符合判断server所在区域的性能和server可用性选择服务器.
6.如何替换掉默认的负载均衡算法?
需要注意的是,下面这个配置类不能被springboot扫描到(springboot会默认扫描主启动类所在的包以及其子包),所以该配置类不能放到与主启动类的同级目录下,否则该负载均衡规则将对所有服务生效
@Configuration
public class MyRule{			//同理通过配置类,把自定义的IRule实现添加到容器中
    
    @Bean
    public IRule myRule(){
	return new RandomRule();
    }
}

@SpringBootApplication
@EnableEurekaClient
@RibbonClient(name="[CloudName]",configuration=MyRule.class)
public class Application{

}
所有在主启动类上添加@RibbonClient注解对特定的服务调用使用特定的负载均衡算法(加载特定的配置类),所以该注解还是对于调用者而言的,切记负载均衡算法是对于调用者而言的!
通过这种方式指定特定的服务名和配置类,代表对于不同的服务可以使用不同的负载均衡算法(这也是为什么上面说直接被SpringBoot扫描会对所有服务生效的原因)


负载均衡之OpenFeign:
1.之前说过最原始的远程调用是通过RestTemplate来完成的,实际上这是配合@LoadBalanced注解以及Ribbon的负载均衡算法完成的.
而使用OpenFeign时,服务之间的调用就不是使用RestTemplate而是使用接口+注解来实现的
2.和Ribbon一样OpenFeign也是针对服务的调用者而言的(首先得是Client然后才是服务调用者),所以被调用者是不感知的.
3.Client调用者引入依赖(因为负载均衡是调用者的事)
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-openfeign</artifactId>
</dependency>
4.在调用者主启动类上添加@EnableFeignClients注解,开启OpenFeign的自动配置(这里可以不添加@EnableEurekaClient注解)
5.具体使用方法,之前在使用RestTemplate的时候,我们真正去请求的还是controller层对应的接口,现在也差不多
首先我们需要在服务调用者模块编写一个接口,该接口的方法要和被调用的服务的controller层的接口对应,该接口不需要我们编写实现类
@Component				//接口需要添加到容器中
@FeignClient(value = "[CloudName]")	//通过该注解指定微服务名称,表明当前接口要去调用哪个微服务下面的接口
public interface FeiginService{

    UserInfo getUserInfo(@PathVariable("id") Long id);
}

@GetMapping(value = "/get/{id}")		//这个控制层方法就是[CloudName]微服务下的一个接口
public UserInfo getUserInfo(@PathVariable("id") Long id){
	//...
}

最后我们在服务调用者一方,只要自动注入FeiginService,调用它的getUserInfo方法,OpenFeign就会去调用对应的服务
6.在OpenFeign中服务的调用者只等待目标服务1s钟处理时间(也就是说如果目标服务1s以内还没有返回结果,则服务调用者就会报错)但有的时候确实某些服务需要超过1s的时间去处理该怎么办?

服务降级之Hystrix:
1.服务雪崩
产生原因:在一整条服务调用流程上,如果某一环一个服务出现故障,这就会导致整个系统不可用阻塞(换句话说就是无法保证系统的高可用).另外如果目标服务调用响应时间超时,则也可能需要服务降级(这个和前面讲的不矛盾,在约定好的情况下,如果你服务本身要3s而我调用者也可以接受3s,但是现在调用你花费了5s此时就需要降级)
2.服务降级:相当于在服务调用的流程上,如果某一环出现问题,则你要告知系统给一个最终的解决方案(FallBack),而不是让系统就这么等待.所以服务降级就是当程序出问题时去调用指定方法
3.服务熔断:服务熔断包含了服务降级,当服务达到最大访问量之后直接把服务关掉,并且给一个最终解决方案(FallBack).服务熔断只是为了恢复服务
4.服务限流:高峰流量来的时候做限流操作(详情看下面Sentinel,实际上Hystrix也不推荐使用了)
5.服务提供者和服务调用者都可以使用Hystrix,但是一般是用在调用者上.
6.Client依赖
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-hystrix</artifactId>
</dependency>
6.5在Client主启动类上添加@EnableCircuitBreaker注解(开启服务降级/熔断的相关的自动配置)
7.先从服务提供者下手(下面的方法是Service层方法)
@HystrixCommand(fallbackMethod = "getUserInfoHandler",commandProperties = {
    @HystrixProperty(name = "execution.isolation.thread.timeoutInMilliseconds",value = "3000")		//当前方法调用多长时间后未响应就调用降级方法

14.服务熔断比服务降级来得更重量级,服务熔断包含了服务降级,如果出现问题也返回友好提示但服务熔断是直接把服务关掉,但服务熔断厉害的是关掉服务只是为了恢复服务.
出现熔断的原因是:之前在降级的时候如果目标服务每次都无法调用完成,每次都要超时,那调用者每次还就会傻傻地去调用服务.此时熔断的机制是,如果目标服务一段时间内调用失败的次数超过一定的阈值,则直接将目标服务熔断,快速返回失败信息,而不是干等
梳理:所以说服务降级是允许一段的调用时间,真正运行的时候还是会走到服务降级的阈值才会降级,而熔断是如果频繁地服务降级则直接熔断,这样连降级的这段时间都不用走了,直接返回错误信息.
15.服务熔断肯定是针对Client服务提供者而言的,因为不可能你自已的服务出错了让调用者熔断吧......

    @HystrixProperty(name = "circuitBreaker.requestVolumeThreshold",value = "10"),	//请求次数
    @HystrixProperty(name = "circuitBreaker.sleepWindowInMilliseconds",value = "10000"),//时间窗口期[int/ms]
    @HystrixProperty(name = "circuitBreaker.errorThresholdPercentage",value = "60")	//失败率达到多少后熔断服务

    //....
}

    return "服务器错误,请稍后重试!";
}
上面那些配置表明,在10S内如果有10次以上的请求且这些请求中有60%都失败了则触发熔断.

6.流控规则面板详解(也就是我们点击流控后跳出来的面板)
资源名:请求地址/@SentinelResource的唯一标识符

针对来源:

阈值类型(和单机阈值搭配使用):
QPS:每秒请求量
线程数:同时支持最大并发量(传统的servlet中一个请求一个线程)

单机阈值:灵活一点,当阈值类型是QPS的时候该值表示每秒的请求量,当阈值类型是线程数的时候表示支持的最大并发量.

流控模式:
直接:api达到限流条件时,直接限流(不跟你多BB直接限流当前请求地址)
关联:当关联的资源达到阈值时,就限流自已(假设对地址A进行限流,关联的地址是B,则当地址B达到阈值时,地址A会被限流.也就是B达到阈值了,确是A限流.比如支付系统达到阈值了就限流订单系统,这么做是为了防止一个带崩整个)
链路:只记录指定链路上的流量(指定资源从入口资源进来的流量,如果达到阈值,就进行限流)

流控效果:
快速失败:直接失败,抛异常,想前端返回信息
Warm UP(预热):比如现在的QPS是10,但是这里设置一个预热时长(比如10S),当系统刚启动的时候默认此时的QPS是我们设定值除以3(也就代表此时如果1S有三个请求就会限流),当系统运行完全运行到预热时长里的这段时间QPS会慢慢提升,当运行时间达到10S后(也就是完全达到预热时间之后),此时QPS才完全达到我们的设定值10
排队等待(匀速排队):比如阈值设为10,就代表我1S就处理10次请求,也就是每次请求之间间隔100ms.如果100ms之内有两次请求,则请第二次请求等待(排队).排队等待只支持QPS不支持线程.

7.Sentinel服务降级简单介绍,当某个服务调用超时或者异常比例升高,则该服务会被降级(熔断),在接下来的一段时间内(降级时间窗口之内)关闭该服务.等时间窗口过去后再恢复服务.

8.服务降级面板详解
资源名:请求地址/@SentinelResource的唯一标识符

降级策略:
RT:当选择此策略时会出现RT降级时间(代表一段时间内如果所有请求的平均运行时间达到该时间则会服务降级,RT的最大值是4900ms,也就是如果一段时间内请求的数量超过5次并且这些请求处理的平均时间超过4900ms则会服务降级)
异常比例(0-1之间):如果1S内的请求数量大于5,并且这段时间内出现异常的请求超过了设定的异常比例(比如设定为20%,只要5次里面有1次异常)就会服务降级(熔断) 
异常数:一分钟之内如果请求的异常数超过多少后服务降级.注意在该模式下时间窗口设置的时间一定要大于60S 

时间窗口(单位S):当服务被降级(熔断)后多长时间恢复服务.

服务网关之Gateway:
1.Gateway可以做反向代理、鉴权、流量控制、熔断、日志监控
2.已经有的Nginx和负载均衡(Ribbon、OpenFeign)为什么还需要服务网关?
Nginx是服务器负载均衡,是挡在最外面的所有的客户端(浏览器)请求都会先到Nginx,由Nginx选择转发到哪个Tomcat上(这个Tomcat就可以是网关)
Gateway是挡在真正每一个微服务之前的一层,更全的原因详情看第9.1条
Ribbon是本地负载均衡,当内部服务调用服务的时候,会先从注册中心上获取Server的注册列表信息缓存到本地,然后在本地实现RPC远程调用的选择(在本地选择具体调用哪个服务)
3.Geteway采用servlet3.1+、支持异步非阻塞、集成WebFlux,总结一句话就是性能高.
4.Geteway网关的三大概念
路由(Route):
断言(Predicate):做匹配条件定位到真正的服务节点
过滤(Filter):在我们真正业务逻辑之前和之后去做一些操作(有点像AOP)

所以最终的效果就是,虽然Geteway Client并不提供服务,而且我们请求的是Geteway Client的IP和端口,最终却能落实到具体的服务.这样就能隐藏掉真实服务地址(而且服务一般都是单独部署的,IP和端口都各不相同,通过网关就能统一管理,通过访问网关由网关做转发)
所以Geteway Client是做转发的!即如果断言通过,Geteway只是把你请求的端口和IP换成目标服务的端口和IP(当然不绝对,编码的方式可以改成别的请求地址)

		locator:

		uri: lb://[CloudName]		//没错和之前RestTemplate一样,这里直接填上目标服务的服务名(不过注意是以lb开头而不是http)

所以此时对于客户端(一定不是Client的调用者,这里指的就是浏览器或者用户)所有的访问首先经过Nginx,然后由Nginx转发到一个个的服务网关上(还记得之前说过的服务网关也是一个微服务,所以服务网关也要集群部署)

10.predicates是做断言的,之前已经讲过可以对请求路径做一个断言,除了请求路径之外还可以做别的一些断言
- After=2020-09-19T12:11:54.485+8:00[Asia/Shanghai]		//只有当前系统时间在该时间之后断言才成功(目标服务才能访问)
- Before=							//这个就是系统时间在目标时间之前才成功
- Between=2020-09-19T12:11:54.485+8:00[Asia/Shanghai],2020-09-19T12:11:54.485+8:00[Asia/Shanghai]	//系统时间之内
- Cookie=[CookieKey],[CookieValue]			//请求的时候必须带上Cookie逗号,前面指定的是需要带上的Cookie的Key,逗号后面指定的是需要带上的Cookie的value的正则表达式.也就是说你的value必须满足该正则表达式断言才算通过
- Header=[HeaderKey],[HeaderValue]			//带上请求头
- Host=[HostName]					//请求的Host的值必须是什么(这个Host的值也包含在整个请求当中,详情打开浏览器),同样支持Ant风格
- Method=[GET/POST]					//必须是什么请求才允许访问
- Path=[Path]
- Query=[QueryKey],[QueryValue]			//带上请求参数
11.路由和断言已经讲过,现在了解一下过滤器,Gateway默认有三十几种局部过滤器和十几种全局过滤器,过滤器和断言不一样要注意!
spring:
    cloud:
	gateway:
	    discovery:
		locator:
		    enabled: true		
	    routes:				
		- id: [RoutesID]		
		uri: lb://[CloudName]		
		filters: - AddRequestParameter=[RequestParameterKey],[RequestParameterValue]		//添加一对请求头,对于通过断言的请求而言,该过滤器会给当前请求添加请求头(请求头的Key和value就是RequestParameterKey/Value),所以过滤器能做更丰富的事情.
		predicates: - Path= [Path]

服务配置之Config:
1.做到现在随之出现的一个问题,就是每一个模块(服务)都要编写一个application.yml,如果后期服务多了有几百个服务,那么这些配置文件到时候改起来会很麻烦,所以服务配置就是来解决这个问题的,用Config Server(服务配置)来完成集中化的外部配置支持.
2.所以服务配置的思想就是,该抽象出来统一使用的就直接从Config Server中继承,如果需要单独配置的就单独配置.
3.服务配置的好处在于,运行期间如果要更新配置,一次更新底下所有的服务的配置会立即感知(而不是一个一个地去修改).而且服务配置推荐和Git进行整合.
4.服务配置中心本身也是一个服务(模块),需要注册到注册中心中去
5.服务配置分为服务端和客户端,服务端提供配置,客户端读取服务端的配置.
5.Config Server依赖
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-config-server</artifactId>
</dependency>
6.Config Server的application.yml
spring:
    cloud:
	config:
	    server:
		git:
		    uri: 				//项目的远程仓库地址
		    search-paths: - [Repository]	//搜索远程仓库的哪个路径(一般就是填项目名称,注意前面要加个-)
		label: master				//读取哪个分支
上面说过Config Servert是和Git进行整合使用的,所以这里需要读取远程仓库上面的配置(GitHub/gitee),而且服务配置只认远程仓库上的内容.
7.在主启动类上添加@EnableConfigServer注解
8.这个时候去访问服务配置的IP+端口/[branch]/[configName](即IP+端口/[分支]/[配置名称])那么浏览器就会显示该配置的内容.另外根据约定配置文件的命名应遵循:config-(dev/test/prod)这种形式,以config开头然后加一个分号-
9.Config Client依赖
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-config</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
注意spring-boot-starter-actuator这个监控依赖基本上除了服务网关之外一般都要加载,添加该依赖之后就可以暴露监控端点.
10.Config Client的bootstrap.yml
spring:
    cloud:
	config:
	    label: [dev/master/prod]			//当前读取Config Server的哪个分支(也就是读取远程仓库的哪条分支)
	    name: [config]				//第8条说过约定配置文件名要遵守config-[dev/master].yml 那么这里的name就填上远程仓库配置文件分号-的前缀
	    profile: [dev/master/prod]			//这里填上后缀
	    uri: [Config Server URI]			//填入Config Server的URI地址(也就是当前要读取哪个Config Server)
通过以上这些配置就能确定一个唯一的配置文件,注意Config Client从Config Server中读取配置,而Config Server是从远程仓库中读取配置.
11.什么是bootstrap.yml?实际上该配置文件和application.yml配置文件效果一样都是来配置的,只不过bootstrap.yml会先去读取外部的配置(也就是先读取Config Server的配置)然后再读取当前配置文件内的配置.
12.但现在又出现了一个问题,假设现在远程仓库的配置发生改变并提交,Config Server可以理解响应变动,但是Config Client却无法响应这次更改,还是使用之前的那次配置(虽然重启可以解决但是太麻烦)
13.Config Client的bootstrap.yml
management:
    endpoints
	web:
	    exposure:
		include: "*"
14.在所有和配置文件绑定的属性所在的类上添加@RefreshScope注解,因为配置是要和属性绑定的,所以配置的修改就是属性的重置,所以我们要在被绑定的属性的类上添加@RefreshScope注解
@Server
@RefreshScope
public class Service{
    @Value("${userInfo}")
    private String userInfo;
}
15.完成以上这些,现在远程仓库配置更改Config Client还是感知不到,我们还需要向Config Client发送一个请求,该请求就代表告知目标服务配置要刷新了
http://[IP]:[Port]/actuator/refresh			//注意得是POST请求

服务总线之Bus:
1.一般服务总线都是和服务配置搭配使用的,而且服务总线(或者至少说是Bus)也要和消息中间件搭配使用.

在微服务中通常会构件一个主题,并且将所有的微服务都订阅这个主题,由于该主题产生的消息会被所有的监听者(订阅者)消费,所以就它们为消息总线.

消息驱动之Cloud Stream:
1.目前市面用的最多的有四种消息中间件,ActiviteMQ、RabbitMQ、RocketMQ、Kafka.很多时候不同的公司使用不同的MQ(有些时候一个项目可能使用多个MQ),那么对于程序员而言学习成本太高,维护成本高!
此时就可以引入Cloud Strem屏蔽底层中间件的差异,统一消息编程的模型(也就是只操作Cloud Stream就可以了)
2.Cloud Stream仅仅支持RabbitMQ和Kafka
3.Cloud Stream使用绑定器来屏蔽差异,input是消息消费者,output是消息生产者.(Cloud Stream分为生产者和消费者,就是经典的消息中间件那一套)
4.消息发送者模块的依赖,消息生产者本身也是一个服务,也要注册到注册中心中去

11.现在出现一个新问题,假设现在有两个消费者(两个服务/模块),消息生产者生产消息后,两个消费者均能获取到消息,这就存在重复消费的问题(当然也可以理解为现在是主题和订阅模式).
那么怎么避免这种问题?在CloudStream中,可以通过分组的方式来解决该问题.在同一个组中一个消息只能被一个成员消费,当消息生产者生产消息会将该消息发送给所有的组!所以组内成员是竞争消费的关系.所以在CloudName中虽然消费者同属一个主题但每个消费者都是一组(默认)
12.RabbitMQ默认的分组名是一串流水号,实际上可以通过配置自定义分组名
生产者的application.yml
spring:
    cloud:
	stream:
	    bindings:		
		output:
		   group: groupA
消费者的application.yml
spring:
    cloud:
	stream:
	    bindings:		
		output:
		   group: groupB
这样这两个消费者就分别分为组A和组B

假设现在这两个组名相同,此时消息生产者生产就不会产生重复消费的问题了.(即同一个组的服务实例只有一个服务能拿到消息)
13.消息丢失问题,假设现在两个消费者都是组A的,然后这两个消费者宕机后,生产者生产消息.此时让一个消费者的组名改为流水号(也就是之前没有出现的过的组)
另一个消费者不动,现在两个服务重新启动,只有组名还是A的消费者能消费到消息,组名改变的消费者无法消费消息(消息丢失).原因我猜测大概是,虽然两个服务都宕机了
但是消息生产者已经将消息生产到rabbitMQ中的A组了已经持久化了,但是这条消息发出的时候只有组A,所以组A等待消费者消费.所以组A服务再起启动时可以消费消息,但是另外一个消费者
由于已经改名了,也就代表之前消息生产者生产的时候没有生产到这个新组里面(因为当时这个组还不存在),所以新组无法消费消息,消息丢失.

集群和持久化配置之Nacos:
1.之前我们在服务注册中心、服务配置中使用的nacos都是单机版的,那么还是那句老生常谈的问题,为了保证nacos注册中心的高可用,肯定不能只有一台nacos,需要集群部署nacos.
nacos互相直接并不能直接关联,而是需要通过一个VIP(这个VIP不是会员的意思),这个VIP有点类型与Nginx(当然也不好说).
2.nacos上的各种配置,比如服务配置的那些配置是持久化的,也就是说如果关掉nacos再重启这些配置不会消失,nacos默认采用本地数据库来完成该功能的持久化.但是当有
多个nacos的时候,各个nacos之间的数据一致性就可能有问题,所以推荐让所有的nacos的配置全部持久化到mysql上.

接着同样打开该目录下的application.properties配置文件,在里面拉倒最后一行添加如下配置

characterEncoding=utf8&connectTimeout=1000&sockerTimeout=3000&autoReconnect=true

db.password:[PassWord]			//数据库连接的密码

分布式事务之Seata:
1.在大型的系统中,往往不止一个数据库(而且这些数据库都担当不同的功能),比如订单一个库、商品存储一个库、账单一个库.假设现在订单服务要调用商品服务,而这些服务需要和不同的数据库交互.
此时为了保证数据之间的一致性问题,就需要使用分布式事务,保障多个数据库之间的数据的一致性.
2.Seata中有三个组件(概念了解即可)+全局唯一的事务ID来完成分布式事务
TC(事务协调者):维护全局和分支事务的状态,驱动全局事务提交或回滚.(就相当于我们启动Seata这个软件,Seata Server)
TM(事务管理器):定义全局事务的范围,开始全局事务、提交或回滚全局事务(哪个方法被@GlobalTransaction注解修饰哪个方法就是TM,谁是事务的发起方)
RM(资源管理器):管理分支事务处理的资源,与TC交谈以注册分支事务和报告分支事务的状态,并驱动分支事务提交或回滚.(事务的参入方,参数该事务的服务)
Transaction ID:每一个事务都有一个全局的唯一ID
3.TM向TC申请开启一个全局的事务,TC创建全局事务并生成一个全局的唯一ID(Transaction ID),这个ID将贯穿整个服务的调用链路,直到最后TC调度Transaction ID,将完成该ID下管辖的所有事务的提交或者回滚.
4.Seata直接去官网上下就行,也是个软件,以前spring提供了一个@Transaction注解,但该注解只针对本地(一个)数据库,而分布式事务对于多个数据库可以使用@GlobalTransaction注解

16.Seata有如下几种模式

TCC:支持TCC模式并可与AT混用,灵活度更高
SAGA模式:为尝试无提供有效的解决方案
XA模式(开发中):支持以实现XA接口的数据库的XA模式

17.AT模式是如何做到无侵入式的自动补偿事务模式的?
第6条运行那条sql语句之后,seate数据库有三个表:branch_table、global_table、lock_table
第9条在每个数据库中运行那条sql语句之后,会新建一个undo_log表
一阶段:分析SQL语义,找到这些SQL要更新的业务数据(表),在表被更新之前,创建一份该数据的镜像(before image).执行目标SQL(就是执行我们写的SQL),并再对执行完SQL的业务数据(表)再创建一份镜像(after image)(现在有两个镜像,一个是执行之前的,一个是执行之后的).执行完之后最后生成行锁.这些操作全都在一个数据库事务内完成(也就是Seata Server的数据库),保证了该阶段的原子性.
二阶段提交:如果一阶段执行过程中没有产生异常(也就是所有服务全部正常执行完毕),此时直接对要更新业务数据的数据库执行提交(commit)命令.(因为此时SQL已经作用到目标数据库上了).并且将两份镜像以及行锁删除.
二阶段回滚:如果一阶段执行过程中产生异常,则就需要回滚一阶段执行的SQL,还原业务数据.回滚的方式就是通过在一阶段表更新之前创建的那份镜像,在还原之前,首先要检查当前业务数据的内容和after image镜像是否吻合,如果吻合就还原(反向补偿),不吻合代表有脏写,需要人工处理.


雪花算法(snowflake):
1.需求:在分布式高并发的集群部署的系统中,我需要一个全局的唯一性ID.并且存放到数据库中
2.解决方案:
通过JDK自带的UUID类来实现,答案不合适,因为UUID生成出来的字符太长(而且无规律,下一刻生成出来的毫无逻辑可言),不适合放到数据库中加索引(或者作为主键).
通过mysql的主键作为全局唯一性ID,答案不合适,虽然mysql主键保证了唯一性,但是当mysql是集群部署的时候,多台mysql的主键都是从1开始,就不能保证唯一性了(除非你是单机).而且这么做会有性能问题.我们一直在强调的是mysql尽量少干事
通过redis来生成,答案不合适,相较于mysql已经好很多了,但是怎么解决集群redis之间自增的Key不重复?假设现在有5台redis,则第一台redis从1开始每次增加5,第二台从2开始每次也增加5.这样就保证了全局唯一的ID,但现在问题时假如又要新增一台redis怎么办?所以这种方式不利于扩展,每次增加的值要在一开始就确定好.
3.雪花算法中的ID是一个long类型的值(64bit),其中第1位是符号位(这个是java的特性),其中前41bit表明时间戳(时间从1970年开始,毫秒级别),中间10bit表明工作进程位.最后12bit是序列号位.
工作进程ID用来记录工作机器的ID,包括5位datacenterID和5位workerID(可以表示1024个节点(集群机器),也就是每个机器都有自已单独的ID)
序列号ID用来记录同毫秒内产生的不同的ID,一台机器1毫秒能产生4095个序列号ID 
4.雪花算法可以保证所有生成的ID随时间趋势递增,整个分布式系统内不会产生重复ID.生成ID性能高、不依赖数据库等第三方系统(java代码可以直接做)
5.雪花算法缺点:
依赖机器时钟,如果机器时钟回拨会生成重复ID
在单机上是递增的,但由于涉及到分布式环境,每台机器上的时钟不可能完全同步,有时候会出现不是全局递增的情况.(此缺点无所谓)
