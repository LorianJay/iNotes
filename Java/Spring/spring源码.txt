SpringMVC全流程:
Controller层返回值->ModelAndView(Controller层返回值即时ModelAndView)->视图解析器(ViewResolver)->视图(View)
过程中遇到异常->DispatcherServlet委托给HandlerExceptionResolver异常处理解析器链->ModelAndView->视图解析器(ViewResolver)->视图(View)

国际化:先走DispatcherServlet的doService方法->doService方法通过调用request的addAttribute添加LocaleResolver->调用LocaleChangeInterceptor的preHandle方法从而一次设置多次使用(详情可看preHandle方法)->在后续的运行过程中通过RequestContextUtils来得到Locale和TimeZone(如果是扩展的话才有)(能得到是因为doService方法)->最终调用DispatcherServlet的render方法时会得到应用上下文中唯一的LocaleResolver然后调用其resolveLocale方法得到Locale后作为ViewResolver的resolveViewName方法的入参.

视图解析器:每一个ViewResolver的实现类基本上都会间接实现Ordered接口 Sring支持同时定义多个ViewResolver视图解析器,这些视图解析器将会形成一条链 当我们在Controller层返回一个视图名称后 Spring会根据这些ViewResolver(视图解析器)中Order的返回值来依次调用视图解析器链的resolveViewName方法,将回调参数传入(视图名称和Local),如果该方法的返回值不为null则表明当前视图解析器可以解析该视图则终止解析器链.其中InternalResourceViewResolver解析器会最后一个解析(因为它能解析所有视图),但假设由于某种原因这个视图压根就找不到(既然没有就无法被解析器解析),那么Spring就会抛出一个异常也就是页面找不到异常 如果没有AdviceController则会来到/error页面(多次强调不再解释)
视图:每一个View就是最终要输出到浏览器的内容
在SpringMVC启动时每一个@RequestMapping方法都会通过HandlerMethodMappingNamingStrategy分配一个默认的名称,规则是当前方法所在类的所有大写字母+方法名(例如LoginController中有一个loginInfo方法则该方法的名称是LC#loginInfo)如果方法重名可以通过@RequestMapping(name="..")分配一个明确的名称

·WebMvcConfigurerComposite:实现了WebMvcConfigurer接口 Composite翻译为综合
由于一个项目可能存在多个WebMvcConfigurer接口的实现,所以我们应当对每一个WebMvcConfigurer接口的实现进行回调. 给类内部封装一个List<WebMvcConfigurer>旨为所有WebMvcConfigurer接口实现的集合,只需要遍历该集合并调用该接口的方法传入回调的类即可.
addWebMvcConfigurers(List<WebMvcConfigurer>) return addWebMvcConfigurers;		//新增方法 调用该方法传入所有WebMvcConfigurer的实现类集合(该方法被@Autowired注解修饰,所以会自动注入该集合 也就是从容器中拿到List<WebMvcConfigurer>集合)
configureViewResolvers(ViewResolverRegistry) return void;		//该方法入参本身还是回调,只不过在该类中对所有的WebMvcConfigurer接口的实现方法,内部都是遍历List<WebMvcConfigurer>集合并调用集合中的每个元素的对应方法,并把当前方法的入参作为形参传入(这就保证了每一个WebMvcConfigurer接口的实现类的方法的入参对应都是同一个对象)

·WebMvcConfigurationSupport:该类实现ApplicationContextAware、ServletContextAware接口(这两个接口是回调接口)
对于每一个WebMvcConfigurer接口的方法,该类都会提供对应的两个方法
configureViewResolvers(ViewResolverRegistry) return void;		//空方法(被下面的类实现) 那么这条链路该方法的ViewResolverRegistry到底是由谁传进来的?
mvcViewResolver(ContentNegotiationManager) return ViewResolver;		//agrs0:这个参数不用管它(自动注入的) 该方法被@Bean注解修饰 就是由该方法内部来创建ViewResolverRegistry的然后将该对象作为形参传入上面的方法(别的方法也类似) 创建好了之后这个注册器不能浪费要重复利用,但是这里没有直接将其添加到容器中,而是得到ViewResolverRegistry中封装的所有ViewResolver添加到ViewResolverComposite对象中(为什么这么干?我怀疑就是Spring用的时候只需要ViewResolverComposite(视图解析器综合),而暴露给程序员的是ViewResolverRegistry这个类,也有一种原因是不能乱)
requestMappingHandlerMapping(ContentNegotiationManager,FormattingConversionService,ResourceUrlProvider) return RequestMappingHandlerMapping;		//定制路径匹配规则 最终交给RequestMappingHandlerMapping处理

所以大致可以理解为由@Bean注解去驱动WebMvcConfigurer中的每个回调方法,再将注册后的每个组件添加到容器中.

·DelegatingWebMvcConfiguration:该类继承自WebMvcConfigurationSupport 该类内部组合WebMvcConfigurerComposite 该类被@Configuration注解修饰 虽然该类被@Configuration注解修饰,但是该类并不会被扫描到不会添加到Spring容器中,除非通过@Import等别的方式将该配置类能被添加到容器中发挥作用.SpringBoot的自动配置类很方便地完成了这一点
configureViewResolvers(ViewResolverRegistry) return void;		//该类同样提供了WebMvcConfigurer接口中的每一个方法的实现(但这些方法全都是重写的父类WebMvcConfigurationSupport的方法),对于每一个实现内部就是调用WebMvcConfigurerComposite类的实现方法.当前方法的入参作为形参传入


·ViewResolverRegistry:顶级类,用于注册ViewResolver 该注册器正如上面所说,内部封装一个List<ViewResolver>集合
viewResolver(ViewResolver) return void;					//注册一个自定义视图解析器

jsp(String,String) return UrlBasedViewResolverRegistration;		//重载方法 args0:前缀 args1:后置
freeMarker() return UrlBasedViewResolverRegistration;			//内部创建了一个FreeMarkerViewResolver并包装在UrlBasedViewResolverRegistration中,创建完毕后会将ViewResolver添加到集合中
getViewResolvers() return List<ViewResolver>;				//获得当前注册器注册的所有视图解析器ViewResolver

·ViewResolverComposite:实现了ViewResolver、Ordered、InitializingBean、ApplicationContextAware、ServletContextAware接口 Composite翻译成综合 该类内部封装一个List<ViewResolver>为所有视图解析器的集合

·ViewResolver:顶级接口,视图解析器 
每一个ViewResolver的实现类基本上都会间接实现Ordered接口 Sring支持同时定义多个ViewResolver视图解析器,这些视图解析器将会形成一条链 当我们在Controller层返回一个视图名称后 Spring会根据这些ViewResolver(视图解析器)中Order的返回值来依次调用视图解析器链的resolveViewName方法,将回调参数传入(视图名称和Local),如果该方法的返回值不为null则表明当前视图解析器可以解析该视图则终止解析器链.其中InternalResourceViewResolver解析器会最后一个解析(因为它能解析所有视图),但假设由于某种原因这个视图压根就找不到(既然没有就无法被解析器解析),那么Spring就会抛出一个异常也就是页面找不到异常(会抛出ServletException异常) 如果没有AdviceController则会来到/error页面(多次强调不再解释) 也就是DispatcherServlet类的resolveViewName方法
resolveViewName(String,Locale) return View;						//args0:要解析的视图名称() agrs1:支持国际化的Local对象(这个Local是java原生类,用于支持国际化的 这个类可以通过application.yml去配置 该参数Spring会回调传入,Spring会从容器中得到LocaleResolver然后调用其resolveLocale方法得到Local对象)

·View:顶级接口,视图(由视图解析器解析得到)
getContentType() return String;								//返回视图的内容类型 比如ThymeleafView、JstView返回的就是"text/html;charset=ISO-8859-1" 表明当前View视图返回的是一个HTML页面 比如MappingJackson2JsonView返回的就是"application/json"表明当前视图返回的是JSON数据
render(Map<String, ?>,HttpServletRequest,HttpServletResponse) return void;		//springMVC会回调ViewResolver解析出来的View的render方法.该View就可以通过HttpServletResponse将View中的内容输出到浏览器 Spring是在DispatcherServlet类的render方法里调用具体View的render方法

视图类型			视图类型					简介

URL视图资源图		InternalResourceView			将jsp或其他资源封装成一个视图.被视图解析器InternalResourceViewResolver默认使用 和redirect重定向不同的是,如果Controller的逻辑视图返回forward转发则会由该视图View去处理,它会通过ServletRequest的getRequestDispatcher方法得到RequestDispatcher,然后通过调用RequestDispatcher的forward方法进行转发(详情在springboot中说过,所有的转发的都是靠RequestDispatcher完成的)
			JstView					InternalResourceView的子类.在解析的过程中如果发现JSP使用了JSTL国际化标签就会自动转为该视图
			ThymeleafView				ThymeleafView		

文档视图			AbstractExcelView			Excel文档视图的抽象类
			AbstractPdfView				PDF文档视图的抽象类

报表视图			ConfigurableJasperReportsView		常用的JasperReports报表视图
			JasperReportsHtmlView			同上
			JasperReportsPdfView			同上
			JasperReportsXlsView			同上

Json视图			MappingJackson2JsonView			将数据通过Jackson框架的ObjectMapper对象,以Json方式输出(该视图应该是最特殊的了,因为它不需要输入页面,输出的内容就是JSON 这个JSON就封装在当前对象中)
			MappingJackson2XmlView			将数据通过Jackson框架的ObjectMapper对象,以XML方式输出 这两个视图也许是通过HTTPMessageConveter得到的(这里配合Springboot MappingJackson2HttpMessageConverter食用)

特殊视图			RedirectView  				当逻辑视图中返回redirect也就是重定向时,视图解析器会将其解析为RedirectView(代表重定向视图)

这些模板引擎的作用如下:例如部分JSP、FreeMarker、Thymeleaf语法浏览器本身并不支持,浏览器本身只支持HTML、CSS、JS所以势必要把前端写的灵活的标签解析为浏览器能直接显示的HTML或者拼接视图的前缀和后缀以及对"redirect:"、"forward:"逻辑上的支持(所以视图解析器简单说下来就干这个事)最终再通过调用各种View的实现的render方法将内容输出到浏览器.

另外说明的两个类:
UrlBasedViewResolver:该类继承自AbstractCachingViewResolver 基本上Spring自已实现的视图解析(InternalResourceViewResolver、FreeMarkerViewResolver)都继承了该类 通过该类你就可以指定视图解析器要拼接的前缀和后缀以及当前返回的格式是text/html还是JSON等功能
AbstractCachingViewResolver:Thymeleaf支持缓存的原因就是通过继承AbstractCachingViewResolver来完成的(FreeMarker和JSP同理) 之前一直说的视图解析器需要添加到Spring容器中才能使用 对于Thymeleaf而言它需要通过视图解析器注册器向容器中注册ThymeleafViewResolver 再创建视图解析器之前,需要先从application.yml中对应的配置绑定类中读取当前应用程序是否支持缓存(ThymeleafProperties) 然后再通过调用setCache(boolean)方法设置当前视图解析器的缓存支持设置
setCache(boolean) return void;					//设置当前是否允许缓存

·InterceptorRegistry:顶级类,拦截器注册表,正如上述所说内部封装一个List<InterceptorRegistration>
addInterceptor(HandlerInterceptor) return InterceptorRegistration;		//注册一个HandlerInterceptor拦截器 因为返回值是InterceptorRegistration,所以通过返回值就可以进一步定制该拦截器的 拦截路径和排除路径
getInterceptors() return List<Object>;						//HandlerInterceptor没有Composite综合 但是该方法充当了Composite综合类的功能 该方法内部通过遍历List<InterceptorRegistration>并调用每个元素的getInterceptor()方法得到所有的MappedInterceptor实例并返回

·MappedInterceptor:实现HandlerInterceptor接口 内部封装HandlerInterceptor实例以及该实例对应的 拦截目标地址的List<String>、排除的目标地址List<String> 以及一个AntPathMatcher
matches(String,PathMatcher) return boolean;			//当一个请求过来时Spring需要做的就是回调所有的拦截器 在回调之前spring应该判断当前请求能否被当前某个拦截器拦截 所以就需要调用该方法来匹配拦截器 该方法内部通过AntPathMatcher、和两个List<String>组合实现该功能 如果判断成功了那么该拦截器就会被添加到HandlerExecutionChain(拦截器链)中做进一步操作
那么Spring是如何获取到所有MappedInterceptor的呢?(有了MappedInterceptor就相当于有了所有HandlerInterceptor) 之前说过WebMvcConfigurationSupport会通过@Bean方法回调所有WebMvcConfigurer接口的实现类的addInterceptors方法,它会传入InterceptorRegistry实例 当回调完成后Spring会调用该实例的getInterceptors方法得到所有的MappedInterceptor并返回(会封装为List<Object>)添加到容器中 之后spring就通过BeanFactoryUtils的静态方法获得所有的MappedInterceptor实例

·InterceptorRegistration:拦截器注册,内部组合一个HandlerInterceptor实例、一个List<String>所有被会被拦截的地址、List<String>所有不被拦截的地址
addPathPatterns(String...) return InterceptorRegistration;			//添加当前拦截器要拦截的请求 这里相当于使用了一种Builder模式
excludePathPatterns(String...) return InterceptorRegistration;			//添加当前拦截器要排除的请求 这里相当于使用了一种Builder模式
pathMatcher(PathMatcher) return InterceptorRegistration;			//设置路径匹配器
order(int) return InterceptorRegistration;					//同样可以设置当前拦截器的优先级
getInterceptor() return Object;							//将当前对象内部的HandlerInterceptor实例、拦截目标地址的List<String>、排除的目标地址List<String>封装在MappedInterceptor中并返回

·HandlerInterceptor:顶级接口(拦截器),用于在目标请求执行各个阶段执行命令(该接口和InterceptorRegistry、InterceptorRegistration的关系自已得理清楚)
preHandle(HttpServletRequest,HttpServletResponse,Object) return boolean;			//目标方法执行前,如果该方法返回true则继续执行链,否则终止.
postHandle(HttpServletRequest,HttpServletResponse,Object,ModelAndView) return void;		//执行目标方法后(该方法对被@ResponeBody修饰的Controller作用不大)不过你可以实现ResponseBodyAdvice或者直接在RequestMappingHandlerAdapter上进行配置
afterCompletion(HttpServletRequest,HttpServletResponse,Object,Exception) return void;		//完成完整的请求后(此处应该理解为此时Client已经和服务端没有任何瓜葛时的回调?)

AsyncHandlerInterceptor:该拦截器用于支持Servlet3.0的异步响应
afterConcurrentHandlingStarted(HttpServletRequest,HttpServletResponse,Object) return void;	//在启动异步处理的初始请求时调用该方法(具体也有点懵逼,用在DeferredResult上)


·HandlerExceptionResolverComposite:实现HandlerExceptionResolver、Ordered接口 Composite翻译成综合 如上所说内部封装一个List<HandlerExceptionResolver>
对于configureHandlerExceptionResolvers和extendHandlerExceptionResolvers这两个方法比较特殊,它们没有所有的注册器(Registry),Spring回调的时候是直接传入一个List<HandlerExceptionResolver>.
在WebMvcConfigurationSupport回调configureHandlerExceptionResolvers和extendHandlerExceptionResolvers之间Spring会往List<HandlerExceptionResolver>中添加几个默认的解析器,最后将该List封装到HandlerExceptionResolverComposite中并添加到容器中

·HandlerExceptionResolver:顶级接口 如果在请求到整个Controller层中发生异常(包括Controller外的异常比如参数校验异常等) 则DispatcherServlet将会委托HandlerExceptionResolver链来处理该异常
resolveException(HttpServletRequest,HttpServletResponse,Object,Exception) return ModelAndView;		//args2:执行当前异常的处理程序(可能为null,一般是HandlerMethod实例) 对于返回值,如果是自定义异常处理器就可以自已创建一个ModelAndView 然后Spring会通过视图解析器得到该ModelAndView指向的错误视图 如果该ModelAndView返回null则代表当前异常处理解析器无法处理需要继续执行解析器链

AbstractHandlerExceptionResolver:抽象类实现HandlerExceptionResolver、Ordered(也就是说所有继承当前类的子类的异常处理解析器的优先级是最低的)接口

AbstractHandlerMethodExceptionResolver:抽象类继承自AbstractHandlerExceptionResolver 提供了对HandlerMethod的支持

·HandlerMethod:提供对方法参数、方法返回值、方法注释等便捷访问

·DefaultHandlerExceptionResolver:继承自AbstractHandlerExceptionResolver抽象类 Spring会默认创建该异常处理解析器
该类解决了SpringMVC引发的异常,并将错误映射到响应码

·ResponseStatusExceptionResolver:继承自AbstractHandlerExceptionResolver抽象类,实现MessageSourceAware接口 Spring会默认创建该异常处理解析器
该类完成@ResponseStatus注解表明的响应码(@ResponseStatus注解详情可见springboot入门中的异常)

·ExceptionHandlerExceptionResolver:继承自AbstractHandlerMethodExceptionResolver抽象类,实现ApplicationContextAware、InitializingBean接口 该异常处理解析器Spring会默认创建并添加到容器中HandlerExceptionResolverComposite
该类通过调用被@Controller、@ControllerAdvice注解修饰的类中被@ExceptionHandler注解修饰的方法来解决异常(也就是说@ExceptionHandler注解能生效得益于该类)

SimpleMappingExceptionResolver:继承自AbstractHandlerExceptionResolver Spring不会创建使用

·DefaultErrorAttributes:实现了HandlerExceptionResolver接口 这个类很眼熟(详情见springboot入门异常)


·LocaleResolver:顶级接口 一个应用程序中应该只支持有一个LocaleResolver实例 并且通过@Bean向容器中注册的时候保证方法名是localeResolver 因为spring在使用的时候是根据BeanName来得到容器中的LocaleResolver的 详情请看DispatcherServlet的initLocaleResolver方法
resolveLocale(HttpServletRequest) return Locale;			//对于每一个请求spring都会调用国际化解析器的该方法 传入request得到一个Local 该Local就会作为ViewResolver(视图解析器)resolveViewName方法的入参 详情见DispatcherServlet的render方法 记住所有的国际化都是调用该方法得到的Locale完成的(不管是不是LocalContext接口)
setLocale(HttpServletRequest,HttpServletResponse,Locale) return void;	//对一个请求设置Local 这个方法很关键,有定制化的内容

AcceptHeaderLocaleResolver:实现LocaleResolver接口,在springboot中如果我们没有通过@Bean向容器中注册LocaleResolver(国际化解析器)则springboot的自动配置会帮我们注册该Bean 该Bean内部封装一个Locale defaultLocale(默认的国际化)、List<Locale> supportedLocales(支持的国际化集合) 其中默认的国家化可以application.yml来配置

·Callable<T>:相较于DeferredResult<T>而言Callable更简单,Controller直接创建后直接返回,不需要真的去调用Service层,而是将回调接口返回后SpringMVC帮我们回调该接口 而DeferredResult是需要我们自已维护一个结果的处理线程调用其setResult方法 但是DeferredResult更灵活更麻烦 如果想要设置Callback超时回调,可以使用WebAsyncTask代替(Springboot入门中有写)

·DeferredResult<T>:返回该实例就代表是一层包装,这个泛型应该指代返回值,返回该实例代表会从线程异步返回泛型类型实例对象(该对象的构造器看看就懂了)该方法的返回值就和普通的Controller没什么两样 用于支持servlet3.0异步响应支持 这个类可以用于Controller层的返回(springboot入门有介绍) 
DeferredResult处理过程如下:Controller返回DeferredResult并包装在一个队列中->SpringMVC调用request.startAsync()->DispatcherServlet和所有过滤器退出请求处理线程->应用程序从某个线程(我们自已维护的线程)调用DeferredResult的setResult(),SpringMVC将请求分派回Servlet容器->再次调用DispatcherServlet并使用异步产生的返回值恢复处理(调用getResult得到返回值)但是这一步将返回值写回给浏览器时还是阻塞的!
关于我们自已维护的线程到底该怎么维护?
public DeferredResult<String> getWashHistory() {   
    DeferredResult<String> defResult = new DeferredResult<String>();
    线程池.execute();// 这里通过线程池去执行一个Task(任务)所以最终还是很类似于Callable用一个线程去执行任务 只不过这里执行的时候并不会阻塞(也就是说execute调用完毕后这里应该会立即执行下一行代码即返回DeferredResult) 那么我们只要在这个任务里调用setResult(T)方法就能接上上面说的步骤了 更灵活的地方在于DeferredResult能设置超时回调等等
    return defResult;
}

当请求 /static/style.css 时，返回的头信息中会多两条信息:
Cache-Control:max-age=600, private
Last-Modified:Sun, 04 Oct 2016 15:08:22 GMT
浏览器会将该信息连同资源储存起来，当再次请求该资源时,会取出Last-Modified并添加到在请求头If-Modified-Since中:
If-Modified-Since:Sun, 04 Oct 2016 15:08:22 GMT
Spring MVC 在收到请求,发现存在If-Modified-Since,会提取出来该值,并与资源的修改时间比较,如果发现没有改变,则仅仅返回状态码 304,无需传递资源内容.浏览器收到状态码304,明白资源从上次请求到现在未被改变,http 缓存依旧可用.

如果没有终止信号(没有onComplete和onNext)就是一个无限数据流,不是代表订阅者没有这两个方法,而是发布者不会调用者两个方法.
如果发布者没有发出任何一个数据,而是直接调用完成信号(没有onComplete和onNext),表示这是一个空数据流.数据链还是会先subscribe->request->再按照数据流从上到下
很重要的一个思想,在订阅之前什么都不会发生!!!!
但是一个数据流被订阅者消耗完后并不代表该数据流就没有了,如果这时候再调用subscribe方法设置一个新的订阅者会将数据流从头再消费一遍.(这个叫冷发布者)
一个数据流可以同时被多个订阅者订阅(就相当于观察者模式),如果生产者生产完的消息被所有订阅者消费后,再来一个新的订阅者也不会消费被消费的数据(这个叫热发布者)
在Reactor中数据的生产归生产,消费归消费 生产不必刻意关照消费情况 而且生产的速度和消费的速度不一定就是相同的(只不过在同线程中是这样 但不要被迷惑了) 生成和消费可以是两个线程(通过调度器实现/线程池)
Controller异步调用Service->Service正常处理业务逻辑->Service异步调用DAO

FluxSink.OverflowStrategy:枚举类,回压的几种策略
IGNORE:完全忽略下游回压 当队列积压满时会导致IllegalStateException
ERROR:当下游跟不上节奏的时候发出一个错误信号
DROP:当下游没有准备好接收新的元素的时候抛弃这个元素
LATEST:让下游只得到上游最新的元素
BUFFER:缓存下游没有来得及处理的元素(如果缓存不限大小的可能导致OutOfMemoryError) 该策略相当于队列

public void testTransform() {

	    .map(String::toUpperCase);

    Flux.fromIterable(Arrays.asList("blue", "green", "orange", "purple"))

可变的打包操作符:
public void testCompose() {
    AtomicInteger ai = new AtomicInteger();
    Function<Flux<String>, Flux<String>> filterAndMap = f -> {
        if (ai.incrementAndGet() == 1) {
            return f.filter(color -> !color.equals("orange"))
                    .map(String::toUpperCase);
        }
        return f.filter(color -> !color.equals("purple"))
                .map(String::toUpperCase);
    };

    Flux<String> composedFlux =
            Flux.fromIterable(Arrays.asList("blue", "green", "orange", "purple"))
                    .doOnNext(System.out::println)
                    .compose(filterAndMap);

    composedFlux.subscribe(d -> System.out.println("Subscriber 1 to Composed MapAndFilter :" + d));
    composedFlux.subscribe(d -> System.out.println("Subscriber 2 to Composed MapAndFilter: " + d));
}
这个例子中,filterAndMap函数式有一个名为ai的会自增的状态值.每次调用subscribe方法进行订阅的时候,compose会导致ai自增,从而两次订阅的操作链是不同的.
如果把compose方法换成transform方法则两个订阅者最后打印的内容是一致的.

热生产者:

@Test
public void testHotSequence() {
    UnicastProcessor<String> hotSource = UnicastProcessor.create();

    Flux<String> hotFlux = hotSource.publish()
            .autoConnect()
            .map(String::toUpperCase);


    hotFlux.subscribe(d -> System.out.println("Subscriber 1 to Hot Source: "+d));

    hotSource.onNext("blue");
    hotSource.onNext("green");

    hotFlux.subscribe(d -> System.out.println("Subscriber 2 to Hot Source: "+d));

    hotSource.onNext("orange");
    hotSource.onNext("purple");
    hotSource.onComplete();
}
打印结果:

Subscriber 2 to Hot Source: ORANGE
Subscriber 1 to Hot Source: PURPLE
Subscriber 2 to Hot Source: PURPLE
这个例子中,hotSource发出两个元素后第二个订阅者才开始订阅,所以第二个订阅者只能收到之后的元素,这就是热生产者.

之前说过,如果通过@Component、@Repository、@Service、@Controller,spring就会帮我们去实例化并存放到IOC容器中.这个bean的名称默认就是类名的首字母小写(如果不手动指定bean名称的话),首字母小写是spring默认的生成规则,可以实现BeanNameGenerator接口的方法并提供无参构造器来自定义默认的bean名称的指定规则.

作用域:就是这个bean是单例的还是多利的还是session的还是什么.同样的我们可以自定义自已的作用域,只需要实现ScopeMetadataResolver接口.

除了通过方法切入的方式指定动态代理外,还可以通过@ComponentScan注解指定.通过这种方式可以指定非单例bean的动态代理.

只推荐在@Configuration注解中使用@Bean.@Configuration类不能是final的,它们应该拥有空构造.
@Profile("dev")注解配合@Configuration一起使用,通过里面的value值表明当前被@Configuration修饰的配置类在什么环境下生效.该注解同样可以配合@Bean一起使用,代表在什么环境下才向容器中注册该bean.

通过@Bean注入到容器中的Bean,在bean销毁时会自动回调close()或者shutdown()方法.如果不想spring回调这两个方法通过设置@Bean(destroyMethod="")来避免.实际上destroyMethod应该指代销毁时会调用哪个方法.
