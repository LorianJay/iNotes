1.3配置mybatis文件(config.xml):mybatis的配置文件,用于配置mybatis通过SqlSessionFactoryBuilder构建SqlSessionFactory
<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE configuration
        PUBLIC "-//mybatis.org//DTD Config 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-config.dtd">
<configuration>
    <environments default="development">
        <environment id="development">
            <transactionManager type="JDBC">
	    </transactionManager>
            <dataSource type="POOLED">
                <property name="driver" value="com.mysql.cj.jdbc.Driver"/>
                <property name="url" value="jdbc:mysql://localhost:3306/mybatis?serverTimezone=UTC"/>
                <property name="username" value="root"/>
                <property name="password" value="13218526138"/>
            </dataSource>
        </environment>
    </environments>
    <mappers>
        <mapper resource="xyz/entity/personMapper.xml"/>
    </mappers>
</configuration>


值内容:
default值和id值保持一致	暂时不改
transactionManager type:事务处理的方式	JDBC:手动处理	MANAGED:将事务交由其他组件去托管(spring,jobss)用完后会关闭连接.对于MANAGED这种事务处理器,默认情况下它会关闭连接但是有些容器不希望关闭连接,则可以使用以下配置来阻止此行为.在Spring中不必要配置事务管理器,因为Spring的配置会覆盖该配置(Spring是老大)<transactionManager type="MANAGED"><property name="closeConnection" value="false"/></transactionManager>
dataSource type:默认的连接池	POOLED:打开连接池	UNPOOLED:关闭连接池(不推荐)	JNDI:tomcat自带的

当dataSource的type为UNPOOLED时它提供以下属性配置:
driver					驱动的Java类全限定名(下面的都是和上面对应的)
url					数据库的URL地址
username				数据库的同户名
password 				登录数据库的密码
defaultTransactionIsolationLevel 	默认连接的事务隔离级别
defaultNetworkTimeout 			等待数据库完成的默认网络超时时间(等待数据库的超时时间)
driver.encoding				可选配置,还可以设置编码,该配置应该是作为driver下的子集

POOLED提供了上述属性外很多额外的属性:
poolMaximumActiveConnections 		任意时间存在的可用的连接数量(最多多少个连接同时被使用)默认值为10
poolMaximumIdleConnections 		任意时间可能存在的空闲连接数(个人理解为最大的空闲连接数)
poolMaximumCheckoutTime 		在被强制返回之前池中连接被踢出的实现 默认20000毫秒(20S)
poolTimeToWait				如果连接数据库花费了相当长的时间就打印日志并重新获取一个新的连接(避免一直卡死等待) 默认20000毫秒(20S)
poolMaximumLocalBadConnectionTolerance 	新增于3.4.5 如果从连接池中拿到一个坏的连接,则数据源会尝试重新获取一个新的连接.但这个获取的时间不应该超过poolMaximumIdleConnections与poolMaximumLocalBadConnectionTolerance之和 当前配置的默认值为3
poolPingQuery 				发送到数据库的侦测查询,用来检验连接是否正常工作并准备接受请求.默认是"NO PING QUERY SET"
poolPingEnabled 			是否开启侦测查询,如果开启则需要poolPingQuery属性为一个可执行的SQL语句(最好是一个非常快的SQL语句) 默认值为false
poolPingConnectionsNotUsedFor 		配置poolPingQuery的发送频率 可以设置为和数据库连接超时异常来避免不必要的麻烦 默认为0代表所有连接每一刻都被侦测

JNDI用于EJB或应用服务器这类容器中使用,该选项只有两个配置
initial_context 			用来在InitialContext中寻找上下文 这是个可选属性如果忽略则会直接从InitialContext中寻找data_source属性
data_source				这是引用数据源实例位置的上下文路径.提供了initial_context配置时会在其返回的上下文中进行查找,没有提供时则直接在InitialContext中查找
其余的数据源配置类似,可以通过添加前缀env.*把配置传递给InitialContext,比如env.encoding=UTF8会在InitialContext实例化时往构造方法里添加UTF8的encoding属性


需要更改的四个property name的values值:和jdbc的驱动操作类似
driver后面的value=""		你所使用的数据库的Driver接口,注意不要使用废弃的.
url后面的value=""		选择连接的ip数据库名称端口号,还是看jdbc
username后面的value=""		用户名
password后面的value=""		密码	

resource=""			调用的映射文件的全路径

1.4通过编程的方式创建SqlSessionFactory,在Mybatis中SqlSessionFactory是通过SqlSessionFactoryBuilder创建的,而SqlSessionFactoryBuilder可以通过mybatis-config的XML配置或者编码的方式创建.
·SqlSessionFactoryBuilder:该对象的作用域是局部变量,用于创建SqlSessionFactory 一旦创建了SqlSessionFactory就应该及时废弃该对象不应保留 可以通过该对象创建多个不同的SqlSessionFactory(这样就能实现分库的查询)
build(Reader) return SqlSessionFactory;						//直接从一个Reader对象创建SqlSessionFactory
build(InputStream) return SqlSessionFactory;					//直接从一个InputStream输出流创建SqlSessionFactory对象
build(Configuration) return SqlSessionFactory;					//还可以通过一个Configuration对象创建SqlSessionFactory,用Configuration对象创建的SqlSessionFactory就是这里说的基于编码的方式创建的.
build(Configuration,String) return SqlSessionFactory;				//args1:指定当前选择的环境ID
build(Configuration,String,Properties) return SqlSessionFactory;		//args1:指定当前选择的环境ID args2:指定外部的Properties(就类似于上面的Properties标签)


·TransactionFactory:Mybatis的事务管理器(和Spring的事务管理器不冲突)实际上如果使用Spring可以不配置事务管理器,因为Spring会覆盖该配置
setProperties(Properties) return void;						//回调传入config.xml配置文件中的所有properties配置或者传入SqlSessionFactoryBuilder中build方法提供的Properties实例


JdbcTransactionFactory:直接创建该对象,使用JDBC默认的事务管理器
ManagedTransactionFactory:该事务管理器代表交由别的事务管理器去管理事务(选择该事务管理器就可以通过Spring来管理事务)实际上如果使用Spring可以不配置事务管理器,因为Spring会覆盖该配置

·DataSource:Java原生类,代表数据库原信息对象,包含用户名、密码、URL、连接池等信息 该对象就可以通过JDBC的DriverManager类来得到


·Environment:表示环境的对象,一个环境只包含三个信息(这和上面的XML是对应的)transactionManager和dataSource这两个标签包含在environment中还有一个环境ID
Environment(String,TransactionFactory,DataSource);	//创建环境对象(还可以通过Builder构建器对象创建该对象) args0:指定当前的环境ID(ID的作用和下面的切换多个数据库相似) args1:事务管理器 args2:数据源(包含连接的目标数据库的信息)

·Configuration:Mybatis的配置类 该类配置就相当于拥有Configuration标签下所有子标签的功能
Configuration(Environment);				//根据环境创建Configuration

addMapper(Class<T>) return void;			//该方法可以添加一个映射类(这个映射类就是Mybatis中的Mapper接口)
getTypeAliasRegistry() return TypeAliasRegistry;	//得到类型别名注册器 该类也帮我们注册了(Mybatis框架常用的类别名)
getLogImpl() return Class<? extends Log>;		//得到当前的日志实现
setLogImpl(Class<? extends Log>) return void;		//设置日志实现(Log的实现类)
getLogPrefix() return String;				//得到日志的前缀
setLogPrefix(String) return void;			//设置日志的前缀
setMapUnderscoreToCamelCase(boolean) return void;	//是否开启驼峰命
...还有更多的设置,详情见Javadocs


·TypeAliasRegistry:类型别名注册器,内部通过封装一个Map来实现 该类帮我们注册了很多常用的类型别名(该类主要注册JDK自带的类别名) 在Mybatis中基本类型前面要加_int=int 而int=Integer类
registerAliases(String) return void;			//批量注册别名
registerAlias(Class<?>) return void;			//注册单个类型别名,别名为类型的类名大写 如果通过@Alias注解修饰则用该注解内指定的别名
registerAlias(String,Class<?>) return void;		//注册单个类型别名 agrs0:类型别名(字符串) args1:类型
getTypeAliases() return Map<String, Class<?>>;		//得到这个Map

·TransactionIsolationLevel:Mybatis中的事务隔离级别的枚举类
NONE
READ_COMMITTED
READ_UNCOMMITTED
REPEATABLE_READ
SERIALIZABLE

·Executor:执行器,我们的SQL语句最终是通过Executor+MappedStatement完成的

BaseExecutor:Executor的抽象实现

·ExecutorType:预处理语句的执行策略,每一个策略对应一个Executor的实现(和一个BaseExecutor的子类)
SIMPLE				该策略会为每一个语句的执行创建一个预处理语句,默认执行策略(对应的实现类:SimpleExecutor)
REUSE				该策略会复用预处理语句(对应的实现类:ReuseExecutor)
BATCH				该类型的执行器会批量执行所有更新语句,详情见Mybatis高级13节Executor执行器策略之BATCH(对应的实现类:BatchExecutor)


·SqlSessionFactory:顶级接口 该对象的作用域是应用程序 见名知意用于创建SqlSession的工厂 在运行期间该对象一旦创建就没有理由丢弃它或重新创建一个内容相同的实例
注意SqlSessionFactory是根据SqlSessionFactoryBuilder创建的,而SqlSessionFactoryBuilder封装了Configuration而Configuration又封装了Environment环境对象,Environment环境对象中又封装了DateSource 所以SqlSessionFactory中封装了一个DataSource(这个DataSource是JDBC的接口) 通过SqlSessionFactory创建的SqlSession本质上就是通过DataSource获得的Connection连接
openSession() return SqlSession;				//得到SqlSession 1.不自动提交事务 2.将由当前环境配置(Environment)中的DataSouce获得Connection对象 3.事务隔离级别将会使用驱动或数据源的默认设置 4.预处理语句不会被复用,也不会批量处理更新
openSession(boolean) return SqlSession;				//args0:是否自动提交事务 和第一条的第1点的区别是自动提交事务
openSession(Connection) return SqlSession;			//args0:手动传入的链接 和第一条的第2点的区别是使用自定义连接 Mybatis没有提供同时设置是否自动提交事务和自定义连接的方法,这是因为Mybatis会根据传入的Connection来判断是否要启动自动提交事务
openSession(TransactionIsolationLevel) return SqlSession;	//args0:设置事务隔离级别 和第一条第3点的不同
openSession(ExecutorType) return SqlSession;			//args0:设置预处理语句策略 和第一条第4点的不同
openSession(ExecutorType,boolean) return SqlSession;		//可以同时指定是否自动提交事务以及预处理语句的策略
......

getConfiguration() return Configuration;		//得到该SqlSessionFactory中封装的数据库Configuration配置

·SqlSession:该对象是线程不安全的,不能被共享 所以每个请求(线程)都应该创建一个新的SqlSession 所以该对象的作用域是当前线程(对于Web来说作用域是一次HTTP请求回话)
selectOne(String,Object) return T;			//方法作用:执行一条sql语句

参数1:指定执行那条sql语句.你需要指定personMapper.xml文件中对应的语句.例如:"xyz.entity.personMapper.queryPersonById"
参数2:站位符内容
返回值:返回Object或者resultType定义的映射对象(返回值为泛型)

selectList(String,Object) return List<T>;		//用法类似,返回查询到的所有数据(返回值类型:List)
selectList(String,Object,RowBounds) return List<T>;	//用法类似 args2:略过指定数量的记录,并限制返回结果的数量
select(String,Object,RowBounds,ResultHandler) return void;	//用法类似 args3:ResultHandler参数允许自定义每行结果的处理过程(高级用法详情见Mybatis官方文档)
insert(String,Object)					//参数二:映射文件里该sql语句对应的参数类型(可以简单理解为映射对象)
delete(String,Object)					//参数二:占位符内容
update(String,Object)					//参数二:映射文件里该sql语句对应的参数类型,具体使用看例子


getMapper(Class<T>) return T;				//传入配置好的Mapper接口的Class对象,返回值类型:该接口 最常用方法!!!!!!

flushStatements() return List<BatchResult>;		//将ExecutorType设置为ExecutorType.BATCH时,可以使用这个方法清除(执行)缓存在JDBC驱动类中的批量更新语句

如果当前使用的事务管理器是JDBC(JdbcTransactionFactory)事务管理器,则下面的commit和rollback才会生效,如果在创建SqlSession的时候指定了自动提交事务则下面的两个方法也不会生效(因为自动提交)
commit();						//手动提交事务
rollback() return void;					//回滚事务

clearCache() return void;				//清除本地缓存(清除当前SqlSession的一级缓存)

close()							//关闭连接,在每一次查询完成之后都应该关闭该连接


·RowBounds:配合select语句使用,用于告知Mybtais略过指定数量的记录,并限制返回结果的数量
RowBounds(int,int);					//args0:从哪个数据开始 args1:数据量(拿多少数据量) 该对象一旦创建就无法改变

比如现在select * from student 查询出1000个学生,那么就可以通过创建RowBounds(100,20)来告诉Mybatis筛选查询结果 从查询到的数据的第100位开始取,取20个.它就相当于SQL中的limit查询条件

id				在命名空间中唯一的标识符,可以被用来引用这条语句
parameterType			入参的类型的全限定名 如果查询的时候要用到入参类型的属性,比如根据Person的ID查询数据 则#{field}引用的是类的属性
flushCache			将其设置为true后只要语句被调用,都会导致本地缓存和二级缓存缓存被清空 对于insert、update、delete而言默认值是true会清除缓存,但如果你不想这三条语句清除缓存可以设置为false 详情见19节查询缓存
timeout				在抛出异常前,驱动程序等待数据库返回请求结果的秒数 默认未设置
statementType			设置STATEMENT、PREPARED(默认)、CALLABLE其中之一 这会让Mybatis分别使用Statement、PreparedStatement(默认)、CallableStatement 这三个东西是在JDBC官方API中执行SQL语句的
useGeneratedKeys		仅用于insert和update 如果设置为true则Mybatis会使用JDBC的getGeneratedKeys来生成数据库内部的主键(比如Mysql的主键就不需要我们设置了)
keyProperty			仅用于insert和update 指定能够唯一识别对象的属性,MyBatis会使用getGeneratedKeys的返回值或insert语句的selectKey子元素设置它的值 如果生成列不止一个,可以用逗号分隔多个属性名称
keyColumn			仅用于insert和update 设置生成键值在表中的列名 在某些数据库(像PostgreSQL)中,当主键列不是表中的第一列的时候,是必须设置的.如果生成列不止一个,可以用逗号分隔多个属性名称.
databaseId			详情见14.7Mybatis动态SQL之环境切换
lang				插入脚本语言(详情见14.8)

<insert id="addPerson" useGeneratedKeys="true" keyProperty="id" parameterType="xyz.entity.Person">

4.4.除此之外还可以通过selectKey来指定主键(selectKey顾名思义就是用来指定主键的) 比如上面就可以改成 其实selectKey也可以作为动态SQL的一种
<insert id="addRandomStudent" parameterType="Student">
    <selectKey keyProperty="id" resultType="_int" order="BEFORE">
        SELECT FLOOR((RAND() *1000))
    </selectKey>
    insert into student(id,Name) values (#{id},#{name})
</insert>

其中selectKey标签的意思是帮助当前外部的语句(insert和update)查找主键,所以selectKey标签下存放的是主键生成的语句(生成一个随机主键)该语句执行的结果(输出)将作为底下insert语句的ID主键的输入.
keyProperty			当前selectKey生成的结果应该被设置到的目标属性(比如这里想将结果作为目标的KEY就设置为Key,和占位符匹配) 如果当前selectKey语句返回的属性不止一个则字段之间用逗号,隔开
keyColumn			返回结果集中生成列属性的列名 如果生成列不止一个,可以用逗号分隔多个属性名称	
resultType			当前selectKey生成的结果的类型,一般情况下Mybatis可以推断出来(当然写上更好)
order				可以设置为BEFORE/AFTER 如果设置为BEFORE则会先执行内部的selectKey语句生成主键,如果设置为AFTER则会先执行外部的insert语句再执行selectKey中的语句
statementType			和签名一样 设置STATEMENT、PREPARED(默认)、CALLABLE其中之一

2.自定义类型处理器:实现java类型和JDBC类型的转换
这里举个例子:
在Person类里面有:	boolean		sex	true:代表男	false:女
在Person表里有:		int		sex	1代表男		0代表女

实际上这里就是将java的布尔值转为JDBC的0或1

//1.创建转换器类:
两种方法:
a.实现TypeHandler接口
b.继承BaseTypeHandler类(比较简单)

采用BaseTypeHandler类:
@MappedJdbcTypes(JdbcType.INTEGER)						//该注解可以指定当前类型转换器目标数据库的类型为Integer类型(该注解可以省略不写,通过XML配置文件中的标签指定)
public class BooleanAndIntegerConterver extends BaseTypeHandler<Boolean> {	//这里的泛型就是java的类型
    /*
    set:从java到数据库(boolean->int)
    get:从数据库到java
     */

    /**
     * @param preparedStatement:PreparedStatement对象
     * @param i:PreparedStatement对象操作参数的位置
     * @param aBoolean:java的类型
     * @param jdbcType:JDBC(数据库)的类型
     * @throws SQLException
     */
    @Override
    public void setNonNullParameter(PreparedStatement preparedStatement, int i, Boolean aBoolean, JdbcType jdbcType) throws SQLException {
        if (aBoolean) {
            preparedStatement.setInt(i, 1);
        } else {
            preparedStatement.setInt(i, 0);
        }
    }

    /*
    提供了三种拿值的方式:
    1.通过列名
    2.通过下标(索引)
    3.通过存储函数
     */
    /**
     *通过列名
     * @param resultSet:从数据库返回的数据(ResultSet结果集)
     * @param s:字段名称
     * @return
     * @throws SQLException
     */
    @Override
    public Boolean getNullableResult(ResultSet resultSet, String s) throws SQLException {
        return resultSet.getInt(s) == 1 ? true : false;
    }

    /**
     * 通过下标(索引)
     * @param resultSet:从数据库返回的数据(ResultSet结果集)
     * @param i:字段位置(索引)
     * @return
     * @throws SQLException
     */
    @Override
    public Boolean getNullableResult(ResultSet resultSet, int i) throws SQLException {
        return resultSet.getInt(i) == 1 ? true : false;
    }

    /**
     *通过存储函数
     * @param callableStatement:从数据库返回的数据(ResultSet结果集)
     * @param i:字段位置(索引)
     * @return
     * @throws SQLException
     */
    @Override
    public Boolean getNullableResult(CallableStatement callableStatement, int i) throws SQLException {
        return callableStatement.getInt(i) == 1 ? true : false;
    }

}

总结:管他那么多干什么,占位符和JavaBean属性全部一一对应即可. 

1.#{}于${}的区别之 当输入类型为基本类型+String类型时的区别:

select * from person where id= #{id}这条语句的占位符可以有两种情况:
#{}和$的区别:
a.
#{}	其中括号可以放任意值
${}	其中括号只能放字段名称(id)

b.
#{}自动给String类型加上''	(自动类型转换)
${}原样输出,适合于动态排序	(动态字段)	如果你要放一个String你就需要:select... name='${name}'

动态查询的例子:比如现在要动态的根据某个字段进行查询
<select id="findByColumn" resultType="Student">
   select * from student where ${column} = #{value}
</select>
现在要根据哪个字段进行查询是不确定的,那么在调用该语句的时候就可以把${column}占位符替换为具体的表字段,而表字段一定是String类型的(不过下面的例子是不安全的,会造成SQL注入的风险)
#{column} 用这种形式对于String类型Mybatis会自动添加''则语句变为 select * from student where 'id' = '1';(结果是查询不到)
${column} 而用这种形式会不会对输入的占位符做任何操作 则语句变为 select * from student where id = '1';(数据库查询成功)


c.
#{}可以防止sql注入
${}不可以

#{}和$的相同之处:
a.都可以获取对象的值(嵌套类型对象(组合对象))

模糊查询的例子:
	1:
    <select id="queryPersonFuzzy" parameterType="Person" resultMap="personResult">
        select * from person where id= #{id} or name like #{name}
    </select>    

	2:
    <select id="queryPersonFuzzy" parameterType="Person" resultMap="personResult">
        select * from person where id= #{id} or name like '%${name}%'
    </select>
	第一种情况:传入的Person类应该是:new Person(1,"%张%",null,null);
	此时sql语句变为:select * from person where id=1 or name like '%张%'
	第二种情况:传入的Person类应该是:new Person(1,"张",null,null);
	此时sql语句:select * from person where id=1 or name like '%张%'
	3:这个和#{}${}无关,使用bind标签
    <select id="queryLike" parameterType="Student" resultType="Student">
        select * from student where
            <bind name="queryName" value="'%'+name+'%'"/>
            <if test="name!=null ">
                name like #{queryName}
            </if>
    </select>
	第三种情况:传入的值必须是javabean对象(和返回值要对应)
	bind标签:给传入的值添加一些内容	value=""就是将传入的值添加百分号变为:	%name%	name=""	修改后的值的字段名和if标签里面括号的值对	应
	

结果:虽然sql语句最终相同,但是第一种情况的模糊查询要比第二种模糊查询更不方便

<select id="selectStudentAndTeacherByStudentID" parameterType="_int" resultMap="selectStudentAndTeacherByStudentIDResultMap">

</resultMap>

4.关联查询的多结果集(解决1+N查询):从Mybatis3.2.3开始Mybatis提供了另一种解决1+N查询的方法
实际上这种方法就是通过SQL的存储过程来实现的,首先看存储过程(该存储过程返回两个结果,一定得是两个结果的)
CREATE DEFINER=`root`@`localhost` PROCEDURE `selectStudentAndTeacher`(in student_id int)	//该存储过程需要一个student_id
BEGIN
SELECT * FROM student WHERE id = student_id;			//由于这里不会写SQL语句,实际上这里应该将第一次查询到的Student的teacher_id保存下来,作为下一行查询的入参
SELECT * FROM teacher WHERE id = teacher_id;
END

<select id="selectStudentAndTeacherByStudentIDWithFunction" resultSets="students,teachers" resultMap="studentAndTeacherResultMap" statementType="CALLABLE">
    {call selectStudentAndTeacher(#{id,jdbcType=INTEGER,mode=IN})}			//[id]:id这个字面值是和Mapper接口中方法参数对应的,和存储函数的输入属性名无关,但和存储函数也是对应的 JDBCType:指定该字段在存储函数中的类型(和上面存储函数对应) mode:指定模式(和上面存储函数对应,上面写的就是IN,区分大小写)
</select>

resultSets:上面说该存储过程会返回两个结果,那么用students代表第一个select查询返回的结果,用teachers代表第二个select返回的结果(resultSets为每一个结果集指定一个名称,多个名称使用逗号分开)
statementType:当运行存储过程时该值得是CALLABLE

<resultMap id="studentAndTeacherResultMap" type="Student">
    <id property="id" column="id"/>
    <result property="name" column="name"/>
    <result property="teacher_id" column="teacher_id"/>
    <association property="teacher" javaType="Teacher" resultSet="teachers" column="id" foreignColumn="id">
        <id property="id" column="id"/>
        <result property="name" column="name"/>
    </association>
</resultMap>
resultSet:指定当前嵌套结果集时和上面指定的resultSets的哪个对应
javaType:指定该结果集映射在JavaBean类型的全限定名
column:有待商榷,可能是指定Teacher表的主键也有可能是指定Student表的主键
foreignColumn:有待商榷,可能是指定Teacher表的主键

鉴别器:对查询结果进行分支处理(当某一种情况时查询一种结果,当另外一种情况时查询另外的结果,有点类似于if标签.)

14.Mybatis实现动态Sql:
对于一条sql语句:select * from person where name=#{name} and age=#{age};
去实现根据不同的情况而执行不同的语句:
如果用户传入了name那么我就执行:select * from person where name=#{name};
如果用户只传入了age那么就执行:select * from person where age=#{age};
如果两个参数都有则:select * from person where name=#{name} and age={age};

动态SQL和鉴别器的区别在于:鉴别器是根据不同的查询结果选择不同的字段映射策略,而动态SQL是根据不同的SQL入参执行不同的SQL语句

<select id="selectAllStudentWithSqlArea" resultType="Student">

4.<choose>标签
choose标签有点类似Java的switch关键字,现在考虑一个问题,对于Student而言如果有StudentID就根据StudentID进行查询,如果入参没有StudentID但有StudentName就根据StudentName查,如果都没有就给一个兜底的查询(这就很类似于Java中的switch,一旦某个条件满足就可以break,如果所有条件都不满足还有个default)

choose标签的子标签:
when					这就相当于一个case
otherwise				这就相当于最后的的default

<select id="" resultMap="">
    select * from student where teacher_id = 3
    <choose>
	<when test="id != null">		//多个when条件,相当于case一旦碰到一个判断成功则后续的所有when都不会执行
	    and id = #{id}
	</when>
	<when test="name != null">
	    and name = #{name}
	</when>
	<when test="xxx != null">
	    and xxx = #{xxx}
	</when>
	<otherwise>				//相当于default,如果所有的when都没有判断成功,则最终执行该otherwise的语句
	    and xxx = xxx
	</otherwise>
    </choose>
</select>


5.<where>、<trim>、<set>标签

<select id="" resultMap="">
    select * from student where
    <if test="id != null">
        id = #{id}
    </if>
    <if test="name != null">
        and name = #{name}
    </if>
</select>
假设我们这里把14.3中的select语句中的id也改为动态的条件.那么此时就会出现很多问题 如果<if>标签没有匹配的条件 则这条语句会变为select * from student where 这本身就是一条错误的语句 如果只匹配了第二个条件则查询语句变为select * from student where and name = #{name} 也是条错误的语句
为了解决这个问题,下面使用<where>标签

<select id="" resultMap="">
    select * from student
    <where>
        <if test="id != null">
            id = #{id}
        </if>
        <if test="name != null">
            and name = #{name}
        </if>
    </where>
</select>

<where>标签就对应SQL语句的where判断条件,在<where>标签下如果有一个<if>标签触发成功了,那么才会在原本的SQL语句上拼接where,否则不会拼接(如果这里<where>下没有一个匹配则查询语句为select * from student)
另外在<where>标签中如果第一个判断成功的<if>标签中以and/or开头则<where>标签会自动将它去除

同样对于update语句也有这样的困扰,但使用<set>标签就能做到类似<where>标签的功能,效果如下:

<select id="" resultMap="">
    update student 
    <set>
        <if test="name !=null">name = #{name}</if>
        <if test="teacher_id !=null">teacher_id = #{teacher_id}</if>
    </set>
</select>

<set>标签对应SQL语句的set判断条件,在<set>标签下如果有一个<if>标签触发成功了,那么才会在原本的SQL语句上拼接set 并且对于set标签Mybatis还会删除额外的逗号,(保证是正确的SQL语句)

<trim>标签比where、set标签更加灵活,上述的<where>标签的等价<trim>标签的写法如下:
<select id="" resultMap="">
    select * from student
    <trim prefix="where" prefixOverrides="AND |OR ">
        <if test="id != null">
            id = #{id}
        </if>
        <if test="name != null">
            and name = #{name}
        </if>
    </trim>
</select>

上述的<set>标签的等价<trim>标签的写法如下:
<select id="" resultMap="">
    update student
    <trim prefix="SET" suffixOverrides=",">
        <if test="name !=null">name = #{name}</if>
        <if test="teacher_id !=null">teacher_id = #{teacher_id}</if>
    </trim>
</select>

trim标签下的标签
prefix					一旦当前<trim>标签下有一个<if>标签触发成功了,则在原本的语句SQL语句上拼接该值(这里prefix指定为where)
prefixOverrides				在<trim>标签下如果第一个判断成功的<if>标签中有以该标签指定的值开头的话 则<trim>标签会自动去除它 每个值之间用空格+管道符 |分割(注意空格是必须的)
suffix					
prefixOverrides				在<trim>标签下如果[最后一个]判断成功的<if>标签中有以该标签指定的值[结尾]的话(注意和上面的区别)

6.<foreach>标签(用于遍历集合)
比如:select * from person where age in (18,20);
括号里面的内容可是是:数组、对象数组、容器、对象的属性(该属性就是一个容器,只不过在一个对象里)

·对于数组而言:
执行:select * from person where age in (#{age});
//1.配置Mapper文件(映射文件):

        select * from person
        <where>
            <if test="array!=null and array.length>0">
                <foreach collection="array" open="age in(" close=")" item="age" separator=",">
                    #{age}
                </foreach>
            </if>
        </where>
    </select>


parameterType="int[]"	传入数组类型,一定要加[]

主语句不变:
select * from person

7.数据库环境切换
介绍:之前说到可以在config文件中通过environments default标签来切换数据库环境,但问题是数据库之间包含差异,有些在mysql中定义的sql语句到Oracle中就不能使用,所以我么可以在config文件中定义一个标签,这个标签可以使得Mapper文件知道我们当前使用的哪个数据库,然后再在Mapper文件中根据当前的数据库去执行当前数据库对应的sql语句.
这里比较复杂和神奇,Mybatis帮我们干了很多事情,首先我们引用了0.1中的配置 指定了MySQL的别名为mysql Oracle的别名为oracle.并且假设我们在environment中指定了两个连接环境一个是MySQL一个是Oracle 并且在environments default指定当前环境是MySQL 此时Mybatis会推断出当前数据库连接的环境就是MySQL,此时我们又在Mapper.xml文件中指定如下两个语句.

<select id="selectAllStudent" resultType="Student" databaseId="mysql">
    select * from student
</select>
<select id="selectAllStudent" resultType="Student" databaseId="oracle">
    select * from student
</select>

databaseID="..."			//指定当前SQL语句的ID
也许会觉得很神奇,为什么这两个语句的id重名了?但是由于当前连接的环境Mybatis已经推断出来是MySQL环境了,那么此时我们通过Mapper接口调用selectAllStudent方法时调用的就是上面的MySQL语句
那现在再通过environments default将环境切换到Oracle(oracle) Mybatis会自动推断出环境为oracle此时再执行就会运行下面的语句 我们仅仅通过整体的环境切换就能切换数据库环境(这是Mybatis帮我们推导的)

还可以配合内置变量(_databaseId)来实现动态SQL,例如:

<select id="selectAllStudent" resultType="Student">
    select * from student
    <if test="_databaseId == 'mysql'">
        where id = #{id}
    </if>
    <if test="_databaseId == 'oracle'">
        where teacher_id = #{id}
    </if>
</select>

在不同的环境下<if>标签得到的结果也是不同的

19.查询缓存:
1.一级缓存+开启的细节
2.二级缓存的共享
3.第三方提供的二级缓存
4.二级缓存的缓存引用

1.一级缓存:
当我们第一次查询(比如查询张三)需要访问数据库,查询的结果会放入缓存(保存在sqlSession对象里),如果之后你还用同一个sqlSession对象来找张三就不需要访问数据库,直接从缓存中获得.
直到运行了commit(提交事务)会清除所有缓存对象,将张三这条信息从sqlSession中移除.

如果是同一个sqlSession对象进行多次相同的查询,则直接进入一级缓存.否则如果有二级缓存则进入二级缓存,所以二级缓存是针对sqlSession是不同的对象而言的.

因为一级缓存是针对单个sqlSession对象而言的,如果要清除一级缓存就调用对应的sqlSession对象的clearCache方法

2.二级缓存的共享:mybatis默认没有开启二级缓存,需要手动开启
mybatis自带二级缓存:(同一个namespace)生成的mapper对象共享缓存

如果是同一个sqlSession对象进行多次相同的查询,则直接进入一级缓存.

为什么说namespace生成对象,namespace是Mapper文件的一个标签,该标签拥有Mapper(操作mybatis的接口)接口的引用,标签的值就是接口的包路径.
一个namespace对应一个接口.
PersonMapper personMapper = sqlSession.getMapper(PersonMapper.class);
Person person1 = personMapper.queryPersonByIdWithConverter(1);
我们在代码里面是根据Mapper接口来生成对象的,意思也就是说通过同一个接口对象生成出来的对象(personMapper)是共享二级缓存的.

结论:只要产生的对象来自同一个namespace,则这些对象共享二级缓存(所以缓存的开始是针对mapper.xml文件的,也同时针对一个对于的Mapper接口,因为它们的命名空间相同)

如果:
PersonMapper personMapper1 = sqlSession.getMapper(PersonMapper.class);
PersonMapper personMapper2 = sqlSession.getMapper(PersonMapper.class);

此时personMapper1和personMapper2依然是共享二级缓存的.而且就算不是同一个sqlSession,他们只要来自同一个namespace就还是共享二级缓存的,所以能不能共享二级缓存,你只需要看他们是不是同一个namespace.

·手动开启二级缓存:

//1.在mybatis文件中设置二级缓存开启:
<settings>
    <setting name="logImpl" value="LOG4J"/>
    <!--开启延迟加载-->(详情见18节延迟加载)
    <setting name="lazyLoadingEnabled" value="true"/>

    <!--关闭立即加载-->(详情见18节延迟加载)
    <setting name="aggressiveLazyLoading" value="false"/>
    <!--开启缓存-->
    <setting name="cacheEnabled" value="true"/>
</settings>

//2.在你需要开启二级缓存的namespace文件(Mapper.xml)中确认该namespace开启二级缓存:
<cache/>		添加这么一个标签,但是添加该标签之后是一个Mapper文件的全局标签(所以缓存的开始是针对mapper.xml文件的,也同时针对一个Mapper接口)
该标签做了以下几件事
映射语句文件中的所有select语句的结果将会被缓存。
映射语句文件中的所有insert、update和delete语句会刷新缓存
缓存会使用最近最少使用算法(LRU)算法来清除不需要的缓存
缓存不会定时进行刷新(也就是说，没有刷新间隔)
缓存会保存列表或对象(无论查询方法返回哪种)的1024个引用
缓存会被视为读/写缓存,这意味着获取到的对象并不是共享的,可以安全地被调用者修改,而不干扰其他调用者或线程所做的潜在修改.


cache标签下的标签:
type					
eviction				可用的清除缓存策略 LRU(默认,最长时间不被使用的对象)、FIFO(先进先出 最先被缓存的对象最先被清除,有点类似队列)、SOFT(软引用,基于垃圾回收器的规则和引用移除规则移除对象)、WEAK(弱引用,更积极地基于垃圾回收器状态和弱引用规则移除对象)
flushInterval				刷新缓存的间隔时间单位毫秒,该值可以设置为任意的正整数,默认情况下不设置,缓存仅仅在调用语句时刷新
size					缓存的对象个数,可以设置为任意正整数,要注意预缓存对象和运行环境中可用的内存资源,默认值是1024
readOnly				设置值为true/false,是否只读(默认为false),如果是只读true则每次返回给调用者的都是相同的实例,因此这些对象不能被修改,性能会提高.如果是读写false则会通过序列化返回缓存的拷贝对象,速度上会慢一些但更安全.(详情就在下面//3.二次缓存说到最后....) 虽然是这样但Mybatis在后续又说不要对Mybatis返回的对象作出更改,以防后患.
blocking

·解释为什么需要close?
根据序列化的知识可知,实际上二级缓存是存放在硬盘文件里的,是通过IO放到硬盘里的,但是由于IO也很费性能,当我们读取到数据之后不一定就放到硬盘里面,只有当你close的时候才放到硬盘里面.实际上close就是将一级缓存里面的查询到的内容复制一份放到二级缓存里,只有当close才放是因为一次性放入节省IO性能.

·关于日志文件的0.0和0.5的意思:
这里查询了两次:
第一次:从数据库中查询(比如查到张三),查询后通过close放到二级缓存里面此时由于二级缓存里面没有张三,所以二级缓存命中率为0%
第二次:从二级缓存中查询(必须是查询张三),由于二级缓存已经有了数据,所以命中了数据一共查询两次,一次命中就是1/2	命中率50%(0.5)

·清除二级缓存:
有两种方式:
a.通过commit
b.通过标签

a.通过commit:
和一级缓存一样都是通过commit清除缓存的.
commit清除缓存的原因:防止脏数据

代码分析:
sqlSessionSelect1 = sessionFactory.openSession();
PersonMapper personMapper1 = sqlSessionSelect1.getMapper(PersonMapper.class);
Person person1 = personMapper1.queryPersonByIdWithConverter(1);

if(sqlSessionSelect1!=null){
    sqlSessionSelect1.close();		//通过close开启二级缓存
}

sqlSessionInsert = sessionFactory.openSession();
PersonMapper personMapper = sqlSessionInsert.getMapper(PersonMapper.class);
Person person = new Person(5,"王宝强",35,true);
personMapper.addPersonWithConverter(person);
sqlSessionInsert.commit();
if(sqlSessionInsert!=null){
    sqlSessionInsert.close();
}

sqlSessionSelect2 = sessionFactory.openSession();
PersonMapper personMapper2 = sqlSessionSelect2.getMapper(PersonMapper.class);
Person person2 = personMapper2.queryPersonByIdWithConverter(1);
if(sqlSessionSelect2!=null){
    sqlSessionSelect2.close();
}

分析可知:commit必须要作用到数据库上才会清除二级缓存,如果你是select语句,select本身不需要commit,所以如果你通过selectcommit是不会清楚二级缓存的.这里的代码:先查询的一次,然后通过一个新的insert语句插入一条数据之后通过commit来清除二级缓存(实际上也就是上面说的通过delete、update、insert来清除缓存)


b.通过标签:
在你需要取消二级缓存的查询语句里添加flushCache标签:

<select id="queryPersonByIdWithConverter" parameterType="int" resultMap="personResult" flushCache="true" useCache="true">
    select * from person where id= #{id}
</select>

之后你在执行该sql语句的时候会自动清除二级缓存.

4.二级缓存的缓存引用

我们在19.2节得出的缓存共享的结论是:只要产生的对象来自同一个namespace,则这些对象共享二级缓存(所以缓存的开始是针对mapper.xml文件的,也同时针对一个对于的Mapper接口,因为它们的命名空间相同)
所以说如果两个Mapper(mapper.xml)它们的命名空间不同,那么它们就不会共享缓存,如果我希望两个不同的namespace之间也共享缓存,可以使用cache-ref(缓存引用标签)
<cache-ref namespace="..."/>
namespace:指定要引用的缓存的命名空间(namespace),此时被引用的Mapper的缓存当前Mapper也能使用了!

7.增加null:
·解释:如果你在数据库中插入一行记录其中某一个字段的值为null

oracle:如果插入的字段是null,提示错误:Other 而不是null

mysql:如果插入的字段是null,可以正常执行,其值就为null

原因:
各个数据库 在mybatis中对各种数据类型的默认值不一样
mybatis中,jdbcTypeForNull(如果是null),则默认值是Other.对于Other来说,mysql能够处理Null,但是Oracle不行

解决:
对于oracle:声明Other类型就是null
a.修改具体的Mapper标签:
假如这是一个Oracle的语句:

11.mybatis架构:
·分层:
mybatis大概分为4层分别是:接口层、数据处理层、框架支撑层、引导层


接口层(广义接口):在Mapper接口中定义的方法和测试代码(Demo)中的各种build、openSession方法.对外暴露的

数据处理层:参数处理	SQL解析		SQL执行		处理结果
					(Executor)
在这个过程中每一步都对应一个对象

框架支撑层:事务管理、缓存机制、连接池管理(各种辅助功能)	很多都是配置在config文件中的

引导层:SQL语句方式:XML配置	接口注解		(前面说过执行sql语句有两种方式的)

·分析源码:
主要研究的代码:
Reader reader = Resources.getResourceAsReader("me/config.xml");				1

SqlSessionFactory sessionFactory = new SqlSessionFactoryBuilder().build(reader);	2

SqlSession sqlSession = sessionFactory.openSession();					3

IMapper iMapper = sqlSession.getMapper(IMapper.class);					4

iMapper.xxx();										5
第一步是服务于第二步的,重点研究第二步.

第二步详解:
首先mybatis底层是将config.xml文件(和mapper.xml文件)的所有信息保存到Configuration文件中的.
调用build方法,将字符流传入,通过这个字符流mybatis有一个XMLConfigBuilder是专门解析xml文件的,用解析出来的对象(XMLConfigBuilder)调用parse方法,parse方法可以将解析出来的内容保存到一个Properties对象(文件)中.
在解析过程中会解析config文件的所有标签,也包括mapper文件(映射文件)的解析,由于mapper.xml文件的路径是需要在config文件中定义的,所以解析config的时候就可以直接解析mapper文件.mapper文件的解析最具代表性的也就是增删改查
找到了mapper文件的路径之后再用parse方法对mapper进行解析,解析完了之后调用MapperBuilderAssistant类的addMappedStatement方法将刚才解析到的所有内容传入,返回一个MappedStatement对象.也就是mybatis会将Mapper.xml中的增删改查的标签解析成MappedStatement对象(所以MappedStatement拥有Mapper.xml文件中SQL语句的所有定义信息)
由于是在config中解析mapper标签,然后再调用mapper专门的解析方法,之前说过config.xml文件的所有信息保存到Configuration,而MappedStatement对象又是保存mapper文件的,所以MappedStatement存在于Configuration对象中.

前面说过调用parse方法可以解析config和mapper文件,将解析的结果存放在Configuration对象中,将这个对象返回作为形参放入build方法里,这个方法返回
SqlSessionFactory接口的实现类DefaultSqlSessionFactory,此时经过上面一大堆的步骤之后,终于将DefaultSqlSessionFactory类返回出去,创建了SqlSessionFactory类.SqlSessionFactory-->实现类DefaultSqlSessionFactory-->包含了Configuration类,所以SqlSessionFactory拥有一切内容.

第三步详解:
openSession方法的本质是调用DefaultSqlSessionFactory实现类的实现方法,这些方法最终都指向了一个私有的openSessionFromDataSource()方法,在该私有方法中
通过configuration.newExecutor(tx, execType);创建执行器	-->默认执行器是SimpleExecutor(ExecutorType.SIMPLE)

根据不同的类型ExecutorType(执行器策略,关于执行策略详情见SqlSessionFactory),产生不同的Executor,并且会对执行器产生拦截操作.
Executor executor = this.configuration.newExecutor(tx, execType);
通过装饰模式,将刚才产生的executor包装成一个更强大的executor.这个拦截器是一个链的结构,每一个链都会增强一次执行器.
这个执行器就可以执行增删改查了,通过DefaultSqlSession(configuration, executor, autoCommit(事务))构造方法,创建一个DefaultSqlSession,并返回出去,DefaultSqlSession就是SqlSession的实现类.openSession方法最后返回一个DefaultSqlSession.

第四步详解:
获取接口的class文件(可以使用反射)
包含SqlSession(SqlSession包含Configuration、executor、事务)、代理接口的对象(MapperInterface)、methodCache(存放查询缓存,底层是CurrentHashMap(解决高并发问题,线程问题))

第五步详解:
执行Mapper接口的方法时被MapperProxy类的invoke方法拦截了,MapperProxy实现自InvocationHandler(动态代理接口),用到了代理对象,所以实际上步骤4获取到的代理对象就是MapperProxy对象.代理对象帮我们代理执行增删改查的方法(代理模式)

通过mapperMethod.execute(this.sqlSession, args)实际调用增删改查的方法.依靠了SqlSession中的Configuration和Executor...

通过convertArgsToSqlCommandParam()方法处理增删改查的参数,如果参数是0个,则返回null.如果参数是1个,则返回第一个.如果是多个参数,则放入map中.

查询方法:selectOne又调用selectList,查询多个但只要第一个.通过Configuration.getMappedStatement()即用于获取到用于增删改查的对象.

通过boundSql将我们写的SQL和参数值进行了拼接后的对象,这个boundSql就是正真的sql语句.可以被mysql执行的语句.

mybatis使用的是jdbc的PreparedStatement,底层执行增删改查通过PreparedStatement的execute方法.

mybatis底层在执行CRUD可能会涉及到的四个处理器:
StatmentHandler:对象PreparedStatement处理器
ParamenterHandler:参数处理器--->TypeHandler类型处理器
ResultSetHandler:结果集处理器


整体梳理:
config.xml和mapper.xml文件被加载存放到-->
Configuration对象中,这个对象被包含于SqlSessionFactory对象中,SqlSessionFactory实际上是DefaultSqlSessionFactory实现类-->
通过SqlSessionFactory获得SqlSession,实际上SqlSession是DefaultSqlSession,而DefaultSqlSession又包含Configuration、Executor-->
通过SqlSession就可以执行sql语句了,实际上是通过Executor执行器执行-->
Executor执行增删改查的标签,增删改查的底层实际上就是执行MappedStatement-->
通过MappedStatement对象执行时还需要输入参数 	输入参数:javabean/int/HashMap	
输入完参数之后,再到数据库里面执行			返回值:javabean/int/HashMap/List


你有没有想过为什么Mapper接口能和Mapper.xml文件对应,并且为什么Mapper接口的全限定名要和Mapper.xml中的命名空间对应?
因为在Mybatis中不管是通过调用SqlSession的selectOne或者是调用SqlSession的getMapper返回的Mapper接口去进行查询,最终真正能执行SQL语句的还得是MappedStatement对象.
所以我们通过selectOne指定的mapper.xml的命名空间和通过Mapper接口查询的目的就是想通过一个Key来找到其唯一的一个MappedStatement对象.
也就是说在Mybatis初始化的时候,维护一个Map Map的Key就是命名空间/类全限定名(能理解为什么需要对应了?因为Key得对应上)然后我们调用getMapper方法时,Mybatis就会根据传入类的类全限定名来找到唯一的MappedStatement对象(当然最后返回的还是MapperProxy代理对象)

四个处理:StatmentHandler、ParamenterHandler、ResultSetHandler、TypeHandler(类型转换器,详情见Mybatis基础第4节)
四大核心对象:StatmentHandler、ParamenterHandler、ResultSetHandler、Executor(执行器,详情见Mybatis基础1.4)

四大核心对象的共同特点:1.都涉及到了 拦截器 用于增强 2.都包含了该增强操作

·自定义插件的编写逻辑:根据mybatis规则编写一个拦截器,在拦截器内部加入自定义增强功能

//1.编写拦截器:需要实现Interceptor类

public class MyInterceptor implements Interceptor {
    /**
     * 当我们在测试代码中调用接口方法时,接口方法会去寻找Mapper文件的过程中被拦截
     * @param invocation
     * @return
     * @throws Throwable
     */
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        /*
        invocation.proceed():可以理解为拦截结束
         */
	System.out.println("拦截方法");
        Object proceed = invocation.proceed();		//拦截结束
        return proceed;
    }

    /**
     * 插件方法 该方法由Mybatis回调,会在Configuration的newExecutor、newParameterHandler、newResultSetHandler、newStatementHandler中回调InterceptorChain的pluginAll方法
     * 而pluginAll方法会回调所有拦截器的plugin,Mybatis约定必须在plugin方法中调用Plugin.wrap(o, this)的形式(plugin的默认方法也是调用Plugin.wrap)然后将wrap方法的返回值返回给plugin方法(实际上wrap方法最终返回的对象是JDK的代理对象,通过Proxy.newProxyInstance方法创建.代理角色就是Plugin,Plugin实现了InvocationHandler接口)
     * 所以在Mybatis中是插件嵌套插件的形式(或者说是嵌套代理)所以说plugin方法最终返回的是代理对象,pluginAll方法会将warp方法返回的代理对象作为入参再次传入plugin(wrap)方法中)
     * 所以当我们想执行任何SQL语句的时候,说过不止一次,在Mybatis中都是通过Executor去执行的,而现在我们又对Executor执行器进行了代理,当我们执行Executor这个真实角色的方法时,首先会调用第一层代理方法,在调用代理方法之前由于该代理方法又被代理了,所以会先调用第二层代理方法(我代理了你的代理)无限套娃
     * 那么wrap方法干了什么呢?
     * 首先获取到拦截器上是否有@Intercepts注解以及@Signature注解内的值做进一步操作,最终把当前拦截器的所有信息以及上一个Plugin实例封装到一个Plugin实例中.Plugin就是代理角色
     * 那么Plugin这个代理角色实例中的代理方法干了什么?
     * 很简单就是把当前Plugin实例封装的三个元素封装到Invocation中,即当前Plugin代理的目标对象、当前真实角色调用的方法Method对象、真实角色调用方法传入的参数(后面两个参数就是Plugin实例实现的invoke方法的后两个参数)
     * 
     * 作用:将拦截器中定义的增强功能和原来的核心对象(Executor)合并起来成为最终的核心对象(当我们调用DefaultSqlSessionFactory的openSession时会增强Executor这个核心对象)
     * 
     * @param o 这里回调传入的Object实际上就是运行期间创建根据ExecutorType枚举值创建的Executor执行器对象
     * @return
     */
    @Override
    public Object plugin(Object o) {
        /*
        wrap():
        参数一:你需要增强的四大核心对象
        参数二:当前的拦截器,就是this
        返回值:增强之后的产物
         */
        Object wrap = Plugin.wrap(o, this);
	System.out.println("plugin"+wrap);
        return wrap;
    }

    @Override
    public void setProperties(Properties properties) {				//该方法也可以用来配置ObjectFactory工厂,回调方法 Mybatis会把我们的properties对象传入
	System.out.println("设置属性"+properties);
    }
}

//2.添加注解:
@Intercepts({
        @Signature(type = StatementHandler.class,method = "query", args = {Statement.class, ResultHandler.class})
})
public class MyInterceptor implements Interceptor {
.
.
.
}

解释:
@Intercepts({	//告知当前类是一个拦截器
        @Signature(type = StatementHandler.class,method = "query", args = {Statement.class, ResultHandler.class})
})

@Signature:
type:具体拦截哪个对象(四大核心对象之一)的class文件
method:具体拦截type对应对象的哪个方法,这里拦截的是query(查询)方法
args:拦截的方法的参数,有点时候万一有重载方法既可以通过这种方式来指定具体方法,参数类型是class.注解类型是class

Invocation:
getTarget() return Object;				//得到当前被代理的目标对象
getMethod() return Method;				//得到当前真实角色执行的方法
getArgs() return Object[];				//得到真实角色调用目标方法传入的参数
proceed() return Object;				//执行目标方法(关于这个目标方法具体指代哪个方法,对于第一个代理(Plugin)而言目标方法就是当前Executor执行器想要执行的方法,对于第二个代理而言的目标方法是第一个代表对象的代理方法) 该方法内部实际上就是调用getMethod().invoke(getTarget(),getArgs());

可知拦截器的方法执行的顺序是:setProperties-->plugin-->intercept-->实际的方法执行结果

分析一遍intercept方法,这个方法会执行四次,原因是虽然我们通过注解指定了要增强的是StatementHandler,但是intercept方法还是会尝试包装所有的核心类,理解分析一下这句话什么意思.
对于每一个Plugin而言其内部都封装了一个独立的Map<Class<?>, Set<Method>> signatureMap;Map的Key是当前Plugin拦截的目标类的.class文件 Map的Value是当前拦截器拦截的目标类的Method方法对象.
之前说过在Mybatis中会在newExecutor、newParameterHandler、newResultSetHandler、newStatementHandler方法中调用pluginAll方法
而pluginAll方法并不会检验当前拦截器具体拦截的是哪些核心对象,而是直接根据上述所说返回一个代理对象(但如果某个拦截器没有标注@Intercepts注解的话是会报错的),所以当Mybatis调用newExecutor、newParameterHandler......这四个方法的时候,看源码可知它们都会遍历当前所有的拦截器然后调用它们的plugin方法,用于增强当前的四大核心对象.
所以每一个核心对象都会被Plugin代理,每一个Plugin都会间接代理四大核心对象,只不过在真正执行的代理方法中(Plugin的invoke方法)会通过signatureMap来判断当前拦截器到底有没有想要对当前核心对象做增强的欲望.
流程:调用任意一个核心对象的真实方法->经过每个拦截器->调用拦截器的代理方法,在代理方法中进行判断是否对当前真实对象进行拦截->如果要增强则调用拦截器的intercept方法,否则直接调用真实对象的方法.

引言:
当我们插入某条记录时,如果主键为null 可以通过Mybatis-Plus制定的规则来帮助我们生成这个ID


5.1自动ID

假设这里有一张Teahcer表:在Entity实体类的主键ID上,用@TableID注解进行标注
public class Teacher {
    @TableId(type = IdType.ASSIGN_ID)
    private Long id;
    private String name;
}

@TableId:
String value() default "";				//指定当前被标注的属性对应的数据库中的字段名称
IdType type() default IdType.NONE;			//主键生成类型


IdType:枚举类 规定了主键ID的生成类型
AUTO						//数据库自增ID
NONE						//该值表示当前主键ID未指定生成类型 其效果约等于INPUT类型
INPUT						//用户输入ID 也就是插入记录的时候实体类的主键ID得程序员自已指定(相当于Mybatis-Plus不做任何操作)
ASSIGN_ID					//使用自定义ID生成器生成ID
ASSIGN_UUID					//使用自定义ID生成器生成UUID

5.2自定义ID生成器

首先根据5.1的内容可知,如果是通过自定义生成器生成ID则@TableId注解中要指定为IdType.ASSIGN_ID 如果要生产UUID则指定为IdType.ASSIGN_UUID

将我们的自定义的ID生成器添加到SpringBoot容器中(将IdentifierGenerator接口的实现添加到容器中)
@Bean
public IdentifierGenerator idGenerator() {
    return new IdentifierGenerator(){...}
}

IdentifierGenerator:顶级接口 用于生成自定义主键ID
nextId(Object) return Number;					//当主键生成类型指定为IdType.ASSIGN_ID时会调用该方法 args0:回调传入当前需要生成主键的Entity对象 返回值为该Entity对象的主键(返回值为Number)
nextUUID(Object) return String;					//当主键生成类型指定为IdType.ASSIGN_UUID时会调用该方法 该方法是个默认方法(可以不重写)默认方法的算法就是雪花算法 args0:回调传入当前需要生成主键的Entity对象 返回值为该Entity对象的主键,因为已经特指为UUID了所以主键应该就是UUID


6.扩展插件之逻辑删除

引言:何为逻辑删除?有些时候为了方便数据恢复和保护数据价值等情况下并不需要数据真正得被删除,而是给一个标记 意思是从逻辑上删除了(虽然数据并没有被真正删除)此时该记录就不会参与到任何事件中就好像透明一样

假设这里有一张表Teacher:在标记属性上标注@TableLogic注解(那么通过该字段我们就可以判断当前记录有没有被逻辑删除)
public class Teacher {
    private Long id;
    private String name;
    @TableLogic
    private Integer deleted;
}

根据Mybatis-Plus的约定,当标记字段类型指定为Integer的时候 0代表未删除 1代表该记录已经被逻辑删除 另外最好指定deleted的默认值为0

我们可以调用BaseMapper的deleteById方法根据实体类的ID来删除一条记录,删除后查询数据库会发现对应的记录并没有被删除,只不过被删除的记录的deleted字段被设置成了1
然后再调用BaseMapper的selectList方法并传入Wrapper,最终会发现deleted值为1的记录不会被查询到.

强调:不管是删除还是查询 一定得是调用BaseMapper的方法才能体现逻辑删除的效果,如果直接调用原生的Mybatis Mapper接口是不会生效的.

除了通过@TableLogic注解来标注标记字段外,还可以通过application.yml定义全局的逻辑删除规则:
mybatis-plus:
  global-config:
    db-config:
      logic-delete-field: flag				//将所有Entity类的flag作为标记字段(此时就不需要@TableLogic来标记字段)
      logic-delete-value: 1 				//逻辑已删除值(当逻辑删除时标记字段的值,默认为1)
      logic-not-delete-value: 0 			//逻辑未删除值(当逻辑未删除时标记字段的值,默认为0)



7.通用枚举

引言:通用枚举有什么用?假设这里有一张表Teacher,其中有个属性值是Evaluation(该字段表示对教师的评价) 在设计数据库的时候该字段可能是int类型的 例如0代表优秀 1代表一般 2表示差劲 如果该实体类的字段是Integer类型的,那么就不能保证传入的值一定是0 1 2 这三个数当中的值.
现在这里把Evaluation定义为一个枚举,通过枚举中的code来和数据库表中的那个int类型的字段对应上,效果如下:

public class Teacher {
    private Long id;
    private String name;
    @TableLogic
    private Integer deleted;
    private Evaluation evaluation;			//evaluation和表字段对应		
}

public enum Evaluation {
    GOOD(0, "优秀"),
    General(1, "一般"),
    BAD(2, "差");
    Evaluation(int code, String message) {
        this.code = code;
        this.message = message;
    }

    @EnumValue						//通过@EnumValue来标记和数据库表中对应的字段(但这里要注意,这里的属性名无任何含义只要类型对应上即可,数据库字段名和Entity类属性名的对应是在Entity类中的,也就是上面的Teacher类,所以和之前一样evaluation要和表字段对应)
    private final int code;					
    private final String message;
}

最后在application.yml中配置扫描枚举值:
mybatis-plus:
  type-enums-package: com.github.enums			//配置当前枚举类的包路径(只有配置了被扫描到后才能生效)
 
现在这就相当于,数据库中int类型的evaluation表字段->和Java中的Evaluation枚举对应->对应是通过@EnumValue注解修饰的属性的值对应的




8.字段类型处理器之JSON字段

现在有一张Teacher表,其中包含一个Address属性,只不过这个Address不是单独出来的一张表,并没有连接查询而是直接以JSON字符串的形式存储在Teacher表中VARCHAR类型,存储和读取的时候直接序列化和反序列化该字段.

@TableName(autoResultMap = true)					//Entity实体类必须标注该注解
public class Teacher {
    private Long id;
    private String name;
    @TableField(typeHandler = JacksonTypeHandler.class)			//在要JSON序列化和反序列化的字段上标注@TableField并指定typeHandler(类型处理器) 如果是JSON的话可以指定JacksonTypeHandler或者FastjsonTypeHandler
    private Address address;

}

public class Address {
    private String country;
    private String detailedAddress;
}

此时插入新的记录就会发现Address是以JSON的形式存放在数据库中的.另外在Jackson中是通过ObjectMapper来进行序列化和反序列化的,如果想自定义ObjectMapper还可以这么做:
@Component
public class MpJsonConfig implements CommandLineRunner {
    @Override
    public void run(String... args) throws Exception {
        JacksonTypeHandler.setObjectMapper(new ObjectMapper());
        GsonTypeHandler.setGson(new Gson());
    }
}

CommandLineRunner接口是Spring的接口,它会在系统启动的时候回调该接口,那么我们就可以在run方法内调用JacksonTypeHandler的setObjectMapper方法来自定义ObjectMapper.


@TableField:
Class<? extends TypeHandler> typeHandler() default UnknownTypeHandler.class;			//设置字段的类型处理器

9.1分页插件
首先思考一个问题,在Mybatis中如果想要在特定的情况下更改Plugin的行为该怎么做?
在Mybatis中Mybatis会自动帮助我们创建Interceptor拦截器对象,然后调用Interceptor对象的setProperties方法传进Properties对象,这个Properties对象就包含了Mybatis-config.xml配置文件中指定的配置.
然后再回调plugin方法,根据回调的Properties来定制插件的功能.

如果是在SpringBoot中还可以通过ConfigurationCustomizer接口回调传入的Configuration(Mybatis原生类)来动态添加插件Plugin

//1.
所以在Mybatis-Plus中也有一套动态添加插件的方式,代码在如下Configuration类中:
@Configuration
public class Configuration {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();						//MybatisPlus拦截器 直接new
        PaginationInnerInterceptor paginationInnerInterceptor = new PaginationInnerInterceptor(DbType.MYSQL);
        interceptor.addInnerInterceptor(paginationInnerInterceptor);
        return interceptor;
    }

    @Bean
    public ConfigurationCustomizer configurationCustomizer() {
        return new ConfigurationCustomizer() {
            @Override
            public void customize(MybatisConfiguration configuration) {

    }

10.自动填充功能(要和主键生成区分下来)

如果我们在插入/更新某个记录时,对于该Entity对象的某个字段值我们并没有指定具体的值而是希望能有一个全局的方法来帮助我们自动填充某些字段的默认值.只有调用BaseMapper的方法才会生效,记住在Mybatis-Plus中但凡是涉及到Mybatis-Plus注解的相关内容十有八九都必须调用BaseMapper接口的方法才会生效.

public class Teacher {
    private Long id;
    private String name;
    @TableField(fill = FieldFill.INSERT_UPDATE)				//通过@TableField的fill属性来表明当前字段需要自动填充
    private LocalDateTime dateTime;					//例如当前这个字段 我们在insert或者update的时候并不指定该值
}

@Component
public class xxx implements MetaObjectHandler {				//编写一个MetaObjectHandler接口的实现类,并将其添加到容器中 或者通过@Bean添加
    @Override
    public void insertFill(MetaObject metaObject) {			//当被@TableField注解标记的字段未指定值时并且是插入语句时会调用该方法
        this.strictInsertFill(metaObject, "dateTime", LocalDateTime.class, LocalDateTime.now());		//固定写法 args1:待填充的目标字段的属性名 args2:目标字段的类型 args3:具体填充的默认值
    }

    @Override
    public void updateFill(MetaObject metaObject) {			////当被@TableField注解标记的字段未指定值时并且是更新语句时会调用该方法
        this.strictInsertFill(metaObject, "dateTime", LocalDateTime.class, LocalDateTime.now());		//固定写法 args1:待填充的目标字段的属性名 args2:目标字段的类型 args3:具体填充的默认值
    }
}


FieldFill:填充模式的枚举类
DEFAULT				//不填充
INSERT				//插入记录时填充
UPDATE				//更新记录时填充
INSERT_UPDATE			//插入和更新记录时填充



11.自定义BaseMapper(扩展BaseMapper功能)+SQL注入器


引言:由于在Java中是单继承,假设现在BaseMapper不能满足需求了,该如何去扩展BaseMapper?

//1.编写BaseMapper的子接口
public interface MyBaseMapper<T> extends BaseMapper<T> {
    Integer deleteAll();					//我们希望调用该方法后删除当前Mapper对应的表的所有数据
}

//2.我们原先的Mapper接口是直接继承自BaseMapper的,现在改为继承MyBaseMapper

//3.编写AbstractMethod抽象类的子类 该抽象类只有一个抽象方法injectMappedStatement(Class<?>,Class<?>,TableInfo) return MappedStatement; 该方法的入参都是回调,最终Mybatis-Plus希望你返回一个MappedStatement 还记得在Mybatis中每一个SQL语句最终的执行都是通过Executor执行器+MappedStatement来执行的 所以我们每自定义一个SQL都需要返回一个MappedStatement 都需要创建一个AbstractMethod的子类
public class DeleteAll extends AbstractMethod {
    @Override
    public MappedStatement injectMappedStatement(Class<?> mapperClass, Class<?> modelClass, TableInfo tableInfo) {
        // 执行SQL动态SQL参考类SqlMethod
        String sql = "delete from " + tableInfo.getTableName();
        // mapper接口方法名一致
        String method = "deleteAll";
        SqlSource sqlSource = languageDriver.createSqlSource(configuration, sql, modelClass);		//构建SQL元数据
        return this.addDeleteMappedStatement(mapperClass, method, sqlSource);				//args1:该值必须和我们自定义的方法名一致
    }
}

AbstractMethod:这个抽象类里还有很多方法,详情看源码
addDeleteMappedStatement(Class<?>,String,SqlSource) return MappedStatement;				//如果自定义SQL是删除语句则调用AbstractMethod的当前方法返回MappedStatement
addUpdateMappedStatement(Class<?>,Class<?>,String,SqlSource) return MappedStatement;			//更新语句
addInsertMappedStatement(Class<?>,Class<?>,String,SqlSource,KeyGenerator,String,String) return MappedStatement;		//插入语句
addSelectMappedStatementForOther(Class<?>,String,SqlSource,Class<?>) return MappedStatement;		//查询语句
addSelectMappedStatementForTable(Class<?>,String,SqlSource,TableInfo) return MappedStatement; 		//查询语句


//4.添加SQL注入器

·ISqlInjector:顶级接口 自动注入接口
inspectInject(MapperBuilderAssistant,Class<?>) return void;				//检查SQL是否已经注入

AbstractSqlInjector:实现了ISqlInjector接口
getMethodList(Class<?>) return List<AbstractMethod>;							//Mybatis会回调传入当前正在查询的Mapper的Class对象 Mybatis-Plus希望你返回一个List<AbstractMethod>集合 这个AbstractMethod正巧就是上面第三条我们继承的抽象类 可以预想到Mybatis-Plus拿到这个集合后会回调injectMappedStatement方法来得到MappedStatement实例 从而最终配合执行器来执行我们自定义的SQL语句

DefaultSqlInjector:继承自AbstractSqlInjector
getMethodList(Class<?>) return List<AbstractMethod>;							//当前这个类默认实现了该方法 在这个默认实现中,Mybatis-Plus往List中注册了很多默认的AbstractMethod实现 而这些AbstractMethod实现就正好一一对应BaseMapper中的所有的抽象方法!!!!!! 所以可想而知,BaseMapper的每个抽象方法的实现就是通过这一个一个的AbstractMethod实现的
													
public class MyLogicSqlInjector extends DefaultSqlInjector {
    @Override
    public List<AbstractMethod> getMethodList(Class<?> mapperClass) {
        List<AbstractMethod> methodList = super.getMethodList(mapperClass);				//我们在这里调用了父类的getMethodList方法,那么我们就能得到所有BaseMapper的所有方法的实现了 如果这里不调用,而是直接创建一个新的List 则BaseMapper接口的所有方法都将失效
        methodList.add(new DeleteAll());								//在BaseMapper(DefaultSqlInjector)注册的AbstractMethod的基础上添加我们第三条编写的自定义的SQL语句
        return methodList;
    }
}

//5.把我们编写的自动注入接口添加到容器中(beanName=sqlInjector)
@Configuration
public class Configuration {
    @Bean
    public ISqlInjector myLogicSqlInjector() {
        return new MyLogicSqlInjector();
    }
}
