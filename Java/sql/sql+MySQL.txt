21.mysql正则表达式:regexp关键字
正则表达式一般配合where去寻找一个或多个指定的数据

例如:select * from student where sname regexp '^王';		where后面跟字段名,字段名后面跟regexp

mysql的正则表达式规则:

模式			描述
^		匹配输入字符串的开始位置。如果设置了 RegExp 对象的 Multiline 属性，^ 也匹配 '\n' 或 '\r' 之后的位置。
$		匹配输入字符串的结束位置。如果设置了RegExp 对象的 Multiline 属性，$ 也匹配 '\n' 或 '\r' 之前的位置。
.		匹配除 "\n" 之外的任何单个字符。要匹配包括 '\n' 在内的任何字符，请使用象 '[.\n]' 的模式。
[...]		字符集合。匹配所包含的任意一个字符。例如， '[abc]' 可以匹配 "plain" 中的 'a'。
[^...]		负值字符集合。匹配未包含的任意字符。例如， '[^abc]' 可以匹配 "plain" 中的'p'。
p1|p2|p3	匹配 p1 或 p2 或 p3。例如，'z|food' 能匹配 "z" 或 "food"。'(z|f)ood' 则匹配 "zood" 或 "food"。
*		匹配前面的子表达式零次或多次。例如，zo* 能匹配 "z" 以及 "zoo"。* 等价于{0,}。
+		匹配前面的子表达式一次或多次。例如，'zo+' 能匹配 "zo" 以及 "zoo"，但不能匹配 "z"。+ 等价于 {1,}。
{n}		n 是一个非负整数。匹配确定的 n 次。例如，'o{2}' 不能匹配 "Bob" 中的 'o'，但是能匹配 "food" 中的两个 o。
{n,m}		m 和 n 均为非负整数，其中n <= m。最少匹配 n 次且最多匹配 m 次



22.mysql处理重复数据:
1.放置表中出现重复的数据:
在表中指定主键约束或者唯一约束

2.三种插入方式对于重复数据的处理方式:
insert into student values();
	如果你插入了重复的数据,会报错

insert ignore into student values();
	使用这条插入语句,如果你插入了重复的数据,会产生警告,但不会报错

replace into student values();
	这条语句会替换原来的数据

3.如果这个表本身没有定义约束,现在要你去重你该怎么办?
1.对表进行分组从而达到去重的目的:
select id,name from student group by id,name;
2.创建一个新表,这个表的内容就是去重后的表:
create table temp select id,name from student group by id,name;
3.删除原来的表:
drop table student;
4.将表命名会原来的表:
alter table tmp rename to student;


23.mysql及sql注入:
1.永远不要信任用户的输入。对用户的输入进行校验，可以通过正则表达式，或限制长度；对单引号和 双"-"进行转换等。
2.永远不要使用动态拼装sql，可以使用参数化的sql或者直接使用存储过程进行数据查询存取。
3.永远不要使用管理员权限的数据库连接，为每个应用使用单独的权限有限的数据库连接。
4.不要把机密信息直接存放，加密或者hash掉密码和敏感的信息。
5.应用的异常信息应该给出尽可能少的提示，最好使用自定义的错误信息对原始错误信息进行包装
6.sql注入的检测方法一般采取辅助软件或网站平台来检测，软件一般采用sql注入检测工具jsky，网站平台就有亿思网站安全平台检测工具。MDCSOFT SCAN等。采用MDCSOFT-IPS可以有效的防御SQL注入，XSS攻击等

select * from student into outfile 'D:/mysql/testout/my.txt';		导出的目录必须由你设置好父目录

数据导出的形式:
select * from student into outfile 'D:/mysql/testout/my.txt'
fields terminated by ',' enclosed by '"'				每个数据用,隔开,每个数据加上"
lines terminated by '\r\n';						每列数据之间/r/n


自定义导出的格式:好处是,你可以导出一个xml文件或者json文件被别的程序方便读取

注意事项:
LOAD DATA INFILE是SELECT ... INTO OUTFILE的逆操作，SELECT句法。为了将一个数据库的数据写入一个文件，使用SELECT ... INTO OUTFILE，为了将文件读回数据库，使用LOAD DATA INFILE。

SELECT...INTO OUTFILE 'file_name'形式的SELECT可以把被选择的行写入一个文件中。该文件被创建到服务器主机上，因此您必须拥有FILE权限，才能使用此语法。

输出不能是一个已存在的文件。防止文件数据被篡改。

你需要有一个登陆服务器的账号来检索文件。否则 SELECT ... INTO OUTFILE 不会起任何作用。

在UNIX中，该文件被创建后是可读的，权限由MySQL服务器所拥有。这意味着，虽然你就可以读取该文件，但可能无法将其删除。


关于cmd导出sql文件会报错.


25.mysql函数:

mysql函数分为:

1.字符串函数:
ASCII(s):返回字符串s的第一个字符的 ASCII 码:
	select ASCII(sname) from student;

CHAR_LENGTH(s):返回字符串 s 的字符长度
	select char_length(sname) from student;

character_length(s):返回字符串 s 的字符数
	select character_length(sname) from student;	和上面的函数没差

concat(s1,s2...sn):字符串 s1,s2 等多个字符串合并为一个字符串(合并多个字符串)
	select concat(sname) from student;

concat_ws(x, s1,s2...sn):同concat,但是每个字符串之间会加上x这个分隔符.x代表分隔符
	select concat_ws("-", "sql", "tutorial", "is", "fun!");
	
field(s,s1,s2...):返回第一个字符串 s 在字符串列表(s1,s2...)中的位置
	select field("c", "a", "b", "c", "d", "e");		返回c在这个字符串数组里的位置

find_in_set(s1,s2):返回在字符串s2中与s1匹配的字符串的位置
	select find_in_set("c", "a,b,c,d,e");

format(x,n):函数可以将数字 x 进行格式化 "#,###.##", 将 x 保留到小数点后 n 位，最后一位四舍五入。
	select format(250500.5634, 2);
	将小数点之前的数按照#,##,###来格式化,取决于小数点之前的数的位数,将小数点后面的数按n来保留

insert(s1,x,len,s2):字符串 s2 替换 s1 的 x 位置开始长度为 len 的字符串
	select insert("google.com", 1, 6, "runnob");		输出：runoob.com
	
locate(s1,s):从字符串 s 中获取 s1 的开始位置
	select locate('st','myteststring');  输出 5

lcase(s):将字符串 s 的所有字母变成小写字母
	select lcase('RUNOOB');	输出 runoob

left(s,n):返回字符串 s 的前 n 个字符
	select left('runoob',2) -- ru

lower(s):将字符串 s 的所有字母变成小写字母
	select lower('RUNOOB') -- runoob

lpad(s1,len,s2):在字符串 s1 的开始处填充字符串 s2，使字符串长度达到 len
	select lpad('abc',5,'xx') -- xxabc

ltrim(s):去掉字符串 s 开始处的空格
	select ltrim("    runoob");-- runoob

mid(s,n,len):从字符串 s 的 start 位置截取长度为 length 的子字符串，同 SUBSTRING(s,n,len)
	select mid("runoob", 2, 3); -- uno

position(s1 IN s):从字符串 s 中获取 s1 的开始位置
	select position('b' in 'abc') -- 2

·mysql建表约束
1.主键约束(复合主键+自增主键)
2.自增约束
3.唯一约束
4.非空约束
5.默认约束
6.外键约束

1.主键约束:字段不重复不为null,但只能定义一个主键.不管联合主键还是自增主键都属于主键约束.只能添加一个
它能够唯一确定一张表中的一条记录,也就是我们通过给某个字段添加约束,就可以使得该字段不重复,且不为空.(类似于set)

create table user(id int primary key,name varchar(20));		int后面跟约束,而且只能定义一个主键

添加复合主键:

create table user2(id int ,name varchar(20),password varchar(20) ,primary key(id,name));	在最后丢入需要添加约束的名称

复合主键:只要丢入的所有字段的值不重复就可以了,这里也就是id和name加起来不重复就就可以了.但同样任何一个字段都不能为null

添加主键约束的第三种方式:通过alter命令



2.自增约束:和主键约束搭配使用,也属于主键,只能添加一个

添加自增约束:
create table user3(
	id int primary key auto_increment,			key后面加auto_increment代表这个值不给定就自增
	name varchar(20)
);

插入语句:
insert into user3 (name) values ('张三');			表名后面跟小括号,里面的字段,代表当前命令只该字段传值
								这里由于是自增约束,id会默认自增.



3.唯一约束:约束修饰的字段的值不可以重复,可以添加多个唯一约束

在创建表的时候添加:
create table user6(id int,name varchar(20),unique(name));	unique后面添加字段

另外一种方式:
create table user7(id int,name varchar(20) unique);		直接在字段后面添加unique

如果:你创建这样一个表的话,就不是复合的了,这两个字段要单独计算.
create table user7(id int unique,name varchar(20) unique);

第三种添加方式:在上面,通过alter命令.

添加复合唯一约束:
create table user6(id int,name varchar(20),unique(id,name));	和复合主键一样,只要id+name不重复就可以了



4.非空约束:修饰的字段不能为null,就算这个字段有默认值,你也不能传入一个null

在创建表的时候添加:
create table user9(id int,name varchar(20) not null);

第二种添加方式:在上面,通过alter命令


5.默认约束:当我们插入字段值的时候,如果没有传入值,就会使用默认值.

创建表的时候添加:
create table user10(id int,name varchar(20),age int default 10);		你任然可以在10后面添加别的约束

第二种方式:在上面,通过alter命令

6.外键约束:较为复杂
--涉及到父表和子表(主表/副标)

班级:
create table classes(id int primary key,name varchar(20));

学生表:
create table students(id int primary key,name varchar(20),class_id int,foreign key(class_id) references classes(id));

后面的key后面跟(子表字段) references后面跟父表+括号里面的父表字段.前面的内容来自后面的内容

往班级里插入数据:
insert into classes values(1,'一班');
insert into classes values(2,'二班');
insert into classes values(3,'三班');
insert into classes values(4,'四班');

往学生表里插入数据:
insert into students values(1001,'张三',1);
insert into students values(1002,'李四',2);
insert into students values(1003,'王五',3);

解释一下外键的作用,此时students表里面的class_id字段的值必须来自classes表中的id.也就是说class_id的值只能取id的值.

使用alter改变表的字段约束:
not null:
alter table tablename modify column1 columntype not null;
alter table tablename modify column1 columntype null;
unique:
alter table tablename add unique (column1);
alter table tablename add constraint indexname unique (column1,column2);
alter table tablename drop index indexname;		//根据索引名称删除索引
primary key:
alter table tablename add primary key(column1);
alter table tablename add constraint constraintname primary key (column1,column2)
alter table tablename drop primary key;
foreign key:
alter table tablename add foreign key (childColumn) references fathertablename(fatherColumn) on delete ... on update ...;
alter table tablename add constraint indexname foreign key (childColumn) references fathertablename(fatherColumn) on delete ... on update ...;
alter table tablename drop foreign key indexname;
check:
alter table tablename add check (column_name1>number and/or ...)
alter table tablename add constraint constraintname check (column_name1>number and/or ...);
alter table tablename drop check constraintname;
default:
alter table tablename alter column1 set default value;
alter table tablename alter column1 drop default;
auto_increment:
alter table tablename modify column1 columntype auto_increment;
alter table tablename modify column1 columntype;	
对于自增删除的解释:直接作用于列的只有 not null、default、auto_increment.如果是这种暴力更改会不会使得表紊乱?
1.由于自增必须作用在主键上,而主键保证了not null,所以这种暴力更改不会影响列是not null这点.
2.自增和默认值不可以同时作用在一列上.

2.第二范式:必须是满足第一范式的前提下,大前提!!!!!!!!!!!
第二范式要求,除主键外的每一列都必须完全依赖于主键.
如果要出现不完全依赖,只可能发生在联合主键的情况下.

订单表:
create table myorder(
	product_id int,
	customer_id int,
	product_name varchar(20),
	customer_name varchar(20),
	primary key(product_id,customer_id)
);

-问题:
这里的product_name依赖于product_id,customer_name依赖于customer_id
除主键以外的其他列,值依赖于主键的部分字段.此时不满足第二范式
此时需要进行拆表设计

create table myorder(
	order_id int primary key,
	product_id int,
	customer_id int
);

create table product(
	id int primary key,
	name varchar(20)
);

create table customer(
	id int primary key;
	name varchar(20)
);

分成了三个表之后,就满足了第二范式的设计.



3.第三范式:必须先满足第三范式!!!大前提
除了主键列的其他列之间不能有传递依赖关系

错误例子:
create table myorder(
	order_id int primary key,
	product_id int,
	customer_id int,
	customer_phone varchar(15)
);

-问题:我们可以通过customer_phone来找到customer_id,也就是说这两个列之间存在某种依赖关系(联系),customer_id是依赖于主键没错,但是除了主键之外其他列之间还有依赖关系的话就不满足第三范式.

正确例子:应该直接在用户这个表里面添加phone

create table customer(
	id int primary key;
	name varchar(20),
	phone varchar(15)
);

45.exist、not exist关键字,类似于in关键字但不相同
select Sname from student where exist (select * from sc where sno = student.sno and cno = '1');
带有exist谓词的子查询不返回任何数据,只产生逻辑值true或false.
由exist引出的子查询,其目标列表达式通常都用 *,因为带exist的子查询只返回真值或假值,给出列名无实际意义.
这个相关子查询的处理过程是:首先取外层查询中student表的第一个元组,根据它与内层查询相关的属性值(sno值)处理内层查询,若where子句返回值为真,则取外层查询中该元组的Sname放入结果集;
然后再取student表的下一个元组,重复这一过程直至外层student表全部检查完为止.
所以其实这里并没有什么比较,是一种比较特殊的筛选方式.

1.mysql默认是开启事务的(自动提交)

select @@autocommit;		显示当前事务对应的模式代码,这会显示1,默认就是1,自动提交数据

默认事务(1)开启的作用是:当我们去执行一个sql语句的时候,效果会立即体现出来,且不能回滚.

什么是回滚?rollback;

举个例子:
create database bank;

create table user(
	id int primary key,
	name varchar(20),
	money int
);

插入数据:
insert into user values(1,'a',1000);

回滚就是撤销事物语句的效果.

这里插入玩数据之后,即使我调用了rollback,但是刚才插入的语句仍然生效.


如何让回滚语句生效?
调用 set autocommit=0;			设置事务为0

此时再插入数据:
insert into user values(2,'b',1000);

rollback;
调用回滚,刚才的插入语句失效,因为此时的数据库还没有提交

重新插入数据并提交数据库:
insert into user values(2,'b',1000);
commit;

此时再调用
rollback;刚才插入的数据将不会回滚,上面事务为1的时候,调用回滚不生效是因为自动提交了事务.而这里则需要手动提交.



所以如果我们使用手动提交:

update user set money=money-100 where name='a';
update user set money=money+100 where name='b';

rollback;

如果我们在ab转账的时候发生了什么错误,我们家可以使用rollback回滚,当我们确认没有问题时,我们可以手动提交.



2.开启手动事务:
	首先将事务模式设置回set autocommit=0;		0

手动开启事物:
begin或者start transaction

无法回滚:
进行转账操作:
update user set money=money-100 where name='a';
update user set money=money+100 where name='b';
此时调用rollback;将不能回滚

当模式为1时如何回滚?:
begin;
update user set money=money-100 where name='a';
update user set money=money+100 where name='b';
rollback;					回滚结束,begin和rollback之间的语句将失效

只需要在之前加入begin语句,就相当于手动开启了事务.你还是可以通过rollback来进行回滚,但和上面一样,一切止步于提交commit之前.

commit执行提交语句之后,就无法回滚




3.事务的四大特性:
A 原子性:事务是最小的单位,不可在分割
C 一致性:事务要求,同一事物中 sql语句,必须保证同时成功或同时失败
I 隔离性:事务1和事务2之间是具有隔离性的,也就是事务1和事务2互不影响
D 持久性:事务一旦结束,就不可以返回


事务开启:
	1.修改默认提交 set autocommit=0;
	2.begin;
	3.start transaction;

事务手动提交:
	commit;

事务手动回滚:
	rollback;

事务是针对多个事务的情况:
脏读:一个事务读取到另一个事务还没有提交的数据.事务T1将某一值修改,然后事务T2读取该值,此后T1因为某种原因撤销对该值的修改,这就导致了T2所读取到的数据是无效的,值得注意的是,脏读一般是针对于update操作的.
不可重复读:事务A在同一个事务中读取的数据前后不一致.(原因是因为事务B修改了数据)
幻读:事务B已经对表中的数据进行添加操作,并且commit了.但是事务A并不能察觉到数据库数据个数发生改变(此时如果事务B也添加了这行记录就会报错)

不可重复读和幻读的区别:一个是读取的值不同,一个是读取的数据量不同.
不可重复读:读过的数据再读一次发现两次结果不一致
幻读:第一次和第二次读取的数据量不同.(实际上幻读是不可重复读的一种)

关于事务的隔离性:事务由低到高(虽然隔离性越来越好,但是处理的能力越来越差,所以实际工作中要视情况而定).
				脏读			不可重复读		幻读
1.read uncommitted;	 	可能			可能			可能
2.read committed;			不可能			可能			可能
3.repeatable read;			不可能			不可能			可能
4.serializable;			不可能			不可能			不可能

此时无论是在张全蛋还是王尼玛的表里面,都是修改过后的数据了.即使王尼玛还没有提交

接着张全蛋接着操作:
start transaction;				正常执行
insert into user values(8,'小猪',1000);		此时,执行该语句时,会阻塞.

只有当王尼玛那边提交数据之后,这边才可以正常执行下去.
当然如果王尼玛这边提交的时间间隔太长还是会报错 Lock wait timeout exceeded; try restarting transaction

当user表被其它另外一个事务操作的时候(以start transaction;语句来判断是否被其它事务操作).其它事务的里面的写操作是不可以进行的.

从第一步开始分析,当张全蛋开始事务时,计算王尼玛也开始了事务,但是由于张全蛋是先开的,所以王尼玛就无法对相同的表进行写操作,除非张全蛋提交了表.
那么张全蛋没提交时,王尼玛这边会进入排队状态(串行化)直到张全蛋那边提交.


串行化的问题就是:处理数据太慢,性能太差.

2.sql优化:SQL优化是概率事件
	原因:性能低、执行时间太长、SQL语句欠佳(连接查询)、索引失效、服务器参数设置不合理

	a.SQL语句:分为两个过程
	
	编写过程:
		select dinstinct ... from ..join ..on ..where .. group by .. having .. order by .. limit
	解析过程:
		from.. on.. join .. where group by .. having.. select dinstinct ..order by limit..

		1.From:当涉及多个表的时候，左边表的输出会作为右边表的输入，之后会生成一个虚拟表VT1.
		(1-J1)笛卡尔积
		计算两个相关联表的笛卡尔积(CROSS JOIN),生成虚拟表VT1-J1。
		(1-J2)ON过滤
		基于虚拟表VT1-J1这一个虚拟表进行过滤,过滤出所有满足ON 谓词条件的列,生成虚拟表VT1-J2.
		(1-J3)添加外部列
		如果使用了外连接(Left,Right,Full(当然mysql不支持Full)),主表(保留表/原本的表)中的不符合ON条件的列也会被加入到			VT1-J2中,作为外部行,生成虚拟表VT1-J3.(就算不满足条件,最后也是会显示的,不过内容为null,具体可看之前的外连接特性)

		2.Where:对VT1过程中生成的临时表进行过滤，满足WHERE子句的列被插入到VT2表中。
		注意：
		此时因为分组，不能使用聚合运算；也不能使用SELECT中创建的别名；
		与ON的区别:
		如果有外部列,ON针对过滤的是关联表,主表（保留表）会返回所有的列,
		如果没有添加外部列，两者的效果是一样的.
		应用:
		对主表的过滤应该放在WHERE,
		对于关联表，先条件查询后连接则用ON，先连接后条件查询则用WHERE.
		
		3.GROUP BY:这个子句会把VT2中生成的表按照GROUP BY中的列进行分组.生成VT3表。
		注意:
		其后处理过程的语句,如SELECT,HAVING,所用到的列必须包含在GROUP BY中,对于没有出现的,得用聚合函数.
		原因:
		GROUP BY改变了对表的引用,将其转换为新的引用方式,能够对其进行下一级逻辑操作的列会减少
		我的理解是
		根据分组字段,将具有相同分组字段的记录归并成一条记录,因为每一个分组只能返回一条记录,除非是被过滤掉了,而不在			分组字段里面的字段可能会有多个值,多个值是无法放进一条记录的,所以必须通过聚合函数将这些具有多值的列转换成单值
					
		4. HAVING:这个子句对VT3表中的不同的组进行过滤，只作用于分组后的数据，满足HAVING条件的子句被加入到VT4表中。
			
		5. SELECT:这个子句对SELECT子句中的元素进行处理，生成VT5表。
		(5-J1)计算表达式 计算SELECT 子句中的表达式,生成VT5-J1
		(5-J2)DISTINCT
		寻找VT5-1中的重复列,并删掉,生成VT5-J2
		如果在查询中指定了DISTINCT子句,则会创建一张内存临时表（如果内存放不下,就需要存放在硬盘了）.这张临时表的表结			构和上一步产生的虚拟表VT5是一样的,不同的是对进行DISTINCT操作的列增加了一个唯一索引,以此来除重复数据.

		6.ORDER BY:从VT5-J2中的表中,根据ORDER BY 子句的条件对结果进行排序,生成VT6表.
		注意：
		唯一可使用SELECT中别名的地方.
		
		7.LIMIT:LIMIT子句从上一步得到的VT6虚拟表中选出从指定位置开始的指定行数据.
		注意：
		offset和rows的正负带来的影响
		当偏移量很大时效率是很低的,可以这么做:
		采用子查询的方式优化,在子查询里先从索引获取到最大id,然后倒序排,再取N行结果集
		采用INNER JOIN优化,JOIN子句里也优先从索引获取ID列表,然后直接关联查询获得最终结果

	b.SQL优化,主要就是在优化索引:
		索引:相当于书的目录,比如我们要查询一个数据,如果没有索引,就需要一个一个的遍历,很浪费时间,有了索引就可以直接跳转到你需要查的内容对应的地方
		索引:index是帮助mysql搞笑获取数据的数据结构.索引是数据结构(树:B树(默认)、Hash树)

		索引的弊端:
			   1.索引本身很大,可以存放在硬盘/内存(通常为硬盘)		
			   2.索引不是所有情况均适用:a.少量数据 b.平凡更新的数据	c.很少使用的列
			   3.索引会降低增删改的效率:因为如果一个数据有索引,当我们更改这个数据的时候,不但要更改这个数据还要更改索引
		索引的优势:
			   1.提高查询效率(降低了IO使用率)
			   2.降低CPU使用率(orde by age desc):如果没有索引,我需要把这一堆数据全部拿出来然后排序,但如果有索引根据B数的特性,就不需要排序了,因为已经排好了
			   			
	
	B树:和二叉树一样,小的放左边,大的放右边.实际语法上只要大于2(节点大于2)的都叫B树
	B数:不光有二叉还有三叉树
	三叉树:			10	50	
			5	    12		60
	比10小的放左边,比50大的放右边.介于10和50的放中间.然后在再子节点放置一个小于10的数,让5个另一个小于10的数,形成一个新区间
	三叉数:可以存放上百万条数据
	B树:一般都是指B加数,对于B加数来说数据全部存放在B节点中.拿上面的来举例子.10和50只是作为一个分界线,真正存放数据的是在最	后的叶节点
	B树中查询任意的数据的次数:n次(B树的高度)

9.Extra:额外的信息:
	会出现的数值:	
		     1.using filesort:性能消耗大,需要额外的一次查询.一般出现在order by中
		     2.using temporary:性能损耗大,用到了临时表,一般出现在group by语句中
		     3.using index:性能提升,索引覆盖.原因:不读取源文件,只从索引文件中获取数据,不需要会表查询
		     4.using where:和using index相反.using where需要回表查询
		     5.impossible where:where子句永远为false
		     6.Using join buffer:表示mysql引擎使用了连接缓存	mysql认为查询语句太差了		
	
1.using filesort:性能消耗大,需要额外的一次查询.一般出现在order by中
再次之前我们需要先理解一个概念:排序之前需要先查询

注意:下面的这些字段全是单索引
查询执行计划:
explain select * from teacher where tname='' order by tname;
结果:null
原因:先查询tname,查完了然后在排序tname很合理.

explain select * from teacher where tname='' order by tid;
结果: Using filesort
原因:先查询tname,查完了按tid排序的时候还需要再查询一次.	

小结:对于单索引,如果分组排序的是同一个字段,则不会出现using filesort.


注意:接下来的字段是符合索引:
复合索引:不能跨列(最佳左前缀)
假设我这里有个表,里面有a1,a2,a3这个三个字段,并且他们是一个复合索引
查询执行计划:
explain select * from mytb where a1='' order by a3;		
结果:using filesort
原因:不能跨列a1和a3跨了一个a2,所以出现了using filesort

查询执行计划:
explain select * from mytb where a2='' order by a3;	
结果:using filesort
原因:跨了a1

查询执行计划:
explain select * from mytb where a1='' order by a2;
结果:不会出现

小结:避免 where 和 order by 按照复合索引的顺序使用,不要跨列或无序使用


2.using temporary:性能损耗大,用到了临时表,一般出现在group by语句中.如果已经有表了,但不适用,必须再来一张表.需要额外使用一张表

查询执行计划:
explain select * from teacher where tname='王境泽' group by tname;
结果:null
原因:先按照tname查出来,然后再按照tname分组.没有问题

查询执行计划:
explain select * from teacher where tname='王境泽' group by tid;
结果:Using temporary
原因:先查找tname,然后再按照tid分组.此时sql就需要再查询一次
避免:查询那些列,就根据那些列group by

a.查询执行计划:
explain select * from test03 where a2=2 and a4=4 group by a2,a4;
结果:没有using temporary
原因:根据解析过程,是先where再group by.这里先根据a2a4筛选了一次,再根据a2a4分组一次,就不需要新表了.这里where和group是同一张表
b.查询执行计划:
explain select * from test03 where a2=2 and a4=4 group by a3;			
结果:using temporary
原因:where先筛选了a2a4,但又要根据a3来分组,就需要用到新的表.

·另外group by 两个字段的情况比较复杂,需要分a2、a4是否为无索引列.单值索引列和唯一索引列来讨论.
总的来说只要a2 a4有一个是唯一索引就不需要用到临时表


3.using index:性能提升,原因:不读取源文件,只从索引文件中获取数据,不需要会表查询

注意接下来的是单值索引:
查询执行计划:
explain  select tname from teacher where tname='王境泽';
结果:Using index
原因:只要使用到的列 全部都在索引中,就是索引覆盖

注意接下来的是符合索引:
假如这里有个表,里面有一个复合索引(a1,a2,a3)
查询执行计划:
explain select a1,a2 from mytb where a1='' or a2='';
结果:Using index
原因:你用的a1、a2全部被包含在复合索引里面.和上面一样是索引覆盖.

现在这个表里面有一个复合索引(a1,a2)
查询执行计划:
explain select a1,a2 from mytb where a1='' or a3='';
结果:不出现Using index
原因:因为复合索引a1、a2不能覆盖a1和a3

查询执行计划:
explain select a1,a2 from mytb;
结果:Using index
原因:你还是用到了被索引覆盖的两个列

另外:对比当前这个和上上面那个.一个有where一个没有.那么会造成什么影响?
如果用到了索引覆盖(using index时),会对possible_key和key造成影响
a.如果没有where,则索引只出现在key中
b.如果有where,则索引出现在key和possible_keys中

4.using where:和using index相反.using where需要回表查询:
查询执行计划:
explain select cname from course where cname='java';
结果:Using where
原因:cname没有索引需要回原表查询

5.impossible where:where子句永远为false
查询执行计划:
explain select * from teacher where tname='蔡徐坤' and tname='王境泽';
结果:impossible where
原因:作为精确查询,tname不可能即等于蔡徐坤又等于王境泽




5.优化示例:SQL优化是概念事件
比如一个表拥有一个复合索引:a1,a2,a3,a4

select a1,a2,a3,a4 from where a1=1 and a2=2 and a3=3 and a4=4;		
推荐写法,因为索引使用过的顺序(where后面你的顺序)和复合索引的顺序一致.

select a1,a2,a3,a4 from where a4=1 and a3=3 and a2=2 and a4=4;		
虽然编写的顺序和索引顺序不一致,但是在sql真正执行前 进过了sql优化器的调整 结构上与上一条sql是一致的
 
以上的两个语句使用了全部的复合索引,ken_len是最大值

select a1,a2,a3,a4 from where a1=1 and a2=2 and a4=4 order by a3;
Extra结果:Using where,Using index
	原因:a1 a2 是符合顺序的	但是a4跨列了需要回表查询.
	没有出现Using filesort的原因是:对于Using filesort而言where 和 order by不要出现跨列使用.这里是全部用到了
ken_len:只有a1 和 a2的大小

select a1,a2,a3,a4 from where a1=1 and a4=4 order by a3;
key_len:只有a1的大小	因为a4跨列了
Extra结果:Using where,Using index,Using filesort
	原因:本质原因是不要跨列查找.对于Using filesort的跨列使用的意思:where 和 order by 拼起来不要跨列使用.这也是为什么上面没		有出现Using filesort的原因

select a1,a2,a3,a4 from where a1=1 and a4=4 order by a2,a3;
Extra结果:Using where,Using index
key_len:只有a1的大小

总结:
1.如果(a,b,c,d)复合索引和使用的顺序完全一致(且不跨列使用),则复合索引全部使用.如果部分一致(且不跨列使用),则使用部分索引
2.where 和 order by不要出现跨列使用



6.优化案例:
	分类:单表优化、两表优化、三表优化.	这三种表优化的原理互相适用

1.单表优化:主要针对索引位置和索引失效的练习
创建表:
bid:书本id
name:名称
authorid:作者
publicid:公共id
typeid:类型id

	create table book(
		bid int primary key,
		name varchar(20) not null,
		authorid int not null,
		publicid int not null,
		typeid int not null
	);

插入数据:
insert into book values(1,'java',1,1,2);
insert into book values(2,'c',2,1,2);
insert into book values(3,'c#',3,2,1);
insert into book values(4,'c++',4,2,3);

查询authorid=1 且 typeid为2或3的bid信息

这里为了方便演示,先按照这样查询:select bid from book where typeid in(2,3) and authorid=1
查询执行计划:
explain select bid from book where typeid in(2,3) and authorid=1 order by typeid desc;
结果:
type:all
possible_key:null
key:null
Extar:Using where Using filesort

可以看到这条语句是必须优化的.性能很低.

开始优化:
	加索引:
		alter table book add index idx_bta(bid,typeid,authorid);
	查询执行计划:
	type:index
	possible_key:null
	key:idx_bta
	Extar:Using where Using filesort Using index

	根据SQL实际解析顺序调整,调整索引的顺序.
	为什么要调整?表面上来看这里字段的顺序是bid,typeid,authorid.但是sql解析的时候是从where后面开始解析的,最后(在分组之前)才		执行select
	
	调整索引:	
		alter table book drop index idx_bta;
		alter table book add index idx_bta(typeid,authorid,bid);	
		虽然可以回表查询bid,但是将bid放到索引中 可以提升使用Using index.就是bid的时候也不需要回原表,也根据索引来查
	查询执行计划:
	type:index
	possible_key:null
	key:idx_bta
	Extar:Using where Using index		与上面不一样的地方在于少了 Using filesort
	
接着优化:优化type
	explain select bid from book where typeid in(2,3) and authorid=1 order by typeid desc;
	形成index的原因:如果想要达到range.注意in这个关键字是特殊的,有时会失效.对于我们的索引而言,一旦typeid失效,那么authorid也	可能会失效.所以把authorid放在typeid前面,计算typeid失效了,authorid的索引也不会失效.
	
	调整索引:
		alter table book drop index idx_bta;
		alter table book add index idx_bta(authorid,typeid,bid);

	查看执行计划:需要调整查询语句
		explain select bid from book where authorid=1 and typeid in(2,3) order by typeid desc;
	type:range
	possible_key:null
	key:idx_bta
	Extar:Using where Using index Backward index scan	

	小结:
	    a.索引不能跨列使用,保持索引的定义和使用的顺序一致性
	    b.索引需要逐步优化
	    c.将含in的范围查询放到where的最后,防止失效.
	
	本例中出现Using where的原因是:typeid索引失效需要回原表查询.
	如果把in换成=号
	explain select bid from book where authorid=1 and typeid =3 order by typeid desc;
	结果:
	type=ref
	Extra:Using index
	讲道理此时key_len也会变大


2.两表优化:针对左右表的练习
老师表:

课程表:

insert into course2 values(2,'python');
insert into course2 values(3,'c');

左连接查询:
	select * from teacher2 t left join course2 c on t.cid=c.cid where c.cname='java';

	索引往那张表加?解决结论:小表驱动大表	索引建立在经常使用的字段
小表驱动大表:
	小表:10
	大表:300

	where 小表.x = 大表.y;
	如果小表是10,大表是300.那么小表每拿一次数据大表需要查询300次.	一共是10*300=3000
	
	where 大表.x = 小表.y;
	此时大表每查一次需要查询小表10次 				一共是300*10=3000

	这两个查询最终都会执行3000次,但是对于双层循环来说一般建议将数据小的循环放外层 数据大的循环放内层.
	一般来说,外层循环越小,内层循环越大对于程序的性能是越高的.

7.避免索引失效的一些原则:
	a.复合索引,不要跨列或无序使用(最佳左前缀)
	b.复合索引,尽量使用全索引匹配
	c.不要在索引上进行任何操作	比如对索引进行加减乘除计算、函数计算、类型转换.否则索引失效
	正确:select ... where A.x=..		--假设A.x是一个索引
	错误:select ... where A.x*3=...		
	d.复合索引不能使用不等于(!= > <) 或 is not null (is null) ,否则自身以及右侧的索引全部失效.
	查询计划执行:
	explain select * from book where authorid=1 and typeid=2;	这里authorid和typeid是两个单值索引
	结果:
	possible_key:预测都使用到了authorid和typeid
	key:实际上只用到了authorid的索引
	key_len:4	
	原因:SQL优化是一种概率层面的优化.至于实际是否使用了优化,需要通过explain进行推测.

	查询计划执行:
	explain select * from book where authorid!=1 and typeid=2;
	结果:
	possible_key:authorid和typeid
	key:typeid		这里改变了
	key_len:4
	原因:这里authorid使用了不等于造成了索引失效,虽然authorid失效了,但是typeid并没有失效

体验概率情况:原因是服务层中有SQL优化器,可能会影响我们的优化.
	假如这里有一个表里面有(a,b)一个复合索引
	select * from mytb where a=1 and b=1;		结果:索引全部使用
	select * from mytb where a>1 and b=1;		结果:索引全部失效,通过上述结论可得
	select * from mytb where a=1 and b>1;	结果:索引全部使用.此时和结论相违背.就是证明结论适合大部分情况,优化是概率事件


	e.补救方法:尽量使用索引覆盖
	f.like尽量以"常量开头",不要以'%'开头,否则索引失效
	select * from mytb where name='%x%';	即使name有索引,如果以%开头索引也会失效.
	解决方法:使用索引覆盖
	select tname from mydb where tname='%x%';此时索引是不会失效的.		查询的tname包含在索引里面
	
	g.尽量不要使用类型转换(显示、隐式),否则索引失效
	select * from teacher where tname='123';
	select * from teacher where tname=123;		tname是一个字符串,你传123底层会进行类型转换,造成索引失效

	h.尽量不要使用or,否则索引失效   or很强大,强大到可以把索引左边的都失效
	假如有一个表,有两个索引a,b
	select * from mytb where a=1 and b=2;		此时两个索引全部用到
	select * from mytb where a=1 or b=2;		用了or之后两个索引全部失效,即使b在右边
	
8.其它的优化方法:
	a.exist和in什么时候用?
	如果主查询的数据很大,则用in
	如果子查询的数据很大,则用exist
	
	exist语法:将主查询的结果,放到子查询结果进行条件校验(是否有数据,如果有数据则校验成功).如果符合校验,则保留数据.和in相反
	select tname from teacher where esists(select * from teacher);先从外表检查数据,将查到的结果放到子查询再查询.
	--等价于select tname from teacher;
	
	b.order by 优化:order by经常会出现Using filesort
	Using filesort有两种算法:双路排序、单路排序.(根据IO的次数)
	双路排序:通过IO扫描两次文件(1.从磁盘读取排序字段 读完立即排序 (在buffer中进行排序) 2.扫描其他字段)
		缺点:性能消耗大
	单路排序:通过IO扫描一次文件(只读取了一次,读取全部字段放在buffer中,然后在buffer中找到需要排序的内容进行排序)
		隐患:不一定真的是一次IO,有可能多次IO.而且单路排序比双路排序占用更多的buffer	原因:文件太大,buffer放不下
		在使用单路排序时,如果数据太大,可以考虑调大buffer的容量大小.	通过set max_length_for_sort_data=1024 单位byte
		如果max_length_for_sort_data值太低,则mysql会自动从单路变为双路.(太低:需要排序的列的总大小超过了		max_length_for_sort_data值定义的字节数)
	MySQL4.1之前默认使用双路排序
	MySQL4.1之后默认使用单路排序
	
	提高order by 查询的策略:
	a选择使用单路还是双路,调整buffer的容量大小
	b.避免select * ...	不要用*,大部分情况下,你用什么就写什么.如果一个表只有abcd这四个字段.这行的效率没有下面的效率高
	select a,b,c,d 
	c.复合索引不要跨列使用,避免Using filesort
	d.保证全部的排序字段 排序的一致性(都是升序或降序)

11.锁机制:解决因资源共享 而造成的的并发问题
	示例:买最后一件衣服
	A:看到衣服	先买了衣服
	B:看到衣服	再买了衣服
	问题:如果不加锁,系统没有及时更新的话,有可能导致B也买到了衣服.线程同步那一章.
	
	解决方法:
	A:看到衣服	买:衣服加锁->执行后续的操作->解锁
	B:看到衣服	买:衣服已消失

	分类:操作类型、操作范围
	
操作类型:又分为:读锁、写锁
	a.读锁(共享锁):对同一个数据,多个线程进行读取操作可以同时进行互不干扰
	b.写锁(互斥锁):如果当前读写操作(任意操作)没有结束,则其余线程对当前数据无法进行读写操作

操作范围:
	a.表锁:一次性对一张表进行加锁,如MyISAM存储引擎(上面),开销小、加锁快,但锁范围大,容易发生锁冲突、并发度低.
	b.行锁:一次性多一行数据进行加锁,开销大加锁慢.容易出现死锁,锁的范围较小,不易发生锁冲突,并发度高(很小概率 发生高并发问题	:脏读 幻读 不可重复读 丢失更新)
	c.页锁:

·加读锁:这个锁只针对当前的这个cmd.
locak table tablelock read;
	select * from tablelock;		--读(查),可以
	delete from tablelock where id=1;	--写(增删改),不可以

	select * from emp;			--读,不可以
	delete from emp where eid=1000;		--写,不可以

另外一个cmd中:
读可以
写,会进行等待,等待上面的cmd将锁释放(通过unlock tables;或者事务解锁(提交和回滚))


总结:
某一个会话 对A表加了read锁,则该会话可以对A表进行读操作但不能进行写操作.且会话不能对其它表进行读、写操作.

在另外一个cmd中对A表可以读,但是写的时候会进行等待.


·加写锁:
locak table tablelock write;

当前cmd:可以对加了写锁的表进行任何操作,但是不能操作别的表.

另外的cmd:对加了锁的表可以进行操作,但是前提是要等上一个cmd释放锁

·10.主从复制读写分离

1.主从复制:

假如现在有两个数据库A和B.
影响MySQL-A数据库的操作,在数据库执行后,都会写入本地的日志系统A中.
假设,实时的将变化了的日志系统中的数据库事件操作,在MYSQL-A的3306端口,通过网络发给MYSQL-B.
MYSQL-B收到后,写入本地日志系统B,然后一条条的将数据库事件在数据库中完成.
那么,MYSQL-A的变化,MYSQL-B也会变化,这样就是所谓的MYSQL的复制,即MYSQL replication.
在上面的模型中,MYSQL-A就是主服务器,即master,MYSQL-B就是从服务器,即slave.
日志系统A,其实它是MYSQL的日志类型中的二进制日志,也就是专门用来保存修改数据库表的所有动作,即bin log.(注意MYSQL会在执行语句之后,释放锁之前,写入二进制日志,确保事务安全)
日志系统B,并不是二进制日志,由于它是从MYSQL-A的二进制日志复制过来的,并不是自己的数据库变化产生的,有点接力的感觉,称为中继日志,即relay log.
可以发现,通过上面的机制,可以保证MYSQL-A和MYSQL-B的数据库数据一致,但是时间上肯定有延迟,即MYSQL-B的数据是滞后的.
(即便不考虑什么网络的因素,MYSQL-A的数据库操作是可以并发的执行的,但是MYSQL-B只能从relay log中读一条,执行下.因此MYSQL-A的写操作很频繁,MYSQL-B很可能跟不上.)

2.主从复制的几种方法:
同步复制:
所谓的同步复制,意思是master的变化,必须等待slave-1,slave-2,...,slave-n完成后才能返回.
这样,显然不可取,也不是MYSQL复制的默认设置.比如,在WEB前端页面上,用户增加了条记录,需要等待很长时间.

异步复制:
如同AJAX请求一样.master只需要完成自己的数据库操作即可.至于slaves是否收到二进制日志,是否完成操作,不用关心.MYSQL的默认设置.

半同步复制:
master只保证slaves中的一个操作成功,就返回,其他slave不管.
这个功能,是由google为MYSQL引入的.

3.主从复制分析:
问题1:master的写操作,slaves被动的进行一样的操作,保持数据一致性,那么slave是否可以主动的进行写操作?

答:假设slave可以主动的进行写操作,slave又无法通知master,这样就导致了master和slave数据不一致了.因此slave不应该进行写操作,至少是slave上涉及到复制的数据库不可以写.实际上,这里已经揭示了读写分离的概念.

4.读写分离:
由于slave数据库无法进行增删改操作,所以slave数据库专门用来进行读操作.
那么master数据库就用来进行写操作.

问题2:主从复制中,可以有N个slave,可是这些slave又不能进行写操作,要他们干嘛?

答:可以实现数据备份.
类似于高可用的功能,一旦master挂了,可以让slave顶上去,同时slave提升为master.
异地容灾,比如master在北京,地震挂了,那么在上海的slave还可以继续.
主要用于实现scale out,分担负载,可以将读的任务分散到slaves上.
很可能的情况是,一个系统的读操作远远多于写操作,因此写操作发向master,读操作发向slaves进行操作)


问题3:随着应用的日益增长,读操作很多,我们可以扩展slave,但是如果master满足不了写操作了,怎么办呢?

答:scale on ?更好的服务器? 没有最好的,只有更好的,太贵了...
scale out ? 主从复制架构已经满足不了.
可以分库(垂直拆分),分表(水平拆分).
